Hive Workspace · Split Horizon · revision 4

A workspace built on what Hive can prove.

One cohesive control surface for the Queen's Hive workflow, the SessionHost terminal stack, and the Communication Fabric—grounded in Hive’s current menus, settings, feed, names, status, attention, and daemon ownership.

Implementation target

Third transition: the native AppKit Workspace, evolved.

Split Horizon is built after the terminal foundation (T0–T7) and the Queen's Hive hierarchy, inside the existing native AppKit Workspace: one selected live HiveTerminalKit (Ghostty) workbench attached to hive-sessiond, a persistent hierarchy rail, and typed daemon contracts. SwiftTerm is already deleted by the terminal transition (T7/TG7); the pane-grid topology is what Split Horizon deletes — the native product itself evolves and remains.

This document specifies product behavior, missing contracts, and the legacy deletion boundary. It does not claim the new Workspace already exists.

01 · Reality boundary

The UI is the third change, not a source of truth.

Split Horizon composes two completed transitions plus the Communication Fabric contract. It must not invent hierarchy from names, derive status from terminal text, or treat a button as proof that an action landed.

Current Hive

Carry forward proven behavior

Keep the daemon-owned semantics and data contracts—not the AppKit windows or pane topology that happen to present them today.

  • Human first names and queen addressing
  • Measured agent status and explicit unknown
  • Attention ordering and acknowledgement
  • Tasks, Models, Usage, and autonomy behavior
  • Project identity and daemon-owned mutation
Queen's Hive + SessionHost + Communication Fabric

Wait for the structural truth

The workflow owns hierarchy and gates; the terminal stack owns process/session reality; the Communication Fabric owns envelopes, delivery receipts, channels, and WorkManifest recovery.

  • Parent, role, owner, run, and generation
  • Owners (queen or lead), reviewers, G1/G2, review state
  • Snapshot/cursor attach and off-screen detach
  • Input ownership
  • Delivery receipts from the Communication projection
  • Scoped queries and lifecycle events
Split Horizon

Compose one honest workspace

The new client makes the two planes legible and actionable without owning either.

  • Persistent hierarchy rail and attention rollups
  • One selected live terminal workbench
  • Integrated settings and native menus
  • Task, review, event, and evidence inspector
  • Explicit target, authority, and destructive previews
Replacement rule. Split Horizon ships as a hard UI cutover. When a successor meets its acceptance gate, the replaced window, panel, command, shortcut, reducer branch, and tests leave the product in the same release. During development the pane-era implementation may remain a reference oracle, but Hive does not ship two workspace modes.
Owned contract. The Queen's Hive keeps specs, tasks, interface contracts, and artifacts as typed durable records (SpecRevision, Task/TaskDetail, InterfaceContract, ArtifactRef), while Communication Fabric owns MessageContentRef, rollup, and WorkManifest references for summaries and overflow. The hierarchy platform contract owns the indexed ArtifactRef/TaskDetail read model, and this document requires that model before those inspector tabs ship. The view layer consumes their projections — it must not crawl the repository or fabricate checklist progress.
Sequence position. Split Horizon is the third transition. It consumes the terminal foundation (T0–T7: locators, SessionHost, sessiond, HiveTerminalKit, input arbiter, status/event primitives), The Queen's Hive (hierarchy records, delegation, review duty, gates, grant-bound landing), and the communication contract (hive-communication.html). It redesigns none of them.

07 · Build order

Ship capabilities in dependency order, then cut over once.

The new workspace begins only after both prerequisite transitions and the Communication Fabric release contracts meet their gates. Each phase has a falsifiable exit and a corresponding deletion entry; users never choose between old and new dashboards.

U0

Freeze contracts and deletions

Turn the ledgers into versioned client types and a hard cutover manifest before drawing production surfaces.

  • Freeze WorkspaceSnapshot v2 and hierarchy authorization.
  • Freeze the HiveTerminalKit attach-bridge contract.
  • Freeze the Communication Fabric queue/receipt/channel/message-content and WorkManifest recovery projections plus their typed actions.

08 · Explicit cuts

What Split Horizon does not promise.

These cuts are part of making the design buildable. They differ from the removal ledger: cuts are features Hive will not build; removals are current paths Hive will delete.

CutNo terminal mosaic

One terminal is live. The hierarchy is not represented by 19 miniature terminal tiles.

CutNo cross-project fleet

The current process owns one project instance. Project aggregation needs separate authenticated supervisor work.

CutNo transcript reconstruction

Provider TUIs remain real terminal content. Status and workflow never come from screen scraping.

CutNo generic approval button

Provider prompts stay in their TUI. G1/G2 name exact workflow gates and revisions.

CutNo fake artifact explorer

Without ArtifactRef authorization and indexing, the tab is omitted—not populated by view-layer filesystem access.

CutNo invented metrics

Unknown remains unknown; quota is percent plus reset; task progress requires explicit criteria state.

CutNo classic mode

No pane-era/classic Workspace fallback, legacy layout toggle, disabled retired commands, or second settings window ships beside Split Horizon — AppKit is the product, not the fallback.

CutNo UI-owned control plane

The dashboard does not grow its own hierarchy database, settings service, terminal arbiter, or filesystem crawler to fill contract gaps.
