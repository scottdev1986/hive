Hive Workspace · Split Horizon · revision 3

A workspace built on what Hive can prove.

One cohesive control surface for the Foreman workflow and SessionHost terminal stack—grounded in Hive’s current menus, settings, feed, names, status, attention, and daemon ownership.

Implementation target

Post-workflow, post-terminal Electron workspace.

The terminal plan’s P6 client is the single target: sandboxed renderer, xterm.js, one selected live attachment, and a narrow main-process bridge. The AppKit/SwiftTerm Workspace is the source for parity requirements, not a second UI to retain.

This document specifies product behavior, missing contracts, and the legacy deletion boundary. It does not claim the new Workspace already exists.

Backed by today’s Swift Workspace, daemon, config, or CLI contracts.

ForemanWorkflow prerequisite

Delivered by the hierarchy, lead, reviewer, gate, and merge-flow change.

SessionHostTerminal prerequisite

Delivered by generation-safe sessions, events, snapshots, and input arbitration.

RemoveShipping surface retired

Current UI or transport code deleted after its successor passes parity; no compatibility mode.

01 · Reality boundary

The UI is the third change, not a source of truth.

Split Horizon composes two completed prerequisites. It must not invent hierarchy from names, derive status from terminal text, or treat a button as proof that an action landed.

Current Hive

Carry forward proven behavior

Keep the daemon-owned semantics and data contracts—not the AppKit windows or pane topology that happen to present them today.

  • Human first names and queen addressing
  • Measured agent status and explicit unknown
  • Attention ordering and acknowledgement
  • Tasks, Models, Usage, and autonomy behavior
  • Project identity and daemon-owned mutation
Foreman + SessionHost

Wait for the structural truth

The workflow owns hierarchy and gates; the terminal stack owns process/session reality.

  • Parent, role, owner, run, and generation
  • Leads, reviewers, G1/G2, review state
  • Snapshot/cursor attach and off-screen detach
  • Input ownership and delivery receipts
  • Scoped queries and lifecycle events
Split Horizon

Compose one honest workspace

The new client makes the two planes legible and actionable without owning either.

  • Persistent hierarchy rail and attention rollups
  • One selected live terminal workbench
  • Integrated settings and native menus
  • Task, review, event, and evidence inspector
  • Explicit target, authority, and destructive previews
Replacement rule. Split Horizon ships as a hard UI cutover. When a successor meets its acceptance gate, the replaced window, panel, command, shortcut, reducer branch, and tests leave the product in the same release. During development the old app may remain a reference oracle, but Hive does not ship two workspace modes.
Known unowned gap. The Foreman design says specs, tasks, interface contracts, summaries, and artifacts live on disk, but neither prerequisite currently defines an indexed read contract for the Workspace. This document requires an ArtifactRef/TaskDetail read model before those inspector tabs ship. The renderer must not crawl the repository or fabricate checklist progress.

07 · Build order

Ship capabilities in dependency order, then cut over once.

The new workspace begins only after both prerequisites meet their release gates. Each phase has a falsifiable exit and a corresponding deletion entry; users never choose between old and new dashboards.

U0

Freeze contracts and deletions

Turn the ledgers into versioned client types and a hard cutover manifest before drawing production surfaces.

  • Freeze WorkspaceSnapshot v2 and hierarchy authorization.
  • Freeze SessionHost Electron bridge messages.
  • Own TaskDetail/ArtifactRef or explicitly cut those tabs.
  • Pin settings fixtures and every legacy path scheduled for deletion.
ExitGolden fixtures cover current, unknown, stale, disconnected, replaced, gated, and unauthorized states; every migrated surface has one removal owner.
U1

Build the shell and settings

Deliver cohesive navigation without depending on terminal rendering.

  • Native menus and one typed command dispatcher.
  • Read-only project/instance context; no ProjectSwitcher.
  • Tasks, Models, Usage, autonomy through the main-process CLI bridge.
  • Loading, error, stale, unknown, and not-metered states.
ExitSettings fixtures match AppKit behavior; old settings and project-switcher paths are deletion-ready; no subprocess blocks the renderer.
U2

Build the outer horizon

Render run and hierarchy truth before adding input authority.

  • Virtualized hierarchy tree with real names and role badges.
  • Run phase, gates, status, branch facts, and attention rollups.
  • Tree focus history and semantic keyboard navigation.
  • Attention drawer with explicit empty state.

U3

U4

  • Typed event history with cursor recovery.
  • Task criteria and artifacts only behind completed U0 contracts.

U5

  • G1/G2 approval with exact revisions.

  • Detach, close-agent, cascade preview, stop-run separation.
  • Delete AppKit/SwiftTerm Workspace code, packaging, menus, and compatibility paths.
