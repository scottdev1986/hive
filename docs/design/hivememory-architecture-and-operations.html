Architecture & Operations

The complete memory system as built on main: every layer, every recall path, every guardrail — plus operator instructions and a proof for every element. Companion to the investor overview (hivememory-system-overview.html).

01Full architecture

Three memory layers behind one daemon. Writes flow down the blue paths; recall is summoned up the violet paths (spawn, wake, trigger) or pulled by agents through the MCP gateway. Nothing crosses project boundaries — isolation is structural, derived from the daemon's own identity.

write / ingest path summoned recall path guardrail / lifecycle

02The recall model: summoned, never hoped-for

The governing design finding (from external research, adopted as plan A1/A2): agents save voluntarily but never recall voluntarily, and prose instructions are recited then ignored. So every recall path is initiated by the system or the human — the agent's own search tools exist, but nothing depends on the agent choosing to use them.

1 · Spawn injection (ranked)

Every agent's first prompt ends with a 30-row memory index: pitfalls first, then articles sharing tokens with the assignment brief, then newest fill. Read from the primary checkout — .hive/memory is gitignored, so worktrees never have it (works identically if a project tracks it; primary wins over stale copies).

2 · Wake deltas

When a message is delivered or an agent resumes, a 300-token block of what changed since that agent's high-water mark (wiki log diff + task-matched pitfalls) rides the send lane. No vendor hooks required — works on Grok. Per-agent marks persist in the episodic store meta table, surviving restarts.

3 · Trigger protocol

Queen/operator messages starting with recall:, note this:, document this: are executed by the daemon — search-and-inject, write an observation, scaffold an article — and the labeled result is delivered instead of the raw text. Agent senders can't trigger; the text passes through verbatim.

4 · Agent-initiated tools

memory_recall (hybrid FTS+vector bundle), memory_query (9 classes, token ceilings), memory_search/memory_read (wiki), memory_digest, memory_pitfall. All server-ceilinged, all labeled with verification status — used for saves and explicit lookups, never relied on for recall.

03Operator usage

Everything below runs against the live daemon (or fixtures, where noted). CLI memory commands talk to the daemon over the same MCP endpoint agents use.

document this: checkpoint format decisions   → curated article scaffolded [unverified], confirmation delivered

The 10-minute end-to-end proof (live, manual)

  1. Health

    hive memory self-test --live → fixture PASS lines plus all [live] PASS lines. This proves the store, search, read-back, dedup, delete-guard — and that the running daemon really has working embeddings, not just the in-process fixtures.

  2. Seed a lesson

    hive memory write "Vendor X quota resets at midnight UTC" --scope repo --topic quota --body "…" --source human --evidence "…" --status verified --supersedes ""

  3. Prove summoned recall at spawn

    Spawn an agent with task "reconcile the quota tables" → its first prompt's memory index ranks the quota article near the top (brief-match), above newer unrelated articles.

  4. Prove the wake delta

    While that agent is idle, write another memory, then hive_send it any message → the delivery carries the 🧠 delta block naming your new article. Send a second message → no block (high-water advanced).

  5. Prove triggers

    As queen/operator, send the agent recall: quota → it receives the labeled bundle. Then note this: quota API rejects empty arrays → confirmation comes back and the article exists as [unverified].

  6. Prove dedup

    Repeat the same note this: → it becomes an update of the same article, not a duplicate.

  7. Prove the learning loop

    Run a session that hits a real failure; kill it. memory_pitfall (search) shows a new [unverified] candidate with event/digest provenance. Verify it (hive memory write --id <pitfall-id> --status verified --supersedes <pitfall-id>).

  8. Prove knowledge sharing

    Spawn a different agent on a matching task → the verified pitfall leads its spawn index. It never searched; Hive summoned it.

  9. Prove guardrails

    Ask a worker agent to call memory_delete → forbidden. As operator, delete an article another article supersedes → refused, blocker named.

  10. Prove digests & drill-down

    memory_digest for the killed session → sections with [eN] pointers; drill down one pointer → the exact source event rows that ground the claim.

Caveats. The deployed single-file binary cannot carry onnxruntime — run hive embeddings install once per machine (probe-verified) so the daemon resolves the external runtime from ~/.hive/tools/embeddings/; until then it degrades to keyword-only with a labeled state visible in hive_status. Real-model embedding tests SIGTRAP under bun test teardown (onnxruntime-node) — always run them targeted with the env flag, never in a flagged full-suite run. The memory corpus is fresh (per the 2026-07-22 fresh-start ruling), so consolidation and hybrid-recall quality get more meaningful as the corpus grows.
