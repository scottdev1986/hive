Hive Workspace · Memory Manager · revision 1

A management surface for the memory the daemon owns.

One management surface for the HiveMemory backend that already ships on main — the curated wiki, the episodic store, the pitfall pipeline, summoned recall, and the semantic leg — plus its health monitoring. The UI renders daemon projections; it never becomes a second memory implementation.

Implementation target

A Memory menu and a Memory Manager window, inside the Split Horizon shell.

The backend ships on main: eleven memory MCP tools, per-project episodic stores, deterministic digests, pitfall harvest, wake deltas, triggers, local embeddings, retention sweeps, and a golden-canary self-test. What is missing is the read/management layer this window needs — list projections, a daemon-side self-test record, consolidation and sweep controls, config write, metrics, and a memory audit feed. Those are named below as contract gaps, not assumed.

This document specifies product behavior and missing contracts. It does not claim the window exists. Backend claims cite the code that landed with the HiveMemory epic (#119–#123, #71, #72).

01 · Reality boundary

The window manages the daemon's memory. It never owns it.

Every row in this window is a daemon projection or tool result. Every mutation rides an existing capability-gated tool. If the UI writes .hive/memory itself, the provenance model breaks.

Current Hive

The backend the window consumes

Landed with the HiveMemory epic (2026-07-22), full suite green:

  • Wiki layer: raw/ + wiki/, statuses, kind: pitfall, dedup layers 1–2, reference-checked delete
  • Episodic store: bi-temporal facts, events, digests, vectors — per project, structurally isolated
  • Summoned recall: ranked spawn injection, 300-token wake deltas, queen/operator triggers
  • Semantic leg: local bge-small, RRF hybrid, labeled degraded states
  • Lifecycle: retention sweeps, stale-demotion, golden-canary self-test (fixture + --live + --strict)
  • Tools: search, read, write, delete, reindex, query, recall, note, digest, pitfall, promote
plan §8–§11 implementation record
Contract gap

Backend work the window requires

The existing tools are agent-shaped: they answer one query at a time. A management UI needs browsable projections and daemon-side verbs:

  • List/browse projections (articles, digests, pitfalls-by-status, facts)
  • Daemon-executed self-test with a last-run record
  • Consolidation report/apply over the daemon, not only the CLI
  • Config read/write with validation and loud changes
  • Sweep trigger + sweep history
  • Metrics: per-class token cost, latencies, index sizes, RSS
  • Memory audit feed (deletes, promotions, triggers, demotions)
  • Recall preview (what would an agent see for this brief?)
Forbidden

Rendering and mutation rules

  • No direct file access — mutations go through daemon tools with capabilities
  • CRUD rides the write contract: edits are self-superseding revisions, never in-place overwrites
  • Raw observations are immutable — viewable, never editable
  • Digests are read-only — correct the source record; the drift audit catches hand edits
  • Degraded is always visible: semantic: degraded:<state> is a banner, never a footnote
  • Absent vs empty is rendered distinctly on every list
  • Unverified is a claim to reconcile, styled as such everywhere
  • Delete and promote are operator-only, preview-first, audited
  • No fabricated counts: unknown renders as unknown
Authority rule. The window holds no capability of its own. Read surfaces use the operator's memory:read; verify/write uses memory:write; delete, promote, and consolidation-apply require memory:delete (queen/operator tier) and show a destructive preview naming exactly what changes before the call is made.
Fresh-corpus honesty. The production memory corpus was reset by the fresh-start ruling (2026-07-22) and is intentionally small. Mockup numbers below are representative of a matured project; the shipped window renders the real counts, including honest zeros and explicit empty states.

05 · The Scribe pattern

Humans draft in natural language; a Scribe agent compiles to spec; approval runs the daemon write path.

This section is the design document for the drafting flow — written to plan and write stories from. It applies identically to creating, editing, and updating memory, and it exists to keep memory writes contract-conformant without requiring the operator to know the contract.

Current Hive

Why this fits the memory model

The system already separates raw input from compiled knowledge: immutable observations feed compiled articles through a contract. The Scribe pattern puts the human at the top of that same pipeline:

  • The user's natural language is the raw input — stored verbatim as the raw observation
  • The Scribe agent is the compiler — the same role the hive-memory skill prescribes
  • Human approval is the verification act — nothing enters the wiki unapproved
  • The daemon write path is the only way in — the Scribe holds no memory capability and cannot bypass a single gate

The pattern adds no new memory semantics. It composes the spawn infrastructure, the memory skill, and the write contract that already exist.

Contract gap

What the Scribe does (and never does)

Does:

  • Turns plain-language input into contract-conformant Markdown — current truth first, durable/non-derivable/load-bearing test applied
  • Proposes topic, kind, status, and tags with reasons
  • Sharpens evidence into the required form without inventing any
  • Runs the dedup check before drafting and links similar candidates instead of merging
  • Extracts exact values (SHAs, paths, error strings) into typed phrasing
  • For edits: produces a minimal diff against the current revision

Never does: write to memory, set its own status to verified without the human's date, discard the user's wording (the verbatim input is the raw observation), bypass dedup/date/supersede gates.

UI work

The flow, state by state

  • 1 · Draft — user fills scope (dropdown), title, natural-language body, natural-language evidence. No other fields exist.
  • 2 · Compiling — the daemon spawns the Scribe (cheap model category); the UI shows the job's live state, not an indeterminate spinner
  • 3 · Proposal — the compiled article with proposed fields, dedup findings, and (for edits) a diff; every proposed field is overridable before approval
  • 4 · Approve — executes memory_write through the daemon; all gates re-run server-side; raw + article + index + log + embedding land in one call
  • Revise / Discard — back to draft with the proposal attached, or gone entirely; neither leaves a trace in the wiki
