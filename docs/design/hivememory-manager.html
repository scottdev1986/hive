Hive Workspace · Memory Manager · revision 1

A memory system you can see, judge, and tune.

One management surface for the HiveMemory backend that already ships on main — the curated wiki, the episodic store, the pitfall pipeline, summoned recall, and the semantic leg — plus the monitoring that proves it stays healthy. The UI renders daemon truth; it never becomes a second memory implementation.

Implementation target

A Memory menu and a Memory Manager window, inside the Split Horizon shell.

The backend is real: eleven memory MCP tools, per-project episodic stores, deterministic digests, pitfall harvest, wake deltas, triggers, local embeddings, retention sweeps, and a golden-canary self-test. What is missing is the read/management layer this window needs — list projections, a daemon-side self-test record, consolidation and sweep controls, config write, metrics, and a memory audit feed. Those are named below as contract gaps, not assumed.

This document specifies product behavior and missing contracts. It does not claim the window exists. Backend claims cite the code that landed with the HiveMemory epic (#119–#123, #71, #72).

01 · Reality boundary

The window manages the daemon's memory. It never owns it.

Every row in this window is a daemon projection or tool result. Every mutation rides an existing capability-gated tool. The moment the UI writes .hive/memory itself, the provenance model is dead.

Current Hive

The backend the window consumes

Landed with the HiveMemory epic (2026-07-22), full suite green:

  • Wiki layer: raw/ + wiki/, statuses, kind: pitfall, dedup layers 1–2, reference-checked delete
  • Episodic store: bi-temporal facts, events, digests, vectors — per project, structurally isolated
  • Summoned recall: ranked spawn injection, 300-token wake deltas, queen/operator triggers
  • Semantic leg: local bge-small, RRF hybrid, labeled degraded states
  • Lifecycle: retention sweeps, stale-demotion, golden-canary self-test (fixture + --live + --strict)
  • Tools: search, read, write, delete, reindex, query, recall, note, digest, pitfall, promote
plan §8–§11 implementation record
Contract gap

What the window needs that nothing serves yet

Today's tools are agent-shaped: they answer one question at a time. A management UI needs browsable projections and daemon-side verbs:

  • List/browse projections (articles, digests, pitfalls-by-status, facts)
  • Daemon-executed self-test with a last-run record
  • Consolidation report/apply over the daemon, not only the CLI
  • Config read/write with validation and loud changes
  • Sweep trigger + sweep history
  • Metrics: per-class token cost, latencies, index sizes, RSS
  • Memory audit feed (deletes, promotions, triggers, demotions)
  • Recall preview (what would an agent see for this brief?)
Forbidden

Honesty rules the window must keep

  • No direct file access — mutations go through daemon tools with capabilities
  • CRUD rides the write contract: edits are self-superseding revisions, never in-place overwrites
  • Raw observations are immutable — viewable, never editable
  • Digests are read-only — correct the source record; the drift audit catches hand edits
  • Degraded is always visible: semantic: degraded:<state> is a banner, never a footnote
  • Absent vs empty is rendered distinctly on every list
  • Unverified is a claim to reconcile, styled as such everywhere
  • Delete and promote are operator-only, preview-first, audited
  • No fabricated counts: unknown renders as unknown
Authority rule. The window holds no capability of its own. Read surfaces use the operator's memory:read; verify/write uses memory:write; delete, promote, and consolidation-apply require memory:delete (queen/operator tier) and show a destructive preview naming exactly what changes before the call is made.
Fresh-corpus honesty. The production memory corpus was reset by the fresh-start ruling (2026-07-22) and is intentionally small. Mockup numbers below are representative of a matured project; the shipped window renders the real counts, including honest zeros and explicit empty states.

04 · Contracts to build

The backend work this window requires.

Eight additions, all daemon-owned, all capability-gated. None changes the memory model; they expose and extend what exists. CRUD and search need no new contract — they ride the existing memory tools; only browsing (C3) is a gap. Ordered by dependency, each with its acceptance note.

ContractWhat it providesBuilt onAcceptance
C1 · memory_overview

One projection: wiki counts by status/kind/scope, episodic counts, vector counts, store footprint, embedding state, latest sweep report, last self-test verdict.

Parts exist

hive_status embedding section, adapter discover, episodic rowCounts.

Unknown ≠ zero; absent surfaces reported as absent; fixed cost, no scans on the hot path.

C2 · daemon self-test run

Runs the existing probe daemon-side (fixture + live tiers) and persists the last run (per-assertion results, timestamp, strict flag).

Probe exists

memory-self-test.ts + --live tier.

Same assertion names as the CLI; a FAIL is retrievable until superseded by a newer run.

C3 · memory_list

Browsable, filterable, paginated lists: wiki articles (scope/topic/status/kind), pitfall inbox, digest sessions, episodic facts — plus persisted drift-audit verdicts on digests.

Parts exist

discoverMemoryFacts, digestFor.

Server-side paging; verification labels on every row; absent-vs-empty discriminators.

C4 · consolidation verbs

Daemon-side report (identical/similar buckets with scores) and apply (identicals only, per-pair, operator tier).

CLI pass exists

memory-consolidate.ts.

UI and CLI share one implementation; applies are audited; similar bucket has no apply path.

C5 · memory_preview

Read-only composition: spawn index or wake delta for an arbitrary brief, plus the recall bundle — with no delivery and no high-water mutation.

Composers exist

buildMemoryIndex, composeMemoryDelta, recall bundle.

Byte-identical to what a real spawn/delivery would produce for the same inputs.

C6 · config + sweep verbs

Validated read/write of [memory]/[memory.retention]; sweep trigger; persisted sweep history.

Schema exists

schemas/config.ts strict parsing; runRetentionSweep.

Invalid values rejected with the zod message; invariants not writable; changes logged loudly; sweep history queryable.

C7 · memory_audit

Unified feed: capability-audited deletes/promotes, trigger executions, wiki log entries (ingest/demote/delete), consolidation applies.

Sources exist

capabilities audit, episodic memory-trigger events, wiki log.md.

Chronological, filterable by actor/action; reads stay unlogged (deliberate).

C8 · memory_metrics

Runtime counters: per-class query token totals, recall/embed latencies, index sizes, vector counts — plus the scripted-corpus measurement results when the epic's outstanding token-cost AC lands.

New

Bench script only today.

Counters are cheap and always-on; measurement-run results are labeled with corpus + date.
