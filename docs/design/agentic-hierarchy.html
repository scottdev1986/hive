Agentic hierarchy redesign

The Queen’s Hive

Hive grows from a flat queen-to-workers fan-out into a governed hierarchy the queen scales to the work: the engineer decides, the queen orchestrates and owns direct work, leads integrate crews when a run earns them, and workers execute.

Core operating principle

Delegate execution.
Bound every authority.

Operational capabilities attenuate at every hop; the engineer keeps accountability and the G1/G2 gates. Parallelism is earned by file-disjoint work and explicit interfaces.

The human engineer remains in charge.
01 / GROUND Clarify before building

Grill the task, scout the real code, and earn an approved spec before decomposition.

02 / CONTAIN Promote through the current owner

Candidates promote through their current owner — lead crews only when approved — and the daemon’s serialized queue alone moves integration refs and main.

03 / CONTROL Records carry context

Specs, tasks, and interface contracts are durable records with ArtifactRefs—not chat relays or “latest” files on disk.

01 · The hierarchy

Four responsibilities, one revisioned ownership chain.

The queen is the permanent task root. The run’s TopologyDecision attaches each worker directly to the queen or inserts a lead owner above a crew; either way, every candidate crosses an independent review and a daemon-serialized promotion before it joins its next integration stage.

Authority · Human

Engineer

Holds final authority over the spec, architecture choices, irreversible actions, and final land. Receives periodic and on-demand updates from the queen.

Orchestration · No code

Queen

Runs the grill-me interview, sends read-only scouts into the code, maintains the spec, decomposes the work, spawns direct workers and/or leads as the approved topology requires, and handles policy and escalation. The queen never writes code.

Constitution→Specify→Plan→Tasks
Integration · Lead worker

Lead Worker

Owns one daemon-mutated IntegrationStage whose ref only the serialized promotion queue moves — the lead coordinates and inspects, never authors on it. Plans worker-sized tasks; spawns, monitors, and closes its own crew; requests independent review and promotion for each candidate. A lead worker who must code takes an ordinary scoped task branch with independent review, like any worker.

Execution · Scoped

Worker

Takes one small Task in a worktree branched from its grant-bound base — its assigned integration stage, whichever owner holds it. Holds a file-disjoint path lease where eligible, honors InterfaceContract semantics, and returns a bounded summary plus ArtifactRefs—never a full transcript.

Quality · Temporary duty

Reviewer

A temporary assignment, not a hierarchy tier: reviews every author candidate before its next integration stage, with fresh context, no authored commit in the candidate’s diff range (merge-base..head), no overlapping author task, and a different identity from every author. The verdict binds the exact patch digest and reviewed base; a parent-only identical-patch rebase preserves the semantic verdict only after automated validation at the exact new synthetic SHA — any patch change requires a new review. Cross-vendor review is preferred, not proof; reviewer sessions are transient capacity charged to the assigning run.

02 · Adaptive topology

The hierarchy is capacity, not mandate.

The queen is the permanent root and never writes code. The lead-worker tier appears only when it reduces coordination load for a genuinely separable build. Every run chooses the smallest shape that can do the work, and the engineer can override that choice at G1.

DIRECT · QUEEN → 1

One direct worker

Use for one coherent change, tightly shared context, or work whose integration cost would exceed its parallel benefit. The queen specifies, monitors, and escalates; one worker authors and verifies.

FLAT · QUEEN → N

Direct worker team

Use for a small set of independent tasks under one shared design. Today’s queen-to-workers system remains a first-class run shape, normally with 2–4 direct workers where the decomposition supports it.

FULL HIVE · QUEEN → LEADS → CREWS

Lead-worker crews

Use when the run contains multiple coherent domains that can own separate interfaces and integration refs. A crew normally has 2–3 workers. Lead workers coordinate, integrate, and report; they do not become ambient-authority managers. Full hive may mix lead-worker subtrees with queen-owned direct workers for cross-cutting tasks or independent review; G1 approves the exact tree, not a uniform tier template.

Decision rule. Shared context and interdependent edits push the run flatter. Homogeneous, isolated fan-out and stable interfaces justify a lead tier. Nineteen concurrent agents is a forward load target, not a default team, a spawn grant, or evidence that every run needs hierarchy. All workers, reviewers, and lead workers still fit inside the daemon’s global 32-session admission cap and the run’s smaller budget. Any mid-run TopologyDecision change is engineer-approved, not merely revisioned.

The middle tier must earn its keep

  • Record lead tokens, wall time, wakeups, queue wait, escalations, and rework separately from worker execution.
  • Measure accepted promotions, conflicts caught before promotion, interface violations, idle worker time, and critical-path change.
  • Compare the observed fan-out and coordination cost with the G1 prediction in a terminal TopologyReview.
  • Use the evidence to recommend a future shape; never rewrite the approved shape mid-run without a revisioned decision.

No vendor-native nesting assumption

Every lead worker and worker is an ordinary top-level provider session. Hive supplies the hierarchy through durable records, scoped credentials, isolated worktrees, routing policy, and daemon enforcement. A provider may offer a native subagent feature, but correctness and recovery never depend on it.

Lead pathologies are system failures: premature “done,” stale task state, doing worker code directly, or vague delegation. The daemon counters them with typed state, exact evidence, mandatory delegation records, independent review, and measured gates—not stronger role-play.

03 · Operating process

Seven phases, two human gates.

Autonomy increases after the spec is approved, then narrows again at land. G1 prevents premature execution; G2 prevents an assembled feature from silently becoming production policy.

P0

Intake & Grill

The queen interviews until scope, acceptance criteria, and non-goals are pinned. Read-only scouts ground the task in real code. Output: a proposed SpecRevision plus preliminary PlanRevision/dependency DAG, TopologyDecision, and RunBudget/gate policy — the exact package G1 approves.

P1

Decompose & Route

Materialize Tasks, DelegationSpecs, nodes, grants, stages, and path leases within the G1-approved boundaries. Parallelize only file-disjoint, independent parts; use one agent for tightly coupled decisions. A broader plan, topology, or spec change returns to the engineer.

P2

Build

Workers implement small scoped tasks against their grant-bound target. Event-driven freshness requires a rebase onto the assigned stage before review and whenever overlapping files change.

P3

Independent Review & Promotion

Every candidate gets an independent digest-bound review and automated validation at the predicted target SHA. The daemon’s serial CAS promotion alone moves a candidate into its next integration stage.

P4

Assemble & Validate

Direct and flat candidates promote into the queen-owned run integration stage; full hive assembles lead stages first, then the run stage. The speculative validation train runs the whole repo at exact SHAs; an interface conflict becomes a typed RunDecision, not a chat thread.

P5

Land

After G2 the daemon revalidates epochs and moves the exact approved run-stage SHA to main — no other actor touches the ref. Public integrated history is never rebased.

P6

Close-out

The current owner closes its workers as soon as work merges and reports upward; finalizers and retention run before teardown. Ephemeral branches are deleted; durable lessons are persisted to memory.

G1
Engineer CAS-approves the exact proposal package

SpecRevision, preliminary PlanRevision, TopologyDecision, and RunBudget/gate policy together. No spawn or implementation before G1.

G2
Engineer approves the exact assembled run-stage SHA

The approval binds SHA, digest, evidence, and target base. No main movement before the integrated result is accepted.

04 · Git flow

Branches mirror accountability.

Each merge moves one level upward. Workers never land directly to main, and hive-next is a disposable integration sensor—not a release branch.

CI gets broader at every merge boundary.

Fast local feedback stays close to the worker. Integration and release gates progressively widen until the exact production candidate has crossed the full stack.

Worktree invariants

Isolation is real, but it is not a separate repository.

  • Worktrees share refs, objects, config, and hooks.
  • Each worktree has its own index.
  • Create active worktrees with --lock.
  • The daemon serializes worktree prune operations.

Freshness is event-driven

A timer cannot tell whether review assumptions are still valid.

  • Workers rebase before review and whenever the lead changes an overlapping path, API, or dependency.
  • Work running longer than two hours gets a checkpoint warning—not a blind timed rebase.
  • New pushes dismiss stale approvals.
  • Lead branches target promotion to main in under 24 hours.

Some files cannot be path-leased.

Repository-wide coordination points are serialized across the whole hierarchy, even when their surrounding work is otherwise file-disjoint.

05 · Communication

Local conversation, durable context.

Status stays hub-and-spoke. Leads go peer-to-peer only to negotiate interfaces, and the outcome becomes a written contract. Workers speak only to their own lead.

Allowed channels

  • Engineer → queen: G1/G2 approvals, architecture decisions, and stop or redirect.
  • Queen ↔ leads: periodic and on-demand status envelopes.
  • Lead ↔ lead: interface negotiation through written contracts.
  • Lead ↔ workers: tasking, monitoring, review, and lifecycle.
  • Queen → engineer: periodic updates and answers whenever requested.

Context protocol

Artifacts—not transcripts—are the unit of coordination.

  • Living spec and shared task list stay on disk.
  • Cross-lead interfaces are versioned files.
  • Workers return a 1–2k-token summary plus artifacts.
  • Peer routes are default-deny; the only exceptions are owner/queen-authorized task-scoped channels with explicit participants, purpose, budget, and expiry, plus a rate-limited worker→queen emergency escalation — see the communication fabric.
The lead worker owns normal crew closure and closes each worker promptly after merge. Daemon safety systems remain the last-resort floor: idle-reap warnings, teardown, and the resource watchdog stay active, and when a lead dies its crew’s ownership returns to the queen while daemon reaping continues.

The queen remains special

The queen root has no agents-table row and no coding or landing right. A hierarchy snapshot projects the reserved root identity above run nodes; it does not manufacture a fake agent. Root provider generations may change while the queen’s run authority and audit trail remain durable.

Role, assignment, and capability differ

Role describes organizational position. Assignment kind describes current work such as author, reviewer, researcher, or lead coordination. Capability authorizes an operation. An engineer is outside the tree; a reviewer does not become a new tier; and a lead-worker label never grants spawn, write, message, review, or promotion authority by itself.

Specification is stable; plans may move

G1 approves an exact spec revision, acceptance IDs, topology, outward-effect policy, gates, and budget. PlanRevision may change task decomposition, ordering, or assignee within those boundaries by compare-and-swap. Any broader change returns to the engineer; agents cannot smuggle scope expansion through a child task or message.

Authority is monotonic attenuation

A grant chain may only narrow actions, descendants, tasks, paths, branches, time, and budget. The daemon validates the complete chain at each sensitive operation. Revocation is append-only; new authority is a new grant under the current revisions and epochs.

07 · Execution contract

Agents propose. The daemon admits, fences, and promotes.

The hierarchy expands judgment without distributing ambient authority. Every state-changing operation names its durable record, expected revision, and evidence; the protected service checks all current fences before it acts.

Integration refs: serial mutation, speculative validation

Write ownership

  • Each authoring task owns one isolated worktree and source branch.
  • Each full-hive lead stage has a dedicated integration ref logically owned by the lead worker but mutated only by the daemon.
  • The lead coordination session may inspect its integration ref; it never edits or commits on that ref.
  • If a lead worker must author code, it receives an ordinary scoped task branch, a different reviewer, and the same promotion path.
  • Lead → main promotion uses the exact final candidate approved at G2; no floating branch or caller-selected target lands.

Validation train

For ordered candidates C1…Cn against integration ref R, the daemon creates ephemeral synthetic states S1 = merge(R,C1), S2 = merge(S1,C2), and so on. Required CI may run speculatively on those exact SHAs. Ref movement remains strictly serial: after S1 passes and all records are current, the daemon compare-and-swaps R → S1, then advances the next already-valid state.

If a train fails, the daemon bisects the failing group, preserves and requeues innocent candidates against the last good state, and returns each culprit to its author with exact evidence. A stale base, source digest, patch digest, path lease, InterfaceContract, review, CI environment, hierarchy revision, or epoch invalidates the affected predicted states.

Review is exact—and fallible

The semantic verdict binds to the patch/diff digest and reviewed base. A parent-only rebase with an identical patch may preserve that verdict only after automated checks validate the exact new synthetic SHA. Any patch change requires a new review. Reviewers consume the assigning run and lead’s capacity and the selected provider’s quota. G2 remains the engineer’s backstop; a green review is evidence, not infallibility.

Interfaces are admission state

Promotion rechecks the cross-crew InterfaceContract registry, not merely the source paths. A task depending on a conflicted, stale, or superseded contract cannot promote. Path leases constrain admission and landing, while contracts cover semantic coupling that paths cannot see. “Dangerous autonomy” may broaden exploration only by explicit engineer grant; it never broadens promotion scope.

Recovery and supervision

One failure, one bounded repair

Independent worker failure defaults to one-for-one recovery. A dependency or accepted-interface failure pauses only affected downstream tasks or subtree. Repeated same-cause restarts spend a daemon-owned RecoveryBudget; exhaustion escalates worker → lead worker → queen → engineer instead of looping. No provider restart proves task recovery until state and evidence are re-measured.

Lead-worker loss

The daemon freezes subtree promotions, captures pending tasks, reservations, contracts, dirty/stranded work, reviewed candidates, and queue state in RecoveryCheckpoint, and returns logical ownership to queen. Queen may issue OwnershipTransfer to a replacement binding. Existing review survives only when candidate, patch, base, environment, leases, contracts, and policy remain current.

Teardown finalizers. Before cleanup, Hive must capture handoff, classify worktree state, preserve unmerged candidates and artifacts, settle quota reservations, revoke grants/channels, and record process reaping. Leads own prompt normal closure; the daemon still owns safe process teardown, the idle/orphan reaper, and crash recovery.

Queen succession formalizes today’s backup-root protocol

Checkpoint before context becomes a cliff

Near 60–65% of measured or estimated root context, Hive requests a RunCheckpoint sync; by roughly 80%, it requires a handoff before further delegation or promotion. Provider estimates are shown as estimates. The checkpoint records state by revision and digest, not a prose claim of remembered reasoning.

Fresh root, explicit attestation

A replacement declares backup status, receives the prior exit reason plus bounded live-agent snapshot, loads the latest checkpoint, sends non-destructive recovery requests, then re-reads status and inbox. It attests the checkpoint digest before authority resumes. If no checkpoint exists, it explicitly attests “no checkpoint,” reconstructs from SQLite plus measured replies, writes the first checkpoint, and continues—bootstrap never deadlocks behind a file that was never created.

No memory fiction. The stable queen root identity survives; provider generation and conversational context do not. QueenSuccession records both generations, checkpoint or no-checkpoint proof, replies, discrepancies, and attestation. Contradictions remain visible until resolved.

Budget is policy, not advice

Reserve before spawn

Every worker, lead worker, and reviewer reserves session and provider quota against RunBudget before launch. Per-run and per-lead admission prevents one crew from consuming all 32 global slots. Cross-vendor review charges the reviewer provider pool as well as the assigning run/lead.

Circuit breakers outside the model

Soft thresholds stop new spawns and ask queen to replan. Hard token/cost/wall-time/session/CI thresholds atomically fence the run and begin bounded quiescence. Rate anomalies—token or tool-call spikes, repeated identical failures, restart churn, or runaway spawning—trigger the same daemon-owned safety path. Queen may explain or request; only engineer-authorized budget revision resumes.

Hierarchy can outperform solo work

Anthropic reports +90.2% for lead-plus-subagents, ~15× chat token use for multi-agent and ~4× for single-agent systems, with parallel research time cut by up to 90%.

O

Shared decisions resist isolation

Cognition’s influential “Don’t Build Multi-Agents” argues for full decision context in artifacts and warns that parallel isolation creates conflict on coupled decisions.

E

Isolation plus dependencies matters

CAID 2026 combines worktree/branch isolation with dependency-aware delegation, improving PaperBench by 26.7 points and Commit0 by 14.3 points.

M

Cross-agent conflicts are elevated

An early July 2026 preprint reports textual merge conflicts at 41.7% across agents versus 19.8% within an agent—directionally important, but not yet mature evidence.

E

Review improves through aggregation

SWR-Bench finds multi-review aggregation improves review F1 by up to about 44%; the single automated review baseline is roughly 10–15% F1.

E

Passing is not merge-worthy

METR found roughly half of SWE-bench-passing PRs were not merge-worthy; maintainer merge rate ran about 24 points below automated pass. That gap justifies G2.

M

Clarification rarely happens unaided

Agents ask clarifying questions in only about 1–3% of turns while users push back around 44% of the time. Mandatory grill-me turns an exception into protocol.

10 · Failure controls

Design for the failure, not the demo.

The hierarchy is justified only if it reduces coordination risk as deliberately as it adds parallel capacity.

Duplicated work

Grill-me clarification + dependency DAG + explicit path leases.

Merge conflicts

File-disjoint leases + hierarchy-wide serialization for shared hot paths + hive-next early composition + event-driven rebase.

Drift off-task

Lead supervision + regular on-task checks against the written task.

Dead agents

Lead liveness monitoring; the lead closes workers promptly after merge.

Telephone-game context loss

Specs, interface contracts, task lists, and results live as artifacts on disk—not chat relay.

Cost blowup (~15×) E

Reserve the hierarchy for high-value, parallelizable work. Use a solo agent otherwise; see Evidence.

False CI confidence

Synthetic merge testing validates exact worker→lead and protected-main merge-result SHAs.

Review rubber-stamping

Cross-model reviewers + stale-approval dismissal + strict size and file-count limits.

Definition of done

  • G1 displays direct, flat, or full-hive choice, evidence, budget, and engineer override; queen never codes in any shape.
  • A lead worker attempts to mutate its integration ref or code directly on it: both fail; the same agent can author only through a separate scoped task branch.
  • A speculative three-candidate train passes, fails in the middle, bisects, promotes the first, requeues the innocent third, and returns exact evidence for the culprit without concurrent ref mutation.
  • A source rebase preserves patch digest: semantic review survives only after exact-SHA automated revalidation. A changed patch invalidates review.
  • A stale hierarchyRevision, runEpoch, or capabilityEpoch reaches the protected ref writer: each is independently rejected.
  • Queen succession works with a valid checkpoint, a corrupt/stale checkpoint, and no checkpoint; measured replies and contradictions remain visible.
  • Budget hard-limit quiesces at turn boundaries; deadline fallback marks mid-turn worktrees unverified and cannot claim the whole run paused until every process is measured.
  • Lead loss preserves reviewed candidates and dirty work, freezes only the affected subtree, and transfers ownership through revisioned records.
  • Closing a node runs teardown finalizers and the daemon safety reaper remains effective; no role prompt can bypass capture-before-destroy.

Transition · Terminal complete → hierarchy → UI

Introduce authority before asking the UI to depend on it.

Hierarchy is the second major transition. It begins only after Ghostty/sessiond reaches T7 under the flat policy and ends with stable daemon projections that Split Horizon can render.

H0 · SCHEMA

Shadow records

Add Run, Plan, Task, node/binding, grant, budget, decision, checkpoint, review, contract, and promotion records; project current flat assignments without changing their authority.

H1 · DIRECT + FLAT

New-run control plane

Enable G1/G2, DelegationSpec, stable identity, scoped tasks, independent review, budget fencing, and daemon-only promotion for new direct and flat runs.

H2 · FULL HIVE

Optional lead tier

Enable lead-worker delegation, integration refs, interface contracts, pair channels, subtree rollups, recovery, and TopologyDecision under strict attenuation.

H3 · THROUGHPUT

Promotion trains

Add synthetic predicted states, speculative exact-SHA validation, failure bisect/requeue, and promotion-time contract/policy revalidation.

H4 · RECOVERY

Succession and run control

Exercise lead loss, queen checkpoint/no-checkpoint succession, bounded quiesce, SIGSTOP fallback, abort retention, teardown finalizers, and anomaly circuit breakers.

H5 · HANDOFF

Stable projections

Freeze WorkspaceSnapshot v2, TaskDetail, ArtifactRef, queue/receipt, gate, budget, and incident schemas. Only then begin Split Horizon’s hierarchy rail and workbench UI.

Migration rule. New runs opt into one versioned hierarchy contract. Active legacy runs drain flat. No live tree is reparented, no queued message is rebound by name, and no UI feature becomes an authority source.
