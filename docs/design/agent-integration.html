Agent integration · reliability contract

An agent you can always reach, read, and trust.

This document defines how provider agents (Claude, Codex, Grok) are integrated into Hive — launch, identity, status, input, permission modes, and lifecycle — and the reliability contract an engineer can hold Hive to. It is written against the designs at HEAD: Terminal Stack Transition, The Queen’s Hive, Communication Fabric, and Split Horizon. Where older documents disagree, those four win.

01 · The reliability contract

Four guarantees, each enforced by a mechanism — never by hope.

The user’s bar is agent uptime an engineer can build on. Every guarantee below names the mechanism that enforces it and the direction it degrades in. The degradation rule is uniform: unknown degrades to open and visible, never to locked and silent.

01 / TYPE A rendered pane is a typable pane

Keyboard, paste, and programmatic input into an agent’s terminal never branch on identity, write authority, pause state, activity status, feed health, or liveness. Those states render in the pane header as information. The only thing that cannot be typed into is a pane with no attached terminal — a physical absence, not a policy. App-server panes meet this through the pane-input bridge (§05); TUI panes meet it natively.

02 / READ Status is measured at provider boundaries

Turn state comes from each vendor’s own lifecycle surface (hooks, native notifications, or session artifacts), and the maintenance sweep records what it observed including nothing: an unreadable measurement overwrites a stale verdict with unknown rather than letting a fiction stand — for every vendor, not only Codex.

03 / KNOW Identity is observed, and carries its confidence

Every running agent’s row carries a provenance verdict — matching, drift, unknown, or unattested — comparing what was launched against what a real provider surface observes, for all three vendors. A display-grade observation is never presented — or consumed — as attestation-grade. Where a vendor exposes no surface, the row says so and this document says so explicitly below.

04 / FAIL HONESTLY A refusal names its true cause

A healthy installed vendor must never read as nonexistent. Launch refusals carry the failing probe’s own reason into the route audit; discovery retries once before declaring a vendor unavailable; a lost hook boundary leaves a durable trace instead of a silence; a policy refusal is never the cover story for a mechanism that silently failed to engage.

Where safety actually lives. None of these guarantees weaken containment, because input was never the enforcement point. Landing requires a current capability epoch through the daemon’s gate; Codex mutations are brokered per-turn on the app-server driver against the exact durable holder, re-proven after every await; write revocation acts on the daemon’s authority records without touching the conversation. A human typing into a revoked writer’s pane converses with a process that still cannot land or mutate anything.

02 · Launch

One spawn path, three vendors, immutable launch identity.

A spawn resolves a category through the user’s routing policy, passes the launch gate (catalog resolution, enablement, quota, containment), creates a worktree and tmux session, and wires the vendor’s signal path. The launch identity — vendor, model, effort — is frozen on the row; everything observed later is compared against it, never merged into it.

The driver is a per-incarnation launch fact. A Codex agent’s row records the driver its current process launched with, and every incarnation transition (spawn fallback, crash recovery, critical-control restart) re-records the driver it actually launches, because telemetry, identity maintenance, and the mutation gate dispatch on it. A Codex writer auto-engages the app-server whenever it is available — it is the only surface a writer is admissible on, and with the pane-input bridge its pane types like any other. Readers keep the TUI by default with an explicit config opt-in for app-server. A refused Codex writer names the true blocker in its error and route audit: manager unconfigured, or the availability probe failed — never a policy text covering a broken mechanism.

04 · Status

Turn state from the vendor’s own boundary, staleness bounded by the sweep.

Status words (spawning, working, idle, awaiting-approval, control-paused, terminal states) are derived from provider boundary evidence: Claude and Codex TUI hooks, app-server native notifications, Grok’s updates.jsonl turn records. The ~30s maintenance sweep is the fallback that keeps rows honest when a push signal is lost, and a failed vendor read never skips the row: the verdict goes to unknown rather than letting the last confident word stand. A hook boundary the daemon rejects is retried once and then traced durably to ~/.hive/hook-event-failures.jsonl — a lost boundary says so on disk while the sweep heals it.

Acting is not being. Hive’s recurring bug class proxied “did it call a tool” for “is it alive.” The rule that replaced it: status is turn state read from a boundary record; liveness is a measured process fact; neither is inferred from the other, from artifact timestamps, from elapsed time, or from a screen redraw. A lost signal reads unknown — visibly — instead of holding the last confident word.

05 · Input

The keyboard is the human’s, unconditionally.

Every agent pane in the Workspace is a real terminal attached to the agent’s tmux session (plain attach-session, writable). Viewer attachment binds on workspace instance + agent UUID + process incarnation + tmux session — never on provider conversation identity. Windows are viewers, not containers: closing a pane detaches a client and never kills an agent.

The pane-input bridge — how an app-server pane types. The app-server host renders the event feed, and its child’s stdin is the JSON-RPC stream, so raw pane bytes must never go there. Instead the host reads its own PTY stdin (streaming UTF-8 decode) and forwards each submitted line — and Ctrl-C — to the daemon’s /pane-input route, serialized through one chain so order holds and a failed POST renders visibly with the undelivered line rather than retrying blindly (an accepted-but-lost POST retried would deliver the same prompt twice). The daemon routes text onto the manager’s live session — idle starts a turn, an active turn is steered — through one serialized lane per exact holder; input typed before the session is boot-ready queues and flushes in order after the initial prompt. Ctrl-C is a turn interrupt, confirmed before the next line (it formerly killed the entire app-server via the host’s SIGINT handler), and it settles any mutation approval still waiting inside the interrupted turn without closing the conversation. The route authenticates conversation-grade: the host’s original launch token keeps typing alive across authority rotation (pause/resume reissue, critical epoch advance) while the exact holder binding — agent id + incarnation + name + app-server driver — still refuses any replacement. This route can approve, attest, and authorize nothing; its ceiling is steering a conversation the same credential could already message.

06 · Permission modes

Sandboxed and dangerous change authority, never observability.

The autonomy dial (sandboxed default, dangerous opt-in) is daemon-owned, live state — never an agent claim. It governs vendor sandboxes and approval prompting. It does not participate in input, status, or identity: every guarantee in §01 holds identically in both modes, for readers and writers.

07 · Lifecycle

Incarnations are explicit; recovery re-records the facts it changes.

A row is one durable holder of a name; a process incarnation is one launch of it. Every transition that allocates a new incarnation — spawn fallback, crash recovery, critical-control restart — resets the observed identity, re-records the driver it actually launches with, and reissues capability at a fresh epoch. Pane conversation survives that rotation for the same process (§05); a replacement incarnation’s pane starts its own. Teardown captures artifacts before destruction; an idle merged agent is reaped by the daemon on its own timer.

Viewer restart without machine loss is the contract — and today’s Workspace does not yet meet it. Per Terminal Stack Transition and Split Horizon, windows are viewers over daemon-owned state, so quitting a viewer must never destroy the machine. Today, quitting the Workspace app stops the whole instance, and an ordinary relaunch selects a fresh one: a restart is teardown plus cold start, losing live agents and pane history. The designed fix path is the terminal transition itself — sessiond-owned persistent sessions with Ghostty (or any viewer) attaching and detaching freely — which removes the app from the machine’s lifetime with minimum further change. Until that lands, daemon-side fixes activate only on a deliberately scheduled restart, and nothing in this document assumes the app can be bounced casually.
