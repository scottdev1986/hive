Agent integration · reliability contract

An agent you can always reach, read, and trust.

This document defines how provider agents (Claude, Codex, Grok) are integrated into Hive — launch, identity, status, input, permission modes, and lifecycle — and the reliability contract an engineer can hold Hive to. It is written against the designs at HEAD: Terminal Stack Transition, The Queen’s Hive, Communication Fabric, and Split Horizon. Where older documents disagree, those four win.

01 · The reliability contract

Four guarantees, each enforced by a mechanism — never by hope.

The user’s bar is agent uptime an engineer can build on. Every guarantee below names the mechanism that enforces it and the direction it degrades in. The degradation rule is uniform: unknown degrades to open and visible, never to locked and silent.

01 / TYPE A rendered pane is a typable pane

Keyboard, paste, and programmatic input into an agent’s terminal never branch on identity, write authority, pause state, activity status, feed health, or liveness. Those states render in the pane header as information. The only thing that cannot be typed into is a pane with no attached terminal — a physical absence, not a policy.

02 / READ Status is measured at provider boundaries

Turn state comes from each vendor’s own lifecycle surface (hooks, native notifications, or session artifacts), and the maintenance sweep records what it observed including nothing: an unreadable measurement overwrites a stale number with unknown rather than letting a fiction stand.

03 / KNOW Identity is observed, and carries its confidence

Every running agent’s observed model/effort comes from a real provider surface, labeled with its provenance. A display-grade observation is never presented — or consumed — as attestation-grade. Where a vendor exposes no surface, the row says unknown and this document says so explicitly below.

04 / FAIL HONESTLY A refusal names its true cause

A healthy installed vendor must never read as nonexistent. Launch refusals carry the failing probe’s own reason into the route audit; discovery retries once before declaring a vendor unavailable; a policy refusal is never the cover story for a mechanism that silently failed to engage.

Where safety actually lives. None of these guarantees weaken containment, because input was never the enforcement point. Landing requires a current capability epoch through the daemon’s gate; Codex mutations are brokered per-turn on the app-server driver; write revocation and pause act on the daemon’s authority records. A human typing into a reader’s pane converses with a process that still cannot land or mutate anything.

02 · Launch

One spawn path, three vendors, immutable launch identity.

A spawn resolves a category through the user’s routing policy, passes the launch gate (catalog resolution, enablement, quota, containment), creates a worktree and tmux session, and wires the vendor’s signal path. The launch identity — vendor, model, effort — is frozen on the row; everything observed later is compared against it, never merged into it.

The driver is a per-incarnation launch fact. A Codex agent’s row records the driver its current process launched with, and every incarnation transition (spawn fallback, crash recovery, critical-control restart) re-records the driver it actually launches, because telemetry and identity maintenance dispatch on it. The app-server driver remains an explicit config opt-in — including for writers, deliberately: the app-server pane cannot yet accept human keyboard input (§05), and auto-engaging it would replace an honest refusal with a live-looking pane that eats typing. A refused Codex writer names the true blocker in its error and route audit: manager unconfigured, availability probe failed, or the config opt-in not set with the reason why.

04 · Status

Turn state from the vendor’s own boundary, staleness bounded by the sweep.

Status words (spawning, working, idle, awaiting-approval, control-paused, terminal states) are derived from provider boundary evidence: Claude and Codex TUI hooks, app-server native notifications, Grok’s updates.jsonl turn records. The ~30s maintenance sweep is the fallback that keeps rows honest when a push signal is lost, and it writes “nothing observed” as unknown rather than skipping it — a null that is skipped is a stale number that stands forever.

Acting is not being. Hive’s recurring bug class proxied “did it call a tool” for “is it alive.” The rule that replaced it: status is turn state read from a boundary record; liveness is artifact activity; neither is inferred from the other, from elapsed time, or from a screen redraw. A lost signal reads unknown — visibly — instead of holding the last confident word.

05 · Input

The keyboard is the human’s, unconditionally.

Every agent pane in the Workspace is a real terminal attached to the agent’s tmux session (plain attach-session, writable). Viewer attachment binds on workspace instance + agent UUID + process incarnation + tmux session — never on provider conversation identity, which the Codex path deliberately never binds. Windows are viewers, not containers: closing a pane detaches a client and never kills an agent.

Known residual — the app-server pane, and the writer wave it gates. The default Codex driver is the TUI, whose pane is fully interactive. A configured app-server agent’s pane runs the host’s event feed, and the host does not yet route pane bytes anywhere: typed input has no path to turn/start/turn/steer. Reaching an app-server agent today means hive_send (steer lands mid-turn through the same connection). Codex writer admission is deliberately held behind the config opt-in until this bridge exists. The committed writer-wave design, in one landing: (1) the host reads its own PTY stdin and translates it into steers; (2) the mutation broker reads the autonomy dial — full autonomy auto-decides allow after the same exact holder/thread/turn/identity checks, sandboxed mode queues the human approval; (3) identity failure on an app-server writer revokes mutation/landing authority but keeps the conversation running — physical SIGSTOP suspension is reserved for unbrokered legacy processes, because a stopped host is an unreachable agent, and unreachable is the failure this whole contract forbids.

06 · Permission modes

Sandboxed and dangerous change authority, never observability.

The autonomy dial (sandboxed default, dangerous opt-in) governs vendor sandboxes and approval prompting. It does not participate in input, status, or identity: every guarantee in §01 holds identically in both modes, for readers and writers.

07 · Lifecycle

Incarnations are explicit; recovery re-records the facts it changes.

A row is one durable holder of a name; a process incarnation is one launch of it. Every transition that allocates a new incarnation — spawn fallback, crash recovery, critical-control restart — resets the observed identity, re-records the driver it actually launches with, and reissues capability at a fresh epoch. Teardown captures artifacts before destruction; an idle merged agent is reaped by the daemon on its own timer.

Viewer restart without machine loss is the contract — and today’s Workspace does not yet meet it. Per Terminal Stack Transition and Split Horizon, windows are viewers over daemon-owned state, so quitting a viewer must never destroy the machine. Today, quitting the Workspace app stops the whole instance, and an ordinary relaunch selects a fresh one: a restart is teardown plus cold start, losing live agents and pane history. The designed fix path is the terminal transition itself — sessiond-owned persistent sessions with Ghostty (or any viewer) attaching and detaching freely — which removes the app from the machine’s lifetime with minimum further change. Until that lands, daemon-side fixes activate only on a deliberately scheduled restart, and nothing in this document assumes the app can be bounced casually.
