# Graphify integration

Hive agents answer "where is the thing that does X" by grepping, and they do it badly on unfamiliar repos: broad reads, wrong files, burned context. Graphify ([Graphify-Labs/graphify](https://github.com/Graphify-Labs/graphify), MIT) builds a queryable knowledge graph of a repository — locally, with tree-sitter, zero LLM calls in code-only operation — and exposes it through a CLI and an MCP server with 10 tools. This document specifies how Hive adopts it: installed only with explicit consent, pinned and hash-verified, confined to the repo, attached to agent spawns through the existing MCP scoping machinery, and measured rather than assumed to be used. The implementation lives in `src/adapters/graphify.ts` (install, builds, digest, hygiene), `src/daemon/graphify-service.ts` (the per-repo server), and `src/cli/graphify.ts` (the consent surface).

Two principles govern everything below. First, **graph context is a hint, never an authority**: upstream's own published QA accuracy is 45–76%, so nothing Hive does may treat a graph answer as ground truth, and no Hive operation — spawn, landing, recovery — may ever block on graphify being present, healthy, or fast. A broken graph means agents run without graph context, loudly noted, never failed. Second, **the integration must be losslessly declinable**: a user who says no, or a machine without Python tooling, gets exactly the Hive that exists today.

## What we are integrating, verified

Facts checked against the live docs and PyPI on 2026-07-11, then against the pinned CLI itself — which disagreed with the docs in both directions, a concrete measure of how fast upstream moves. The wrapper code, not this doc, owns exact flag spellings.

- PyPI package **`graphifyy`** (double y; the CLI binary is `graphify`), v0.9.12 as of 2026-07-10, MIT, pre-1.0 with multiple releases per day.
- **Provenance smells confirmed**: the PyPI project page lists its homepage as `safishamsi/graphify`, not the `Graphify-Labs` org, and the maintainer is the alias "captainturbo". None of this proves malice — it does mean install-time verification is mandatory, and it blocks on-by-default (see Rollout).
- Extraction is two-tier: **code is parsed locally with tree-sitter and never leaves the machine**; docs and media (`.md .mdx .qmd .html .txt .rst .yaml .yml`, PDFs, images, video) are sent to an LLM backend, and headless extraction with doc files present and no backend configured **fails with an error** rather than skipping — the fail-closed property the design leans on. The pinned CLI has an explicit **`extract <path> --code-only`** flag ("index code (local AST, no API key) and skip doc/paper/image files"); the web docs claim no such flag exists. They lag the near-daily releases — the first of several places the live binary, not the docs, had to be the source of truth.
- Output lands in `graphify-out/` (`graph.json`, `graph.html`, `GRAPH_REPORT.md`). Incremental re-extraction is `graphify update <path>` in the pinned CLI ("re-extract code files, no LLM needed") — the docs spell it `--update`, a second docs-vs-binary drift. Community *naming* during clustering is another LLM path (`--backend`, `--no-label`), distinct from extraction.
- The MCP server is the venv's `graphify-mcp` entry point (equivalently `python -m graphify.serve`), stdio by default, HTTP via `--transport http --host --port`, serving MCP at `/mcp`. It import-errors unless the **`graphifyy[mcp]` extra** is installed. Probed live, it exposes **10 tools**, not the docs' "8 (7 named)": `query_graph`, `get_node`, `get_neighbors`, `get_community`, `god_nodes`, `graph_stats`, `shortest_path`, `list_prs`, `get_pr_impact`, `triage_prs`.
- Installer behavior is version-specific and was re-probed against pinned **0.9.12** with separate throwaway repositories, empty fake homes, and the scrubbed environment. Bare `graphify install` is documented upstream to default to the user's global assistant profile; Hive did not probe that unsafe default and still never invokes it. `graphify claude install` and `graphify codex install` were measured repo-local with zero fake-home writes: respectively `CLAUDE.md` + `.claude/settings.json`, and `AGENTS.md` + `.codex/hooks.json`. Their `--project` variants were also measured: they additionally wrote the packaged skill and references under `.claude/skills/graphify/` or `.codex/skills/graphify/` (plus `.claude/CLAUDE.md` for Claude), still with zero fake-home writes. None solves Hive's delivery problem: these untracked files do not propagate into linked worktrees, and this repo ignores `.claude/` and `.codex/`. Hive therefore invokes none of the installers; it adopts the hook mechanism through daemon-written worktree config. One more pinned-binary divergence is load-bearing, and it inverted between codex releases: the installed Codex hook calls `graphify hook-check`, whose 0.9.12 implementation deliberately emits nothing on the claim that Codex rejects Claude's `PreToolUse.additionalContext` shape. Measured against codex 0.144.1 with a local mock model provider, the truth is now the opposite — a PreToolUse `{"systemMessage": …}` is parsed and then silently dropped (the text never reaches a model request), while the Claude-style `hookSpecificOutput.additionalContext` **is** injected, as a developer message, in both `exec` and the TUI. Hive shipped the `systemMessage` shape first and no Codex agent ever received a nudge; the hook now emits the one shape both CLIs honor.

## Install: consent, pin, and hashes

Graphify ships as a **Hive-built frozen bundle**: a standalone artifact Hive freezes from the pinned closure, publishes on its own release tag, and verifies against a sha256 embedded in the Hive binary (`src/adapters/graphify-artifacts.ts`; the feasibility measurements and build pipeline live in `graphify-bundling.md`). The user needs no uv, no Python, no PyPI — the whole toolchain question exists only on Hive's build machines. Vendoring upstream's Python *source tree* into Hive's repo remains rejected (it would import upstream's code and churn into Hive's own provenance); the bundle is different in kind: upstream's code never enters Hive's repo, and upstream's churn reaches a user only when a Hive developer deliberately bumps the pin and ships new artifacts. An earlier shipped design installed through `uv pip install --require-hashes` into a Hive-owned venv — chosen then because uv could fetch a managed Python and hashes made the PyPI fetch trustworthy — and it lost to the bundle on the grounds the risk register had already named: it required a third-party toolchain on every user machine ("graphify broke because my Python is weird" was a standing support surface), fetched from an origin Hive does not control, and dented the single-binary promise. The bundle closes all three: one origin (Hive's releases, the same trust as `hive update`), one verification (an embedded hash over the exact published bytes), zero toolchain.

Consent is decided at **`hive init`**, and the tension it resolves is worth recording. This section previously ruled that no Hive CLI command prompts — non-interactivity is a design principle init's own doc comment states — and rejected an interactive y/n for exactly that reason. The product decision is that a human running init should be *offered* graphify (it is the recommended configuration, and a decision buried in a help page is not much of an offer). Both are right, and the resolution is layered so neither gives ground: explicit flags (`hive init --graphify` / `--no-graphify`) always win and never prompt; with no flag, a TTY is asked once — the question names exactly what will be installed (Hive's bundle, sha256-verified, into `$HIVE_HOME/tools`) and that code is parsed locally with nothing leaving the machine — and the answer either way is persisted so it is never re-asked; non-interactive with no flag **declines for that run** and prints one line naming the enable command, because a script that did not ask for graphify must not acquire it as a side effect. `hive init` in CI behaves byte-for-byte as before: no prompt, no install, no hang. `hive graphify enable` remains the scriptable spelling everything resolves to, and running it is still consent in itself. A platform with no published bundle is never asked — there is nothing installable to consent to — and gets one honest line instead.

The mechanics, in order:

1. The Hive repo carries `graphify.lock`: a fully hash-pinned requirements file (`graphifyy[mcp]` at the exact pinned version **plus its entire transitive closure**, every artifact with a sha256 — the `[mcp]` extra because the MCP server import-errors without it), generated with `uv pip compile --generate-hashes` and reviewed like any other dependency change. The lock is what the bundle is *built from* — the freeze input — so a pin bump is still a reviewable diff of exactly which artifacts changed. The pin embedded in the binary still comes from this lock; the per-platform artifact hashes ride beside it in `graphify-artifacts.ts`.
2. Enable resolves this platform's artifact from the embedded registry, downloads it from Hive's release tag, verifies the sha256 of the downloaded bytes against the embedded constant, and only then unpacks into `$HIVE_HOME/tools/graphify/<pin>/` — one immutable directory per pin, so a bump can never layer onto a stale install. A hash mismatch refuses loudly and leaves nothing behind; there is no half-installed state for anything to trip over. Both entry points are probed (`--help`) before the install reports success.
3. The daemon invokes the tool only by absolute path (`$HIVE_HOME/tools/graphify/<pin>/graphify`, and `…/graphify-mcp` for the server). Nothing goes on `PATH`, and a `graphify` the user installed themselves is neither touched nor trusted.

The remaining rejected alternatives: `uv tool install` (no `--require-hashes` — verified against uv's CLI — and it pollutes the user's tool list); `pipx` (second toolchain, same hash gap); embedding the bundle inside the Hive binary itself (rejected on measured numbers in `graphify-bundling.md`: ~40% binary growth re-downloaded on every Hive update for a pin that deliberately moves rarely).

`hive graphify disable` stops the per-repo server and all injection; the installed bundle stays for cheap re-enable. Worktree-local hook scripts are derived Hive runtime config: without a live server they exit zero with no output, and any subsequent spawn/configuration removes them. `disable --purge` deletes `$HIVE_HOME/tools/graphify/` and the repo's `graphify-out/`; **`hive uninstall --repo`** is the full repo-level uninstall (skills, worktrees, exclude entries, derived state — see `src/cli/uninstall.ts` for the audited inventory), and plain `hive uninstall` removes Hive from the machine. The uninstall story staying this small is the acceptance test for the whole repo-local posture, and the bundle made it smaller: no venv, no uv cache, no managed Python anywhere on the user's machine. Enable/disable persist the per-repo state first (a `graphify.toml` under Hive's project state dir, never in the repo) and then poke a running daemon's `POST /graphify` route best-effort — the state file is the single source of truth, the daemon converges on it, and an unreachable daemon costs only immediacy because its next start reads the same file.

## Repo-local guarantees

**Code-only, enforced twice.** Lock one: every build runs `extract <path> --code-only`, the pinned CLI's own zero-LLM switch — code is tree-sitter-parsed locally, doc/paper/image files are skipped entirely. Lock two, the fail-closed backstop: the daemon runs every graphify invocation with a **scrubbed allowlist environment** — `PATH=/usr/bin:/bin`, a `HOME` pointed into Hive's own tools dir (so upstream's `~/.graphify` global state is never read or written), `TMPDIR`, and nothing else. An allowlist rather than a `*_API_KEY` scrub, so a provider key Hive has never heard of still cannot leak. Upstream's behavior without a backend key is to **error rather than degrade**, so if a pin bump ever drifts what `--code-only` covers, the failure mode is a loud broken-graph degradation (agents run graphless, noted), not silent exfiltration. An earlier revision of this design had Hive writing a `.graphifyignore` excluding every enrichment extension, because the docs claimed no code-only flag existed; the pinned binary proved otherwise, and the ignore file lost — it made Hive owner of an extension list that upstream should own, and dropped a repo-root file this posture is trying not to create. Enrichment is not exposed by any Hive surface; a future opt-in would be its own consent decision, not a flag.

**Ignore hygiene.** `graphify-out/` and `.graphifyignore` are appended to `.git/info/exclude`, not `.gitignore`: graphify enablement is a per-machine choice, and Hive editing a tracked file the whole team shares — today Hive writes no `.gitignore` anywhere in `src/` — was rejected as uninvited repo mutation. The entry goes to the git *common* dir, so one line covers every linked worktree. The enable step verifies the exclusion took effect with `git check-ignore --no-index graphify-out/probe .graphifyignore`; plain `check-ignore` consults the index and reports "not ignored" for anything already tracked, so only the `--no-index` form on a fresh name is evidence.

**Vendored code stays out of the graph.** Measured on this repo, 51% of all graph nodes were vendored Swift under `workspace/.build/checkouts/`, and the damage landed at query start-node selection: "spawning" anchored on a vendored `AgentFeed.swift`, `.attach()` on SwiftTerm's `BufferLine.swift`, so every traversal started in code no agent asks about. (The flood was independently heavy enough that an unanchored content search over the tree OOM-killed a 12 GB-budget search process.) The pinned binary honours gitignore rules **only at the scan root** — `_load_graphifyignore` walks ancestors, never descendants — so the nested `workspace/.gitignore: .build/` that git itself respects was invisible to extraction, and `extract` has no `--exclude` flag: exclusion is a `.graphifyignore` question, and the binary reads that file nowhere but the repo root. Hive therefore generates `<root>/.graphifyignore` before every build (`writeGraphifyIgnore`, `src/adapters/graphify.ts`): a short static floor of commonly *committed* vendored dirs (`vendor/`, `third_party/`, `Pods/`, `.build/`, …) plus the directories the repo's own gitignore machinery already excludes, taken verbatim from `git ls-files --others --ignored --exclude-standard --directory` — the team's own declaration of "not our code", per-repo and ecosystem-free, which is what makes the rule general instead of a hand-maintained ecosystem list. The file is a repo-root file only because the binary accepts no other location; the `.git/info/exclude` entry above keeps it out of `git status`, `disable --purge` removes it, and a `.graphifyignore` not starting with Hive's marker line is user-authored and never touched or deleted. Over-exclusion being a silent failure, the build detail names what was excluded out loud. Rebuilt with the generated file, this repo's graph went from 10,033 nodes (51.3% vendored, 25.3% `src/`) to 4,895 nodes (0% vendored, 51.8% `src/`), and the same query's start nodes stopped resolving into SwiftTerm.

**One server per repo, owned by the daemon.** The MCP server runs as a single daemon-managed process per repository (`GraphifyService`, `src/daemon/graphify-service.ts`): the venv's `graphify-mcp` in HTTP mode, bound to `127.0.0.1` on an OS-assigned free port, serving MCP at `/mcp`, scrubbed like every other invocation. The child is *held*, not detached — it must die with the daemon rather than leak — with a readiness poll before it is ever advertised and an exit watcher so a crashed server can never be a URL that spawns still attach. Started on daemon start (and via `POST /graphify`) when the repo opted in, restarted after each successful rebuild so every agent queries the current graph. The server is pointed at a **serving snapshot** under Hive's project state dir, never at the live `graphify-out/graph.json`: the serve process re-reads its graph file from disk on every query (upstream hot-reload), and `graphify update` rewrites the live file in place, so serving it directly opened a "graph.json not found" window during every post-landing rebuild — measured 2026-07-12, when a freshly spawned agent's first `query_graph` call landed inside one. Each restart refreshes the snapshot (tmp+rename) before the new process comes up; "the old server keeps serving the last good graph" is only true when the file it serves is one no rebuild mutates. The alternative — a stdio server spawned per agent, the shape of the existing `hive-channel` bridge (`src/adapters/tools/claude.ts:59`) — was seriously considered for its simplicity (no port, no lifecycle, dies with the agent) and rejected because N concurrent agents would each hold a full copy of the graph in memory, and a long-lived agent would keep serving a graph from before every landing since its spawn; the daemon-owned instance gets restart-on-rebuild for free. The cost is port and health management, both patterns the daemon already carries.

## Lifecycle: build, update, degrade

The graph is built at enable time, in the enable command's own foreground with progress printed — a consent command that says "building the graph" and then does it visibly beats a background job the user has to poll, and the adoption path for existing repos is the same command, with no separate migration machinery. (This repo, ~3,600 nodes, builds in about seven seconds; a much larger repo waits minutes, in plain sight.) A daemon that starts enabled but finds no graph — the state file survived, `graphify-out/` did not — rebuilds in the background before bringing the server up; agents spawned before it completes simply spawn graphless, noted in their brief.

Incremental updates hook the one choke point every landing already passes through: the daemon's land path, immediately after the fast-forward merge succeeds (`landAgent` in `src/daemon/server.ts`). A fire-and-forget `scheduleRebuild()` follows the precedent of `void this.rebuildMemoryIndex().catch(...)` on daemon start: never awaited by the land response — the merge result is already on the wire. The rebuild is the pinned CLI's `graphify update <root> --force` (code files only, no LLM): `--force` because upstream refuses a rebuild that shrinks the graph, and landings legitimately delete code; the exit-0 gate below is what protects against a corrupt shrink instead. Rebuilds serialize and coalesce (any burst of landings during a rebuild queues exactly one follow-up), and the server is restarted only after an update exits zero, so a failed rebuild leaves the previous good graph serving rather than nothing. Out-of-band history movement (a human force-push, a rebase outside Hive) is caught lazily: staleness is tolerated until the next landing or daemon restart, because chasing every ref update buys accuracy the 45–76% source material cannot deliver anyway.

The degradation rule is absolute and worth stating as the contract it is: **spawn and landing never block on graphify.** Absent (not installed, consent declined, no bundle published for this platform), broken (extract failing, server unhealthy), or slow (build in progress, query timeout): all three collapse to the same behavior — the agent runs without graph context, its brief says so in one line, `hive graphify status` and the `POST /graphify` response report the server state, and the daemon log carries the cause. No spawn failure, no land failure, no retry loop an agent can observe. "Loudly noted" is load-bearing: a silently missing graph would look identical to a repo with nothing to find, and per Hive's own protocol an absent field is unknown, never false.

## MCP attachment

Hive already scopes MCP per spawn; graphify rides that machinery rather than growing new config surface.

For Claude agents, scoping is an allowlist: the daemon writes a worktree-local `.mcp.json` and passes `--strict-mcp-config`, so an agent has exactly the servers that file names (`claudeMcpConfigPath` in `src/daemon/spawner-impl.ts`; the file is written by `writeClaudeAgentConfig`, `src/adapters/tools/claude.ts`). When graphify is enabled and healthy at spawn time, the daemon adds a `graphify` entry of type `http` pointing at the per-repo server's loopback port, alongside the existing `hive` and `hive-channel` entries — and when it is not, the writer also *removes* any stale `graphify` entry a previous spawn merged into that file, because a dead URL would cost every agent a connect-timeout. For Codex agents, scoping is an exclude-list over inherited servers (`buildCodexMcpExclusionArgs`, `src/adapters/tools/mcp-scope.ts`); graphify is attached symmetrically, as a `-c mcp_servers.graphify.url` config argument, and while attached, `graphify` joins the exclusion keep-set beside `HIVE_MCP_SERVERS` — otherwise a user's own inherited server of the same name would be disabled by the very pass whose url override Hive just claimed. The spawner reads the server URL exactly once per spawn, synchronously, before prompt assembly, so the prompt's directive, the digest, and the MCP config all describe the same observation; null attaches nothing anywhere. Read-only control restarts (a revoked agent's replacement, which must not act) attach nothing by design, for the same reason they detach the human's own servers: pure context cost on a process that exists only to acknowledge. One known gap: Codex spawns driven through the experimental app-server host take a different launch path and do not yet attach graphify; they still get the layer-1 digest.

## Making agents actually use it

Hive's own telemetry says shipped guidance is not followed guidance: when the karpathy-guidelines skill relied on progressive disclosure, 22% of agents opened it. A graph nobody queries is pure cost, so the design assumes noncompliance and layers accordingly.

**Layer 1 — value with zero agent compliance.** The daemon injects graph-derived context directly into the spawn brief, as a third arm of the `Promise.all` that already builds the memory index and the scoped brief before `buildAgentPrompt` (`src/daemon/spawner-impl.ts`). The arm is **Hive's own locate over `graph.json`** (`buildTargetedGraphBrief`, `src/adapters/graphify.ts`), not the binary's `query`: `query` anchors its BFS on its own keyword matcher and accepts no explicit start nodes, and that matcher is the measured failure — on the acceptance question ("where does the daemon attach the graphify MCP server to a spawning agent") it anchored "spawning" on vendored Swift and, even de-poisoned and de-truncated, ranked the graphify implementation cluster above the files that answer. Hive's locate scores files by IDF-weighted name/symbol/path match, expands one hop through the graph's own edges (hub-normalized, so `db.ts`-shaped files cannot win on connectivity), and adds the matched-symbol rule — a seed imports a symbol whose *name* matches the task, so the file *defining* it surfaces — which is what finds the config writer no locate-question ever names. Output stays in the binary's NODE/EDGE grammar (file:line citations, provenance tags), ~6KB, with a kept-N-of-M line so truncation never reads as an empty graph. Measured across six real orientation questions on this repo, every answer file surfaces, including all three the acceptance question demands; ~10ms per brief with no subprocess. The old `query --budget 40000` + `selectGraphBrief` selection path (which fixed the original zero-edges truncation: all NODE lines serialize before the first EDGE, so budget 1200 + a 6KB head cap delivered 51 nodes/0 edges every spawn) remains the fallback for malformed, matchless, or >50MB graphs, still subprocess-time-boxed. One fixed preamble sentence states that the context is advisory, 45–76% accurate upstream, and must be verified against source before anything load-bearing. (An earlier sketch added a second call for summary statistics; the query's own traversal header already carries node counts, and a digest that is mostly statistics is a digest nobody reads.) The arm is time-boxed at 3 seconds; measured against this repo it returns in ~250ms. Repos that never opted in get silence. Opted-in repos with a sick graph get one loud line — "Graph context: unavailable (<cause>); proceeding without it." — never a failed or slower spawn. Prompt delivery is already file-based (`writeLaunchPrompt`, `src/daemon/launch-prompt.ts`), so the tmux 16KB command ceiling that binds spawn commands does not bind this.

**Layer 2 — tools, directive, and harness nudge.** The MCP attachment above puts the 10 tools in the agent's hands, and the spawn prompt directive is **graph-first** (a product decision: when graphify is on, agents use it, and grep only when they have to): start from the brief's Graph locate leads, walk with `get_neighbors`/`shortest_path`, use `query_graph` with `token_budget: 16000` for sweeps — the schema default of 2000 provably returns zero of the cited, tagged edges. The mandate stays honest by naming its own limits concretely: raw search remains correct for exact string/error-message hunts, files the graph does not index, and graph leads that verify empty. `query_graph` alone still loses to grep on locate-questions (its BFS anchoring, measured), so mid-task locate-questions go to **`graph_locate`**, a tool on Hive's own MCP server (not graphify's — that surface is pre-1.0 and not ours): the same Hive-side locate the brief runs, reading the serving snapshot first so a rebuild can never yank the file mid-answer, replying in the same cited NODE/EDGE grammar, and answering "no strong leads — use content grep" honestly when the question's vocabulary does not match the code's names, instead of returning confident look-alikes. Measured byte-identical to the spawn-brief path on all six acceptance questions through the MCP layer (8–46ms; one parsed graph cached per rebuild), and every failure — not enabled, not built, corrupt, oversized — degrades to an honest answer, never an error. The layer-3 adoption count includes it alongside the `mcp__graphify__*` calls, so graph-first usage is measured wherever it happens. A healthy-server control proved prompt text alone insufficient: neutral capability-tracing agent Arash finished with `graphifyCalls=0` while positive controls April and Anton recorded 2 and 1. Hive therefore writes a small worktree-local `PreToolUse` hook beside each agent's existing runtime config, delivered to both CLIs through the same `hookSpecificOutput.additionalContext` shape — Claude before Bash searches and Read/Glob, Codex before Bash searches through the command-line hook overrides it already uses (never a project `.codex/config.toml`, which that launch does not load; Codex normalizes its shell tool to the name "Bash" in hook input, so one matcher and one search filter serve both). Codex's alternative `{"systemMessage": …}` output is parsed and silently discarded by codex 0.144.1 — measured with a mock provider; see the hook-check note above — which is why the first shipped nudge never reached a single Codex agent. The hook only nudges; it never queries, blocks, or gates landing. It is emitted only with a ready server URL, probes that loopback endpoint under a 30ms deadline, and exits zero with no stdout or stderr when the server is absent, dead, slow, or the intercepted command is irrelevant. Measured warm-path cost on this repo was 14.81ms median / 16.10ms p95 for a matching healthy search and 5.11ms / 5.64ms for irrelevant Bash.

**Layer 3 — measure, don't assume.** The telemetry sweep already tails each agent's durable artifacts (`src/daemon/tool-telemetry.ts`); counting rides it. `readGraphifyCalls` advances an incremental cursor per agent — only bytes appended since the last sweep are read, never a partial line — counting Claude `tool_use` entries named `mcp__graphify__*` and Codex `mcp_tool_call_end` events naming the `graphify` server (both shapes verified against real artifacts). Claude reads are keyed by `toolSessionId`, never "newest file in the directory" — path-keyed readers are a known Hive bug class that inherits a dead predecessor's transcript across respawns; the Codex rollout is still discovered per worktree and carries that known aliasing, bounded by a changed-path count reset. Counts live in daemon memory on purpose: the artifacts are durable, so a restart recounts from offset zero instead of trusting a stale number. `hive_status` surfaces the per-agent `graphifyCalls` (null is unknown, never zero) on daemons running graphify. This is what makes the rollout decision below empirical: if agents demonstrably don't call the tools, layer 1 is the whole product and layers 2–3 told us so cheaply.

There is deliberately **no land-time enforcement** — no "did you consult the graph" gate. It would be unverifiable in exactly the way Hive's protocol warns about (an MCP call proves an act, not that the answer informed anything) and would put a 45–76%-accurate oracle in the landing path, violating the never-block rule for no measurable gain.

## Rollout and version policy

Graphify ships **recommended, per-repo, by consent**: `hive init` offers it to a human at a terminal as the recommended choice (default yes on Enter), and never installs it for a script that did not ask. This supersedes the earlier default-off posture, which gated any recommendation on upstream reaching 1.0, resolved PyPI provenance, and proven agent adoption — conditions that guarded against upstream code reaching users through an origin Hive did not control. The bundle changes what the gate was protecting: users now run only bytes Hive built, vetted, and signed, fetched from Hive's own release, so recommending graphify is recommending Hive's own artifact, and a breaking or malicious upstream change is caught by Hive developers at bump time, before it can reach a user. What remains deliberately conservative: nothing installs without a yes (a flag, a prompt answer, or an explicit `enable`), a recorded decline is never re-asked, and the layer-3 telemetry question — do graph tools earn their context cost — still stands as the evidence that would justify pushing the recommendation harder or retiring the feature.

**Cross-repo validation of the locate brief (2026-07-12).** The layer-1 locate was tuned against six orientation questions on this repository, so before any wider default it was validated blind against four repos it was never tuned on, chosen for diversity: Archon (Python, flat layout plus in-tree version snapshots), FieldLedger (C# + Next.js `apps/` monorepo), SyllabusAI (FastAPI + React full-stack), and wayfaer (Java, 20,271 nodes / 62,401 edges — four times this repo). Method: ground-truth answer files for five real orientation questions per repo were pinned by direct inspection *before* any locate ran. Result: **16/20 strict**, with every constant (seed/expansion counts, damps, caps) untouched across a 28× node-count range, and 57–65ms per brief on the largest graph. Of the four misses, two carried correct answer files the pinned truth was too narrow to credit (wayfaer's `ColumnGenerator.java` for terrain generation, `CommandExecutor.java` for keybinds), one surfaced the answer's direct neighbors (SyllabusAI's chat page and RAG agent but not the route file), and one is the real, stated limit: locate is **name-anchored**, so when a question uses ecosystem vocabulary the repo's file names don't (Archon's "langgraph" names `langgraph.json` config files, not `archon_graph.py`), it anchors on textual look-alikes — the same place name-based grep fails, and content-level search remains the right fallback, exactly as the directive says. Second stated limit: repos that keep historical copies of themselves in-tree (Archon's `iterations/` is 71% of its own graph) dilute locate, and correctly cannot be ignore-ruled away — they are tracked, first-party code. No constants were changed in response to these results; a flip-one-question tweak is overfitting with extra steps.

The pin moves on Hive's cadence, not upstream's: Hive does not chase daily releases. A bump is a deliberate act per Hive release at most — regenerate `graphify.lock` from its compile input, run the bundle pipeline (freeze, smoke-test extract/serve/query against a fixture repo, publish the per-platform artifacts, land the CI-built sha256s in `graphify-artifacts.ts`; the pipeline is specified in `graphify-bundling.md`), and adapt the wrapper if flags moved (pre-1.0, they do: the transport flags are already documented in two different spellings). A security fix upstream justifies an out-of-band bump through the same steps. The wrapper treats every graphify flag as pinned-version-specific; nothing outside the wrapper module spells a graphify flag.

## Risks carried forward

- **Hive owns the whole runtime now.** The frozen bundle removed the uv/Python support surface, and what replaced it is responsibility: every graphify crash on a user machine is Hive's artifact misbehaving, with no vendor toolchain to point at, and fixing one means shipping new artifacts through the bump pipeline. Contained by the pin-bump smoke suite and the degradation rule (a broken bundle collapses to "graphless, loudly noted"), but the escape hatch of "wait for an upstream binary distribution" is spent — Hive is the upstream binary distribution.
- **Accuracy churn.** 45–76% QA accuracy means wrong answers are routine, and pre-1.0 extraction changes can shift quality between pins without notice. The hint-never-authority framing, the advisory preamble in layer 1, and the small token budget are the mitigations; the residual risk is an agent trusting a wrong graph answer anyway, which no design can fully remove.
- **Supply chain.** The hash lock pins exactly what was reviewed, but hashes verify integrity, not intent: the smelly provenance (org/homepage mismatch, alias maintainer, a package name one typo from `graphify`) means the pinned code itself deserves the scrutiny of a vendored dependency at each bump. This risk is why on-by-default is gated on provenance resolution, and it never fully closes while the ownership question stands.
- **Privacy.** Enrichment is off behind two independent locks (`--code-only` + the scrubbed allowlist environment), and the backstop fails closed. Residual exposure is upstream changing what `--code-only` covers in a way that survives both locks; the pin-bump smoke test explicitly checks that a doc-bearing fixture repo extracts keyless without error or egress.

## Known limitations

- Codex spawns driven through the experimental app-server host do not yet attach the graphify MCP server (the default TUI driver does); they still receive the layer-1 digest. Close when that launch path grows config-override parity.
- The Codex call count inherits the worktree-scoped rollout discovery's known aliasing across respawns, bounded to one rollout by the changed-path reset. It heals when rollout reads move to a session-keyed lookup, which is a pre-existing Hive issue rather than a graphify one.
- The capability rights matrix (`capability-rights-matrix.md`) does not yet list the `/graphify` route — nor the older `/autonomy` it is modeled on; both carry operator-only write actions in `src/daemon/capabilities.ts`, which is the binding source.
