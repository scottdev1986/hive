# Graphify integration

Hive agents answer "where is the thing that does X" by grepping, and they do it badly on unfamiliar repos: broad reads, wrong files, burned context. Graphify ([Graphify-Labs/graphify](https://github.com/Graphify-Labs/graphify), MIT) builds a queryable knowledge graph of a repository — locally, with tree-sitter, zero LLM calls in code-only operation — and exposes it through a CLI and an MCP server with 10 tools. This document specifies how Hive adopts it: installed only with explicit consent, pinned and hash-verified, confined to the repo, attached to agent spawns through the existing MCP scoping machinery, and measured rather than assumed to be used. The implementation lives in `src/adapters/graphify.ts` (install, builds, digest, hygiene), `src/daemon/graphify-service.ts` (the per-repo server), and `src/cli/graphify.ts` (the consent surface).

Two principles govern everything below. First, **graph context is a hint, never an authority**: upstream's own published QA accuracy is 45–76%, so nothing Hive does may treat a graph answer as ground truth, and no Hive operation — spawn, landing, recovery — may ever block on graphify being present, healthy, or fast. A broken graph means agents run without graph context, loudly noted, never failed. Second, **the integration must be losslessly declinable**: a user who says no, or a machine without Python tooling, gets exactly the Hive that exists today.

## What we are integrating, verified

Facts checked against the live docs and PyPI on 2026-07-11, then against the pinned CLI itself — which disagreed with the docs in both directions, a concrete measure of how fast upstream moves. The wrapper code, not this doc, owns exact flag spellings.

- PyPI package **`graphifyy`** (double y; the CLI binary is `graphify`), v0.9.12 as of 2026-07-10, MIT, pre-1.0 with multiple releases per day.
- **Provenance smells confirmed**: the PyPI project page lists its homepage as `safishamsi/graphify`, not the `Graphify-Labs` org, and the maintainer is the alias "captainturbo". None of this proves malice — it does mean install-time verification is mandatory, and it blocks on-by-default (see Rollout).
- Extraction is two-tier: **code is parsed locally with tree-sitter and never leaves the machine**; docs and media (`.md .mdx .qmd .html .txt .rst .yaml .yml`, PDFs, images, video) are sent to an LLM backend, and headless extraction with doc files present and no backend configured **fails with an error** rather than skipping — the fail-closed property the design leans on. The pinned CLI has an explicit **`extract <path> --code-only`** flag ("index code (local AST, no API key) and skip doc/paper/image files"); the web docs claim no such flag exists. They lag the near-daily releases — the first of several places the live binary, not the docs, had to be the source of truth.
- Output lands in `graphify-out/` (`graph.json`, `graph.html`, `GRAPH_REPORT.md`). Incremental re-extraction is `graphify update <path>` in the pinned CLI ("re-extract code files, no LLM needed") — the docs spell it `--update`, a second docs-vs-binary drift. Community *naming* during clustering is another LLM path (`--backend`, `--no-label`), distinct from extraction.
- The MCP server is the venv's `graphify-mcp` entry point (equivalently `python -m graphify.serve`), stdio by default, HTTP via `--transport http --host --port`, serving MCP at `/mcp`. It import-errors unless the **`graphifyy[mcp]` extra** is installed. Probed live, it exposes **10 tools**, not the docs' "8 (7 named)": `query_graph`, `get_node`, `get_neighbors`, `get_community`, `god_nodes`, `graph_stats`, `shortest_path`, `list_prs`, `get_pr_impact`, `triage_prs`.
- `graphify install` writes the user's **global** assistant configuration (skills, rules files, hooks that intercept tool use). **Hive never invokes it, under any circumstances.** Everything graphify-shaped that Hive does is repo-local and daemon-mediated.

## Install: consent, pin, and hashes

Graphify is **installer-fetched, never vendored**. Vendoring a pre-1.0 Python tree into a project that ships as a single binary was rejected outright: it imports upstream's churn into Hive's release surface and its supply-chain exposure into Hive's own provenance. Consent is expressed by running **`hive graphify enable`**: the command states exactly what it is about to do — install `graphifyy==<pin>` hash-verified into a Hive-owned environment, build a local code-only graph under `graphify-out/`, nothing sent anywhere — and running it is the authorization, the same contract `hive init` already lives by. An interactive y/n prompt was considered and rejected: no Hive CLI command prompts (init's own doc comment makes non-interactivity a design principle), and a flag-plus-persisted-state consent is scriptable where a prompt is not. `hive init` reports where the decision stands — enabled, or one line naming the command — and never makes it. Not running `enable` is declining; nothing nags, and everything else in Hive works identically.

The task brief sketched `uv tool install graphifyy==<pin>` with hash verification. That exact shape is not buildable: `uv tool install` supports version pinning and `--constraints`, but has **no `--require-hashes`** (verified against uv's CLI). Pin-without-hashes is precisely the wrong posture for a package with provenance smells — a compromised maintainer can re-upload nothing, but can publish `0.9.13` and wait for a careless bump; hashes are also the only defense against index-level substitution. So the install runs through `uv pip`, which does support hash enforcement:

1. The Hive repo carries `graphify.lock`: a fully hash-pinned requirements file (`graphifyy[mcp]` at the exact pinned version **plus its entire transitive closure**, every artifact with a sha256 — the `[mcp]` extra because the MCP server import-errors without it), generated with `uv pip compile --generate-hashes` and reviewed like any other dependency change. The lock is inlined into the compiled binary through the same text-import mechanism that ships skills (`src/skills/shipped.ts`), because a user's machine has no Hive checkout to read it from. Hive owns the pin; it moves only when Hive deliberately bumps it (see Rollout).
2. Enable creates `$HIVE_HOME/tools/graphify/venv` fresh (`uv venv`, removing any previous venv so a pin bump never layers) and runs `uv pip install --require-hashes -r` against the embedded lock. uv fetches a managed Python if the machine lacks one, which keeps "Python absent" from being a separate failure mode. The install step alone runs with the caller's real environment — uv needs its cache and proxies, and the hashes, not the environment, are what make the fetch trustworthy.
3. The daemon invokes the tool only by absolute path (`$HIVE_HOME/tools/graphify/venv/bin/graphify`, and `…/graphify-mcp` for the server). Nothing goes on `PATH`, nothing appears in the user's `uv tool list`, and a `graphify` the user installed themselves is neither touched nor trusted.

The rejected alternatives: `uv tool install` (no hash verification, pollutes the user's tool list); `pipx` (second toolchain to require, same hash gap); verifying the top-level wheel's sha256 by hand and installing from the local file (leaves the transitive closure unverified — the lock file covers it for free). The cost of the chosen shape is a lock file to regenerate on every pin bump; that is a feature, since a bump becomes a reviewable diff of exactly which artifacts changed.

**When uv itself is absent**, Hive prints the one-line official install command and moves on with the feature disabled. It does not offer to run the installer: piping a vendor's shell script from the network is the same class of act this section exists to avoid, and doing it inside a consent prompt for a *different* tool buries the decision. This dents convenience deliberately — the alternative (Hive silently acquiring and thereafter owning a Python toolchain) was rejected as scope Hive cannot honor from a single binary.

`hive graphify disable` stops the per-repo server and all injection; the environment stays for cheap re-enable. `disable --purge` is the uninstall: it also deletes `$HIVE_HOME/tools/graphify/` and the repo's `graphify-out/` — and nothing else, because nothing else was ever written: no global config, no `PATH` entry, no shell hook, no assistant skill. The uninstall story being one flag is the acceptance test for the whole repo-local posture. Both commands persist the per-repo state first (a `graphify.toml` under Hive's project state dir, never in the repo) and then poke a running daemon's `POST /graphify` route best-effort — the state file is the single source of truth, the daemon converges on it, and an unreachable daemon costs only immediacy because its next start reads the same file.

## Repo-local guarantees

**Code-only, enforced twice.** Lock one: every build runs `extract <path> --code-only`, the pinned CLI's own zero-LLM switch — code is tree-sitter-parsed locally, doc/paper/image files are skipped entirely. Lock two, the fail-closed backstop: the daemon runs every graphify invocation with a **scrubbed allowlist environment** — `PATH=/usr/bin:/bin`, a `HOME` pointed into Hive's own tools dir (so upstream's `~/.graphify` global state is never read or written), `TMPDIR`, and nothing else. An allowlist rather than a `*_API_KEY` scrub, so a provider key Hive has never heard of still cannot leak. Upstream's behavior without a backend key is to **error rather than degrade**, so if a pin bump ever drifts what `--code-only` covers, the failure mode is a loud broken-graph degradation (agents run graphless, noted), not silent exfiltration. An earlier revision of this design had Hive writing a `.graphifyignore` excluding every enrichment extension, because the docs claimed no code-only flag existed; the pinned binary proved otherwise, and the ignore file lost — it made Hive owner of an extension list that upstream should own, and dropped a repo-root file this posture is trying not to create. Enrichment is not exposed by any Hive surface; a future opt-in would be its own consent decision, not a flag.

**Ignore hygiene.** `graphify-out/` is appended to `.git/info/exclude`, not `.gitignore`: graphify enablement is a per-machine choice, and Hive editing a tracked file the whole team shares — today Hive writes no `.gitignore` anywhere in `src/` — was rejected as uninvited repo mutation. The entry goes to the git *common* dir, so one line covers every linked worktree. The enable step verifies the exclusion took effect with `git check-ignore --no-index graphify-out/probe`; plain `check-ignore` consults the index and reports "not ignored" for anything already tracked, so only the `--no-index` form on a fresh name is evidence.

**One server per repo, owned by the daemon.** The MCP server runs as a single daemon-managed process per repository (`GraphifyService`, `src/daemon/graphify-service.ts`): the venv's `graphify-mcp` in HTTP mode, bound to `127.0.0.1` on an OS-assigned free port, serving MCP at `/mcp`, scrubbed like every other invocation. The child is *held*, not detached — it must die with the daemon rather than leak — with a readiness poll before it is ever advertised and an exit watcher so a crashed server can never be a URL that spawns still attach. Started on daemon start (and via `POST /graphify`) when the repo opted in, restarted after each successful rebuild so every agent queries the current graph. The alternative — a stdio server spawned per agent, the shape of the existing `hive-channel` bridge (`src/adapters/tools/claude.ts:59`) — was seriously considered for its simplicity (no port, no lifecycle, dies with the agent) and rejected because N concurrent agents would each hold a full copy of the graph in memory, and a long-lived agent would keep serving a graph from before every landing since its spawn; the daemon-owned instance gets restart-on-rebuild for free. The cost is port and health management, both patterns the daemon already carries.

## Lifecycle: build, update, degrade

The graph is built at enable time, in the enable command's own foreground with progress printed — a consent command that says "building the graph" and then does it visibly beats a background job the user has to poll, and the adoption path for existing repos is the same command, with no separate migration machinery. (This repo, ~3,600 nodes, builds in about seven seconds; a much larger repo waits minutes, in plain sight.) A daemon that starts enabled but finds no graph — the state file survived, `graphify-out/` did not — rebuilds in the background before bringing the server up; agents spawned before it completes simply spawn graphless, noted in their brief.

Incremental updates hook the one choke point every landing already passes through: the daemon's land path, immediately after the fast-forward merge succeeds (`landAgent` in `src/daemon/server.ts`). A fire-and-forget `scheduleRebuild()` follows the precedent of `void this.rebuildMemoryIndex().catch(...)` on daemon start: never awaited by the land response — the merge result is already on the wire. The rebuild is the pinned CLI's `graphify update <root> --force` (code files only, no LLM): `--force` because upstream refuses a rebuild that shrinks the graph, and landings legitimately delete code; the exit-0 gate below is what protects against a corrupt shrink instead. Rebuilds serialize and coalesce (any burst of landings during a rebuild queues exactly one follow-up), and the server is restarted only after an update exits zero, so a failed rebuild leaves the previous good graph serving rather than nothing. Out-of-band history movement (a human force-push, a rebase outside Hive) is caught lazily: staleness is tolerated until the next landing or daemon restart, because chasing every ref update buys accuracy the 45–76% source material cannot deliver anyway.

The degradation rule is absolute and worth stating as the contract it is: **spawn and landing never block on graphify.** Absent (not installed, consent declined, uv missing), broken (extract failing, server unhealthy), or slow (build in progress, query timeout): all three collapse to the same behavior — the agent runs without graph context, its brief says so in one line, `hive graphify status` and the `POST /graphify` response report the server state, and the daemon log carries the cause. No spawn failure, no land failure, no retry loop an agent can observe. "Loudly noted" is load-bearing: a silently missing graph would look identical to a repo with nothing to find, and per Hive's own protocol an absent field is unknown, never false.

## MCP attachment

Hive already scopes MCP per spawn; graphify rides that machinery rather than growing new config surface.

For Claude agents, scoping is an allowlist: the daemon writes a worktree-local `.mcp.json` and passes `--strict-mcp-config`, so an agent has exactly the servers that file names (`claudeMcpConfigPath` in `src/daemon/spawner-impl.ts`; the file is written by `writeClaudeAgentConfig`, `src/adapters/tools/claude.ts`). When graphify is enabled and healthy at spawn time, the daemon adds a `graphify` entry of type `http` pointing at the per-repo server's loopback port, alongside the existing `hive` and `hive-channel` entries — and when it is not, the writer also *removes* any stale `graphify` entry a previous spawn merged into that file, because a dead URL would cost every agent a connect-timeout. For Codex agents, scoping is an exclude-list over inherited servers (`buildCodexMcpExclusionArgs`, `src/adapters/tools/mcp-scope.ts`); graphify is attached symmetrically, as a `-c mcp_servers.graphify.url` config argument, and while attached, `graphify` joins the exclusion keep-set beside `HIVE_MCP_SERVERS` — otherwise a user's own inherited server of the same name would be disabled by the very pass whose url override Hive just claimed. The spawner reads the server URL exactly once per spawn, synchronously, before prompt assembly, so the prompt's directive, the digest, and the MCP config all describe the same observation; null attaches nothing anywhere. Read-only control restarts (a revoked agent's replacement, which must not act) attach nothing by design, for the same reason they detach the human's own servers: pure context cost on a process that exists only to acknowledge. One known gap: Codex spawns driven through the experimental app-server host take a different launch path and do not yet attach graphify; they still get the layer-1 digest.

## Making agents actually use it

Hive's own telemetry says shipped guidance is not followed guidance: when the karpathy-guidelines skill relied on progressive disclosure, 22% of agents opened it. A graph nobody queries is pure cost, so the design assumes noncompliance and layers accordingly.

**Layer 1 — value with zero agent compliance.** The daemon injects graph-derived context directly into the spawn brief, as a third arm of the `Promise.all` that already builds the memory index and the scoped brief before `buildAgentPrompt` (`src/daemon/spawner-impl.ts`). The arm is one local CLI call, `graphify query "<task text>" --budget 1200` against the built graph — the query's own token budget does the trimming, a 6KB character cap backstops it, and one fixed preamble sentence states that the context is advisory, 45–76% accurate upstream, and must be verified against source before anything load-bearing. (An earlier sketch added a second call for summary statistics; the query's own traversal header already carries node counts, and a digest that is mostly statistics is a digest nobody reads.) The arm is time-boxed at 3 seconds; measured against this repo it returns in ~250ms. Repos that never opted in get silence. Opted-in repos with a sick graph get one loud line — "Graph context: unavailable (<cause>); proceeding without it." — never a failed or slower spawn. Prompt delivery is already file-based (`writeLaunchPrompt`, `src/daemon/launch-prompt.ts`), so the tmux 16KB command ceiling that binds spawn commands does not bind this.

**Layer 2 — tools plus one directive.** The MCP attachment above puts the 10 tools in the agent's hands, and the spawn prompt carries exactly one added directive — prefer one `query_graph` call over broad grep sweeps when orienting, treat answers as leads to verify, never as authority — emitted only when the server is actually being attached to that spawn, so the prompt never advertises tools the agent does not have. One sentence in the spawn prompt — the channel the 22% lesson proved agents actually read — not a skill, not a doc pointer.

**Layer 3 — measure, don't assume.** The telemetry sweep already tails each agent's durable artifacts (`src/daemon/tool-telemetry.ts`); counting rides it. `readGraphifyCalls` advances an incremental cursor per agent — only bytes appended since the last sweep are read, never a partial line — counting Claude `tool_use` entries named `mcp__graphify__*` and Codex `mcp_tool_call_end` events naming the `graphify` server (both shapes verified against real artifacts). Claude reads are keyed by `toolSessionId`, never "newest file in the directory" — path-keyed readers are a known Hive bug class that inherits a dead predecessor's transcript across respawns; the Codex rollout is still discovered per worktree and carries that known aliasing, bounded by a changed-path count reset. Counts live in daemon memory on purpose: the artifacts are durable, so a restart recounts from offset zero instead of trusting a stale number. `hive_status` surfaces the per-agent `graphifyCalls` (null is unknown, never zero) on daemons running graphify. This is what makes the rollout decision below empirical: if agents demonstrably don't call the tools, layer 1 is the whole product and layers 2–3 told us so cheaply.

There is deliberately **no land-time enforcement** — no "did you consult the graph" gate. It would be unverifiable in exactly the way Hive's protocol warns about (an MCP call proves an act, not that the answer informed anything) and would put a 45–76%-accurate oracle in the landing path, violating the never-block rule for no measurable gain.

## Rollout and version policy

Graphify ships **opt-in**: default off, enabled per repo by explicit consent. Pre-1.0 near-daily churn, unresolved provenance, and unproven agent adoption each independently justify that posture. Flipping to on-by-default requires all of: upstream reaches 1.0 with a stabilized CLI/MCP surface; the PyPI provenance question is resolved (package ownership verifiably tied to Graphify-Labs, or Hive re-pins to a fork it controls); and Hive's own layer-3 telemetry shows graph tools earning their context cost in real sessions. Any one failing keeps it opt-in.

The pin moves on Hive's cadence, not upstream's: Hive does not chase daily releases. A bump is a deliberate act per Hive release at most — regenerate `graphify.lock`, diff the artifact set, smoke-test extract/serve/query against a fixture repo, adapt the wrapper if flags moved (pre-1.0, they do: the transport flags are already documented in two different spellings). A security fix upstream justifies an out-of-band bump through the same steps. The wrapper treats every graphify flag as pinned-version-specific; nothing outside the wrapper module spells a graphify flag.

## Risks carried forward

- **The single-binary promise is dented.** Hive installs as one binary; this feature requires uv (and through it a Python). Contained by making the dependency opt-in, lazily acquired, and invisible when declined — but a support surface ("graphify broke because my Python is weird") now exists that Hive did not have. If that surface grows disproportionate, the escape hatch is dropping to CLI-only integration or a future upstream binary distribution.
- **Accuracy churn.** 45–76% QA accuracy means wrong answers are routine, and pre-1.0 extraction changes can shift quality between pins without notice. The hint-never-authority framing, the advisory preamble in layer 1, and the small token budget are the mitigations; the residual risk is an agent trusting a wrong graph answer anyway, which no design can fully remove.
- **Supply chain.** The hash lock pins exactly what was reviewed, but hashes verify integrity, not intent: the smelly provenance (org/homepage mismatch, alias maintainer, a package name one typo from `graphify`) means the pinned code itself deserves the scrutiny of a vendored dependency at each bump. This risk is why on-by-default is gated on provenance resolution, and it never fully closes while the ownership question stands.
- **Privacy.** Enrichment is off behind two independent locks (`--code-only` + the scrubbed allowlist environment), and the backstop fails closed. Residual exposure is upstream changing what `--code-only` covers in a way that survives both locks; the pin-bump smoke test explicitly checks that a doc-bearing fixture repo extracts keyless without error or egress.

## Known limitations

- Codex spawns driven through the experimental app-server host do not yet attach the graphify MCP server (the default TUI driver does); they still receive the layer-1 digest. Close when that launch path grows config-override parity.
- The Codex call count inherits the worktree-scoped rollout discovery's known aliasing across respawns, bounded to one rollout by the changed-path reset. It heals when rollout reads move to a session-keyed lookup, which is a pre-existing Hive issue rather than a graphify one.
- The capability rights matrix (`capability-rights-matrix.md`) does not yet list the `/graphify` route — nor the older `/autonomy` it is modeled on; both carry operator-only write actions in `src/daemon/capabilities.ts`, which is the binding source.
