# Graphify integration

Hive agents answer "where is the thing that does X" by grepping, and they do it badly on unfamiliar repos: broad reads, wrong files, burned context. Graphify ([Graphify-Labs/graphify](https://github.com/Graphify-Labs/graphify), MIT) builds a queryable knowledge graph of a repository — locally, with tree-sitter, zero LLM calls in code-only operation — and exposes it through a CLI and an MCP server with 8 tools. This document specifies how Hive adopts it: installed only with explicit consent, pinned and hash-verified, confined to the repo, attached to agent spawns through the existing MCP scoping machinery, and measured rather than assumed to be used.

Two principles govern everything below. First, **graph context is a hint, never an authority**: upstream's own published QA accuracy is 45–76%, so nothing Hive does may treat a graph answer as ground truth, and no Hive operation — spawn, landing, recovery — may ever block on graphify being present, healthy, or fast. A broken graph means agents run without graph context, loudly noted, never failed. Second, **the integration must be losslessly declinable**: a user who says no, or a machine without Python tooling, gets exactly the Hive that exists today.

## What we are integrating, verified

Facts checked against the live docs and PyPI on 2026-07-11 (upstream moves near-daily; the wrapper code, not this doc, owns exact flag spellings):

- PyPI package **`graphifyy`** (double y; the CLI binary is `graphify`), v0.9.12 as of 2026-07-10, MIT, pre-1.0 with multiple releases per day.
- **Provenance smells confirmed**: the PyPI project page lists its homepage as `safishamsi/graphify`, not the `Graphify-Labs` org, and the maintainer is the alias "captainturbo". None of this proves malice — it does mean install-time verification is mandatory, and it blocks on-by-default (see Rollout).
- Extraction is two-tier: **code is parsed locally with tree-sitter and never leaves the machine**; docs and media (`.md .mdx .qmd .html .txt .rst .yaml .yml`, PDFs, images, video) are sent to an LLM backend. There is no `--code-only` flag; a code-only corpus simply requires no API key and runs fully offline. Headless extraction with doc files present and no backend configured **fails with an error** rather than skipping — a property the design exploits below.
- Output lands in `graphify-out/` (`graph.json`, `graph.html`, `GRAPH_REPORT.md`). Incremental re-extraction of changed files only via `--update`. File filtering via a `.graphifyignore` in the repo root (gitignore syntax, merged with `.gitignore`, which is respected automatically).
- The MCP server is `python -m graphify.serve graphify-out/graph.json`, stdio by default, HTTP with a port flag. Docs name 7 of the 8 tools: `query_graph`, `get_node`, `get_neighbors`, `shortest_path`, `list_prs`, `get_pr_impact`, `triage_prs` (the eighth is unnamed in both the docs and the README — an open question).
- `graphify install` writes the user's **global** assistant configuration (skills, rules files, hooks that intercept tool use). **Hive never invokes it, under any circumstances.** Everything graphify-shaped that Hive does is repo-local and daemon-mediated.

## Install: consent, pin, and hashes

Graphify is **installer-fetched, never vendored**. Vendoring a pre-1.0 Python tree into a project that ships as a single binary was rejected outright: it imports upstream's churn into Hive's release surface and its supply-chain exposure into Hive's own provenance. Instead, `hive init` (and `hive graphify enable`, the path for repos already under Hive) asks for consent with a prompt that states exactly what will happen: install `graphifyy==<pin>` into a Hive-owned environment, build a local graph under `graphify-out/`, no code or file content sent anywhere. Declining leaves every Hive feature working and is remembered; Hive does not re-ask on every init.

The task brief sketched `uv tool install graphifyy==<pin>` with hash verification. That exact shape is not buildable: `uv tool install` supports version pinning and `--constraints`, but has **no `--require-hashes`** (verified against uv's CLI). Pin-without-hashes is precisely the wrong posture for a package with provenance smells — a compromised maintainer can re-upload nothing, but can publish `0.9.13` and wait for a careless bump; hashes are also the only defense against index-level substitution. So the install runs through `uv pip`, which does support hash enforcement:

1. The Hive repo carries `graphify.lock`: a fully hash-pinned requirements file (graphifyy at the exact pinned version **plus its entire transitive closure**, every artifact with a sha256), generated at Hive release time with `uv pip compile --generate-hashes` and reviewed like any other dependency change. Hive owns the pin; it moves only when Hive deliberately bumps it (see Rollout).
2. Enable runs `uv venv "$HIVE_HOME/tools/graphify"` then `uv pip install --require-hashes -r graphify.lock` into it. uv fetches a managed Python if the machine lacks one, which keeps "Python absent" from being a separate failure mode.
3. The daemon invokes the tool only by absolute path (`$HIVE_HOME/tools/graphify/bin/graphify`, or that venv's `python -m graphify.serve`). Nothing goes on `PATH`, nothing appears in the user's `uv tool list`, and a `graphify` the user installed themselves is neither touched nor trusted.

The rejected alternatives: `uv tool install` (no hash verification, pollutes the user's tool list); `pipx` (second toolchain to require, same hash gap); verifying the top-level wheel's sha256 by hand and installing from the local file (leaves the transitive closure unverified — the lock file covers it for free). The cost of the chosen shape is a lock file to regenerate on every pin bump; that is a feature, since a bump becomes a reviewable diff of exactly which artifacts changed.

**When uv itself is absent**, Hive prints the one-line official install command and moves on with the feature disabled. It does not offer to run the installer: piping a vendor's shell script from the network is the same class of act this section exists to avoid, and doing it inside a consent prompt for a *different* tool buries the decision. This dents convenience deliberately — the alternative (Hive silently acquiring and thereafter owning a Python toolchain) was rejected as scope Hive cannot honor from a single binary.

`hive graphify disable` stops the per-repo server and all injection; the environment stays for cheap re-enable. Uninstall is `disable` plus deleting `$HIVE_HOME/tools/graphify/` and the repo's `graphify-out/` — and nothing else, because nothing else was ever written: no global config, no `PATH` entry, no shell hook, no assistant skill. The uninstall story being one `rm -rf` and two ignore-file lines is the acceptance test for the whole repo-local posture.

## Repo-local guarantees

**Code-only, enforced twice.** Since upstream has no code-only flag, Hive constructs the guarantee. Lock one: on enable, Hive writes a `.graphifyignore` in the repo root excluding every extension upstream routes to LLM enrichment (the doc list above, plus PDF/image/audio/video). Only code and its tree-sitter parse remain, which upstream processes fully locally. Lock two, the fail-closed backstop: the daemon runs every graphify invocation with a scrubbed child environment — no `*_API_KEY` variables, no `--backend`, no credentials of any kind. Upstream's documented behavior is that enrichment without a backend **errors rather than degrades**, so if the ignore list ever drifts out of sync with upstream's extension routing, the failure mode is a loud broken-graph degradation (agents run graphless, noted), not silent exfiltration of repo content to an LLM API. Enrichment mode is not exposed by any Hive surface; a future opt-in would be its own consent decision, not a flag.

**Ignore hygiene.** `graphify-out/` and `.graphifyignore` are appended to `.git/info/exclude`, not `.gitignore`: graphify enablement is a per-machine choice, and Hive editing a tracked file the whole team shares — today Hive writes no `.gitignore` anywhere in `src/` — was rejected as uninvited repo mutation. The enable step verifies the exclusion took effect with `git check-ignore --no-index graphify-out/probe`; plain `check-ignore` consults the index and reports "not ignored" for anything already tracked, so only the `--no-index` form on a fresh name is evidence. (If a user later commits `.graphifyignore` to share the filter, nothing breaks — the exclude entry is simply redundant.)

**One server per repo, owned by the daemon.** The MCP server runs as a single daemon-managed process per repository: the venv's `python -m graphify.serve graphify-out/graph.json` in HTTP mode, bound to `127.0.0.1` on a daemon-chosen ephemeral port. It follows the detached-child pattern the daemon already uses for itself (`Bun.spawn(..., { detached: true })` + readiness poll, `src/daemon/lifecycle.ts:143`): started on enable and on daemon start when enabled, health-checked, restarted after each graph rebuild so every agent queries the current graph. The alternative — a stdio server spawned per agent, the shape of the existing `hive-channel` bridge (`src/adapters/tools/claude.ts:59`) — was seriously considered for its simplicity (no port, no lifecycle, dies with the agent) and rejected because N concurrent agents would each hold a full copy of the graph in memory, and a long-lived agent would keep serving a graph from before every landing since its spawn; the daemon-owned instance gets restart-on-rebuild for free. The cost is port and health management, both patterns the daemon already carries.

## Lifecycle: build, update, degrade

The graph is built at enable time (whether that is `hive init` on a fresh repo or `hive graphify enable` on one already under Hive — the adoption path for existing repos is the same command, and there is deliberately no separate migration machinery). The build runs daemon-side in the background; on a large repo the first extraction is minutes, and nothing waits for it — agents spawned before it completes simply spawn graphless, noted in their brief.

Incremental updates hook the one choke point every landing already passes through: the daemon's land path, immediately after the fast-forward merge succeeds (`src/daemon/server.ts:1613`, where `landAgent` returns `this.land(...)`). A fire-and-forget rebuild (`graphify extract --update`, changed files only) follows the precedent of `void this.rebuildMemoryIndex().catch(...)` on daemon start (`src/daemon/server.ts:822`): spawned, logged on failure, never awaited by the land response. There is today no HEAD watcher or post-land hook in the daemon — this is a new side effect at that line, not a reuse — and landing latency must not change by a measurable amount when graphify is enabled. Rebuilds are serialized (a landing during a rebuild queues one follow-up); the server is restarted only after a rebuild exits zero, so a failed rebuild leaves the previous good graph serving rather than nothing. Out-of-band history movement (a human force-push, a rebase outside Hive) is caught lazily: staleness is tolerated until the next landing or daemon restart, because chasing every ref update buys accuracy the 45–76% source material cannot deliver anyway.

The degradation rule is absolute and worth stating as the contract it is: **spawn and landing never block on graphify.** Absent (not installed, consent declined, uv missing), broken (extract failing, server unhealthy), or slow (build in progress, query timeout): all three collapse to the same behavior — the agent runs without graph context, its brief says so in one line, `hive_status` shows the graph state, and the daemon log carries the cause. No spawn failure, no land failure, no retry loop an agent can observe. "Loudly noted" is load-bearing: a silently missing graph would look identical to a repo with nothing to find, and per Hive's own protocol an absent field is unknown, never false.

## MCP attachment

Hive already scopes MCP per spawn; graphify rides that machinery rather than growing new config surface.

For Claude agents, scoping is an allowlist: the daemon writes a worktree-local `.mcp.json` and passes `--strict-mcp-config`, so an agent has exactly the servers that file names (`claudeMcpConfigPath`, `src/daemon/spawner-impl.ts:607`; the file is written by `writeClaudeAgentConfig`, `src/adapters/tools/claude.ts:425`, servers object at `:523`). When graphify is enabled and healthy at spawn time, the daemon adds a `graphify` entry of type `http` pointing at the per-repo server's loopback port, alongside the existing `hive` and `hive-channel` entries. For Codex agents, scoping is an exclude-list over inherited servers (`buildCodexMcpExclusionArgs`, `src/adapters/tools/mcp-scope.ts:110`, applied at `src/adapters/tools/codex.ts:170`); graphify is attached symmetrically, as `-c mcp_servers.graphify.*` config arguments at launch. In both cases the decision happens where per-spawn MCP scoping already happens (`src/daemon/spawner-impl.ts:1758` fresh, `:1044` recovery), and `graphify` joins the always-kept set beside `HIVE_MCP_SERVERS` (`src/adapters/tools/mcp-scope.ts:18`) so the scoper never strips it. An agent spawned while the graph is unhealthy gets no graphify entry at all — a dead server in the config would cost every agent a connect-timeout on startup.

## Making agents actually use it

Hive's own telemetry says shipped guidance is not followed guidance: when the karpathy-guidelines skill relied on progressive disclosure, 22% of agents opened it. A graph nobody queries is pure cost, so the design assumes noncompliance and layers accordingly.

**Layer 1 — value with zero agent compliance.** The daemon injects graph-derived context directly into the spawn brief, where prompt assembly already happens (`src/daemon/spawner-impl.ts:1692`, the `Promise.all` that builds `memoryIndex` and `buildScopedBrief` before `buildAgentPrompt`). A third parallel arm runs two local CLI calls against the built graph: the graph's summary header (entity/edge counts, top communities — the `GRAPH_REPORT.md` highlights) and `graphify query "<task text>"` for the task-relevant subgraph. The combined digest is **capped at 1,500 tokens** — comparable to the scoped brief itself, small against a 200k window, and honest about source quality; a hint does not deserve more — truncated subgraph-first, and prefixed with one fixed sentence stating that graph context is advisory, may be wrong, and must be verified against source before load-bearing use. The whole arm is time-boxed at 3 seconds; on timeout, error, or no graph, the brief instead carries the one-line "graph context unavailable (<cause>)" note. Prompt delivery is already file-based (`writeLaunchPrompt`, `src/daemon/launch-prompt.ts:29`), so the tmux 16KB command ceiling that binds spawn prompts does not bind this.

**Layer 2 — tools plus one directive.** The MCP attachment above puts the 8 tools in the agent's hands, and the spawn prompt carries exactly one added directive: prefer `query_graph`/`get_neighbors` over broad grep sweeps when orienting in unfamiliar code, and treat results as leads. One sentence in the spawn prompt — the channel the 22% lesson proved agents actually read — not a skill, not a doc pointer.

**Layer 3 — measure, don't assume.** The telemetry sensor already tails each agent's durable transcript (`readClaudeTelemetry`, `src/daemon/tool-telemetry.ts:156`, keyed by `toolSessionId`; the Codex rollout reader at `:178`). Counting is an extension of that machinery: scan the same JSONL for tool-use entries naming `mcp__graphify__*` (and the Codex equivalent), keep a per-session count, surface it in `hive_status` per agent and as a per-repo adoption number. The count must stay keyed to the tool session, not the worktree path — path-keyed readers are a known Hive bug class that inherits a dead predecessor's transcript across respawns. This is what makes the rollout decision below empirical: if agents demonstrably don't call the tools, layer 1 is the whole product and layers 2–3 told us so cheaply.

There is deliberately **no land-time enforcement** — no "did you consult the graph" gate. It would be unverifiable in exactly the way Hive's protocol warns about (an MCP call proves an act, not that the answer informed anything) and would put a 45–76%-accurate oracle in the landing path, violating the never-block rule for no measurable gain.

## Rollout and version policy

Graphify ships **opt-in**: default off, enabled per repo by explicit consent. Pre-1.0 near-daily churn, unresolved provenance, and unproven agent adoption each independently justify that posture. Flipping to on-by-default requires all of: upstream reaches 1.0 with a stabilized CLI/MCP surface; the PyPI provenance question is resolved (package ownership verifiably tied to Graphify-Labs, or Hive re-pins to a fork it controls); and Hive's own layer-3 telemetry shows graph tools earning their context cost in real sessions. Any one failing keeps it opt-in.

The pin moves on Hive's cadence, not upstream's: Hive does not chase daily releases. A bump is a deliberate act per Hive release at most — regenerate `graphify.lock`, diff the artifact set, smoke-test extract/serve/query against a fixture repo, adapt the wrapper if flags moved (pre-1.0, they do: the transport flags are already documented in two different spellings). A security fix upstream justifies an out-of-band bump through the same steps. The wrapper treats every graphify flag as pinned-version-specific; nothing outside the wrapper module spells a graphify flag.

## Risks carried forward

- **The single-binary promise is dented.** Hive installs as one binary; this feature requires uv (and through it a Python). Contained by making the dependency opt-in, lazily acquired, and invisible when declined — but a support surface ("graphify broke because my Python is weird") now exists that Hive did not have. If that surface grows disproportionate, the escape hatch is dropping to CLI-only integration or a future upstream binary distribution.
- **Accuracy churn.** 45–76% QA accuracy means wrong answers are routine, and pre-1.0 extraction changes can shift quality between pins without notice. The hint-never-authority framing, the advisory preamble in layer 1, and the small token budget are the mitigations; the residual risk is an agent trusting a wrong graph answer anyway, which no design can fully remove.
- **Supply chain.** The hash lock pins exactly what was reviewed, but hashes verify integrity, not intent: the smelly provenance (org/homepage mismatch, alias maintainer, a package name one typo from `graphify`) means the pinned code itself deserves the scrutiny of a vendored dependency at each bump. This risk is why on-by-default is gated on provenance resolution, and it never fully closes while the ownership question stands.
- **Privacy.** Enrichment is off behind two independent locks (ignore list + scrubbed keyless environment), and the backstop fails closed. Residual exposure is upstream changing what counts as locally-parseable in a way that survives both locks; the pin-bump smoke test explicitly checks that a doc-bearing fixture repo extracts with zero network egress.

## Open questions

- The eighth MCP tool's name: both the docs and the README enumerate only seven. Resolve when the pin's wrapper is written, from the pinned version's actual tool list.
- Whether `graphify.serve` re-reads `graph.json` on change or must be restarted to pick up a rebuild. The design assumes restart (the safe choice); if the pinned version hot-reloads, the restart step simplifies away.
- Whether `.graphifyignore`'s merge semantics let a user-committed copy conflict with the Hive-written one (upstream documents merging `.gitignore` and `.graphifyignore`, not two competing intents in one file). Test at wrapper-implementation time.
