# Graphify bundling: shipping the graph tool as a Hive-built artifact

Hive will ship graphify so it works out of the box: no uv, no Python, no toolchain on the user's machine, with the pin moving only when a Hive developer deliberately bumps it. That decision is made; this document answers *how*, and whether the how actually works. The answer is measured, not argued: **a PyInstaller-frozen graphify — both the CLI and the MCP server — was built and run against the exact pinned closure on all four platform slices (darwin-arm64, darwin-x64, linux-arm64, linux-x64), of which the two darwin slices are the ones Hive actually ships today.** Extraction, query, and MCP `query_graph` over Streamable HTTP all produce output identical to the venv-installed baseline on every slice. The artifact is 25–35 MB compressed per platform (123–174 MB installed), and the chosen distribution is a per-platform asset on a Hive-owned GitHub release, fetched by `hive graphify enable` and verified against a sha256 embedded in the Hive binary. The build recipe that produced every number here ships as the real release step at `scripts/graphify/` (published by `.github/workflows/graphify-artifacts.yml`); the Linux measurements remain at `prototypes/graphify-bundling/`.

One relationship to state up front: `graphify-integration.md` describes the shipping uv-based install and rejected "vendoring" — but what it rejected was vendoring upstream's Python *source tree* into Hive's repo. This design ships a Hive-built, Hive-signed *binary artifact* built from the same hash-pinned lock; upstream's code never enters Hive's repo, and upstream's churn reaches users only through a deliberate Hive bump. The integration doc's install and uninstall sections get rewritten to match this design when the runtime work lands; until then it describes what ships today, and this doc describes what was proven and decided.

## The gate: does freezing survive tree-sitter? (measured: yes)

Graphify is the hard case for freezing. It parses code through tree-sitter, which means 26 separate grammar packages each carrying a native `_binding.abi3.so`, loaded not by `import` statements but by `importlib.import_module(config.ts_module)` (`graphify/extractors/engine.py`, L2122 in 0.9.12) — invisible to any static analyzer. The closure adds more native extensions (pydantic-core, rpds-py, cffi, cryptography, numpy, rapidfuzz) and the MCP server drags in a full ASGI stack (uvicorn, starlette). If freezing was going to fail, it would fail here.

It doesn't, but not by default. A naive PyInstaller build would miss every grammar; the spec must collect them explicitly. Three collection requirements were identified and are encoded in `scripts/graphify/graphify.spec`:

1. **Every `tree_sitter_*` package via `collect_all`**, discovered from the build venv's installed distributions rather than hardcoded, so a pin bump that adds a grammar cannot silently ship a bundle missing it.
2. **`collect_all("graphify")`** — the package ships data files (`skill-*.md`, `always_on/*.md`) beside its code.
3. **`copy_metadata("graphifyy")`** — the tool calls `importlib.metadata.version()` at runtime and dies without its dist-info.

Both upstream console scripts matter (`graphify` = `graphify.__main__:main`, `graphify-mcp` = `graphify.serve:_main`), and freezing each separately would ship the 123 MB library tree twice. Instead one EXE fronts a busybox-style dispatcher (`entry.py`): invoked through the `graphify-mcp` symlink it runs the server, otherwise the CLI. The alternatives — PyInstaller's MERGE machinery (fragile, poorly maintained) and two full bundles (double size) — lose to a nine-line dispatcher whose only cost is that the symlink must survive packaging, which tar preserves.

What was **measured** on this machine (macOS arm64, Darwin 25.3.0; CPython 3.12.8 from uv's standalone builds; PyInstaller 6.21.0; `graphifyy==0.9.12` installed `--require-hashes` from `graphify.lock`), with the build venv renamed away so nothing could fall back to it:

- `graphify update` on a nine-language fixture (Python, TypeScript, Go, Rust, Java, Ruby, Kotlin, Swift, C++): 33 nodes / 31 edges, every language contributing symbols, byte-identical results to the venv baseline. The dynamic grammar loading works frozen.
- `graphify query`: identical traversal output to the venv baseline.
- `graphify-mcp --transport http --stateless --json-response`: MCP initialize handshake completes, `tools/call query_graph` returns real graph content, all over the loopback Streamable HTTP transport Hive's daemon actually uses.
- Startup: 60–70 ms frozen vs 30 ms venv — noise against a seven-second graph build; irrelevant for a long-running server.
- Freeze time: ~17 s per architecture on top of the venv install.

The same suite passed on the darwin-x64 slice, built on this same arm64 machine (uv's `cpython-3.12.8-macos-x86_64-none` under Rosetta 2, PyInstaller producing a genuine x86_64 Mach-O) and smoke-tested under Rosetta — and on both Linux slices in containers (see the platform matrix below). Emulated-versus-bare-metal x86_64 is the **inferred** link that remains; CI can close it by building or testing on GitHub's Intel runners (`macos-15-intel`) instead.

## The one real portability finding: cryptography ≥ 49

The x64 build initially failed, and the failure is a first-class finding about the *lock*, not the freeze: **`cryptography==49.0.0` publishes no macOS x86_64 wheel** — arm64 only (verified against PyPI's file list). uv falls back to the sdist, whose Rust build then demands an x86_64 OpenSSL to link against; on the arm64 build machine that cross-build dies in `openssl-sys`, and even on an Intel runner it would cost a Rust toolchain and minutes of compile per build. cryptography reaches the closure via pyjwt (MCP's auth stack) — graphify's own extraction never touches it, but the import graph doesn't care.

The fix is one constraint line, and it is measured: cryptography 46–48 ship `universal2` wheels covering both architectures. Recompiling the lock with `cryptography<49` selects 48.0.1, after which the entire closure installs from wheels on **both** arches — no Rust, no OpenSSL, no sdist builds — and the x64 freeze and full smoke suite pass. The constraint lives in a committed compile input (`scripts/graphify/graphify.in`), not in anyone's memory, and `graphify.lock` is regenerated only from it. The cost: Hive holds cryptography one major version behind until upstream restores Intel wheels or Hive drops the x64 slice — acceptable for a transitive dependency of an auth path Hive doesn't exercise, and re-evaluated at each bump like every other pin. The alternative — building cryptography from its hash-pinned sdist on an Intel CI runner — was rejected while a pure-wheel option exists: it adds a Rust toolchain to the build's trusted surface and minutes to every bump for zero functional gain.

`graphify.lock` is compiled from that input, pins 48.0.1, and the diff against the unconstrained lock is exactly one package — the reviewable-bump property doing its job on the first bump that mattered. The constrained lock was verified on arm64 as well — universal2 covers both.

## Distribution shape: a Hive-owned release asset, not an embedded blob

The size numbers decide this. Measured: 123 MB installed / **25 MB tar.zst** (30 MB tar.gz) for darwin-arm64; 128 MB / **27 MB** for darwin-x64. The Hive binary itself is 63 MB.

**Rejected: embedding in the Hive binary** (the `src/skills/shipped.ts` pattern that was right for kilobyte-scale skills). Even embedding only the matching platform's artifact grows every Hive binary ~40%, and the recurring cost is worse than the static one: Hive releases often and the graphify pin deliberately moves rarely, so every `hive update` would re-download 25 MB of unchanged graphify bytes inside an 88 MB binary. Embedding buys exactly one thing — zero-network enable — and buys it for every user including the majority who never enable graphify.

**Chosen: per-platform artifacts on a dedicated Hive-owned GitHub release**, tagged independently of Hive versions (e.g. `graphify-v0.9.12-hive.1`, the suffix counting Hive's own rebuilds of the same upstream pin — a spec fix or re-sign without an upstream bump). The Hive binary embeds three constants per platform: the tag, the artifact name, and the sha256 of the exact bytes. `hive graphify enable` downloads the one matching artifact from that release, verifies the hash before unpacking, and refuses on any mismatch. This keeps the trust story identical to `hive update` — the user fetches Hive-built bytes from Hive's own release origin, verified against a hash that shipped inside a binary they already trusted — and it decouples the two release cadences: a graphify bump uploads one new asset set once, and every Hive version referencing that tag shares it. No PyPI, no third-party origin, no toolchain.

The variant worth naming: attaching the artifacts to each Hive release instead of a dedicated tag. Simpler (no second tag to manage), but it re-uploads ~52 MB of unchanged assets on every Hive release and welds the cadences back together at the publishing layer; the dedicated tag costs one extra `gh release create` per bump and was chosen for exactly the reason the whole design exists — Hive bumps graphify deliberately and rarely.

The download rides consent, never precedes it: it happens inside `hive graphify enable`, whether the user typed that command or answered yes to init's graphify question (which resolves to the same enable — `graphify-integration.md` has the consent model). Users who never opt in never spend the 25 MB or the network round-trip. An unconditional init-time prefetch was considered and rejected: it buys offline-enable for the rare machine that installs online but enables offline, at the price of 25 MB on every install. An offline enable fails with one line naming the artifact URL and retries next time; nothing degrades below the integration doc's floor.

The artifact ships as tar.zst (the 5 MB win over gzip repeats on every enable across the fleet; `zstd -d` rides in the bundle's own download path — macOS ships `tar` with zstd support since Ventura, and Hive's updater already handles archive unpacking). It unpacks to `$HIVE_HOME/tools/graphify/<pin>/`, replacing the venv the current design puts there. The daemon's absolute-path invocation contract is unchanged in spirit: `$HIVE_HOME/tools/graphify/<pin>/graphify` and `…/graphify-mcp`, nothing on PATH, a user's own graphify installation neither touched nor trusted. **Uninstall is unchanged and still the acceptance test:** `hive graphify disable --purge` deletes `$HIVE_HOME/tools/graphify/` and the repo's `graphify-out/`, and that is the whole footprint — the frozen bundle leaves no uv cache, no managed Python, no venv anywhere on the user's machine, which is strictly less residue than the design it replaces.

## The bump pipeline

A Hive developer bumps graphify in one reviewable motion; `graphify.lock` is not replaced — it is demoted from "embedded in the Hive binary" to "input to the freeze," which is a promotion in honesty since it now describes exactly what the artifact was built from:

1. Edit `scripts/graphify/graphify.in` (the version pin and the cryptography constraint live there) and run `uv pip compile --generate-hashes --python-version 3.12 scripts/graphify/graphify.in -o graphify.lock`. The lock diff is the reviewable artifact-set change, same as today.
2. Run the freeze (`scripts/graphify/build.sh`: hash-verified install, PyInstaller, sign, smoke test, package). It builds both darwin slices from one arm64 machine — the x64 slice via Rosetta and uv's x86_64 CPython, both measured working — and fails loudly if any smoke step (extract, query, MCP `query_graph`) breaks. The interpreter version and the PyInstaller pin are constants in the script, part of the same review.
3. CI runs the same script on a macOS runner, signs and notarizes the bundle's Mach-Os with the existing release credentials, uploads the assets to the dedicated release tag, and emits the sha256s of the final signed bytes. Those hashes — CI's, never a laptop's, since PyInstaller builds are not bit-reproducible — land in the Hive source constants in the same PR.
4. The pin-bump smoke obligations from the integration doc carry over unchanged: fixture extract/serve/query (now inside `build.sh`), wrapper flag adaptation, and the keyless doc-bearing-fixture privacy check.

Signing is **measured** through Developer ID plus the hardened runtime: `scripts/graphify/build.sh` signed all 48 Mach-O files in each slice with a real Developer ID Application certificate, hardened runtime, and a secure timestamp, and the signed bundles then passed the full smoke suite and `codesign --verify --strict` on the exact packaged bytes. The entitlement set is smaller than Hive's own binary needs and different in kind: CPython 3.12 has no JIT, so no `allow-jit`; what the bundle needs is `com.apple.security.cs.allow-unsigned-executable-memory` for libffi's closure trampolines (cffi is in the closure via cryptography), and nothing else — every dylib is signed with the same identity, so library validation passes without being disabled (`scripts/graphify/entitlements.plist` records the reasoning). One keychain lesson from the measurement: a login keychain holding two certificates with the same common name makes `codesign` refuse as "ambiguous," so a local signer passes the identity's SHA-1 hash; CI's throwaway keychain holds one certificate and is immune. What remains **inferred** is notarization acceptance of the PyInstaller bootloader — the credentials (notary issuer id) exist only as CI secrets — and the publish workflow proves it before any tag is cut: a rejection fails the run inside the same before-the-tag window as Hive's own releases. Notarization submits a zip of the signed bundle (bare Mach-Os can't staple, same as Hive's CLI slices — online CDHash lookup covers first run), and the tar.zst ships the same signed bytes. In practice Gatekeeper may never evaluate these files at all — the `hive` binary downloads them without a quarantine xattr, as with `hive update` today — so the signature is defense in depth, not a launch gate.

## Platform matrix and degradation

What Hive actually ships is defined in code, not in anyone's recollection: `src/release/build.ts` builds exactly two CLI slices — darwin-arm64 and darwin-x64 — and `install.sh` refuses any non-Darwin machine outright ("Hive is macOS-only for now"). **The shipping matrix is those two darwin slices, and both are measured green above.** There is no Linux Hive today; a Linux user cannot reach `hive init` at all, so nothing in this design can regress out-of-the-box behavior for them.

The Linux question was still measured rather than left to faith, because "would this design survive a Linux Hive" is exactly the kind of claim that looks obvious until tree-sitter is involved. The same freeze — same lock (with the `cryptography<49` constraint; 48.0.1 ships manylinux wheels for both arches), same spec, same dispatcher, pip's own `--require-hashes` in place of uv — was built and fully smoke-tested inside `python:3.12-slim` containers (`prototypes/graphify-bundling/linux-measure.sh`): nine-language extract, query, and MCP `query_graph` over HTTP, venv hidden, identical output to every other slice. **linux-arm64: 168 MB installed, 32 MB tar.zst, container running natively on this arm64 host. linux-x64: 174 MB installed, 35 MB tar.zst, container under Docker's x86_64 emulation** — measured under emulation, inferred for bare-metal x86_64 Linux, the same caveat class as the Rosetta-tested darwin-x64 slice. Two Linux-specific facts a future pipeline needs: PyInstaller on Linux hard-requires `objdump` (binutils) — the freeze aborts without it, measured — and the frozen binary inherits the build image's glibc as its floor (the measuring image is Debian 13 / glibc 2.41, fine for a spike, too new to ship from; a real Linux build would use the oldest supported base or a manylinux image). The build cost when Linux joins the matrix is one ordinary Linux CI job per arch — GitHub's ubuntu runners or any container host; no cross-build trick from macOS exists or is needed.

On any platform where the artifact is missing — an unsupported slice, a failed download, a hash mismatch, or a frozen tool that breaks at runtime — the degradation rule from `graphify-integration.md` holds absolutely and unchanged: **no spawn or landing ever blocks on graphify, and neither does `hive init`.** Absent, broken, or slow all collapse to the same behavior — `hive graphify enable` fails with one honest line naming what could not be fetched and why, agents run graphless with one line in their brief, `hive graphify status` and the daemon log name the cause, and nothing retries in any path an agent can observe. There is no half-install to strand a user: the artifact unpacks into place only after its hash verifies, so the outcome is always "graphify present and working" or "graphify cleanly absent and honestly reported." The frozen bundle actually shrinks the failure surface this rule guards: "your Python is weird" and "uv is missing" cease to exist as support categories, leaving download failure and hash mismatch, both of which are loud, attributable to Hive's own release infrastructure, and identical in kind to `hive update` failures Hive already reports well.

## Open questions and honest risks

- **Notarization acceptance of the PyInstaller bootloader is the one unproven signing step.** Developer ID + hardened runtime + the entitlement set are measured (see the pipeline section); notarization could not be exercised locally because the notary issuer id lives only in CI secrets. The proof is one run of `.github/workflows/graphify-artifacts.yml`, which submits before tagging, so a rejection costs a build and never a published version; a user who wants pre-publish certainty can run `scripts/graphify/build.sh` with the `MACOS_NOTARY_*` variables set. Neither failure mode reaches a user as a broken Hive: a rejection fails the workflow, and a hypothetical Gatekeeper kill on a user machine collapses into the degradation rule (graphify reported broken, agents run graphless).
- **Emulation-tested is not bare-metal-tested.** The darwin-x64 measurements ran under Rosetta 2 and the linux-x64 measurements under Docker's x86_64 emulation; both arm64 slices ran native. Converting each to measured is one CI job on real hardware — `macos-15-intel` for darwin, any x86_64 Linux runner for linux — reusing the already-built artifacts. A divergence would be an instruction-level or wheel-selection difference, unlikely for this closure, and if it happened it would surface as graphify failing on that platform, which the degradation rule contains to an honest status line.
- **Upstream can reintroduce the cryptography problem in any dependency.** A future pin could pull a package with no x64 wheel or no universal2 escape hatch. The wheel-coverage gate in `build.sh` catches it at bump time — which is precisely when Hive wants to catch it, before it reaches a user — but the *remedy* in that case is a real decision (constrain, patch, or drop the x64 slice), not a script.
- **PyInstaller and the standalone CPython are new build-time supply-chain surface.** Both are pinned and neither ever runs on a user's machine, but the bump review now covers three pins (graphify, interpreter, PyInstaller) instead of one. The PyInstaller install is version-pinned, not hash-pinned; hash-pinning it the way `graphify.lock` pins the runtime closure is cheap and still worth doing.
- **The bundle is fat and mostly not graphify.** Verilog's grammar alone is 18 MB; numpy 7 MB; the MCP auth stack drags in cryptography for a feature Hive never uses. All of it is upstream's dependency graph, pinned as-is on purpose. Trimming (dropping unused grammars, excluding the auth extras) would shrink the artifact meaningfully but forks Hive's closure away from upstream's tested one; not worth it at 25 MB, worth revisiting if the artifact ever triples.
- **The artifact registry ships null until the first publish.** `src/adapters/graphify-artifacts.ts` carries no hashes until `.github/workflows/graphify-artifacts.yml` runs, publishes the assets, and a reviewed PR lands the CI-built sha256s. Until then `hive graphify enable` reports the honest "no bundle published" line and the init prompt is suppressed — dormant, never broken.
