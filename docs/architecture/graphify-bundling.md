# Graphify bundling: shipping the graph tool as a Hive-built artifact

Hive will ship graphify so it works out of the box: no uv, no Python, no toolchain on the user's machine, with the pin moving only when a Hive developer deliberately bumps it. That decision is made; this document answers *how*, and whether the how actually works. The answer is measured, not argued: **a PyInstaller-frozen graphify — both the CLI and the MCP server — was built and run on this machine, on both macOS architectures Hive ships to, against the exact pinned closure in `graphify.lock`.** Extraction, query, and MCP `query_graph` over Streamable HTTP all produce output identical to the venv-installed baseline. The artifact is 25–27 MB compressed per platform (123–128 MB installed), and the chosen distribution is a per-platform asset on a Hive-owned GitHub release, fetched by `hive graphify enable` and verified against a sha256 embedded in the Hive binary. The build recipe that produced every number here is committed at `prototypes/graphify-bundling/` and reruns as one command.

One relationship to state up front: `graphify-integration.md` describes the shipping uv-based install and rejected "vendoring" — but what it rejected was vendoring upstream's Python *source tree* into Hive's repo. This design ships a Hive-built, Hive-signed *binary artifact* built from the same hash-pinned lock; upstream's code never enters Hive's repo, and upstream's churn reaches users only through a deliberate Hive bump. The integration doc's install and uninstall sections get rewritten to match this design when the runtime work lands; until then it describes what ships today, and this doc describes what was proven and decided.

## The gate: does freezing survive tree-sitter? (measured: yes)

Graphify is the hard case for freezing. It parses code through tree-sitter, which means 26 separate grammar packages each carrying a native `_binding.abi3.so`, loaded not by `import` statements but by `importlib.import_module(config.ts_module)` (`graphify/extractors/engine.py`, L2122 in 0.9.12) — invisible to any static analyzer. The closure adds more native extensions (pydantic-core, rpds-py, cffi, cryptography, numpy, rapidfuzz) and the MCP server drags in a full ASGI stack (uvicorn, starlette). If freezing was going to fail, it would fail here.

It doesn't, but not by default. A naive PyInstaller build would miss every grammar; the spec must collect them explicitly. Three collection requirements were identified and are encoded in `prototypes/graphify-bundling/graphify.spec`:

1. **Every `tree_sitter_*` package via `collect_all`**, discovered from the build venv's installed distributions rather than hardcoded, so a pin bump that adds a grammar cannot silently ship a bundle missing it.
2. **`collect_all("graphify")`** — the package ships data files (`skill-*.md`, `always_on/*.md`) beside its code.
3. **`copy_metadata("graphifyy")`** — the tool calls `importlib.metadata.version()` at runtime and dies without its dist-info.

Both upstream console scripts matter (`graphify` = `graphify.__main__:main`, `graphify-mcp` = `graphify.serve:_main`), and freezing each separately would ship the 123 MB library tree twice. Instead one EXE fronts a busybox-style dispatcher (`entry.py`): invoked through the `graphify-mcp` symlink it runs the server, otherwise the CLI. The alternatives — PyInstaller's MERGE machinery (fragile, poorly maintained) and two full bundles (double size) — lose to a nine-line dispatcher whose only cost is that the symlink must survive packaging, which tar preserves.

What was **measured** on this machine (macOS arm64, Darwin 25.3.0; CPython 3.12.8 from uv's standalone builds; PyInstaller 6.21.0; `graphifyy==0.9.12` installed `--require-hashes` from `graphify.lock`), with the build venv renamed away so nothing could fall back to it:

- `graphify update` on a nine-language fixture (Python, TypeScript, Go, Rust, Java, Ruby, Kotlin, Swift, C++): 33 nodes / 31 edges, every language contributing symbols, byte-identical results to the venv baseline. The dynamic grammar loading works frozen.
- `graphify query`: identical traversal output to the venv baseline.
- `graphify-mcp --transport http --stateless --json-response`: MCP initialize handshake completes, `tools/call query_graph` returns real graph content, all over the loopback Streamable HTTP transport Hive's daemon actually uses.
- Startup: 60–70 ms frozen vs 30 ms venv — noise against a seven-second graph build; irrelevant for a long-running server.
- Freeze time: ~17 s per architecture on top of the venv install.

The same suite passed on the darwin-x64 slice, built on this same arm64 machine (uv's `cpython-3.12.8-macos-x86_64-none` under Rosetta 2, PyInstaller producing a genuine x86_64 Mach-O) and smoke-tested under Rosetta. That equivalence to real Intel hardware is the one **inferred** link in the platform story; CI can close it by building or testing on GitHub's Intel runners (`macos-15-intel`) instead.

## The one real portability finding: cryptography ≥ 49

The x64 build initially failed, and the failure is a first-class finding about the *lock*, not the freeze: **`cryptography==49.0.0` publishes no macOS x86_64 wheel** — arm64 only (verified against PyPI's file list). uv falls back to the sdist, whose Rust build then demands an x86_64 OpenSSL to link against; on the arm64 build machine that cross-build dies in `openssl-sys`, and even on an Intel runner it would cost a Rust toolchain and minutes of compile per build. cryptography reaches the closure via pyjwt (MCP's auth stack) — graphify's own extraction never touches it, but the import graph doesn't care.

The fix is one constraint line, and it is measured: cryptography 46–48 ship `universal2` wheels covering both architectures. Recompiling the lock with `cryptography<49` selects 48.0.1, after which the entire closure installs from wheels on **both** arches — no Rust, no OpenSSL, no sdist builds — and the x64 freeze and full smoke suite pass. The constraint lives in a committed compile input (`prototypes/graphify-bundling/graphify.in`), not in anyone's memory, and `build.sh` refuses loudly to attempt an x64 build against a lock that pins cryptography ≥ 49. The cost: Hive holds cryptography one major version behind until upstream restores Intel wheels or Hive drops the x64 slice — acceptable for a transitive dependency of an auth path Hive doesn't exercise, and re-evaluated at each bump like every other pin. The alternative — building cryptography from its hash-pinned sdist on an Intel CI runner — was rejected while a pure-wheel option exists: it adds a Rust toolchain to the build's trusted surface and minutes to every bump for zero functional gain.

The current `graphify.lock` in this repo pins 49.0.0 (it predates this finding, and the arm64-only venv flow it serves never noticed); the first artifact bump recompiles it from `graphify.in`. The constrained lock was verified on arm64 as well — universal2 covers both.

## Distribution shape: a Hive-owned release asset, not an embedded blob

The size numbers decide this. Measured: 123 MB installed / **25 MB tar.zst** (30 MB tar.gz) for darwin-arm64; 128 MB / **27 MB** for darwin-x64. The Hive binary itself is 63 MB.

**Rejected: embedding in the Hive binary** (the `src/skills/shipped.ts` pattern that was right for kilobyte-scale skills). Even embedding only the matching platform's artifact grows every Hive binary ~40%, and the recurring cost is worse than the static one: Hive releases often and the graphify pin deliberately moves rarely, so every `hive update` would re-download 25 MB of unchanged graphify bytes inside an 88 MB binary. Embedding buys exactly one thing — zero-network enable — and buys it for every user including the majority who never enable graphify.

**Chosen: per-platform artifacts on a dedicated Hive-owned GitHub release**, tagged independently of Hive versions (e.g. `graphify-v0.9.12-hive.1`, the suffix counting Hive's own rebuilds of the same upstream pin — a spec fix or re-sign without an upstream bump). The Hive binary embeds three constants per platform: the tag, the artifact name, and the sha256 of the exact bytes. `hive graphify enable` downloads the one matching artifact from that release, verifies the hash before unpacking, and refuses on any mismatch. This keeps the trust story identical to `hive update` — the user fetches Hive-built bytes from Hive's own release origin, verified against a hash that shipped inside a binary they already trusted — and it decouples the two release cadences: a graphify bump uploads one new asset set once, and every Hive version referencing that tag shares it. No PyPI, no third-party origin, no toolchain.

The variant worth naming: attaching the artifacts to each Hive release instead of a dedicated tag. Simpler (no second tag to manage), but it re-uploads ~52 MB of unchanged assets on every Hive release and welds the cadences back together at the publishing layer; the dedicated tag costs one extra `gh release create` per bump and was chosen for exactly the reason the whole design exists — Hive bumps graphify deliberately and rarely.

The download moment stays at `hive graphify enable`, not `hive init`. Enable remains the per-repo consent act — its subject was never really "fetch some bytes" but "build a graph of this repo's code" — and fetching at enable means users who never opt in never spend the 25 MB or the network round-trip. An init-time best-effort prefetch was considered and rejected: it buys offline-enable for the rare machine that installs online but enables offline, at the price of 25 MB on every install. An offline enable fails with one line naming the artifact URL and retries next time; nothing degrades below the integration doc's floor.

The artifact ships as tar.zst (the 5 MB win over gzip repeats on every enable across the fleet; `zstd -d` rides in the bundle's own download path — macOS ships `tar` with zstd support since Ventura, and Hive's updater already handles archive unpacking). It unpacks to `$HIVE_HOME/tools/graphify/<pin>/`, replacing the venv the current design puts there. The daemon's absolute-path invocation contract is unchanged in spirit: `$HIVE_HOME/tools/graphify/<pin>/graphify` and `…/graphify-mcp`, nothing on PATH, a user's own graphify installation neither touched nor trusted. **Uninstall is unchanged and still the acceptance test:** `hive graphify disable --purge` deletes `$HIVE_HOME/tools/graphify/` and the repo's `graphify-out/`, and that is the whole footprint — the frozen bundle leaves no uv cache, no managed Python, no venv anywhere on the user's machine, which is strictly less residue than the design it replaces.

## The bump pipeline

A Hive developer bumps graphify in one reviewable motion; `graphify.lock` is not replaced — it is demoted from "embedded in the Hive binary" to "input to the freeze," which is a promotion in honesty since it now describes exactly what the artifact was built from:

1. Edit `graphify.in` (the version pin and the cryptography constraint live there) and run `uv pip compile --generate-hashes graphify.in -o graphify.lock`. The lock diff is the reviewable artifact-set change, same as today.
2. Run the freeze (`build.sh`: hash-verified install, PyInstaller, smoke test, package). It builds both darwin slices from one arm64 machine — the x64 slice via Rosetta and uv's x86_64 CPython, both measured working — and fails loudly if any smoke step (extract, query, MCP `query_graph`) or the wheel-coverage gate breaks. The interpreter version and the PyInstaller pin are constants in the script, part of the same review.
3. CI runs the same script on a macOS runner, signs and notarizes the bundle's Mach-Os with the existing release credentials, uploads the assets to the dedicated release tag, and emits the sha256s of the final signed bytes. Those hashes — CI's, never a laptop's, since PyInstaller builds are not bit-reproducible — land in the Hive source constants in the same PR.
4. The pin-bump smoke obligations from the integration doc carry over unchanged: fixture extract/serve/query (now inside `build.sh`), wrapper flag adaptation, and the keyless doc-bearing-fixture privacy check.

Signing is the one pipeline step measured only partway. The bundle contains 48 Mach-O files (the EXE plus 47 dylibs/extensions); all were ad-hoc re-signed and the bundle kept working, so signing per se doesn't break a PyInstaller layout. What is **inferred** and must be proven against a real certificate before CI is trusted — this repo already has the harness for exactly that, `scripts/signing/dry-run.sh` — is Developer ID plus hardened runtime on the PyInstaller bootloader, which may need entitlements Hive's Bun binary does not (ctypes/cffi trampolines are the usual suspect). Notarization submits a zip of the signed bundle (bare Mach-Os can't staple, same as Hive's CLI slices — online CDHash lookup covers first run), and the tar.zst ships the same signed bytes. In practice Gatekeeper may never evaluate these files at all — the `hive` binary downloads them without a quarantine xattr, as with `hive update` today — so the signature is defense in depth, not a launch gate.

## Platform matrix and degradation

Hive ships two platforms — darwin-arm64 and darwin-x64 (`docs/versioning-and-release.md`) — and both are measured green above. There is no Linux Hive today; when one exists, the same recipe on a Linux runner is expected to work (PyInstaller's home turf, all closure packages ship manylinux wheels — **inferred**, unmeasured, with the usual glibc-floor caveat).

On any platform where the artifact is missing, the download fails, the hash mismatches, or the frozen tool breaks at runtime, the degradation rule from `graphify-integration.md` holds absolutely and unchanged: **no spawn or landing ever blocks on graphify.** Absent, broken, or slow all collapse to the same behavior — agents run graphless with one line in their brief, `hive graphify status` and the daemon log name the cause, and nothing retries in any path an agent can observe. The frozen bundle actually shrinks the failure surface this rule guards: "your Python is weird" and "uv is missing" cease to exist as support categories, leaving download failure and hash mismatch, both of which are loud, attributable to Hive's own release infrastructure, and identical in kind to `hive update` failures Hive already reports well.

## Open questions and honest risks

- **Developer ID + hardened runtime on the PyInstaller bootloader is unproven.** Ad-hoc signing is measured working; the real certificate, entitlement set, and notarization acceptance are not. Extend `scripts/signing/dry-run.sh` to the bundle before any of this reaches CI. If the hardened runtime fights the bootloader, the fallback ladder is: add the minimal entitlement, then ship signed-but-not-hardened (the artifact is not a login item or app; Gatekeeper likely never evaluates it), then revisit.
- **Rosetta-tested is not Intel-tested.** Every x64 measurement ran under Rosetta 2. The cheap closure is building/smoking the x64 slice on `macos-15-intel` in CI; until then, Intel-Mac correctness is inferred.
- **Upstream can reintroduce the cryptography problem in any dependency.** A future pin could pull a package with no x64 wheel or no universal2 escape hatch. The wheel-coverage gate in `build.sh` catches it at bump time — which is precisely when Hive wants to catch it, before it reaches a user — but the *remedy* in that case is a real decision (constrain, patch, or drop the x64 slice), not a script.
- **PyInstaller and the standalone CPython are new build-time supply-chain surface.** Both are pinned and neither ever runs on a user's machine, but the bump review now covers three pins (graphify, interpreter, PyInstaller) instead of one. Hash-pinning the PyInstaller install the way `graphify.lock` pins the runtime closure is cheap and should happen when this graduates from `prototypes/`.
- **The bundle is fat and mostly not graphify.** Verilog's grammar alone is 18 MB; numpy 7 MB; the MCP auth stack drags in cryptography for a feature Hive never uses. All of it is upstream's dependency graph, pinned as-is on purpose. Trimming (dropping unused grammars, excluding the auth extras) would shrink the artifact meaningfully but forks Hive's closure away from upstream's tested one; not worth it at 25 MB, worth revisiting if the artifact ever triples.
- **`graphify.lock` in this repo still pins cryptography 49.0.0** and must be recompiled from `graphify.in` as part of the first real artifact build. Until the runtime work lands, the shipping uv-based flow continues to use the current lock unmodified.
