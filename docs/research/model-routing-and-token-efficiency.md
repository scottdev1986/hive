# Model routing and token efficiency

## What this is and why

Hive's routing today answers one question well — "which vendor and model can I safely launch right now?" — and leaves a more expensive question unanswered: "what is the cheapest launch that will still produce work I don't have to redo?" This document is the research pass on that second question: how Hive should classify jobs, choose among the models actually available to it (GPT-5.6 Sol/Terra/Luna via Codex; Fable 5, Opus 4.8, Sonnet 5/4.6, Haiku 4.5 via Claude Code), spend tokens deliberately instead of by default, and know afterward whether the choice was right. It reads the current code (`src/schemas/routing.ts`, `src/daemon/quota.ts`, `src/daemon/spawner-impl.ts`, `src/config/load.ts`, `src/adapters/tools/models.ts`) plus the minimum current provider sources, and ends with an implementable decision matrix, a routing rubric, telemetry, an evaluation plan, and a staged rollout. Verified facts and time-sensitive assumptions are kept separate; the assumptions section at the end is dated.

The one-sentence thesis: **Hive's biggest cost lever is not smarter model choice at the top — it is stopping the silent resolution of routine work onto frontier models, and closing the loop so estimates, tiers, and escalations learn from what runs actually cost.** A day of production traffic has since confirmed the first half and indicted the second: on 2026-07-10 Codex consumed more than half its five-hour allowance while Claude spent 3% of its own, and Hive's ledger could not say why, because it has never recorded a single real measurement. Price per token is one factor of the bill; the other is how many tokens a spawn is handed before it has done anything, and that factor turned out to be dominated by a single habit — telling agents to read whole design documents. Everything below serves that thesis.

## What the 2026-07-10 session actually shows

This section is the diagnosis, drawn only from Hive's own SQLite at `~/.hive/hive.db` — no billable probes. It supersedes guesswork about the burn, including two hypotheses it refutes.

**The ledger is fiction, and that is finding number one.** All 45 `quota_usage` rows carry `source = estimated, confidence = estimated`. `estimatedUnits` has *zero variance* within a tier (cheap 4, standard 10, deep 20, review 8) because it is a constant read from config, not a measurement. `quota_observations` holds **zero rows**. Of 661 `events`, **none** carries a `usageUnits` value. The mechanism is precise and fixable: `observeCodexRateLimits` (`src/daemon/quota.ts:264-268`) opens with `const limit = this.limitFor(...); if (limit === null) return null;`, and `limitFor` matches against `config.limits`, which defaults to `[]` (`src/schemas/quota.ts`) and is only populated by `~/.hive/quota.toml` — a file that does not exist on this machine. The Codex app-server *already returns authoritative rate-limit percentages on every turn*, and Hive drops every one of them on the floor. Nothing downstream can be trusted to explain cost, because nothing upstream ever measured it. This is why the user had to read the burn off the vendors' own CLIs.

**The burn was caused by the model, not the effort.** Every tier's Codex column says model `"default"`, which `resolveConcreteModel` reads from `~/.codex/config.toml` (`src/adapters/tools/models.ts:59-74`). For most of 2026-07-10 that file named `gpt-5.6-sol` — the frontier model — so *every* Codex agent ran Sol, at every tier. The `agents.executionIdentity` column preserves the receipts: cheap-tier agents `nora` and `liam` launched as `{"tool":"codex","model":"gpt-5.6-sol","effort":"low"}`. Sol at low effort is still Sol. Counting turns against the agent rows that survive, `standard`-tier Sol took 18 turns across 8 agents and `cheap`-tier Sol another 4 across 4 — routine work, on the most expensive model available. Claude never had this problem, and the asymmetry is the whole story: Claude's column names concrete aliases (`best`, `sonnet`, `haiku`) that resolve per tier, while Codex's column says `"default"` and inherits whatever the human last picked for their own interactive use. **A tier system that lowers the effort flag but not the model is not a tier system.**

**Refuted: deep-tier `xhigh` did not cause this.** `buildCodexConfigArgs` unconditionally passes `-c model_reasoning_effort=<tier effort>` (`src/adapters/tools/codex.ts:58`), so the route's effort always overrides the config file's. Throughout the burn the efforts were the defaults — `high`/`medium`/`low` by tier. `xhigh` appears in exactly two `executionIdentity` rows, both spawned *after* `~/.hive/routing.toml` landed at 13:21Z and pinned `deep.codex.effort = "xhigh"`. So `xhigh` is a forward-looking risk this document created, not the historical cause. It should still be reconsidered (see the open questions), but fixing it would have saved nothing yesterday.

**Also refuted, for now: retest waste and doc re-reading are not the measured *cause*.** Both are real, and doc re-reading is now measured: embedding `SPEC.md` costs **18,726 input tokens** (Haiku, measured by differencing two `claude -p` runs), which `CLAUDE.md` used to instruct every design-touching agent to pay before writing a line. A full `bun test` likewise runs on every rebase of every landing race. But with zero usage telemetry on the burn itself, attributing any *share of it* to either would be fabrication. Both are bounded now — doc reading by the scoped brief (R7), retesting by R8 — and neither is claimed as the fix.

**The inherited-MCP asymmetry is real as a *surface*, and mostly imaginary as a *cost*.** OpenAI's pricing page states that *"Every MCP you add to Codex adds more context to your messages and uses more of your limit"*, while Anthropic's costs page says Claude Code defers MCP tool definitions by default. Both tools inherit the human's own servers: Codex every `mcp_servers` entry in `~/.codex/config.toml` (here `idea` and `openaiDeveloperDocs`), Claude every server in its user config (here five, including three claude.ai connectors). The natural reading — Codex pays a per-message tool-definition tax that Claude escapes — did not survive measurement. On Codex the tax is *unmeasurable*: a `codex exec` turn that uses no tools costs 11,073 input tokens with both inherited servers attached and 11,073 with both detached, five runs of seven identical to the token. On Claude the tax is real but small: detaching the five inherited servers cuts 590 tokens of cache-creation input from the first turn, and removes 41 of the session's 71 tool names. The servers *are* attached in both — a Codex agent with `idea` enabled can name 39 of its tools, and zero once Hive detaches it — so the reachable-authority argument for scoping them stands on its own. The token argument does not, on Codex, and this document no longer makes it.

## Where the tokens go structurally (verified against the code)

The current pipeline, end to end. The orchestrator classifies a task into a tier (`deep`/`standard`/`cheap`/`review`); `resolveRoute` merges `~/.hive/routing.toml` over `defaultRoutingTable(now)` (`src/config/load.ts:70-119`); the spawner resolves both vendors' concrete models, inserts Opus 4.8 as a same-vendor release valve whenever the Claude candidate is Fable (`src/daemon/spawner-impl.ts:614-627`); `routeAndReserve` scores each candidate by worst post-reservation headroom across the five-hour and weekly windows, ties broken by preferred tool then name, and atomically reserves tier-keyed planning units before any process exists (`src/daemon/quota.ts:516-554`). `cheap` and `standard` spawns must additionally clear a deep-capacity reserve floor (defaults: 15% five-hour, 20% weekly — `src/schemas/quota.ts:36-37`). Unconfigured pools fall into compatibility mode: an unbounded reservation at `missing` confidence and one deduplicated warning (`src/daemon/quota.ts:558-589`).

This is a genuinely good safety design, and three properties of it explain most of Hive's avoidable spend:

**1. The defaults resolve routine work onto frontier models.** The `"default"` Codex model is a pass-through to the user's own interactive preference, so the tier only lowers the effort flag — the mechanism behind the burn above. The mid-tier models that providers explicitly position for exactly this work — GPT-5.6 Terra ("balancing intelligence and cost effectiveness") and Luna ("optimized for cost-sensitive workloads"), Sonnet 5 ("near-Opus quality on coding and agentic work at Sonnet cost") — are unreachable by default routing on the Codex side and only partially used on the Claude side. On the Claude side, `deep` resolves `best` → Fable 5 until the 2026-07-12 cutoff (`FABLE_AUTO_ROUTING_CUTOFF`, `src/schemas/routing.ts:77`), after which it pins `claude-opus-4-8`.

Why the code is this way is documented and real: naming a concrete Codex model in the defaults killed every spawn on a ChatGPT-plan account ("model not supported" — SPEC §6). `"default"` is entitlement-safe by construction, and that property is worth more than the tokens it costs. Escaping the trap without surrendering it is R4's whole job; the user-level escape hatch — pinning concrete models in `~/.hive/routing.toml`, which always wins over defaults — already works today and is what ended the burn at 13:21Z.

**2. Quota-aware routing is inert until someone writes `quota.toml`.** `~/.hive/quota.toml` and `~/.hive/config.toml` do not exist on this machine. With an empty `limits` array every candidate scores the same sentinel, ties resolve to the preferred tool, and reservations are unbounded. The scoring machinery, the deep-reserve floors, the Fable→Opus release valve, *and every authoritative usage observation* — none of it can fire. The cheapest high-leverage change in this entire document is shipping a starter `quota.toml`.

**3. Nothing learns.** Tier estimates are static config (`deep: 20, standard: 10, cheap: 4, review: 8` planning units — `src/schemas/quota.ts:39-44`). Reconciliation exists (a completed turn settles the reservation to provider-reported consumption when a hook supplies it), and provenance is honestly labeled (`authoritative`/`reported`/`estimated`/`missing`/`stale`), but — as the ledger audit above shows — the hook never fires, no loop feeds actuals back into the estimates, no record ties "this tier choice" to "this outcome," and the orchestrator's classification never sees the price of being wrong. The telemetry section below closes this.

Worth naming what the codebase already gets right, because the fix is *not* to rebuild it: minimal spawn briefings (task + file scope + memory index only — `buildAgentPrompt`, `src/daemon/spawner-impl.ts`); index-only memory injection capped at 30 lines with on-demand `memory_read` (SPEC §5); artifact-first messaging ("reference large artifacts instead of pasting them" is in every writer prompt); context recycling at a 65% quality line with structured handoffs instead of lossy `/compact` (SPEC §7); an orchestrator that is idle by default and physically cannot code (SPEC §2, §11); and a landing protocol that bounds retry loops at three attempts. These are token-efficiency mechanisms that most orchestrators lack. This document extends them; it does not replace them.

## What a spawn actually costs, measured

The four levers on an agent's *inputs* — as opposed to its model — are the spawn prompt, the docs it is told to read, the MCP surface it inherits, and whether it is spawned at all rather than followed up with. All four now have numbers, taken on this machine on 2026-07-10. The method is deliberately crude and therefore trustworthy: run `claude -p --output-format json` on a fixed one-word prompt, sum `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`, and difference against a baseline of 24,296. Codex numbers come from `codex exec --json` on `gpt-5.6-luna`, reading `usage.input_tokens`.

| Input | Before | After | Measured saving |
|---|---|---|---|
| `SPEC.md`, whole, as a task says "read SPEC.md" | 18,726 tok | — | — |
| …replaced by a brief naming a section (`SPEC §6`) | — | 1,882 tok | **16,844 tok** |
| …replaced by an outline, when no section is named | — | 421 tok | **18,305 tok** |
| Claude's inherited MCP servers (5 servers, 41 tools) | 590 tok | 0 | **590 tok** (once per session) |
| Codex's inherited MCP servers (2 servers) | 11,073 tok | 11,073 tok | **none detectable** |
| Spawn prompt, `standard` tier | 579 tok | — | — |
| …`cheap` tier, trimmed | — | 398 tok | **181 tok** |

Read the table for its shape, not its digits. **The document lever is two orders of magnitude bigger than the other three combined.** A single naive `SPEC.md` read costs more than the entire spawn prompt, the entire MCP surface, and the memory index put together, roughly thirty times over — and it is paid by every design-touching agent, on every spawn, forever. Everything else on this list is worth doing because it is free, not because it is large. That ordering is the whole reason the brief exists and the reason the prompt diet is a one-line footnote rather than a project.

Two of these numbers are worth distrusting in specific ways. The `SPEC.md` figure measures *embedding* the file in a prompt; an agent told to read it instead pays a comparable input cost plus a tool round-trip, so the brief's advantage is if anything understated. And the Codex row is an absence of evidence: `codex exec` bills an identical 11,073 input tokens whether the user's two servers are attached or detached, and attaching a third, known-connecting stdio server changed nothing either. Codex 0.144.0 appears to expose MCP tools through a registry the model queries rather than by injecting every definition into every message. That contradicts OpenAI's own pricing page, so one of the two is wrong; and because Hive spawns the interactive TUI rather than `codex exec`, this negative result may simply not transfer to the surface Hive actually uses. The scoping ships anyway, on the authority argument below — it just does not get to claim tokens.

## The model landscape (prices as of the sources cited)

Hive spawns subscription CLIs, so the binding constraint is usually plan capacity (five-hour and weekly windows), not API dollars. API prices still matter twice over: they are the best available proxy for how hard a model draws on shared plan capacity, and they become the literal bill wherever usage is metered (Fable 5 after 2026-07-12; any API-keyed account).

| Model | ID | Ctx | $/MTok in→out | Position |
|---|---|---|---|---|
| Claude Fable 5 | `claude-fable-5` | 1M | 10 → 50 | frontier; thinking always on; refusal classifiers; deep-route default until 2026-07-12 |
| Claude Opus 4.8 | `claude-opus-4-8` | 1M | 5 → 25 | top Opus; deep-route default after cutoff |
| Claude Sonnet 5 | `claude-sonnet-5` | 1M | 3 → 15 (intro 2 → 10 through 2026-08-31) | near-Opus on coding/agentic; new tokenizer ≈ +30% token counts |
| Claude Sonnet 4.6 | `claude-sonnet-4-6` | 1M | 3 → 15 | previous Sonnet; current `standard`/`review` alias target |
| Claude Haiku 4.5 | `claude-haiku-4-5` | 200K | 1 → 5 | cheap tier |
| GPT-5.6 Sol | `gpt-5.6-sol` (alias `gpt-5.6`) | 1.05M | 5 → 30 | frontier; this machine's Codex default at xhigh |
| GPT-5.6 Terra | `gpt-5.6-terra` | 1.05M | 2.5 → 15 | mid: "balancing intelligence and cost effectiveness" |
| GPT-5.6 Luna | `gpt-5.6-luna` | 1.05M | 1 → 6 | cost-optimized |

Sources: Anthropic model catalog and pricing (platform.claude.com/docs/en/about-claude/models/overview.md, pricing.md; cached reference dated 2026-06-24, model IDs re-verified against live docs in `src/adapters/tools/models.ts` comments dated 2026-07-09/10). OpenAI lineup fetched live 2026-07-10 (developers.openai.com/api/docs/models).

Three provider mechanics change the economics enough to shape routing, all re-verified against live vendor docs on 2026-07-10.

**Model choice, not effort, is what moves a subscription limit.** OpenAI's Codex pricing page publishes per-model message allowances within the shared five-hour window and says a smaller model *"extends your allowance significantly"*; Anthropic's costs page says *"Opus costs several times more per turn than Sonnet, and Sonnet more than Haiku."* Both vendors meter usage per model against the same window. This is the vendor-side confirmation of the diagnosis above: routing a `cheap` task onto a frontier model spends the limit at the frontier rate no matter how low the effort flag is set. Codex's local and cloud usage draw from the *same* five-hour pool.

**Effort is the second lever, and both defaults are higher than Hive assumes.** Codex's `model_reasoning_effort` accepts `minimal|low|medium|high|xhigh` (the config-file enum; `xhigh` is valid but *"model-dependent"*) and defaults to **`medium`**. Claude Code's `--effort` accepts `low|medium|high|xhigh|max` and defaults to **`high`** on Fable 5, Sonnet 5, and Opus 4.8 — so today *every* Claude agent Hive spawns, including `cheap`-tier ones, reasons at `high`, because Hive's Claude route column has no effort field at all. Vendor guidance is explicit that `xhigh` means *"deeper reasoning at higher token spend"* and `max` *"is prone to overthinking"*. Reasoning tokens bill as output tokens at both vendors, with no documented discount against the five-hour window.

**Prompt caching is a dollar discount; whether it is a *limit* discount is undocumented.** Anthropic publishes cache reads at 0.1× input and 5-minute writes at 1.25× (5-minute TTL, refreshed free on each hit), with invalidation cascading `tools → system → messages` — so changing tool definitions invalidates everything. Codex meters cached input roughly 10× cheaper in credits but publishes no TTL or invalidation rules. Neither vendor states that cache reads are discounted against the *subscription* five-hour or weekly limit, and Anthropic's "cached content doesn't count against your limits" line refers to claude.ai Projects, not the CLI. Resuming a session rather than respawning it (which Hive's crash recovery already does via `claude --resume`/`codex resume`) is therefore well-founded on cost and plausible but unproven on limits — do not spend design budget on cache-prefix engineering until that is measured. Batch APIs (50% off) don't apply to interactive CLIs and can be ignored here.

## Choosing a model per job

The orchestrator's classification instinct ("deep/standard/cheap") is right and stays — an LLM judging task difficulty is durable; an LLM recalling model lineups is not (SPEC §6). What's missing is a rubric that makes the classification honest, and a matrix that resolves it onto the models above.

**Classify on five dimensions, take the max.** A task's tier is driven by whichever dimension is worst, not an average:

- **Risk** — blast radius if the output is subtly wrong. Auth, money, migrations, concurrency, safety-stack code (Hive's own capability/landing machinery) are high regardless of size. Changelogs, formatting, doc stubs are low regardless of size.
- **Complexity** — how much must be held in mind at once: cross-module invariants, protocol state machines, novel design vs. pattern-following edits.
- **Uncertainty** — is the task well-specified, or does the agent have to discover what "done" means? Underspecified work burns tokens on exploration; either spec it tighter (cheaper) or send a stronger model (more expensive but converges).
- **Context load** — how much must be read before writing. High-context tasks favor models that navigate repos efficiently at high effort over cheap models that re-read.
- **Tool-use intensity** — long agentic loops (many test runs, migrations across many files) multiply per-turn cost; loop length matters more than model here, so bound the loop, then pick the model.

**Role modifies tier.** The orchestrator itself should be a capable model — it is one long-lived session whose misjudgments multiply across every spawn, and it is read-only and idle by default so its token draw is modest; economizing here is false economy. Workers take the matrix below. Reviewers need less than authors: review is bounded reading plus judgment, not construction — `review` correctly resolves to mid-tier, cross-vendor from the author (`src/daemon/quota.ts:519-522`). Integrators (conflict merges) are judgment-heavy on small context: standard tier, never cheap.

**The compact decision matrix.** Job archetypes → route. Claude effort assumes the route schema grows an effort field (recommendation R3); until then the Claude effort column is aspirational and the model column alone applies.

| Archetype (examples) | Tier | Claude route | Codex route | Validation | Budget guidance |
|---|---|---|---|---|---|
| Novel design, hard debugging, cross-cutting refactor, safety-stack code | deep | Opus 4.8 @ high–xhigh (Fable only by explicit pin) | Sol @ xhigh | tests + cross-vendor review | full context OK; one agent, no fan-out until design lands |
| Well-scoped feature, ordinary bug fix, test authoring | standard | Sonnet 5 @ medium–high | Terra @ medium | tests; review only if risk-high | brief ≤ 1K tokens; expect ≤ ⅓ of context window |
| Mechanical edits, changelogs, doc formatting, config touch-ups, skill authoring | cheap | Haiku 4.5 @ low | Luna @ low | tests/lint only | hard small budget; escalate rather than grind |
| Code review of another agent's diff | review | Sonnet 5 @ medium | Terra @ medium | n/a (is the validation) | diff + touched files only, not the repo |
| Integration/conflict merge (escalation only) | standard | Sonnet 5 @ high | Terra @ high | tests | scope = the conflict, nothing else |
| Research/summarization, repo Q&A | cheap→standard | Haiku→Sonnet 5 @ low–medium | Luna→Terra @ low–medium | none | read-only agent in main checkout (free of worktree cost) |

The deliberate asymmetry: **deep stays frontier, and everything else moves down one provider price band from today's behavior.** Sonnet 5 at Sonnet prices covering "near-Opus quality on coding" is precisely the standard tier's job description; Terra is OpenAI's same positioning. The savings estimate is straightforward: if the modal spawn today is Sol ($5/$30) or resolves through a frontier default, and the modal spawn after is Terra/Sonnet 5 ($2.5-3/$15), the standard tier — the bulk of spawns — costs roughly half per token, before effort tuning, which compounds another large factor on output-heavy agentic work.

**Escalation and de-escalation.** SPEC §6 already names retry-with-a-bigger-model as the escape hatch; make it the explicit ladder: a cheap agent that fails, stalls, or reports "this is harder than briefed" is killed (worktree preserved) and the task respawns one tier up with the failure note in the brief — cheap→standard→deep, one rung at a time, never skipping to deep on the first miss. Symmetrically, de-escalate deliberately: mechanical follow-ups discovered during deep work (rename sweeps, doc updates) are spawned down at cheap rather than done in the deep agent's expensive context. The escalation attempt cap is 2 (three total attempts across tiers), mirroring the landing protocol's 3-attempt bound; after that the orchestrator surfaces it to the user rather than burning a fourth run. A tier bump is also the answer to *uncertainty*: when the orchestrator cannot spec a task tightly, either spend its own (read-only, cheap) turns reading code to tighten the spec, or send standard with an explicit "report back if scope exceeds X" tripwire — never send cheap into fog.

**Early stopping and validation depth.** Every brief should carry a proportionality clause: budget guidance ("this is a ≤N-step task; if you exceed it, stop and report"), and validation scaled to risk — lint-only for cheap mechanical work, tests for standard (structurally enforced by landing anyway), tests plus cross-vendor review for deep or risk-high diffs. Uniform maximum validation is the hidden token tax of cautious orchestrators; the landing protocol's red-tests-never-merge floor means the *minimum* is already safe, so depth above the floor should be bought only where risk pays for it. API-level `task_budget` is not reachable through the interactive CLIs, so budgets are prompt-level (soft) backed by the context-% watchdog and the resource sweep (hard) — acceptable, because the expensive failure mode is grinding, and a grinding agent trips either the 65% line or the orchestrator's stall detection.

**When multi-agent is justified — and when it isn't.** Fan out only when file scopes are disjoint and the interface between them landed first (SPEC §3), when cross-vendor review adds an independent perspective, or when an integrator is escalated. Sequential tasks split across agents get *worse*, not just slower (SPEC's parallelism-ceiling open question cites 39–70% degradation on sequential tasks), and every extra agent pays a fixed overhead — worktree, spawn, briefing, its own repo reading — that a single agent's warm context amortizes. The default answer to "should this be two agents?" is no; the 2–4 writer ceiling is a cap, not a target. The orchestrator brief should say this in one line: *prefer one agent with a good brief over two agents with a coordination problem.*

**Duplicate-work prevention.** Three existing mechanisms just need consistent use: durable memory (check the injected index before spawning research — a fact like this document's existence should stop a future orchestrator from re-running this exact task; that is what the companion memory entry is for), run history (the daemon logs every task descriptor; near-duplicate descriptors within a session are a prompt to reuse the prior agent or its artifacts), and artifact-first handoffs (an agent that wrote `docs/research/X.md` reports the path, and the next agent reads the file rather than receiving a re-narration through the orchestrator's context — which also keeps the orchestrator's own window lean, protecting its cache prefix).

**Caching discipline.** Two rules cover most of the value. First, prefer resume over respawn whenever the conversation is the asset — crash recovery already does this, R9 makes it the orchestrator's default for ordinary follow-ups, and context-line recycling deliberately does not (there, the handoff is the compression). Second, keep spawn-prompt prefixes stable: `buildAgentPrompt` interpolates name and task first, so no cross-agent prefix sharing is possible — harmless today at one spawn per agent, but any future template growth should put static protocol text (the landing protocol, tool instructions) ahead of per-agent content, and never interpolate timestamps or IDs early. The CLIs manage their own caching internally; Hive's job is merely not to defeat it.

**Safe fallback behavior.** The existing posture is correct and should be preserved verbatim in any refactor: an explicit user tool choice is never silently overridden — an unsafe explicit spawn returns remaining capacity, reset times, and the recommended fallback instead of a substitution (`src/daemon/quota.ts:645-675`); control restarts pin the immutable execution identity and fail closed rather than guess (`src/daemon/spawner-impl.ts:296-310`); exhausted vendors mean no process starts. The one addition: when the Fable→Opus valve or any future cost-based downgrade fires, the decision `reason` already flows back through spawn — surface it in the orchestrator's envelope so downgrades are visible, not silent.

## The routing rubric, as an algorithm

What the orchestrator does (judgment, in the brief):

```
classify(task):
  risk, complexity, uncertainty, context_load, tool_intensity ∈ {low, med, high}
  tier = max-dimension mapping:
    any high in {risk, complexity}            → deep
    uncertainty high                          → tighten spec, else standard + tripwire
    all low and mechanical                    → cheap
    otherwise                                 → standard
  review needed = (risk ≥ med) or (diff will be large) → schedule review tier, cross-vendor
```

What the machine does (deterministic, extending `routeAndReserve`):

```
resolve(tier, route, request):
  candidates = [claude(route), codex(route)]            # concrete models
  if claude == fable: insert claude/opus-4.8 after it   # existing valve
  # R2 below generalizes the valve: each tier's route may list a same-vendor
  # downshift chain; append chain entries after the primary.
  filter by explicitTool if present
  score = min(post-reservation headroom, 5h and weekly) # existing
  sort by score, then preferred tool, then name         # existing
  reserve first candidate clearing floors               # existing
  effort = route[tool].effort                           # Codex today; Claude after R3
  on failure: report capacity + reset + safe fallback; never substitute an explicit choice
```

The rubric deliberately adds no new scoring inputs to the deterministic half — headroom remains the only machine-judged quantity, because it is the only one Hive can measure honestly today. Cost-optimality lives in the *table* (which model each tier names), not in a runtime cost model; that keeps the resolver boring, auditable, and immune to stale price data. A runtime $/token optimizer was considered and rejected: subscription capacity is the real constraint, prices staleness-decay, and a wrong price silently rerouting work is worse than a legible table the user can read and override.

## Recommendations, concretely

- **R1 — Ship a starter `quota.toml` and make compatibility mode loud.** This is the top recommendation by a wide margin, because `limits: []` does not merely disable scoring — it makes `limitFor` return null, which makes `observeCodexRateLimits` discard the authoritative usage percentages Codex hands Hive on every turn. Until a pool exists, Hive cannot measure its own spend, and every other recommendation here is unfalsifiable. Provide a commented template (Claude pool, Codex pool, five-hour/weekly allowances) and have `hive claude` nudge once per session while `limits` is empty. Effort: small; config plus one warning path. **This one needs the user's judgment before it lands**: allowances and `estimates` together decide when spawns start getting refused, and a template with wrong numbers converts a measurement gap into a work stoppage. A safe first cut is to set allowances to `100` so units read as percent-of-limit, land observation-only, and tune `estimates` from the first week of real data.
- **R2 — Generalize the Fable→Opus valve into per-tier downshift chains.** Replace the hardcoded `if claudeModel === CLAUDE_BEST_MODEL` splice with an optional `fallbacks: [model...]` list on each route entry, same semantics (appended after primary; ties keep the primary). This lets standard declare Sonnet 5 → Sonnet 4.6, or Sol → Terra, without touching resolver logic. The existing valve becomes data.
- **R3 — Add `effort` to the Claude route column** (`ClaudeRouteSchema`). This is worth more than it first appears: with no effort field, every Claude agent at every tier inherits the CLI's `high` default, so Hive's `cheap` Claude tier reasons as hard as its `deep` one. `--effort <low|medium|high|xhigh|max>` is verified present on claude 2.1.206 here, and per-model support is machine-discoverable at zero cost via the stream-json `initialize` control request (`supportsEffort`, `supportedEffortLevels` — see `docs/model-selection.md`). It is *not*, however, "plumbed the same way Codex's already is": effort is part of `ExecutionIdentity`, the immutable record a control restart replays. Adding a Claude effort means a durable-schema change and a migration for identities already stored, and a half-plumbed version — spawn honors effort, restart silently drops it — is exactly the silent divergence the immutable-identity design exists to prevent. Land the schema, the adapter flag, and the identity together, defaulting every tier to unset (today's behavior) so the change ships inert and the user opts in per tier.
- **R4 — Point default routes at mid-tier models where entitlement-safe.** Claude column: `standard`/`review` → `sonnet` alias already resolves to the current Sonnet (entitlement-adaptive, correct); no change needed beyond effort. Codex column: `"default"` cannot be dropped safely (SPEC §6's entitlement lesson), so use R2's chain inverted — attempt `gpt-5.6-terra` (standard) / `gpt-5.6-luna` (cheap) first, and let the existing 15-second launch-failure monitor (`monitorReadiness`, which already detects "model not supported"-class failures) trigger a respawn on `"default"`. One extra failed launch per unsupported account per session is the cost; frontier-by-default forever is the alternative.
- **R5 — Put the classification rubric and the fan-out caution into `ORCHESTRATOR_BRIEF`.** Today the brief says "classify each task as deep, standard, or cheap" with no criteria. Add the five dimensions, the max rule, the escalation ladder, the "prefer one well-briefed agent" line, and the budget-tripwire habit. This is the highest-leverage prompt edit in the system: every routing decision flows through it.
- **R6 — Close the estimate loop.** Reconciliation already writes actuals when telemetry supplies them; add a periodic job (or `hive quota calibrate`) that compares reserved estimates to reconciled actuals per tier and proposes updated `estimates` values. Keep it propose-not-apply, matching the skills rule-of-three posture. Blocked on R1: there are no actuals to calibrate against until a pool exists.
- **R7 — Scope the agent's inputs at spawn: brief, MCP surface, prompt (implemented).** Four changes, in descending order of what they are worth.

  **The scoped brief** (`src/adapters/brief.ts`) is the one that matters. At spawn the daemon reads the docs the task names — `SPEC §6`, `SPEC.md sections 6 and 7`, `docs/research/x.md`, a quoted heading — extracts exactly those sections, and embeds them verbatim with `path:line` ranges, followed by an outline of every section it did *not* embed. A task that names no section gets the outline alone; a doc under 4K chars is embedded whole because that is cheaper than the tool call to open it. Budgets are hard (6K chars a section, 12K a brief) and overflow degrades to a pointer, never to silence. `CLAUDE.md` no longer tells agents to read `SPEC.md` before design work; it tells them to read it by section. The rejected alternative was retrieval — embed the docs, pull the *k* nearest chunks — which loses the one property that makes this safe: a section boundary is a unit the author wrote, and a `path:line` pointer is a promise the agent can check. A chunk is neither.

  **Per-spawn MCP minimization.** Claude launches with `--mcp-config <worktree>/.mcp.json --strict-mcp-config`, which drops the user's servers and keeps Hive's; measured, that is five servers and 41 tool names down to one and zero. Codex gets one `-c mcp_servers.<name>.enabled=false` per inherited server, computed by *reading* `~/.codex/config.toml` and never writing it. Two facts about the Codex path were found by experiment and are load-bearing: `-c` **deep-merges**, so neither `-c 'mcp_servers={}'` nor a replacement inline table detaches anything — `enabled=false` is the only override that works; and `-c` splits dotted paths on `.` with no TOML quoting, so `mcp_servers."odd.name".enabled=false` silently creates a *new*, transport-less server and Codex then refuses to start. Hive therefore emits overrides only for bare-key-safe names and leaves the rest attached rather than risk a launch failure. `hive` is never detachable. This is R7's original recommendation inverted: the old advice was for the *user* to prune their global Codex config, which was always the wrong place to fix a per-agent concern.

  **The spawn-prompt diet.** The `cheap` tier gets a rewritten landing protocol — 181 tokens shorter, and a rewrite rather than a subset, because every rule in it is safety-critical and a small model is precisely the one that must not infer a missing step. A test asserts each rule survives.

  **A continuous-execution clause**, in every tier's prompt: after reporting a landing or milestone, continue with the next authorized piece of the assignment in the same session. Agents were repeatedly observed idling at their prompt while still holding approved work. This is the mirror of R8's escalate-don't-grind tripwire — grind means escalate, idle-with-work means continue — and it is also a session-reuse mechanism, since the cheapest place to do the next piece of work is the session that already has the context.

- **R9 — Prefer follow-up over respawn (implemented as policy).** A respawn re-pays the brief, the memory index, the spawn prompt, and whatever repo reading the agent does to orient; a `hive_send` to a live agent pays one message. `ORCHESTRATOR_BRIEF` now makes reuse the default: before `hive_spawn`, check for a live agent that already owns the area, reuse it when its status is live and `contextPct` is under 65, and spawn fresh only when no live agent fits, the file scopes would collide, or the agent is near the recycle line. No `reusable` flag was added to the agent row, and that is deliberate: `status` and `contextPct` already answer the question, and a second field asserting reusability is a second source of truth that can disagree with the first. The cost of the policy is that a stale-but-live agent can absorb work it is badly suited to; the 65% line and the orchestrator's own judgment are the only guards, and if that proves insufficient the answer is a better `hive_status` summary, not a new flag.
- **R8 — Bound the two obvious wastes (implemented).** `buildLandingProtocol` permits skipping the post-rebase retest when `git diff --name-only ORIG_HEAD..HEAD` shows the rebase pulled in only `.md` files that no test reads; a landing race between eight agents otherwise reruns the whole suite once per rebase per agent to absorb a docs change. The escape hatch is deliberately narrow — it skips the *rerun*, never the fast-forward gate and never the red-tests-never-merge rule — and it is sound only because the agent's own pre-rebase run was green and no test in this repo reads a committed `.md`; a repo where tests assert on documentation must not adopt it. The rejected alternative was having the daemon decide and skip on the agent's behalf, which would put a correctness judgment about *this repo's* test suite inside generic spawn code. `buildAgentPrompt` tells agents to search rather than read large files whole, to reuse existing artifacts, and to escalate rather than grind when a task exceeds its brief. Unlike the rest of this list, these two cost nothing to be wrong about.

## Telemetry

Record per spawn, in the daemon's existing SQLite (most fields already exist on the agent row or reservation ledger; the delta is joining and keeping them): tier, requested vs. resolved tool/model/effort, decision `reason`, estimate vs. reconciled actual (with provenance confidence), wall time, turn count if the hook path supplies it, terminal outcome (landed / failed / killed / escalated / recycled), escalation lineage (which agent this respawn continues), landing attempts, and review verdict where a review tier ran. Derived weekly views: planning units by tier×model (where the money goes), estimate error by tier (feeds R6), escalation rate by tier (misclassification signal — a high cheap→standard rate means the rubric or the brief is wrong), first-attempt landing rate by model (quality signal), and review catch rate (whether paid validation is finding anything). Provenance stays first-class exactly as the quota store does it: an `estimated` cost row must never be presented with `authoritative` confidence.

## Evaluation plan

Three layers, cheapest first. **(1) Offline replay:** the daemon logs every task descriptor; hand-classify a sample of ~50 historical descriptors against the rubric and compare to the tiers the orchestrator actually chose — this validates R5 before it ships and needs no spawns. **(2) Shadow scoring:** for two weeks after R1/R2 land, log what the new table *would* have chosen alongside what the old table did choose, and diff projected planning units; no behavior change, pure measurement. **(3) A/B by session:** alternate sessions between old and new defaults on real Hive-builds-Hive work (the v1.1 dogfood gate is the natural venue) and compare the telemetry views above — cost per landed task, first-attempt landing rate, escalation rate, and user interventions. Success criteria: standard-tier planning units per landed task down ≥30% with first-attempt landing rate within 5 points of baseline. If landing rate degrades more than that, the mid-tier models are not carrying the standard tier and the matrix moves that archetype back up — the telemetry makes the retreat cheap and legible.

## Staged rollout

1. **Measure and arm (no routing change):** R1 starter `quota.toml` + compatibility-mode nudge; telemetry joins; offline replay of the rubric. R7 and R9 already landed here — they shrink what a spawn reads and reaches without touching which model runs, so they were safe to ship ahead of any routing change and their effect is independent of it.
2. **Route (reversible by config):** R2 downshift chains, R3 Claude effort, R4 mid-tier defaults with launch-failure fallback, R5 brief update — each independently revertible via `routing.toml` overrides, which always win over defaults. Shadow-score for two weeks, then A/B.
3. **Learn (propose, never auto-apply):** R6 estimate calibration; escalation-rate feedback into the rubric text; eventually the curated routing manifest (SPEC §6) carries the matrix so new model generations update the table without a Hive release. Auto-applying learned changes is explicitly deferred — a self-tuning router that drifts silently is the failure mode the immutable-identity and provenance work exists to prevent.

## How a fresh orchestrator should use this

Don't re-run the research — the durable memory entry pointing here exists precisely so this pass isn't repeated. Use the classification rubric and the decision matrix when tiering tasks; treat the matrix's model names as defaults the routing table may already encode, not values to pass explicitly (`hive_spawn` resolves models through the table; its optional `model` field is reserved for the user naming one — see `docs/model-selection.md`). Write task briefs that cite their sources by section (`SPEC §6`, not "read SPEC.md") — the spawner extracts what you name and nothing else, and the difference is measured in the table above. Before recommending routing changes to the user, check which recommendations (R1–R9) have already landed by reading `src/schemas/routing.ts` and `~/.hive/quota.toml` — this document describes the state on its date, and the code outranks it afterward. The time-sensitive assumptions section lists exactly what to re-verify before acting on a price or a CLI behavior.

## Open questions

- **Does the review tier pay for itself?** Telemetry's review-catch-rate view will answer whether cross-vendor review finds enough real defects to justify its planning units, feeding the SPEC's open self-land-vs-gate question with data instead of instinct.
- **Is `deep.codex = xhigh` worth its cost?** `~/.hive/routing.toml` pins the deep Codex tier to `xhigh` — two effort steps above Codex's own `medium` default, on Sol, the model that spends the five-hour limit fastest. Vendor guidance says only that `xhigh` buys *"deeper reasoning at higher token spend"*, and that it is model-dependent. No Hive data supports or refutes the pin, because the deep tier ran it exactly twice. This is the single highest-cost unexamined setting Hive has, and it is the user's to keep or drop; the A/B should sweep `high` against `xhigh` on deep-tier work before anyone defends it. Symmetrically, Claude's `cheap` tier reasons at `high` because Hive never sets effort at all (R3).
- **Does an attached MCP server cost a Codex TUI turn anything?** `codex exec` says no, repeatedly and to the token, against OpenAI's own published claim that it does. Hive spawns the TUI, not `exec`, and the TUI's per-turn token usage is not reachable from a script here — the app-server driver's rate-limit events are the obvious place to settle this once R1 lands and Hive is recording them. Until then Hive's Codex MCP scoping is justified by authority, not by tokens: an agent that cannot see the human's IDE bridge cannot be talked into using it.
- **How often does a task name its sections?** The scoped brief pays 421 tokens instead of 18,726 when a task names a doc, and 1,882 when it names a section — but only if the orchestrator writes briefs that cite sources. `ORCHESTRATOR_BRIEF` now instructs it to; nothing measures whether it does. The cheapest check is a count of spawn task descriptors containing a `§` or a `.md` path, which the daemon already logs.
- **Can classification be assisted without being automated?** A cheap-model pre-classifier suggesting a tier to the orchestrator was considered and deferred: it adds a spawn to save a judgment the orchestrator makes adequately, and the telemetry will show whether misclassification is even the binding problem.
