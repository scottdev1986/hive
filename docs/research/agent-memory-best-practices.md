# Agent memory: what the field actually does

Every claim here was checked against a live vendor page, doc, or paper in July 2026 — not against training memory, which on a topic moving this fast is wrong more often than it is right. Sources are inline as URLs; where a number is self-reported marketing without a public method, it is flagged as such, because a spec that cites a marketing figure as evidence inherits the lie.

This document records what the industry and the literature *do*. What hive does with it — the design decisions, the token-cost accounting, the rejected alternatives — lives in [SPEC.md decision 5](../../SPEC.md#5-what-memory-actually-means) and is not duplicated here. Read this to understand the landscape; read the spec to understand hive's choices.

## The one thing everyone agrees on

Every mature coding-agent vendor now ships **two tracks**, and they are different in kind, not degree:

- A **human-authored, always-injected instruction file** — `CLAUDE.md`, `AGENTS.md`, `.cursor/rules`, `.github/copilot-instructions.md`. Committed, shared, deliberate. Loaded in full every session, bounded only by size caps.
- An **agent-authored, auto-accumulating memory store** — Claude Code auto memory, Cursor Memories, Windsurf/Cascade Memories, GitHub Copilot agentic memory. Cheap, per-user or per-repo, written as the agent works.

Codex is the lone outlier: `AGENTS.md` and nothing else, no native agent-written memory (https://learn.chatgpt.com/docs/agent-configuration/agents-md.md). Everyone else runs both tracks. The consensus split is worth stating plainly because it is the frame for every decision below: *deliberate + shared + committed* is one mechanism, *auto-captured + cheap + local* is a different mechanism, and conflating them is the classic mistake.

`AGENTS.md` is the emerging cross-tool lingua franca — over 60,000 open-source repos ship one, natively read by 20+ agents including Codex, Cursor, Copilot, Windsurf, Aider, Gemini CLI, Devin (https://www.tembo.io/blog/agents-md). Its universal merge rule is **root-down concatenation, nearest-file-wins-by-position**. The one gap that matters for a multi-vendor tool: **Claude Code reads `CLAUDE.md`, not `AGENTS.md`** (https://code.claude.com/docs/en/memory, https://www.tembo.io/blog/agents-md) — bridged only by a `CLAUDE.md` that does `@AGENTS.md`, or a symlink. Any cross-vendor design that assumes one filename will silently starve the other vendor's agents.

## Vendor-native mechanisms

### Claude Code

Two separate systems, both loaded at conversation start (https://code.claude.com/docs/en/memory):

- **`CLAUDE.md`** (human-written): a scope hierarchy — managed-policy → user (`~/.claude/`) → project (`./CLAUDE.md`) → local (`./CLAUDE.local.md`) — **concatenated, not overridden**, root-down so more-specific-and-later wins by recency. On launch Claude walks *up* the tree loading every file in full; subdirectory files load lazily when a file there is read. `@path` imports expand at launch (max depth 4) and do **not** save tokens. Guidance: keep each file **under 200 lines** — longer files "consume more context and reduce adherence." Content is delivered as a *user message after the system prompt* — soft guidance, not enforcement; hard rules belong in a PreToolUse hook.
- **Auto memory** (agent-written, v2.1.59+, on by default): Claude decides per turn whether something is worth remembering; it does not save every session. Stored per-git-repo at `~/.claude/projects/<project>/memory/`, machine-local. The recall design is the load-bearing detail: a **`MEMORY.md` index** (only the **first 200 lines or 25 KB**, whichever comes first, loaded at session start) plus **topic files that are not loaded at startup** and are read on demand with normal file tools (https://code.claude.com/docs/en/memory). This is index-plus-retrieval built to bound token cost — the same shape hive landed.
- **`.claude/rules/`**: modular instruction files; rules with a `paths:` glob are **path-scoped** and enter context only when a matching file is read — conditional injection to save tokens.

Separately, the **Claude memory tool** (API-level, `memory_20250818`, now GA on the Messages API) is a client-side tool: Claude *requests* file ops against a `/memories` prefix and the application executes them against storage it controls, with six commands (`view`/`create`/`str_replace`/`insert`/`delete`/`rename`) (https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool). When present, the API auto-injects a system-prompt instruction to `view` the memory directory first, then read specific files on demand — **just-in-time retrieval, not an up-front dump**. This is a build-your-own-agent tool, distinct from Claude Code's auto memory.

### Codex

`AGENTS.md` only. Global `~/.codex/AGENTS.md`, then repo-root down to cwd; all discovered files **concatenated root-down, joined by blank lines**, nearest-to-cwd last and therefore highest-precedence (https://learn.chatgpt.com/docs/agent-configuration/agents-md.md). `AGENTS.override.md` *replaces* a level's file rather than merging. Concatenation stops at `project_doc_max_bytes` (**default 32 KiB**). Pure always-injected full-dump; no retrieval, no agent-written memory.

### Cursor

- **`.cursor/rules`** (`.mdc` files, frontmatter `description`/`globs`/`alwaysApply`): four activation modes — **Always** (`alwaysApply:true`), **Agent-requested** (agent reads the description and pulls when relevant), **Auto-attached** (glob-matched file in context), **Manual** (`@rule-name`) (https://cursor.com/docs/rules). Only Always rules cost tokens every request; the rest are conditional. Precedence Team → Project → User. Size guidance: **keep rules under 500 lines** — the firmest public per-file number in the field.
- **Cursor Memories** (agent-written): a background model observes conversations and **proposes** a memory that the **user approves before it is saved**; stored per-project, per-user (https://forum.cursor.com/t/0-51-memories-feature/98509). *(Cursor's own Memories doc renders client-side and could not be fetched as text; mechanics here are from changelog/forum, so treat the exact recall path as implementation detail, not a guaranteed contract.)*

### GitHub Copilot

- **Instruction files**: repo-wide `.github/copilot-instructions.md` (all requests) plus path-specific `.github/instructions/NAME.instructions.md` with an `applyTo` glob; both apply when both match. Also reads `AGENTS.md`/`CLAUDE.md`, nearest-in-tree wins (https://docs.github.com/copilot/customizing-copilot/adding-custom-instructions-for-github-copilot).
- **Agentic memory** (public preview 2026-01, on-by-default for Pro 2026-03): agents auto-capture "tightly scoped insights" as **repository facts stored with citations pointing to the supporting code**, plus per-user preferences (https://docs.github.com/en/copilot/concepts/agents/copilot-memory). Two mechanisms make it the most rigorous staleness handling shipped anywhere: **before use, each memory's citations are re-validated against the current branch** — only validated facts are applied, and if code contradicts a memory the agent is prompted to store a corrected version; and **any unused fact is auto-deleted after 28 days**, the timer resetting on validated use (https://github.blog/ai-and-ml/github-copilot/building-an-agentic-memory-system-for-github-copilot/). Citation-validation-before-use is the pattern to steal for anything load-bearing.

## Open frameworks

### Letta (formerly MemGPT) — and MemFS, the closest analog to a file-backed coding memory

Letta's OS-inspired hierarchy: **core memory** (always-in-context blocks, direct access), **recall memory** (append-only chronological log of raw messages, searchable), **archival memory** (vector DB of distilled facts, query-only, no size limit) (https://docs.letta.com/guides/agents/context-hierarchy/, https://docs.letta.com/guides/core-concepts/memory/archival-memory). Self-editing is LLM-driven via a function schema (`core_memory_replace`, `memory_rethink` for full-block rewrite "keeping all information not outdated or inconsistent", empty-string-replace = delete) with no hard-coded write trigger — the agent decides (https://raw.githubusercontent.com/letta-ai/letta/main/letta/functions/function_sets/base.py, https://ar5iv.labs.arxiv.org/html/2310.08560).

The coding-agent product, **MemFS**, is the single closest published analog to hive's design: "a git-backed memory filesystem … projecting memory into markdown files with git version history and conflict resolution." Files in `system/` are always loaded into the prompt; everything else shows **only path + description in a 'memory tree' and loads full content only when relevant** (https://docs.letta.com/letta-agent/memory). Markdown with YAML frontmatter (`description:`). Commands `/init` (bootstrap memory by inspecting the project), `/remember`, `/doctor` (audit token usage). **Dream subagents** review recent conversations and write lessons into memory; reorganization subagents split/merge/restructure using **git worktrees so they write concurrently without blocking** the main agent. Git-backed markdown + always-loaded index folder + lazy full-content is exactly the hive shape, arrived at independently.

### LangGraph / LangMem

Short-term = thread-scoped checkpointers; long-term = a cross-thread key-value **Store** with namespaces (https://docs.langchain.com/oss/python/langgraph/persistence). LangMem documents the memory taxonomy explicitly: **semantic** (facts — "what", stored as a profile or a collection), **episodic** (successful interactions as few-shot examples — schema `observation`/`thoughts`/`action`/`result`, note *successful* only), **procedural** (behavior rules, saved as updated prompt instructions) (https://langchain-ai.github.io/langmem/concepts/conceptual_guide/). Retrieval spans direct key lookup, embedding semantic search, and injection. The write pattern is **retrieve-`existing`-then-decide** with `enable_inserts`/`enable_deletes` gates; the system "must reconcile new information … either deleting/invalidating or updating/consolidating existing memories." Consolidation is split into **conscious/hot-path** (inline, adds latency) vs **subconscious/background** (reflect after the fact, higher recall), with a **ReflectionExecutor** that debounces — cancels and reschedules pending consolidation as new messages arrive so it processes complete context, not fragments. Gaps flagged: no built-in TTL staleness, and no automatic episodic→semantic distillation (documented consolidation is within-type).

### Zep / Graphiti — the staleness gold standard

Graphiti builds three subgraphs: **episodic** (raw, lossless episodes), **semantic entity** (facts/relationships), **community** (clustered summaries) — a clean episodic-vs-semantic-vs-consolidated split (https://arxiv.org/html/2501.13956v1, https://github.com/getzep/graphiti). Its distinguishing feature is a **bi-temporal model**: four timestamps per edge — `valid_at`/`invalid_at` (when the fact was true in the real world) and `created_at`/`expired_at` (when hive's DB learned/unlearned it) (https://blog.getzep.com/beyond-static-knowledge-graphs/, https://help.getzep.com/v2/searching-the-graph). On a contradicting new fact, an LLM invalidation prompt marks the old edge expired and **rewrites its text to past tense** — nothing is deleted, so point-in-time reconstruction ("what did we believe at commit X") survives. Benchmarks (LongMemEval, facts that change over time): Zep gpt-4o-mini 63.8% acc / ~1.6k tokens / 3.2s vs full-context 55.4% / ~115k tokens / 31.3s (https://arxiv.org/html/2501.13956v1). The staleness argument is the real lesson: a plain vector store retrieves the most-similar chunk regardless of currency; a validity-window model retrieves the *currently-true* fact. *(Caveat: Zep-vs-Mem0 head-to-heads seen in secondary write-ups were not confirmable on either vendor's primary pages; the paper compares to a full-context baseline, not to Mem0.)*

### Mem0 — the sharpest write gate

Two-phase pipeline: **extraction** (each new message pair, against a rolling summary + last-10 window, excluding "greetings, filler, vague acknowledgments") → **update** (retrieve top-10 semantically-similar existing memories, then an LLM function-call emits one of **ADD / UPDATE / DELETE / NOOP** — ADD only when no equivalent exists, UPDATE mutates in place with stable IDs, DELETE on contradiction) (https://arxiv.org/html/2504.19413v1, https://github.com/mem0ai/mem0/blob/main/mem0/configs/prompts.py). Retrieving similar memories *before* writing is genuine dedup-before-write, and it is the reference design for it. Storage is a vector DB, retrieval is embedding top-k search-on-demand. Staleness is contradiction-driven only — no validity intervals (a documented gap vs Zep). *(Caveat: two Mem0 benchmark regimes coexist — the paper's LLM-as-Judge 66.88% and a 2026 marketing "token-efficient" LoCoMo 92.5 at a different scale; always cite a Mem0 number with its system and source.)*

## Industry knowledge stores

**Devin Knowledge** (Cognition): each item is a **trigger description** ("Devin uses this when…") + injected content, recalled "when relevant, not all at once or all at the beginning" — selective trigger-matched injection, not a dump (https://docs.devin.ai/product-guides/knowledge). Primarily human-curated in a Library UI, with auto-suggestion from chat feedback that the **user edits before saving**. Scoped org / enterprise / repo. Staleness is weakly specified — docs only advise "make a habit of updating" — a real gap.

**Windsurf/Cascade Memories** (now Cognition): **auto-captured, local-only** (`~/.codeium/windsurf/memories/`, not committed, not shared, no credit cost), retrieved contextually — explicitly the ephemeral "stop re-asking" tier, with the guidance that **durable shared conventions belong in a Rule or `AGENTS.md` instead** (https://docs.devin.ai/desktop/cascade/memories). The cleanest statement in the field of the auto-captured-local vs curated-shared-committed split.

## The patterns that matter

**Episodic vs semantic.** Everyone separates a raw event log from distilled facts; only Zep and LangMem name it. Episodic = what happened / what worked (LangMem's `observation/action/result`, Letta recall, Zep episodes); semantic = distilled facts (Letta archival/core, Zep entities, Mem0 memories). For a coding tool the useful cut is: **store the distilled lesson, not the transcript** — Reflexion's central result, that natural-language *explanations* of failures beat raw trajectories by ~8 points on HumanEval (https://arxiv.org/abs/2303.11366, https://www.alphaxiv.org/overview/2303.11366v4).

**Write policy — what earns a memory.** The dominant pattern is **retrieve-similar-then-decide, before writing** (Mem0 ADD/UPDATE/DELETE/NOOP; Zep entity/edge dedup + invalidation; LangMem `existing`-gated managers). Exclusion rules matter — Mem0 explicitly refuses filler. Auto-capture is universally **suggestion or local-only, never a silent write into the shared tier**: Devin and Cursor route through human approval; Windsurf keeps it local; Copilot validates against code before applying. A spec that lets agents silently write shared, load-bearing facts is out of step with every shipped product.

**Retrieval — three strategies, choose per memory kind.** Always-injected (Letta core blocks / MemFS `system/`, Claude `CLAUDE.md`, LangMem procedural) for identity and always-relevant rules; search-on-demand tool call (Letta archival, Mem0 top-k, Zep hybrid, Claude memory tool) for the long tail; embedding similarity under the search paths. The hybrid worth copying for a file-backed tool is **Letta MemFS / Claude auto-memory**: a tiny always-loaded index (path + description / title) with full content pulled only on match. This keeps the window lean while staying discoverable — you cannot pull what you cannot see exists, so the index is what makes search-on-demand actually fire.

**Staleness — the sharpest differentiator and the biggest shipped gap.** Zep's bi-temporal supersede-don't-delete is the strongest primitive; Copilot's citation-validation-before-use + 28-day TTL is the strongest *shipped* one. Everyone else handles staleness only by contradiction-driven overwrite (Mem0, Letta, LangMem), losing the audit trail; Devin and Windsurf have essentially none. The load-bearing rule across the rigorous systems: **a recalled fact must be re-checked against current reality before it drives an action** — Copilot re-validates citations, Zep re-scopes validity. Facts age; a recalled fact is a claim, not truth.

**Token budget is also an accuracy budget.** The strongest argument for a small injected context is not price — it is that models degrade *inside* their advertised window. Chroma's "context rot" study tested 18 frontier models (GPT-4.1, Claude 4, Gemini 2.5, Qwen3) and found all of them degrade non-uniformly as input grows, driven more by semantic similarity between the target and distractors than by raw length; models even scored *better* on shuffled haystacks than logically ordered ones (https://www.trychroma.com/research/context-rot). Reported magnitudes (secondary aggregations): 30–50% accuracy drops before the documented limit, and an Oct-2025 result found performance falling 13.9–85% as input grew **even with 100% perfect retrieval** — long context degrades reasoning, not just recall (https://www.morphllm.com/context-rot). Generative Agents scores retrieval by **recency (0.995/hour decay) × importance (LLM 1–10) × relevance (cosine)** and stores derived reflections so recall skips raw history (https://ar5iv.labs.arxiv.org/html/2304.03442). The convergent guidance: keep the always-injected surface small and semantically distinct — Cursor's "<500 lines", Claude's "<200 lines", Devin/Windsurf's "inject on trigger, never front-load everything." *(The often-quoted AGENTS.md "150 lines / 20–23% inference cost" figure has no public methodology — do not cite it as evidence; Cursor's <500 is the firm public number.)*

**Consolidation — background, not in the hot path.** The heavy dedup/reorg/distill pass is deferred everywhere: Letta sleep-time/dream subagents (git worktrees for non-blocking concurrent writes), LangMem's debounced ReflectionExecutor, Mem0's async summary module, Zep's map-reduce community summaries. The recurring design is cheap in-loop writes for urgent context, expensive consolidation deferred to a background agent. Genuine cross-item episodic→semantic distillation is rarer than it looks — only Letta sleep-time and Zep communities do it; most "consolidation" is within-type dedup.

## What this means for hive

The patterns hive should adopt and the ones it should decline — with token-cost accounting and rejected alternatives — are argued in [SPEC.md decision 5](../../SPEC.md#5-what-memory-actually-means). In one breath: hive already sits on the industry-consensus shape (committed markdown facts + always-injected index + search-on-demand, the Letta-MemFS/Claude-auto-memory design arrived at independently); the growth is a **write policy** (Mem0's dedup-before-write, suggestion-not-silent-shared-write), **provenance + verification** (Copilot's re-validate-before-load-bearing, without the knowledge-graph weight of Zep because git-committed markdown already gives point-in-time reconstruction for free), and a **repo profile** that front-loads the structured facts an agent would otherwise burn a cold exploration discovering. The one thing hive must not inherit from any single-vendor design is a filename assumption: `CLAUDE.md` and `AGENTS.md` are read by different tools, and a multi-vendor tool that hardcodes either starves the other.

## Confidence caveats

- Cursor Memories mechanics are from changelog/forum, not the (client-rendered) official page — exact recall path is implementation detail.
- Zep-vs-Mem0 head-to-head numbers were not confirmable on either vendor's primary pages; both publish self-favoring figures. Use each system's own-vs-full-context number.
- Mem0 has two coexisting benchmark regimes at different scales (paper 66.88% J vs 2026 marketing LoCoMo 92.5) — cite with system + source.
- The AGENTS.md "150 lines / 20–23%" figure has no public methodology; treated as directional only.
- Context-rot's precise "30–50%" and "300k–400k token" figures are secondary aggregations of Chroma's qualitative curves, not verbatim from the report.
- Letta sleep-time architecture page 404'd; those mechanics came from the official blog + forum guide.
- All vendor mechanics were current as of July 2026 and are protocol/product surfaces, not stable contracts — re-verify before trusting against a newer release.
