# Dynamic model routing: capability-aware, effort-first, release-independent

## What this is and why

This is a design draft, research only — nothing here is implemented. It answers one question: what has to change for Hive's router to assign the best model *and* effort level per task, without a Hive release standing between the user and a model that shipped yesterday.

Today's router is a shipped table with a user override file. That shape has three structural gaps, and each is a required property of the overhaul:

1. **Capability-awareness.** Nothing in the resolver knows what a model *is*. The tier table encodes capability implicitly (`cheap` names `haiku`), but the moment routing becomes dynamic — quota valves, downshift chains, derived defaults — there is no floor stopping a coding task from landing on a Haiku-class model. The gate that refuses that spawn does not exist.
2. **Effort as a first-class knob.** `hive_spawn` takes a `model` but no `effort` (`src/daemon/spawner.ts:8-19`). The Codex route column carries effort; the Claude column does not (`src/schemas/routing.ts:14-29`), so every Claude agent Hive spawns — including cheap-tier ones — reasons at the CLI's `high` default. Effort is half a knob, on one vendor, reachable only through the table.
3. **Release-independence.** The router's knowledge of the model landscape is compiled in: `CLAUDE_BEST_MODEL` and `CLAUDE_OPUS_MODEL` are constants verified by hand against an installed CLI (`src/adapters/tools/models.ts:9,19`), `FABLE_AUTO_ROUTING_CUTOFF` is a date baked into `schemas/routing.ts:77`, the Fable→Opus release valve is an `if (claudeModel === CLAUDE_BEST_MODEL)` splice in the spawner (`src/daemon/spawner-impl.ts:1061-1063`), and `modelVendor` recognizes vendors by regex and a hardcoded alias list (`models.ts:27-39`). Every one of these goes stale the day a lineup turns over, and lineups turned over twice last year. Users should never wait on a Hive release to route to a new model; shipped tables must become the fallback, not the truth.

The thesis: **keep the split SPEC §6 got right — the orchestrator classifies, a table resolves — but stop shipping the table's contents and start deriving them.** The orchestrator's judgment ("this is deep work") is durable. The table's *structure* (tier × tool, override-surviving routes) is durable. The table's *values* — which concrete model is the account's best, whether it takes an effort flag, what to prefer when the first choice is exhausted — are the intersection of two things Hive can obtain at runtime: facts both vendors already publish (the account's models and their effort levels), and curated judgment that must be fetched, not compiled in. Hive already has the machinery pattern for the first half: quota discovery replaced `quota.toml` transcription with provider-read pools, denominated in what the provider actually reports, with shipped behavior as compatibility fallback (SPEC §6). This design applies the same move to capability, and takes the second half — the curated routing manifest SPEC §6 already promises — seriously enough to specify what trusting it requires.

What this doc is not: a vendor-surface catalog (that is `docs/research/router-vendor-surfaces.md`, whose findings this design builds on) and not a cost/tier rubric (that is `docs/research/model-routing-and-token-efficiency.md`, whose R2/R3/R4 this design generalizes).

## Where the router stands today, in one paragraph

The orchestrator classifies a task into one of four tiers; `resolveRoute` merges `~/.hive/routing.toml` over `defaultRoutingTable(now)` per tier and per tool, overrides always winning (`src/config/load.ts:70-119`). The spawner resolves both vendors' concrete models — `best` through a shipped constant, `default` by reading the tool's own user config (`src/adapters/tools/models.ts:79-94`) — splices in Opus 4.8 after Fable as a release valve, and hands the candidate list to the quota resolver, which scores by worst post-reservation headroom, filters by viability, and reserves atomically (`src/daemon/spawner-impl.ts:1036-1079`, SPEC §6). The winning concrete choice is frozen into an `ExecutionIdentity` — `{tool, model}` for Claude, `{tool, model, effort}` for Codex (`src/schemas/agent.ts:28-38`) — which a critical-control restart replays verbatim and fail-closed (`spawner-impl.ts:573-594`). An explicit `hive_spawn` model is a user directive: vendor-bound, launched verbatim, never substituted. All of that machinery is sound and survives this design; what changes is where its inputs come from.

## The capability record: what discovery can tell us

The design's first new object is a per-model **capability record** of discovered facts — read from the provider at runtime, zero-cost, provenance-stamped exactly as quota pools are. It is no longer a proposal: the record store and both discovery feeds landed in `90e550d` (phase 1 — discovery only; routing behaviour, `routing.toml`, and the tier table are untouched). Driving the live binaries to build it **falsified several claims this section asserted in draft**, and the corrections below are the ones phases 3–6 must build on. Everything here was observed 2026-07-11 against claude 2.1.207 and codex-cli 0.144.1.

Claude's stream-json `initialize` control request — the frame Hive already awaits — returns a `models` array whose entries carry `value`, `resolvedModel`, `displayName`, `supportsEffort`, `supportedEffortLevels`, and `supportsAdaptiveThinking`, plus an `account` block with `subscriptionType`. This account enumerates `default` → `claude-opus-4-8[1m]`, `opus[1m]`, `claude-fable-5[1m]`, `sonnet` → `claude-sonnet-5`, and `haiku` → `claude-haiku-4-5-20251001`. On the Codex side the app-server's `model/list` is a full peer, and officially documented: zero-cost, returning the *account-visible* catalog with a per-model `hidden` flag, an `isDefault` flag, a `defaultReasoningEffort`, a `supportedReasoningEfforts` list, input modalities, and service tiers. `config/read` separately reports the effective model and effort an unflagged launch would use. Discovery is a daemon-side act, so record freshness does not depend on which driver later spawns the agent.

**The two surfaces are not symmetric, and the record must not pretend they are.** The draft assumed a shared field set across vendors. The wire says otherwise:

| Fact | Claude control `initialize` | Codex `model/list` |
|---|---|---|
| effort supported | `supportsEffort`, a boolean per model — **absent on Haiku** | **sent for no model** |
| effort levels | `supportedEffortLevels`, a list of **strings** — absent on Haiku | `supportedReasoningEfforts`, a list of **objects** (`{reasoningEffort, description}`) |
| recommended effort | **sent for no model** | `defaultReasoningEffort`, per model |
| vendor-internal | **sent for no model** | `hidden`, a boolean per model |
| entitlement | **sent by neither vendor.** Presence in the account-scoped catalog is the whole of the evidence | |
| CLI version | **carried by neither catalog.** Read separately from `<cli> --version` — free, no session | |

What that asymmetry forces, field by field:

- **`defaultEffort` is Codex-only.** Claude's menu carries no per-model recommended effort at all, so every Claude record's `defaultEffort` is `unknown(surface-silent)`. Code that fills that gap with `medium` is not reading a vendor claim, it is inventing one — and decision 4's effort ladder must be written knowing this rung simply does not exist on Claude.
- **`supportsEffort` is Claude-only.** Codex sends no such boolean for any model, and it must **not** be reconstructed from a non-empty effort list. That inference is exactly the two-fields-into-one merge this section forbids below: it would fabricate a vendor claim that was never made.
- **`hidden` is Codex-only.** Claude has no hidden flag, so a Claude record's `hidden` is `unknown(surface-silent)`, not `false`. The vendor-hidden rule below therefore fires only on Codex today; on Claude it has nothing to read.
- **`entitled` is provided by neither vendor.** It is presence-derived, and therefore only ever `known(true)`: a model the account cannot use is simply *absent* from the catalog, and absence produces no record at all. **There is never a record asserting `entitled: false`**, and nothing downstream may wait for one.
- **Haiku carries no effort fields whatsoever** — `unknown(field-absent)` on both, never `false`.
- **Unknown effort levels survive ingestion.** Sol and Terra advertise `ultra`; Luna, Sol, and Terra advertise `max`. Neither value is in Hive's shipped enums, so raw vendor strings are preserved rather than silently dropped (decision 3).
- **The vendor's default is not the effective default.** Codex's catalog flags `gpt-5.5` as `isDefault`, but `config/read` reports that an unflagged launch *on this machine* actually runs `gpt-5.6-sol` at `xhigh`. `isDefault` is the vendor's catalog recommendation; the effective default is what this account's layered config resolves to. The fallback ladder's "provider's own default" rung (decision 2) means the **effective** one, and the two must never be conflated.

The rule underneath all of that, and the one every consumer of a record must obey: **an absent field is `unknown`, never `false`.** *Which kind* of unknown is itself a fact worth keeping, so the record distinguishes three and never collapses them: `field-absent` (the surface answered for this model and omitted the field — Haiku's effort fields), `surface-silent` (the surface carries the field for *no* model, so its absence says nothing about this one — Claude's `hidden` and `defaultEffort`, Codex's `supportsEffort`), and `malformed` (present but unreadable, which is not a payload that said "none"). Promoting any of the three to `false` manufactures a vendor claim that was never made.

Two further rules govern reading these surfaces, both inherited from the vendor-surfaces doc. **Vendor-hidden entries are not Hive's to route** — an entry flagged hidden (Codex's `codex-auto-review`) is excluded from automatic selection even if a stale manifest still lists it, and stays reachable by explicit pin; when the vendor and the curator disagree about whether a model is internal, the vendor is fresher. Since Claude flags nothing, on that provider the rule is *inert*, not permissive: `unknown(surface-silent)` is an absence of evidence either way, never a licence. **A model name is three facts, not one** — the record keeps the *launch token* (what `--model` receives), the *canonical id* (the join key for quota pools and the manifest), and the *variant* (`[1m]` names a context-window entitlement the CLI appends and the launch flag must never receive). The first draft treated stripping `[1m]` as hygiene; review correctly objected that the variant is an entitlement fact worth keeping, so it is stored, just never launched.

The record, keyed by **(provider, account fingerprint, CLI version, canonicalId + variant)** — account in the key because entitlement and the alias menu are facts about a signed-in account, not about a model, but as a non-PII fingerprint, because the raw frames carry email and organization that a routing store has no business retaining (the vendor-surfaces doc's own caution); CLI version in the key because a catalog read from claude 2.1.207 is not a claim about 2.2's, and an upgrade must not silently overwrite the pre-upgrade truth it will be compared against — and since neither catalog carries the CLI build it came from, the key's version component is read from `<cli> --version`, which prints and exits without opening a session. The shape: `{launchToken, canonicalId, variant, aliases[], entitled, hidden, supportsEffort, supportedEffortLevels, defaultEffort, observedAt}`, where **every discovered fact is a tagged union** — `{state: "known", value, surface, observedAt}` or `{state: "unknown", reason, surface, observedAt}` — rather than a nullable value, so a consumer must branch on `state` before it can read one and *cannot* accidentally treat an undiscovered field as a real one. The type makes the guess impossible rather than merely discouraged. `supportsEffort` and the levels list stay the two raw fields the vendor sent, not merged into one — a merged field collapses "advertised unsupported," "field omitted," and "malformed payload" into one null, and those are three different facts. Provenance is per *field*, not per record, because one record is assembled from more than one read and the two providers do not answer the same questions. Records refresh at daemon start and on the maintenance tick; both surfaces are zero-cost and one is already awaited. A record older than a configured freshness TTL (defaulting to one maintenance-tick interval plus slack) no longer supports *derivation* — a spawn consulting it falls down the fallback ladder below — but still supports *validation*, with staleness named in the warning. Cached in SQLite next to the quota pools it will be consulted beside.

What discovery cannot produce is judgment: nothing in either catalog says which model should carry a coding tier. That is the manifest's job.

## The manifest: ordered candidates, and the trust they require

The first draft carried judgment as a class ontology — every model assigned `frontier|mid|cheap` plus a rank within its class, tiers targeting a class and flooring at another. Cross-vendor review proposed the simpler shape, and the simpler shape wins: **the manifest publishes, per tier × vendor, an ordered list of candidate canonical ids, plus one per-model flag: `codingCapable`.** Derivation intersects the list with live discovery — entitled, non-hidden, fresh — and takes the first survivor as the primary; the *remainder of the list* is the downshift chain; the *end of the list* is the floor. There is no class to assign, no rank to reconcile, no target-class and floor-class pair per tier. Two per-entry attributes make the list expressive enough to retire the remaining shipped policy: an optional **effective window** (`validFrom`/`validUntil` per entry — the Fable cutoff becomes an entry that expires on 2026-07-12, shipped in advance in one manifest, instead of a code constant with a cross-verification test), and an **auto-route eligibility flag**, so a curator can list a model for pin-suggestion and chain membership before trusting it as a primary.

The class ontology is retired as the rejected alternative, and honestly: it lost because its load-bearing content had already leaked out. Once review forced `rankInClass` into the manifest (class alone cannot choose between an account's two frontier models), the ordering was doing all the routing work, and class had shrunk to a grouping label for orderings — two indirections (model→class, tier→class) where one list encodes the answer directly. Class boundaries are also the most contestable judgment in the system (is Sonnet 5 "near-Opus" mid or frontier? the vendor's own positioning argues both), whereas "what should a standard-tier Claude spawn try, in order" is precisely the question the curator is answering anyway. What class genuinely bought — a vocabulary for describing *new* tiers or user-defined profiles without enumerating models — is real but speculative, and speculative flexibility is exactly what this repo's guidelines say not to build. The `codingCapable` flag survives because it is not ontology, it is the floor fact itself, consumed by the `kind` gate below. Manifest authors may keep class labels as commentary; routing never reads them.

A manifest this load-bearing cannot be an unspecified fetch, and review was right to make its trust pipeline a blocker rather than a footnote. The requirements, named: an **owner** (Hive's maintainers publish it; publishing is decoupled from binary releases — that decoupling is the release-independence claim, and if in practice manifests only ship with releases, the claim is dead and this section is where to admit it); **signing** (detached signature over the manifest, verified against a key shipped in the binary — a routing table is an instruction about where code executes, and an unsigned fetched one is an injection vector); **schema versioning** (unknown fields preserved verbatim so old Hives pass through what newer manifests say; unknown *major* version rejected); **expiry** (a `validUntil` past which the manifest still validates but no longer drives derivation — an expired judgment degrades like a stale record, loudly, rather than governing silently forever); **atomic install with rollback** (the previous manifest is kept; a failed verify or parse changes nothing); a **kill switch** (one config flag reverts routing to the shipped table, because the day the manifest is wrong is the day the user has no patience for editing pins per tier); an **audit trail** (every routing decision's `reason` records the manifest version it consulted, so a bad route is attributable to the judgment that produced it); **promotion evidence** (a list change that alters routing ships with the shadow-eval evidence that vetted it — the same criteria the migration's shadow phase defines — so "the curator felt like it" is never a publishable justification); and a **refresh SLO** (a stated cadence commitment, because the open question below is exactly whether anyone will operate this, and an SLO makes the failure measurable instead of ambient). The migration gate is absolute: routing does not flip to manifest-driven before this pipeline exists — a fetched-but-unsigned interim would be strictly worse than the shipped constants it replaces.

## The five decisions

### 1. Task classification: keep the four tiers; add a capability floor, not more tiers

The temptation is to replace `deep/standard/cheap/review` with something richer — the five-dimension rubric (risk, complexity, uncertainty, context load, tool intensity) from the token-efficiency research, or free-form requirements, or letting the orchestrator name models outright. The last is rejected permanently and for the founding reason: an LLM's model knowledge froze at training (SPEC §6). The dimension vector was seriously considered and loses on a subtler point: the research doc's own rubric collapses the five dimensions to a tier by a max rule *before* routing — the dimensions are how the orchestrator should *think*, not what the machine needs to *receive*. Passing them through the spawn API adds five judgment surfaces to misjudge, five fields to validate, and zero routing power the tier doesn't already carry. More tiers (six? eight?) fail the same way: every added tier is a new classification boundary to get wrong, and the 2026-07-10 burn analysis showed the failure was never "four tiers is too few" — it was the values behind the tiers.

What the four tiers genuinely cannot express is the capability floor. `cheap` conflates "small coding task" with "non-coding chore," and only the latter is safe on a model beneath coding capability. So the spawn request grows one small facet, not a taxonomy: **`kind: coding | review | research | mechanical`**, orchestrator-supplied, defaulting by tier (`review` tier → review, everything else → coding — the conservative default, because over-flooring wastes money and under-flooring wastes work, and only one of those gets corrected). The router turns `kind` into an eligibility predicate over the manifest's per-model flag: `coding` **and `review`** require `codingCapable` — the review gap was review-caught and real, since a model that cannot write code cannot judge it either — while `mechanical` and `research` accept any listed model. Review's proposed alternative was a complete tier×kind minimum-capability matrix, and it is worth saying why the design declines it: the composition already *is* that matrix — the tier's ordered list bounds quality (deep's chain cannot leave deep's list, so "deep downshifts frontier→mid" is impossible by construction, the concern the matrix existed to catch), and the kind flag bounds capability — so sixteen explicit cells would restate `list ∩ flag` with sixteen new places for the two judgments to disagree. If a kind ever needs a floor the flag can't express, that is a new flag, not a matrix. That is the entire capability-awareness mechanism — a filter, two enum fields, no scoring. Tier stays the *cost* judgment; kind becomes the *floor* judgment; the two are orthogonal and both are things an LLM classifies well without knowing any model's name.

Cost: one more field the orchestrator can get wrong, and a `mechanical` misclassification routes real coding to a bottom-of-list model exactly as today. Telemetry (escalation rate by kind) is the check, and the default-to-coding rule bounds the blast radius to tasks explicitly marked down.

### 2. Where model knowledge lives: discovery ∩ manifest, with a fallback ladder that never quietly guesses

The alternatives considered for where knowledge lives: keep compiling it into constants (status quo — fails the release-independence requirement by definition); trust discovery alone (fails because discovery has no judgment: it can say a model exists and lacks effort support, not that it shouldn't write code); trust the manifest alone (fails because the manifest can't know an *account* — entitlement, hidden flags, and alias resolution are per-account facts only discovery has). The composition is the answer, and its shape is now literally an intersection: derivation picks from `manifest list ∩ discovered(entitled, non-hidden, fresh)`. Each of those three predicates has to be read the way the records actually work, or it will be implemented as a boolean test against a field that does not exist. **`entitled` means a record exists at all** — no vendor sends an entitlement flag, so presence in the account-scoped catalog *is* the predicate, and an unusable model fails it by being absent rather than by being marked. **`non-hidden` means `hidden` is not `known: true`** — Claude sends no hidden flag, so every Claude record's `hidden` is `unknown`, and treating unknown as hidden would exclude the entire vendor. **`fresh` means within the TTL**, and is the only one of the three that is a plain comparison. A model absent from every manifest list is *structurally* unroutable by Hive's own choice — the unknown-model rule ("never auto-route what nobody vetted") stops being a rule to enforce and becomes a property of the data shape. Guessing a new model upward is how a coding task lands on something that can't do it; guessing downward silently wastes frontier quota; the manifest-gap cost is real and named — when a vendor ships a new lineup, auto-routing cannot use it until a manifest refresh arrives, and users who want it sooner pin it in `routing.toml`, the escape hatch working exactly as designed.

When the intersection is empty — no manifest, no discovery, or nothing survives — the first draft fell straight back to today's shipped `DEFAULT_ROUTING`, and review caught the contradiction: the shipped table names models no record vouches for, which is exactly the "unknown, auto-routed" case the intersection exists to prevent. The fallback is therefore a **ladder, most-recently-verified first**: (1) the last-known-good derived snapshot — the most recent non-empty intersection, cached with its timestamps, used past its TTL only here and labeled stale; (2) the **provider's own *effective* default** — launch with no model flag at all, which runs whatever the account is entitled to by construction (this is today's Codex `"default"` posture, extended to Claude, whose no-flag launch uses the account's `default` menu entry); *effective* is load-bearing, because the catalog's `isDefault` is a different fact: Codex flags `gpt-5.5` as `isDefault` while `config/read` reports this machine's unflagged launch as `gpt-5.6-sol` at `xhigh`, so this rung reads `config/read`, never the catalog flag. Like today's `"default"`, it yields an incomplete execution identity and inherits that path's existing restrictions; (3) the shipped concrete table, as a **loud compatibility exception** — one deduplicated warning naming which rung failed and why, because a router silently running on compiled-in guesses is the status quo this design exists to end, acceptable only as the floor under a fresh install that has never once discovered anything.

Concrete retirements this funds, each today a release-coupling: `CLAUDE_BEST_MODEL` — retired by obsolescence, not by discovery, and the distinction matters: the menu *omits* `best` entirely (its `default` entry resolves to Opus 4.8), so no vendor surface can confirm what `best` launches and the constant is a cache of a fact only a billable probe establishes. Derived routes select from manifest lists of canonical ids and never emit an unlisted alias, so nothing load-bearing needs the constant; it survives only as a deprecated shim for user-pinned `best`, which `docs/model-selection.md` already warns against pinning. Likewise `modelVendor`'s alias list (names join against discovered catalogs; the regexes remain only as a last-resort parse for names discovery has never seen); the direct file-reads in `resolveConcreteModel` for Codex `"default"` (`~/.codex/config.toml` misses layered/profile config — the app-server's `config/read` answers with the *effective* unflagged default); and `FABLE_AUTO_ROUTING_CUTOFF` with `POST_FABLE_CUTOFF_DEEP_MODEL` (a billing-date route change is manifest data if anything is — the next such cutoff should be an edit to the fetched candidate lists, not a code change with a cross-verification test).

### 3. Effort: a first-class, discovery-validated knob, plumbed end to end or not at all

Effort becomes a peer of `model` at every layer it exists today for Codex and is absent for Claude: `ClaudeRouteSchema` grows an optional `effort`; `SpawnRequest` grows an optional `effort` (a directive, like `model`: passed only when the caller means it; routine spawns take the resolution ladder in decision 4); the Claude adapter passes `--effort <level>` (verified present, claude 2.1.207); and the Claude arm of `ExecutionIdentity` grows an effort field, because a knob that spawn honors and control-restart drops is precisely the silent divergence the immutable identity exists to prevent (the token-efficiency doc's R3 called this out, and it is right).

Three design points the plumbing has to get right:

**Effort values are vendor data, not a Hive enum.** Hive's schemas currently hardcode the levels as closed enums — `minimal|low|medium|high|xhigh` in `CodexRouteSchema` and the Codex identity arm (`src/schemas/routing.ts:22-28`, `src/schemas/agent.ts:36`) — and the live catalogs show those enums are *already* wrong in both directions: no listed Codex model advertises Hive's `minimal`, while Sol and Terra advertise `ultra` and Luna, Sol, and Terra advertise `max` — three models carrying values the schema would reject. Unknown levels are preserved verbatim through ingestion and persistence rather than dropped, because a level Hive discards is a level a critical restart cannot replay. A closed enum here is `CLAUDE_BEST_MODEL` in another costume — a shipped constant caching a fact the vendor publishes — and it fails worse, because a schema rejection doesn't misroute a spawn, it refuses a valid one. So effort becomes an open string at every layer (route, spawn request, identity), sanity-bounded at ingestion (length-capped, `[a-z0-9-]` charset — it becomes argv) and validated at spawn time against the resolved model's discovered `effortLevels`, with the known-value list serving only as fallback validation when no record exists. A normalized Hive-only scale mapped per vendor was also considered and rejected: a lossy mapping means the user asks for one thing and gets another — the exact lie the verbatim-model rule exists to prevent.

**Discovery gates the flag, and provenance decides what failure means.** Haiku is the proof case: its menu entry omits the effort fields entirely, so a per-tier effort default naively applied would pass `--effort low` to a model that may not take it.

The draft stated the gate as "a resolved candidate whose record does not *affirmatively* advertise effort support gets no effort flag," and the live surfaces break that phrasing: Codex sends `supportsEffort` for **no model**, so a gate keyed on that boolean would refuse an effort flag to every Codex spawn — silently disabling the one vendor where effort already works. The gate keys on the **levels list**, which both vendors do send, and on the boolean only where it exists. Four states, which are four different facts:

- `supportedEffortLevels` is `known` and contains the value → pass the flag.
- `supportedEffortLevels` is `known` and does *not* contain it → the vendor positively enumerated its levels and this is not one. This is a real "no."
- `supportedEffortLevels` is `unknown` (Haiku's `field-absent`; any `surface-silent` case) → no evidence either way. **Hive-authored** effort passes no flag, because Hive does not guess; **user-authored** effort passes through verbatim with a warning, the CLI's own launch failure being the honest validator — the same treatment an undiscovered model gets, for the same reason. Failing the spawn here would be reading `unknown` as `false`, which is the error this design exists to prevent.
- `supportsEffort` is `known: false` → a positive vendor refusal, and the only state that overrides a levels list. Claude is the only vendor that can currently produce it; Codex never will, and the boolean is never reconstructed from the list to make it look like it could.

When validation fails against a positively-enumerated list, what happens keys on the effort's *provenance*, which the resolution ladder carries with the value — the first draft keyed it on which config object the value arrived in, and review caught the hole: a `routing.toml` effort pin is user-authored, but it arrives via "the route table," so it would have been clamped as if it were Hive's own judgment. Restated on provenance: **user-authored effort** (a spawn directive *or* a `routing.toml` pin) that a discovered model's record *positively excludes* — its levels are `known` and this is not one of them — *fails the spawn loudly* with the record as evidence, because the record is positive evidence and the user should know their ask can't run. User-authored effort that the record merely *doesn't know about* — an undiscovered model, or a discovered one whose levels are `unknown` — passes through verbatim with a warning: absence is unknown, not refusal, and the CLI's own launch failure is the honest validator. **Hive-authored effort** (a manifest tier default, the model's discovered default, the shipped constant) clamps with the decision `reason` noting it — clamping Hive's own defaults is housekeeping; clamping a human's directive is disobedience, whichever file the human wrote it in.

The same gate fixes a quieter shipped constant, but only on one vendor: several spawner and recovery paths default a missing Codex effort to `medium` (`spawner-impl.ts:1086,1238,1288`), while Codex's catalog publishes a per-model `defaultReasoningEffort` (Spark's is `high`, not `medium`) — so an unset Codex effort falls back to the model's own discovered default, and to the shipped constant only beneath that. **Claude has no such rung.** Its menu carries no per-model recommended effort for any model, so a Claude record's `defaultEffort` is always `unknown(surface-silent)`; there is nothing to fall back *to*, and a `medium` written into that gap would be a Hive guess wearing a vendor's badge. An unset Claude effort therefore resolves through the manifest's per-tier default if one exists and otherwise passes no flag at all — the CLI's own default governs, which is exactly today's behaviour and is honest about being it.

**The identity records what was effective, not what was flagged.** The first draft let a Claude identity omit effort, reading absence as "the CLI default, which is what launched — faithful." Review broke that: CLI defaults *drift* — the same absent flag replayed after a CLI upgrade launches a different effective effort, which is precisely the launch-drift the immutable identity exists to prevent. Claude's live status-line payload resolves the gap without inventing a constant: its `effort.level` is the effective session value whether or not Hive passed `--effort`, and Hive already receives that payload on every render. The first observed value completes and freezes the launch identity; a restart then pins it explicitly. Until that first observation, the identity is honestly incomplete on the effort axis and a faithful restart is unavailable. A model that positively does not support effort records none, and that identity is complete.

The status-line value is current state, not a launch stamp: `/effort` changes it during a session. A later observation that differs from the frozen identity is therefore an **effort drift event**. Hive persists and surfaces the disagreement but never rewrites `ExecutionIdentity` to match it; an immutable field that quietly followed mutable state would be a contradiction disguised as telemetry. This preserves provider defaults for unflagged launches, records the effective launch once it is observable, and makes an in-session departure explicit. It does not discover Claude's provider default — a configured value and a provider-selected value can be identical — and nothing in routing assumes one. Legacy rows without the field remain non-reproducible on this axis and restart without an effort flag as compatibility rather than fidelity; retroactively inventing their launch effort would still be an estimate wearing a measurement's badge.

### 4. Deriving defaults: resolve tier → route through the manifest at spawn time, with routing.toml untouched on top

The resolution order is three layers, strictly ranked: **user pin** (`routing.toml`, format unchanged — per-tier, per-tool, always wins) → **derived route** (the manifest list ∩ discovery intersection of decision 2) → **the fallback ladder** (decision 2, rung by rung). The tier stays the key; what a tier *means* moves from "these strings we shipped" to "this candidate list resolved against what your account can actually reach, right now."

Three points the first draft left implicit, each caught by review, each needing to be explicit because getting them wrong is silent:

**Model resolves first; effort is a function of the resolved model, never a parallel column.** Because `routing.toml` merges per-field within a tier×tool cell (`src/config/load.ts:96-102`), a naive per-field layer stack would let a user's pinned model pick up a derived effort computed for a *different* model. The fix is ordering, not merging: resolve the model through the three layers, then resolve effort *against that model* — spawn-level directive first, then the cell's pinned effort, then the manifest's per-tier effort default (the knob that makes a cheap tier actually reason cheaply), then the resolved model's own discovered `defaultReasoningEffort`, then the shipped constant — with each step validated against the resolved model's record and its provenance carried (decision 3). No cross-model value can survive that ordering. **The discovered-default rung exists only on Codex**: Claude publishes no per-model recommended effort, so on that vendor the ladder falls from the manifest's tier default straight to passing no flag at all, and a shipped `medium` inserted there would be a Hive guess presented as a vendor default (decision 3).

**A pin is a standing user directive, and it interacts with the capability floor the way explicit choices always do in Hive: never silently overridden, never silently obeyed.** A `routing.toml` model pin counts as "explicit" for the unroutable-unknown rule (that rule would be vacuous otherwise — the table is where users pin things), and a pinned model that fails the task's `kind` floor spawns anyway, with the conflict named loudly in the decision reason and the spawn report — the user's standing judgment about their own account outranks the manifest's, exactly as a quarantined route is still used when the user pinned it (SPEC §6). Failing every coding spawn on a tier because of a table entry would convert a judgment disagreement into a work stoppage; hiding the disagreement would convert it into a silent misroute. One tightening for pins of models Hive cannot classify: a spawn-level `model` whose vendor neither discovery nor the name heuristics can establish now *requires* an explicit `tool` and otherwise refuses with guidance — today it silently stays on the tier's routed tool (`spawner-impl.ts:1014`), which is a coin-flip that happens to be labeled a default. Table pins never have this problem: the cell (`[deep.claude]`) names the tool by construction. And because "pin always wins" and "quota may substitute" read as a conflict until the sources are separated, the substitution rules are stated per source: a **spawn-level `model`** is the sole candidate — no chain, no cross-vendor alternative, never substituted, exactly as today. A **`routing.toml` pin** sets its cell's *primary*; the tier's manifest chain still appends after it, and the cross-vendor candidate still competes — which is not a new liberty but today's exact behavior, since the shipped valve already splices Opus after *pinned* Fable ("explicitly or otherwise," SPEC §6) and tier routing has always let quota pick the other vendor's column. A pin narrows what its cell says, not what the tier may do; only the spawn-level directive narrows the tier to one candidate. When a chain entry beats a pinned primary on headroom, the decision `reason` says so — pressure-driven, never silent. Throughout, the resolved route carries provenance **per field** — tool, model, and effort each tagged with the layer that supplied them — because `routing.toml`'s partial-field merge means one cell's values routinely have three different authors, and every downstream rule in this design (loud-compliance, fail-vs-clamp, substitution) keys on who authored the specific field, not the cell.

**A stale record that lies gets one bounded retry, and the transaction is specified.** The first draft claimed a stale-record launch failure would "naturally" fall back via the readiness monitor; review correctly flagged that as not current behavior — `monitorReadiness` quarantines the route (SPEC §6's viability machinery) and reports; nothing respawns. The design makes the retry explicit and minimal: when a *Hive-derived* candidate fails to launch, the failed attempt settles its reservation and records against the route exactly as today, and the spawn request then re-enters resolution **once** — the quarantine it just created excludes the failed candidate, so the intersection yields the next list survivor or the ladder's next rung — under a fresh reservation. One automatic retry, never more, and never for user-pinned models, whose failures report with the capacity-and-fallback message rather than being second-guessed (substituting a user's named model is a lie regardless of why it failed).

The auditability worry about per-spawn derivation is real, and gets a direct answer rather than a materialized table: a `hive routing` surface that prints the currently-derived table with per-cell provenance — pinned / derived (with manifest version and record timestamps) / which fallback rung — because a router whose choices can't be inspected will be distrusted the first time it surprises someone, and *should* be.

### 5. Capability is eligibility; quota is ranking; identity is immutable — three stages that never trade places

The quota resolver is already generic over an arbitrary candidate list, scored by headroom, filtered by viability, gated per-pool (SPEC §6). This design does not touch its internals; it changes who builds the candidate list, and what a candidate *is*. The hardcoded Fable→Opus splice generalizes into the manifest's own ordering: after the primary candidate, the remainder of the tier's list — intersected with discovery like the primary — *is* the downshift chain, and the end of the list is the floor no headroom number may cross. This reproduces today's valve (deep's Claude list is `[claude-fable-5, claude-opus-4-8]` before the cutoff manifest edit, `[claude-opus-4-8]` after) as data instead of code. Order semantics are preserved exactly: primary first, ties keep the primary, so the no-quota-data compatibility case still behaves as today. And a candidate is now a **triple** — `{tool, model, effort}`, the effort resolved and validated per candidate before scoring — because the first draft's pairs left effort to be read off the primary route *after* the resolver picked, so a chain candidate would have launched with an effort chosen for a different model, and the reservation ledger would never learn that an `xhigh` run and a `low` run are different quantities. Making estimates actually condition on effort is deferred to telemetry (inventing effort multipliers is the accurate-or-unknown rule's territory), but the identity of what was reserved must be honest now, or there will be nothing truthful to calibrate against later.

The load-bearing rule is the stage ordering. **The capability floor filters candidates before quota ever sees them**: a Hive-chosen model beneath the task's `kind` floor never enters the list (user pins are the one exception, entering with the loud-compliance rule of decision 4), so no amount of quota pressure can downshift a coding task below the floor — exhaustion of every eligible candidate fails the spawn with the capacity report, exactly as an exhausted explicit pin does today. The rejected alternative was making capability a scoring penalty rather than a filter — tempting because it "uses idle cheap capacity" — and it is rejected because a penalty is a price, and under enough quota pressure any price gets paid; a floor that can be outbid is not a floor. The same argument already lost once inside quota design ("headroom alone ranked a route that could not run" — SPEC §6's viability lesson); capability is the same shape of fact.

One drift mode connects this decision to the quota subsystem and was invisible until cross-vendor review lined the two up. Quota's model-scoped pools bind by joining the pool's provider-given display name against the provider's catalog, and that join "fails safely to unbound" on a rename (SPEC §6) — but capability records join the same catalog on their own schedule, so a renamed or newly-shipped model can become *auto-selectable* (fresh discovery, manifest already lists it) at the exact moment its own meter is *unbound* (stale name on the quota surface). Safe-fail on one side plus eager-adopt on the other composes into the unsafe whole: the router's newest choice is precisely the one whose per-model cap has stopped gating, and the general pool's headroom waves it through — the 2026-07-11 Fable-at-99% incident with a fresh coat of paint. Two rules close it. First, **one join, one code path**: capability records and pool bindings resolve names through the same catalog snapshot, so they cannot disagree about what a name means — they can only both know or both not. Second, **an unbound model-scoped pool quarantines auto-adoption on its provider**: while any model-scoped pool on a provider binds to nothing in the catalog, models on that provider that lack a bound meter of their own are not auto-routable — explicit pin only, warning loudly — because the orphaned pool may be exactly their meter under its old or new name. The quarantine lifts the moment the join heals (next catalog or quota refresh), and it is deliberately provider-scoped rather than global: a Codex rename should not stop Claude routing.

Execution identity and control restarts constrain all of this from below, and the constraint is welcome: **manifest and records are consulted at launch time only, and never at restart**. The identity — `{tool, model, effort}` with effort effective-at-launch (decision 3) — remains the frozen, complete instruction for reproducing the launch; a critical-control restart replays it verbatim even if the manifest has since delisted or the catalog renamed the model, because a faithful restart of the interrupted agent is the point and the recorded-model-no-longer-launchable case already fails closed with a durable alert (SPEC §6). Nothing dynamic may be needed to restart an agent: this is why effort lives *in* the identity rather than being re-derived, why derived routes must resolve to concrete launch tokens *before* the identity is written (they do — the resolver's output is concrete by construction), and why the `live_model` observation stays a separate field rather than mutating `model`. The dynamic router makes launch-time choice smarter; it must leave the restart path exactly as dumb as it is.

## Migration from today's tiers

The four tier names, the `routing.toml` format, and every existing pin remain valid throughout — there is no user-visible migration, only the defaults beneath the overrides getting smarter. Staged, each step inert until the next:

1. **Record, change nothing.** ✅ *Landed 2026-07-11 in `90e550d`* — the capability-record store and both discovery feeds (Claude's `initialize`, already awaited; Codex's `model/list`, riding the app-server handshake), provenance-stamped per field, observation only, routing still reading the shipped table. Building it against the live binaries is what falsified the capability-record section's draft claims, above. The `hive routing` inspection surface this step also called for ✅ *landed 2026-07-11* alongside the derivation engine below.

1b. **Derive, route on nothing.** ✅ *Landed 2026-07-11* — the resolution order (pin → derived → ladder), the ladder's three rungs, the per-field provenance, and `hive routing`, which prints the derived table with the layer that authored each cell. `resolveRoute()` is untouched: the engine has exactly one caller, the inspection surface, and no spawn consults it. Two facts the build established that the design could only assume. The ladder's second rung reads Codex `config/read` (`model`, `model_reasoning_effort` — snake_case, off the live wire) and confirms the split it depends on: this machine's unflagged launch is `gpt-5.6-sol@xhigh` while the catalog still flags `gpt-5.5` as `isDefault`. And Claude's rung-2 model is discoverable (the menu's `default` entry names `claude-opus-4-8`) while its *effort* is not — every derived Claude cell prints an unknown effort, which is the open question below staying open rather than being closed with a constant. The last-known-good rung keeps its provenance **per cell**: a run where one provider is unreachable carries that provider's remembered cells forward at their true age instead of overwriting them, because a memory a single failed probe can erase is not a memory, and the ladder would silently drop to the compiled-in table without anyone deciding that.
2. **Plumb effort inert.** Schema fields (`ClaudeRouteSchema.effort`, `SpawnRequest.effort`, identity effort as open strings), adapter flag, discovery-gated validation — all defaulting to unset, so shipped behavior is bit-identical until a user sets a value.
3. **Stand up the manifest pipeline.** Signing, expiry, atomic install, rollback, kill switch, audit (the trust section above) — plus the first manifest, whose candidate lists byte-for-byte reproduce today's shipped table and whose per-tier effort defaults are deliberately absent — automatic effort stays at provider defaults until effort-sensitive telemetry exists to justify anything else — so that adopting the mechanism changes no route and no effort. This step is a hard prerequisite of step 5, not an optional hardening after it.

    **Trust half ✅ *landed 2026-07-11*.** No second signing scheme was built, and none was needed: a routing manifest and a release manifest are the same kind of claim (*this document is Hive's*), so the routing manifest is verified by `release/manifest.ts`'s existing Ed25519 `verifyManifest` against the same build-time-inlined key list, and signed by the same offline key through the existing `scripts/signing/sign-manifest.ts`, which already signs the exact bytes of any file. That reuse quietly settles the release-independence question in the right direction: because the signer is a standalone script and not a build step, **publishing a manifest does not require cutting a release** — the decoupling this design claims is a property of the pipeline, not merely a promise about it. Verification is fail-closed in the strong sense — a tampered manifest, one signed by the wrong key, one whose `.sig` was *stripped*, one with an unknown schema *major*, and one arriving at a build with no embedded key are all **rejected**, and the compiled-in manifest stands. A rejection installs nothing and routes nothing; unknown *fields* still survive verbatim, so a newer minor governs an older Hive. The kill switch (`config.toml: routingManifest = "off"`) disowns even a validly signed manifest, and disowns the last-known-good snapshot with it — the snapshot is manifest-derived judgment, and replaying it would route on exactly what the switch was thrown to reject. Per-cell provenance already carries the manifest revision, which is the audit trail this section asks for. **Still outstanding in this step:** the fetch itself, with its atomic install and rollback (there is no publishing endpoint to fetch from yet, so the transaction has nothing to be atomic about), promotion evidence, and the refresh SLO — the last two being operational commitments no code can conjure.
4. **Shadow-derive.** ✅ *Landed 2026-07-12.* Compute the derived route per spawn, log derived-vs-shipped divergence, route on shipped. The flip criteria are concrete, not vibes: a minimum spawn volume before the window counts at all (a quiet fortnight proves nothing — the window extends until the volume exists); over that volume, zero `kind`-floor violations by the derived choice; zero derived candidates that discovery later contradicted (launch or entitlement failures had they run); a ladder-fallback rate below threshold (derivation that routinely can't produce an answer is not ready to govern, however good its answers are when it has them); every divergent cell attributable to a discovered or manifest fact rather than a resolution bug; and projected planning-unit cost for the same task mix no worse than shipped. Any criterion failing restarts the window after the fix.

    As built, the observation is inert by construction — it runs only after a launch is past the transport, takes the launch as an argument rather than producing one, and swallows its own failures, because an observer that can kill the thing it observes is not an observer. `hive routing shadow` grades every criterion, and **an `unknown` is never a pass**: the gate cannot be cleared by not looking. Two criteria are honestly ungradable today and say so instead of self-certifying — whether a divergence is sound judgment or a resolution bug is not a thing a machine decides, and *nothing prices a planning unit per model*, so the cost criterion has no projection to make. The rollback trigger comes out half-measured, which is the finding rather than an omission: the pre-flip **launch-failure baseline is real** (failures past the transport, over the shipped routes actually taken), while the **escalation rate is measured nowhere in Hive** — "escalation" is prompt wording, nothing counts one, and until something does, that half of the trigger cannot fire and no honest number can be printed for it.

    **It earned its keep within minutes.** The first real shadowed spawns ran just after `FABLE_AUTO_ROUTING_CUTOFF` (2026-07-12T00:00:00Z) passed, and the deep tier diverged immediately: the shipped table had moved to Opus, while the manifest went on deriving **Fable** — because the cutoff was a *code constant the manifest could not see*. Had routing already flipped, every deep-tier Claude spawn would have kept landing on the model that had just moved to usage-only billing off the user's plan. The fix is the one this section already prescribed and step 3's manifest had simply never used: an **effective window** on the candidate entry (`validUntil = FABLE_AUTO_ROUTING_CUTOFF`), so the cutoff is data rather than a branch, the manifest and the table read the same constant and cannot disagree about the date, and after it Fable is not a downshift target but not a candidate at all. This is the migration's first evidence that the mechanism does what it claimed: a lineup change became a manifest edit instead of a code change — and it is also exactly the class of silent misroute the flip gate exists to catch *before* it governs.
5. **Flip the middle layer.** Derived routes govern between pins and the fallback ladder; the manifest's list remainders replace the hardcoded valve; the `kind` floor gates candidates. Reversible per-tier by a `routing.toml` pin, globally by the manifest kill switch; the post-flip rollback trigger is equally concrete — launch-failure or escalation rate rising above the pre-flip baseline reverts first and investigates second.
6. **Retire the constants.** `CLAUDE_BEST_MODEL`, the cutoff pair, the alias list in `modelVendor` — each deleted only once its manifest/discovery replacement has carried a real lineup change (the 2026-07-12 Fable cutoff is, conveniently, an immediate live test: encode it as a manifest list edit in step 3 and watch it govern).

## Open questions

- **How stable are the discovery surfaces as contracts?** The two vendors sit at different epistemic tiers here: Codex's `model/list` is officially documented (OpenAI's app-server docs direct clients to call it before rendering model selectors — see `router-vendor-surfaces.md`), while Claude's `initialize` model menu is undocumented, the same tier as `get_usage`. Capability records carry per-field provenance and CLI version for exactly this reason, but the design has no answer yet for a surface that *silently* drops a field rather than disappearing — and note that Hive cannot currently tell that case apart from `surface-silent` at all: both look like "nobody sent it."
- **Who operates the manifest pipeline, and how fast is fast enough?** The trust section specifies the mechanism; it cannot conjure the operational commitment. "Hours-to-days, not a release" assumes someone publishes on that cadence; if manifest refresh in practice rides releases, the release-independence claim quietly dies. The structural unroutability of unlisted models makes the failure visible (new models are pinnable but not auto-routed) rather than dangerous, but visible-and-slow is still a broken promise.
- **Is `kind` defaulting to `coding` too conservative?** It routes ambiguous cheap chores onto coding-capable models, spending real money to avoid misrouting. Escalation-rate and cost telemetry by kind decides; the reverse default is indefensible without it.
- **Effort defaults per tier are judgment with no data.** The manifest schema has a slot for one, but the shipped manifest leaves every slot empty. Nothing yet measures whether a given effort step pays for itself, and the token-efficiency doc's A/B machinery should run before the manifest gets opinionated.
