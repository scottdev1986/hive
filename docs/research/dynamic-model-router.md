# Dynamic model routing: capability-aware, effort-first, release-independent

## What this is and why

This is a design draft, research only — nothing here is implemented. It answers one question: what has to change for Hive's router to assign the best model *and* effort level per task, without a Hive release standing between the user and a model that shipped yesterday.

Today's router is a shipped table with a user override file. That shape has three structural gaps, and each is a required property of the overhaul:

1. **Capability-awareness.** Nothing in the resolver knows what a model *is*. The tier table encodes capability implicitly (`cheap` names `haiku`), but the moment routing becomes dynamic — quota valves, downshift chains, derived defaults — there is no floor stopping a coding task from landing on a Haiku-class model. The gate that refuses that spawn does not exist.
2. **Effort as a first-class knob.** `hive_spawn` takes a `model` but no `effort` (`src/daemon/spawner.ts:8-19`). The Codex route column carries effort; the Claude column does not (`src/schemas/routing.ts:14-29`), so every Claude agent Hive spawns — including cheap-tier ones — reasons at the CLI's `high` default. Effort is half a knob, on one vendor, reachable only through the table.
3. **Release-independence.** The router's knowledge of the model landscape is compiled in: `CLAUDE_BEST_MODEL` and `CLAUDE_OPUS_MODEL` are constants verified by hand against an installed CLI (`src/adapters/tools/models.ts:9,19`), `FABLE_AUTO_ROUTING_CUTOFF` is a date baked into `schemas/routing.ts:77`, the Fable→Opus release valve is an `if (claudeModel === CLAUDE_BEST_MODEL)` splice in the spawner (`src/daemon/spawner-impl.ts:1061-1063`), and `modelVendor` recognizes vendors by regex and a hardcoded alias list (`models.ts:27-39`). Every one of these goes stale the day a lineup turns over, and lineups turned over twice last year. Users should never wait on a Hive release to route to a new model; shipped tables must become the fallback, not the truth.

The thesis: **keep the split SPEC §6 got right — the orchestrator classifies, a table resolves — but stop shipping the table's contents and start deriving them.** The orchestrator's judgment ("this is deep work") is durable. The table's *structure* (tier × tool, override-surviving routes) is durable. The table's *values* — which concrete model is the account's best, whether it takes an effort flag, what the cheap class is called this quarter — are exactly the facts both vendors already publish at runtime, and Hive already has the machinery pattern for consuming such facts: quota discovery replaced `quota.toml` transcription with provider-read pools, denominated in what the provider actually reports, with shipped behavior as compatibility fallback (SPEC §6). This design applies the same move to capability.

What this doc is not: a vendor-surface catalog (caleb owns discovery mechanics, especially Codex's) and not a cost/tier rubric (that is `docs/research/model-routing-and-token-efficiency.md`, whose R2/R3/R4 this design generalizes).

## Where the router stands today, in one paragraph

The orchestrator classifies a task into one of four tiers; `resolveRoute` merges `~/.hive/routing.toml` over `defaultRoutingTable(now)` per tier and per tool, overrides always winning (`src/config/load.ts:70-119`). The spawner resolves both vendors' concrete models — `best` through a shipped constant, `default` by reading the tool's own user config (`src/adapters/tools/models.ts:79-94`) — splices in Opus 4.8 after Fable as a release valve, and hands the candidate list to the quota resolver, which scores by worst post-reservation headroom, filters by viability, and reserves atomically (`src/daemon/spawner-impl.ts:1036-1079`, SPEC §6). The winning concrete choice is frozen into an `ExecutionIdentity` — `{tool, model}` for Claude, `{tool, model, effort}` for Codex (`src/schemas/agent.ts:28-38`) — which a critical-control restart replays verbatim and fail-closed (`spawner-impl.ts:573-594`). An explicit `hive_spawn` model is a user directive: vendor-bound, launched verbatim, never substituted. All of that machinery is sound and survives this design; what changes is where its inputs come from.

## The capability record: what discovery can and cannot tell us

The design's central new object is a per-model **capability record**. It has two halves with different epistemic status, and keeping them separate is the whole trick.

**Discovered facts** — read from the provider at runtime, zero-cost, with provenance and freshness stamped exactly as quota pools are. Verified live on this machine, 2026-07-11, claude 2.1.207: the stream-json `initialize` control request Hive already awaits returns a `models` array in which every entry carries `value`, `resolvedModel`, `displayName`, `supportsEffort`, `supportedEffortLevels`, and `supportsAdaptiveThinking`, plus an `account` block with `subscriptionType`. Concretely, this account enumerates `default` → `claude-opus-4-8[1m]`, `opus[1m]`, `claude-fable-5[1m]`, `sonnet` → `claude-sonnet-5`, `haiku` → `claude-haiku-4-5-20251001` — and Haiku's entry *omits* the effort fields entirely. Absent, not `false`: as `docs/research/router-vendor-surfaces.md` puts it, omission may mean unsupported, rollout-gated, or simply missing from this protocol version, and the record must preserve that three-way distinction rather than promote it to a boolean. So discovery hands us, per account and per model: existence, entitlement, the alias→concrete mapping for listed aliases, and effort support where affirmatively advertised. Two sharp edges carry over from `docs/model-selection.md`: the menu is not exhaustive (bare `claude-opus-4-8` and `best` launch but are unlisted — the menu is "what exists," not "what parses"), and `resolvedModel` may carry a `[1m]` context-variant suffix the CLI appends and `--model` must never receive. On the Codex side the app-server's `model/list` is the equivalent surface, and caleb's live verification (codex 0.144.1, `docs/research/router-vendor-surfaces.md`) confirms it is a full peer: zero-cost, returning *account-visible* models — entitlement included — with per-model default and supported reasoning efforts, hidden/default flags, input modalities, and service tiers. Both vendors therefore publish models *and* effort levels at runtime. Discovery is a daemon-side act (the app-server handshake answers `model/list` regardless of which driver later spawns the agent), so record freshness does not depend on the spawn path. One finding of caleb's is load-bearing enough to shape a decision below: the live catalogs already contain effort values (`max`, `ultra`) that Hive's current closed enums reject — effort-value drift is real today, not hypothetical.

**Judgment facts** — the things no vendor publishes machine-readably: that Haiku is a cheap-class model unfit for coding tasks, that Sonnet carries a standard coding tier, that Terra is Sol's mid-tier sibling. Discovery cannot produce these; someone's judgment must, and the design must be honest about whose. The options:

- *Name heuristics shipped in code* (`haiku` → cheap, `sonnet` → standard…) — this is `modelVendor()` grown teeth: release-coupled for every new family name, and a wrong guess routes work onto a model that can't do it.
- *Price as proxy* — the cleanest ordinal signal, but neither CLI exposes price at runtime; it would come from a shipped table, which is the release coupling again, now with money attached.
- *The curated routing manifest* — SPEC §6 already commits to it: "provider model-list APIs give you names, not judgment — curation is the value," fetched with Hive releases but *decoupled from them*, local overrides always winning. Class judgments are exactly the judgment the manifest exists to carry.

The manifest wins, with one hard rule for the gap it leaves: **a model with no judgment record is routable only by explicit user pin, never by auto-routing.** Guessing a class upward is how a coding task lands on a cheap model; guessing downward silently wastes frontier quota. Unknown means unknown — the same accurate-or-unknown rule the quota subsystem bled for (SPEC §6, memory `accurate-numbers-only-rule`). The cost is real and named: when a vendor ships a whole new lineup, auto-routing cannot use it until a manifest refresh arrives (hours-to-days, not a release), and users who want it sooner pin it in `routing.toml` — which is a one-line file edit, the escape hatch working exactly as designed.

The record, then: `{provider, concreteModel, aliases[], entitled, effortLevels|null, class: frontier|mid|cheap|null, codingCapable: bool|null, provenance, observedAt}` — discovered half from the providers at daemon start and on the maintenance tick (both surfaces are zero-cost and one is already awaited), judgment half joined from the manifest, cached in SQLite next to the quota pools it will be consulted beside.

## The five decisions

### 1. Task classification: keep the four tiers; add a capability floor, not more tiers

The temptation is to replace `deep/standard/cheap/review` with something richer — the five-dimension rubric (risk, complexity, uncertainty, context load, tool intensity) from the token-efficiency research, or free-form requirements, or letting the orchestrator name models outright. The last is rejected permanently and for the founding reason: an LLM's model knowledge froze at training (SPEC §6). The dimension vector was seriously considered and loses on a subtler point: the research doc's own rubric collapses the five dimensions to a tier by a max rule *before* routing — the dimensions are how the orchestrator should *think*, not what the machine needs to *receive*. Passing them through the spawn API adds five judgment surfaces to misjudge, five fields to validate, and zero routing power the tier doesn't already carry. More tiers (six? eight?) fail the same way: every added tier is a new classification boundary to get wrong, and the 2026-07-10 burn analysis showed the failure was never "four tiers is too few" — it was the values behind the tiers.

What the four tiers genuinely cannot express is the capability floor. `cheap` conflates "small coding task" with "non-coding chore," and only the latter is safe on a model whose class is beneath coding. So the spawn request grows one small facet, not a taxonomy: **`kind: coding | review | research | mechanical`**, orchestrator-supplied, defaulting by tier (`review` tier → review, everything else → coding — the conservative default, because over-flooring wastes money and under-flooring wastes work, and only one of those gets corrected). The router turns `kind` into an eligibility predicate over capability records: a `coding` task requires `codingCapable: true`; `mechanical` and `research` accept any entitled model. That is the entire capability-awareness mechanism — a filter, two enum fields, no scoring. Tier stays the *cost* judgment; kind becomes the *floor* judgment; the two are orthogonal and both are things an LLM classifies well without knowing any model's name.

Cost: one more field the orchestrator can get wrong, and a `mechanical` misclassification routes real coding to a cheap-class model exactly as today. Telemetry (escalation rate by kind) is the check, and the default-to-coding rule bounds the blast radius to tasks explicitly marked down.

### 2. Where model knowledge lives: discovery + manifest, with the shipped table demoted to fallback

Covered above in the capability-record section; recorded here as the decision. The alternatives considered: keep compiling knowledge into constants (status quo — fails the release-independence requirement by definition); trust discovery alone (fails because discovery has no judgment: it can say Haiku exists and lacks effort support, not that Haiku shouldn't write code); trust the manifest alone (fails because the manifest can't know an *account* — entitlement and alias resolution are per-account facts only discovery has). The composition is the answer: discovery supplies existence/entitlement/aliases/effort, the manifest supplies class/coding-capability, the shipped `DEFAULT_ROUTING` and constants remain as the compatibility floor when both are absent — first daemon start, offline machines, a provider that won't answer. This mirrors quota's architecture deliberately: provider-read facts with provenance, curated judgment layered on, manual override supreme, unknown rendered as unknown, compatibility mode loud but functional.

Concrete retirements this funds, each today a release-coupling: `CLAUDE_BEST_MODEL` — retired by obsolescence, not by discovery, and the distinction matters: caleb's probe shows the menu *omits* `best` entirely (its `default` entry resolves to Opus 4.8), so no vendor surface can confirm what `best` launches and the constant is a cache of a fact only a billable probe establishes. Derived defaults select concrete menu entries and never emit an unlisted alias, so nothing load-bearing needs the constant; it survives only as a deprecated shim for user-pinned `best`, which `docs/model-selection.md` already warns against pinning. Likewise `modelVendor`'s alias list (aliases join against discovered catalogs, the regexes remain only as a last-resort parse for names discovery has never seen); the direct file-reads in `resolveConcreteModel` for Codex `"default"` (`~/.codex/config.toml` misses layered/profile config — the app-server's `config/read` answers with the *effective* unflagged default); and `FABLE_AUTO_ROUTING_CUTOFF` with `POST_FABLE_CUTOFF_DEEP_MODEL` (a billing-date route change is manifest data if anything is — the next such cutoff should be an entry in a fetched file, not a code change with a cross-verification test).

### 3. Effort: a first-class, discovery-validated knob, plumbed end to end or not at all

Effort becomes a peer of `model` at every layer it exists today for Codex and is absent for Claude: `ClaudeRouteSchema` grows an optional `effort`; `SpawnRequest` grows an optional `effort` (a directive, like `model`: passed only when the caller — user or orchestrator policy — means it; routine spawns take the route's value); the Claude adapter passes `--effort <level>` (verified present, claude 2.1.207); and the Claude arm of `ExecutionIdentity` grows an *optional* `effort`, because a knob that spawn honors and control-restart drops is precisely the silent divergence the immutable identity exists to prevent (the token-efficiency doc's R3 called this out, and it is right).

Two design points the plumbing has to get right:

**Effort values are vendor data, not a Hive enum.** Hive's schemas currently hardcode the levels as closed enums — `minimal|low|medium|high|xhigh` in `CodexRouteSchema` and the Codex identity arm (`src/schemas/routing.ts:22-28`, `src/schemas/agent.ts:36`) — and caleb's catalog pull shows those enums are *already* wrong: live Codex model records advertise effort values (`max`, `ultra`) the schema would reject. A closed enum here is `CLAUDE_BEST_MODEL` in another costume — a shipped constant caching a fact the vendor publishes — and it fails worse, because a schema rejection doesn't misroute a spawn, it refuses a valid one. So effort becomes an open string at every layer (route, spawn request, identity), validated at spawn time against the resolved model's discovered `effortLevels`, with the known-value list serving only as fallback validation when no record exists. A normalized Hive-only scale mapped per vendor was also considered and rejected: `max` has no image in every catalog, `minimal` in others, and a lossy mapping means the user asks for one thing and gets another — the exact lie the verbatim-model rule exists to prevent. Spawn-level `effort` passes through verbatim; route-level effort is per-tool anyway (the route already has separate `claude` and `codex` columns), so no mapping is ever needed there. The identity arm accepting an open string is safe for the same reason the model field already does: identity records what launched, and what launched was validated at launch.

**Discovery gates the flag.** Haiku is the proof case: its menu entry advertises no effort support, so a per-tier effort default naively applied would pass `--effort low` to a model that may not take it. Rule: a resolved candidate whose record does not *affirmatively* advertise effort support gets no effort flag, full stop — conservative precisely because absence is three-valued (see above). If the effort was an explicit directive, the spawn *fails loudly* with the record as evidence — never silently dropped, mirroring the never-substitute rule for models. If the effort came from the route table (Hive's own derived judgment), it is dropped with the decision `reason` noting it — clamping Hive's own defaults is housekeeping; clamping a human's directive is disobedience. The same gate fixes a quieter shipped constant: several spawner and recovery paths default a missing Codex effort to `medium` (`spawner-impl.ts:1086,1238,1288`), but the catalog publishes a per-model `defaultReasoningEffort` (Spark's is `high`, not `medium`) — an unset effort should fall back to the model's own discovered default, and to the shipped constant only beneath that. Codex's catalog also demonstrates both drift directions at once: no listed model advertises Hive's `minimal`, while Sol and Terra advertise `max` and `ultra` that Hive's enums reject.

Identity migration is additive and safe: existing Claude identities lack the field, and absence means "the CLI's default," which is faithfully what those agents were launched with — a legacy row restarts exactly as it launched. No backfill, no epoch bump.

### 4. Deriving defaults: resolve tier → route through records at spawn time, with routing.toml untouched on top

The resolution order becomes three layers, strictly ranked: **user pin** (`routing.toml`, semantics byte-for-byte unchanged — per-tier, per-tool, always wins) → **derived route** (computed from capability records: for each tier and vendor, the entitled, live, class-matching model — `deep` wants the account's best frontier-class entry, `standard`/`review` the mid class, `cheap` the cheap class — with effort defaults per tier from the manifest) → **shipped table** (today's `DEFAULT_ROUTING`, governing exactly when records are missing or stale). The tier stays the key; what a tier *means* moves from "these strings we shipped" to "this requirement profile resolved against what your account can actually reach, right now."

The alternative — deriving at daemon start and materializing a table — was considered for auditability and rejected for staleness: a table materialized at start is wrong by afternoon when a reset passes or an entitlement changes, and Hive already resolves routes per-spawn (`resolveRoute` re-reads `routing.toml` every time), so per-spawn derivation adds no new architecture. The auditability worry is real, though, and gets a direct answer instead: a `hive routing` surface that prints the currently-derived table with per-cell provenance (pinned / derived-from-discovery / shipped-fallback, with observation timestamps), because a router whose choices can't be inspected will be distrusted the first time it surprises someone, and *should* be.

The Codex column derives like Claude's, because `model/list` enumerates account-*visible* models — the entitlement fact whose absence forced the shipped `"default"` pass-through (SPEC §6's field lesson) is now a discovered fact. `"default"` is not deleted, though; it is demoted to exactly the fallback layer everything else shipped occupies: when no fresh Codex records exist — app-server never answered, first start, signed out — the column falls back to `"default"`, which remains entitlement-safe by construction. The launch-probe posture the token-efficiency doc's R4 proposed (attempt a concrete model, respawn on `"default"` when the 15-second monitor sees "model not supported") was the right design against an entitlement-blind catalog and is superseded by an entitlement-aware one; it survives only as the natural behavior if a stale record lies, since the monitor already exists. All three postures — derive, probe-on-stale, fall back — are one code path with different record availability, which is the property that matters.

### 5. Capability is eligibility; quota is ranking; identity is immutable — three stages that never trade places

The quota resolver is already generic over an arbitrary candidate list, scored by headroom, filtered by viability, gated per-pool (SPEC §6). This design does not touch its internals; it changes who builds the candidate list. The hardcoded Fable→Opus splice generalizes into **downshift chains derived from capability records**: after the primary candidate, append same-vendor, entitled, live models of the same-or-adjacent class in descending class order — which reproduces today's valve (Fable's chain is Opus 4.8) as data instead of code, and gives every tier what R2 asked for without a resolver change. Order semantics are preserved exactly: primary first, ties keep the primary, so the no-quota-data compatibility case still behaves as today.

The load-bearing rule is the stage ordering. **The capability floor filters candidates before quota ever sees them**: a model beneath the task's `kind` floor never enters the list, so no amount of quota pressure can downshift a coding task onto a cheap-class model — exhaustion above the floor fails the spawn with the capacity report, exactly as an exhausted explicit pin does today. The rejected alternative was making capability a scoring penalty rather than a filter — tempting because it "uses idle cheap capacity" — and it is rejected because a penalty is a price, and under enough quota pressure any price gets paid; a floor that can be outbid is not a floor. The same argument already lost once inside quota design ("headroom alone ranked a route that could not run" — SPEC §6's viability lesson); capability is the same shape of fact.

Execution identity and control restarts constrain all of this from below, and the constraint is welcome: **capability records are consulted at launch time only, and never at restart**. The identity — `{tool, model, effort?}` — remains the frozen, complete instruction for reproducing the launch; a critical-control restart replays it verbatim even if the manifest has since reclassified or dropped the model, because a faithful restart of the interrupted agent is the point and the recorded-model-no-longer-launchable case already fails closed with a durable alert (SPEC §6). Nothing dynamic may be needed to restart an agent: this is why effort must live *in* the identity rather than be re-derived, why derived routes must resolve to concrete models *before* the identity is written (they do — the resolver's output is concrete by construction), and why the `live_model` observation stays a separate field rather than mutating `model`. The dynamic router makes launch-time choice smarter; it must leave the restart path exactly as dumb as it is.

## Migration from today's tiers

The four tier names, the `routing.toml` format, and every existing pin remain valid throughout — there is no user-visible migration, only the defaults beneath the overrides getting smarter. Staged, each step inert until the next:

1. **Record, change nothing.** Land the capability-record store and its discovery feeds (Claude's `initialize` is already awaited; Codex's `model/list` rides the app-server handshake), provenance-stamped, visible in `hive routing` — observation only, routing still reads the shipped table.
2. **Plumb effort inert.** Schema fields (`ClaudeRouteSchema.effort`, `SpawnRequest.effort`, optional identity effort), adapter flag, discovery-gated validation — all defaulting to unset, so shipped behavior is bit-identical until a user or the manifest sets a value.
3. **Shadow-derive.** Compute the derived route per spawn, log derived-vs-shipped divergence, route on shipped. Two weeks of divergence data is the go/no-go evidence.
4. **Flip the middle layer.** Derived routes govern between pins and fallback; downshift chains replace the hardcoded valve; the `kind` floor gates candidates. Reversible per-tier by a `routing.toml` pin, globally by config.
5. **Retire the constants.** `CLAUDE_BEST_MODEL`, the cutoff pair, the alias list in `modelVendor` — each deleted only once its manifest/discovery replacement has carried a real lineup change (the 2026-07-12 Fable cutoff is, conveniently, an immediate live test: encode it as manifest data in step 1 and watch it govern).

## Open questions

- **How stable are the discovery surfaces as contracts?** Claude's `initialize` model menu is undocumented (the same epistemic tier as `get_usage`), and Codex's `model/list` is absent from the prose docs — both were verified by driving the CLIs, and either could change shape in a point release. Capability records carry provenance for exactly this reason, but the design has no answer yet for a surface that *silently* drops a field rather than disappearing.
- **Who refreshes the manifest, and how fast is fast enough?** "Hours-to-days, not a release" assumes a publishing pipeline that doesn't exist yet; if manifest refresh in practice rides releases, the release-independence claim quietly dies. The unknown-means-explicit-only rule makes the failure visible (new models are pinnable but not auto-routed) rather than dangerous, but visible-and-slow is still a broken promise.
- **Is `kind` defaulting to `coding` too conservative?** It routes ambiguous cheap chores onto coding-capable models, spending real money to avoid misrouting. Escalation-rate and cost telemetry by kind decides; the reverse default is indefensible without it.
- **The `[1m]` variant suffix.** Discovery reports `claude-opus-4-8[1m]` as a `resolvedModel`; identities and `--model` flags must carry the bare id. Stripping is trivial; the question is whether the variant ever *matters* to record (it names a context-window entitlement that could plausibly gate task assignment someday).
- **Effort defaults per tier are judgment with no data.** The manifest will ship some (`deep: high`, `cheap: low`, presumably) — but the token-efficiency doc's open question stands: nothing yet measures whether a given effort step pays for itself, and the A/B machinery it proposes should run before effort defaults get opinionated.
