## Not tested: SMB and other network volumes

No SMB server was reachable from the machine that produced this log, and standing one up
would have made the result depend on an unrecorded server configuration. Nothing here
should be read as evidence about network volumes. The specific claims that do not transfer:

- **Inode stability.** The refusal logic rests on `dev`/`ino`/`birthtimeMs` changing when a
  directory is replaced. An SMB client synthesizes inode numbers, and a server may reissue
  them across remounts. If it does, `findByEvidence` can both miss a real move and, far worse,
  match an unrelated directory.
- **Bookmark resolution.** The path-first behavior measured below was measured on APFS. A
  bookmark to an unmounted share has no file to resolve at all.
- **Case and normalization behavior.** These are properties of the server's filesystem, not of
  the SMB client. `describeVolume` probes rather than assumes, so it should degrade correctly,
  but that has not been demonstrated.

The experiment that would settle it: export one case-sensitive and one case-insensitive share
from a known server, run the same scenarios against both, then unmount and remount between
each step and re-check that `ino` and `birthtimeMs` survive. Until that runs, a network volume
should resolve to `BLOCKED_CONFIG` rather than silently rely on evidence it cannot trust.
