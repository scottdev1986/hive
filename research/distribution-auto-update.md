# Distribution and automatic updates for Hive

Verified against live documentation and source on 2026-07-10.

## The recommendation

Hive should ship a signed native installer as the primary distribution path: a small `curl | sh` bootstrap that installs the compiled Bun executable into a user-owned, versioned directory, plus `hive update` and an opt-out background checker. The updater should download releases automatically but activate them only at a safe daemon boundary. Homebrew should remain a secondary, package-manager-owned path whose updates are deliberately not performed by Hive.

This is not a cosmetic replacement for `brew upgrade`. It changes who owns the installation, which is the only way Hive can promise that a release reaches installed users without waiting for another package manager or a user command. Claude Code is the closest precedent: its native installer is now the recommended path and updates at startup and periodically while running, while its Homebrew and package-manager installs do not update themselves by default. Anthropic still offers those routes, but it gives the native route the best update semantics ([Claude Code setup](https://code.claude.com/docs/en/setup)).

The price is that Hive becomes responsible for a security-sensitive installer, update protocol, signing keys, staged rollouts, failed-update recovery, and daemon compatibility. That is real product infrastructure. It still wins because Hive is already a single macOS binary and because an agent-control daemon is exactly the kind of software where leaving old security and lifecycle bugs installed indefinitely is the worse trade.

## What Hive already promises

README currently promises `brew install hive`. SPEC makes the same choice part of the technology argument: TypeScript on Bun, compiled with `bun build --compile` into one binary, so users do not need Node or Bun. The daemon starts on demand, persists SQLite state under `~/.hive`, and owns live tmux-backed agents, approvals, delivery, and landing. `launchd` is explicitly deferred as v2 polish.

The runtime choice helps rather than constrains a native installer. Bun documents standalone macOS arm64 and x64 targets that include the runtime and dependencies in one executable ([Bun standalone executables](https://bun.com/docs/bundler/executables)). Bun also documents Developer ID signing for compiled executables, though its JavaScriptCore runtime may require permissive JIT and library-validation entitlements; that exact hardened-runtime/notarization combination must be proven on Hive's final binary rather than assumed from a successful ad-hoc signature ([Bun macOS code-signing guide](https://bun.com/docs/guides/runtime/codesign-macos-executable)).

The daemon is the actual constraint. Replacing a file is easy. Replacing the control plane while agents are writing, approvals are pending, and landing authority is live is not.

## What the self-updating CLIs actually do

The four useful precedents agree on the broad shape and disagree on how much safety they provide.

Rustup bootstraps with `curl ... | sh`, installs a single `rustup` binary, and exposes `rustup self update`. Its automatic mode means that commands such as `rustup update` also check and update rustup; it is not an always-running background service. Users can choose `enable`, `disable`, or `check-only` ([rustup basics](https://rust-lang.github.io/rustup/basics.html)). Rustup's source describes self-update as downloading a new `rustup-init`, running it in self-upgrade mode, then copying and hard-linking tools into place; Windows defers replacement until the running command exits ([rustup self-update source](https://github.com/rust-lang/rustup/blob/main/src/cli/self_update.rs)). Rustup has separate production and beta update roots, which function as two self-update channels ([rustup release process](https://rust-lang.github.io/rustup/dev-guide/release-process.html)). It can also install a pinned rustup version, so rollback means explicitly installing an older known version, not an automatic post-activation health rollback ([rustup changelog](https://github.com/rust-lang/rustup/blob/main/CHANGELOG.md)). The lesson is that bootstrap plus in-binary update is mature, but Rustup does not solve Hive's running-daemon problem.

Deno installs by shell script and exposes `deno upgrade`. Current Deno supports exact versions and `stable`, `alpha`, `beta`, `rc`, and `canary` channels. It caches downloads, verifies published SHA-256 values, verifies both delta patches and their resulting binaries, falls back from a failed delta to a full archive, and runs a downloaded PR binary before replacing the current executable ([Deno upgrade](https://docs.deno.com/runtime/reference/cli/upgrade/)). Rollback is `deno upgrade <older-version>`; the updater does not retain and automatically revive a last-known-good executable. Its updater is explicit rather than a background service. The lesson is to verify the artifact and execute a smoke check before activation, while treating delta delivery as later optimization rather than v1 work.

Bun also uses a shell installer and `bun upgrade`, with stable/canary switching and exact old-version installation through the installer ([Bun installation](https://bun.com/docs/installation)). Its implementation downloads into a temporary version directory, executes the candidate with `--version` or `--revision`, compares the reported version with the expected version, and only then moves it over the running executable. On Windows it first renames the old executable and attempts to restore it if moving the new one fails; on Unix it moves the candidate into place without preserving an automatic last-known-good sibling ([Bun upgrade source](https://github.com/oven-sh/bun/blob/main/src/cli/upgrade_command.zig)). Bun tells Homebrew users to use `brew upgrade bun` instead of self-update, avoiding two owners for one install. The lesson is to detect package-manager ownership and refuse to mutate it.

Claude Code is the strongest model for Hive. Its native installer is recommended, accepts an exact version or a `latest`/`stable` channel, checks at startup and periodically, installs in the background, and activates the update on the next start. `stable` is normally about a week behind and skips releases with major regressions. Users can disable background checks, disable all update paths, or set a minimum version floor. Enterprise policy can impose minimum and maximum versions ([Claude Code setup](https://code.claude.com/docs/en/setup)). The installation retains binaries under `~/.local/share/claude/versions/VERSION` and exposes `~/.local/bin/claude`; Anthropic's uninstall and integrity instructions document those paths. Each release has a SHA-256 manifest signed by Anthropic's GPG key, and its macOS binaries are Developer ID signed and notarized ([Claude Code binary integrity](https://code.claude.com/docs/en/setup#binary-integrity-and-code-signing)). A real 2026 packaging bug left downloaded versions present while the active symlink still targeted an old release, a useful warning that versioned activation is recoverable but not magically correct ([Claude Code issue 24117](https://github.com/anthropics/claude-code/issues/24117)).

None of these tools provides a documented transactional application rollback after a newly activated binary starts and fails. They provide pre-activation verification and an explicit old-version path. Hive should go one step further because it has a daemon with a health endpoint and durable state, making an automatic post-restart check both possible and necessary.

## Why registry-global installation loses

An npm or Bun global package can distribute either JavaScript or a platform-specific optional package containing the same native binary. It gives familiar install commands, registry CDN delivery, versions, and dist-tags for stable/beta rings. An update notifier can check the registry asynchronously, cache the result, and print a message on a later invocation. The widely used `update-notifier` package does exactly that and intentionally stopped auto-installing after automatic mutation proved unpopular ([update-notifier](https://github.com/sindresorhus/update-notifier)).

That is good notification quality and poor automatic-update quality. The notifier cannot reliably apply an update because the global prefix may not be writable, the install may be owned by npm, Bun, nvm, mise, Volta, or a system Node, lifecycle scripts may be disabled, and modifying the native binary behind the package manager makes its receipt lie. npm's own documentation needs a dedicated EACCES recovery guide and recommends a version manager or a user-owned prefix ([npm permissions](https://docs.npmjs.com/resolving-eacces-permissions-errors-when-installing-packages-globally/)). Bun's global default is user-owned, but it is a different directory and package database, and Bun does not run arbitrary lifecycle scripts unless the package is trusted ([Bun global installs](https://bun.com/docs/pm/cli/add)). Supporting both multiplies detection and repair paths.

Claude Code's present design makes the weakness concrete. Its npm package now installs the same native binary through per-platform optional dependencies, yet npm remains an alternative route, the native installer remains recommended, global npm updates can fail when the prefix is not writable, and the documented update command must explicitly request `@latest` because an ordinary global update can respect the old semver range ([Claude Code npm installation](https://code.claude.com/docs/en/setup#install-with-npm)). Moving the runtime into the package did not give npm native-install update semantics.

For Hive, npm/Bun global would also contradict the SPEC's “single binary, no runtime prerequisite” user story unless it were merely a bootstrap wrapper. A bootstrap wrapper adds a registry supply-chain and lifecycle-script layer without improving the final install. It should not be a first-release channel.

## Why Homebrew cannot make the promise by itself

Homebrew has two operations that are easy to conflate. Its built-in auto-update refreshes Homebrew and formula metadata before selected `brew` commands; it does not upgrade installed formulae. `brew upgrade` performs the package upgrade. The default metadata refresh interval is 24 hours, but only when the user invokes a triggering Homebrew command ([Homebrew manual](https://docs.brew.sh/Manpage), [Homebrew FAQ](https://docs.brew.sh/FAQ)). `brew livecheck` finds a newer upstream version for maintainers; it neither changes the formula nor upgrades a user's machine ([Homebrew livecheck](https://docs.brew.sh/Brew-Livecheck)). A tap updates when a user runs `brew update`, and an outdated formula changes only when the user runs `brew upgrade` ([Homebrew tap maintenance](https://docs.brew.sh/How-to-Create-and-Maintain-a-Tap)).

The separate `brew autoupdate` project can install a `launchd` job and, with optional flags, run `brew upgrade` and `brew cleanup`. It can even restrict upgrades to named packages. But it requires the user to install and trust an external tap command, opt into `--upgrade`, and accept Homebrew-wide operational behavior; it moved out of the Homebrew organization in 2023 to reduce Homebrew's maintenance burden ([Homebrew Autoupdate](https://github.com/DomT4/homebrew-autoupdate)). Failures include a sleeping/offline Mac, stale or broken tap metadata, formula lag after the upstream release, pins and environment opt-outs, build/bottle gaps, permissions, unrelated dependency upgrades, and a running daemon that the formula knows nothing about.

Homebrew can be made hands-off on a carefully configured machine, but Hive cannot truthfully promise that behavior from `brew install hive`. It is still valuable as a secondary channel because users trust its receipts, uninstall/upgrade model, and prefix management. The formula or tap should be updated automatically by release CI, and Hive should detect that it is running from a Homebrew-managed path. `hive update` should then say `brew upgrade hive` (or, later, offer an explicit opt-in that invokes exactly that package command, as Claude Code now does) rather than rewriting the Cellar itself.

## Why the macOS-native app route loses

Sparkle is an excellent macOS application updater. It provides appcasts, Ed25519 archive signatures, optional signed feeds, delta updates, key rotation, installer helpers, and a mature consent/relaunch UI. Its setup and distribution model is built around a Cocoa `.app` or another bundle with `Info.plist`, `CFBundleVersion`, an updater object, and helper executables ([Sparkle documentation](https://sparkle-project.org/documentation/)). Tailscale's standalone macOS application uses Sparkle and exposes its auto-update controls through Sparkle preferences and MDM policy ([Tailscale macOS policy](https://tailscale.com/kb/1315/mdm-keys)). That is sensible because Tailscale already ships a GUI app and system extension.

Hive is a CLI and unprivileged user daemon. Wrapping it in a menu-bar app solely to gain Sparkle would add Swift/Xcode build infrastructure, an app lifecycle, helper signing, App Management permissions, and a second control plane. There is no macOS framework equivalent that makes a bare CLI daemon update sane for free. A LaunchAgent can schedule Hive's own updater, but it is scheduling, not an update framework. Sparkle becomes reasonable only if Hive later wants a real menu-bar supervisor for status, approvals, and updates; it should not drive v1 distribution.

## The daemon changes the activation contract

Desktop products confirm that a daemon restart is a product event, not a file-copy detail. Docker Desktop checks automatically, can download updates in the background, and rolls releases out gradually, but full Desktop updates remain restart-oriented; only components such as Compose, Scout, and the CLI can update independently without a full restart ([Docker Desktop settings](https://docs.docker.com/desktop/settings-and-maintenance/settings/), [Docker release FAQ](https://docs.docker.com/desktop/troubleshoot-and-support/faqs/releases/)). Docker Engine's `live-restore` preserves containers across some patch daemon upgrades, but not major upgrades, skipped releases, or incompatible daemon configuration, and management is unavailable while the daemon is down ([Docker live restore](https://docs.docker.com/engine/daemon/live-restore/)).

Tailscale stages stable client updates for roughly a week, schedules them around node activity, and speeds up critical security releases. On macOS its standalone app can use Sparkle; managed package upgrades require an explicit `down-for-update` before replacement and a post-install restart ([Tailscale updates](https://tailscale.com/kb/1067/update), [Tailscale MDM upgrades](https://tailscale.com/docs/integrations/mdm/mac)). OrbStack likewise owns both app and bundled command-line tools, linking its CLIs from an app-managed directory; its public docs do not specify a no-interruption update contract, so it is evidence for single-owner bundling, not evidence that daemon restarts are safe ([OrbStack install](https://docs.orbstack.dev/install)).

Hive should therefore separate download, activation, and daemon restart:

1. A background check may fetch and verify a candidate at any time. It writes only a new immutable version directory and `pending-update.json`; it does not touch the active link or SQLite schema.
2. The daemon declares itself quiescent only when there are no live/spawning agents, no active orchestrator session, no pending approval, landing, critical-control restart, or message injection, and its SQLite transaction queue is drained. A downloaded update waits indefinitely rather than interrupting live work.
3. At quiescence, an external updater helper asks the old daemon to prepare for update. The daemon checkpoints WAL, records its version/protocol/schema and the pending release, closes listeners, and exits cleanly. The helper atomically switches the active symlink and starts the new daemon.
4. The helper requires a bounded health check that opens the existing database read/write, binds the MCP endpoint, reports the expected binary and wire-protocol versions, and performs a no-op transaction. If health fails, it switches the link back and restarts the previous daemon. It keeps both versions and a database snapshot until the new version survives a grace period.

A Unix process continues executing its already-open image after a symlink changes, so atomic link activation does not update the old daemon. That is useful during staging but dangerous after activation: a newly invoked CLI may be version N while the daemon remains N−1. Every CLI/daemon request must therefore start with a version handshake containing binary version, wire-protocol version/range, and database schema range. Hive should guarantee N/N−1 wire compatibility. If compatible, a new CLI can finish a session against the old daemon. If incompatible, it must refuse the operation with an actionable “update staged; finish or stop the current team” message, never silently kill the team.

Agent hooks are another client. A live team should keep using the session's recorded executable path, not whatever `hive` happens to resolve to on `PATH`, or the protocol handshake must cover every `hive event` invocation. Pinning the versioned binary into spawn-time hook configuration is simpler and ensures a team has one control-plane version from birth to quiescence.

Database migrations must follow an expand/contract rule. An automatically activated release may add compatible tables, columns, or indexes, but it must not destroy state the previous binary needs for rollback. Destructive migrations require a later release after the compatibility window or an explicit user-confirmed upgrade. A copied, checkpointed SQLite snapshot protects against implementation bugs, but it is not permission to roll back user work created after activation.

## The release and update mechanism implied by the choice

The release pipeline should have one source build and two macOS artifacts, `darwin-arm64` and `darwin-x64`. A universal binary is not necessary; separate artifacts are smaller and the installer can select with `uname -m`.

For each signed tag, CI should:

1. Run typecheck, unit/integration tests, and updater/rollback fixtures. Build both targets with the exact locked Bun version and embed Hive version, commit, wire protocol, and schema compatibility constants.
2. On a macOS signing runner, sign each final Mach-O with a Developer ID Application certificate, hardened runtime, secure timestamp, and only the Bun entitlements proved necessary. Verify with `codesign --verify --strict` and execute smoke tests on matching hardware or runners.
3. Put each signed executable in a ZIP, submit it with `notarytool`, inspect the notarization log even on success, and verify Gatekeeper behavior on a clean/quarantined test account. Apple requires Developer ID signing, hardened runtime, and secure timestamps for current notarization. Standalone binaries can be notarized inside a ZIP but cannot have a ticket stapled directly, so online ticket lookup and the embedded code signature matter ([Apple notarization requirements](https://developer.apple.com/documentation/security/notarizing-macos-software-before-distribution), [Apple custom notarization workflow](https://developer.apple.com/documentation/security/customizing-the-notarization-workflow)).
4. Generate SHA-256 values and a canonical manifest containing version, channel, publication time, artifact URL/size/hash, supported macOS and architecture, wire/schema ranges, minimum updater version, rollout percentage, and whether the release is security-critical. Sign the canonical manifest with an offline Ed25519 release key whose public key is embedded in Hive. Apple signing authenticates the executable to macOS; the manifest signature authenticates update policy to Hive. Neither should substitute for the other.
5. Create a draft GitHub Release, attach artifacts, checksums, manifest, signature, SBOM, and provenance attestations, then publish with immutable releases enabled. GitHub immutable releases lock the tag and assets and create a verifiable release attestation; artifact attestations connect binaries to the workflow and commit that built them ([GitHub immutable releases](https://docs.github.com/en/code-security/concepts/supply-chain-security/immutable-releases), [GitHub artifact attestations](https://docs.github.com/en/actions/concepts/security/artifact-attestations)).
6. After canary validation, atomically promote a small signed channel document at `updates.hive.example/v1/beta.json` or `stable.json`. GitHub Releases should hold immutable bytes; a small CDN-backed endpoint should express mutable channel and rollout policy. Do not make every client scrape the GitHub “latest” API or trust a moving asset without a signed policy document.
7. Update the Homebrew tap formula from the same immutable asset and SHA. Promotion to stable should happen only after the direct-install ring reaches its health threshold, so Homebrew users inherit the delayed ring rather than becoming canaries.

The installer should be short enough to audit. It detects macOS and architecture, refuses unsupported versions, downloads to a private temporary directory, verifies the manifest signature and artifact SHA, verifies the Developer ID signature/Team ID, executes `hive --version` from the candidate, moves it to `~/.local/share/hive/versions/<version>/hive`, and atomically points `~/.local/bin/hive` at it. Because portable POSIX shell cannot reliably verify Ed25519 on every supported macOS, the bootstrap can use a tiny notarized installer binary or a pinned Apple-native signature check plus SHA from a versioned HTTPS script; the in-binary updater must always verify the embedded release key. The website should offer the script for inspection and a manual download/verification path alongside the one-liner. `curl | sh` is convenient, not a trust argument.

The installed updater should use a per-user lock, private staging directory, `fsync` before rename, no symlink-following writes, and strict file modes. `hive update` should support check-only, explicit version, channel selection, disable/enable, and rollback to the retained previous version. Background checks should be jittered and rate-limited. Stable should be the default, beta opt-in; within stable, a deterministic hash of a locally generated anonymous install ID can implement 5% → 25% → 100% rollout without telemetry. A security release may accelerate the ring but should still never interrupt an active team.

To reach machines when Hive is not running, the native installer may install an opt-out per-user LaunchAgent that invokes a short `hive update --background` job daily. That is the one place the recommendation conflicts with SPEC's current “launchd is v2 polish” line. The lower-cost v1 alternative is to check whenever the daemon or CLI starts and periodically while the daemon runs, exactly like Claude Code; this is automatic for active users but not proactive for dormant installations. Hive must choose which promise it wants. If “releases are pushed/applied without the user running Hive” is literal, the LaunchAgent belongs in v1 and must be disclosed by the installer.

## Route comparison and trust

The native installer has the best automatic-update quality and daemon-restart control, medium-to-high implementation cost, medium macOS friction, and the highest trust burden. Signing, notarization, a transparent script, opt-out controls, delayed stable rollout, retained versions, and visible `hive update status` are what earn that trust. It is the only route where Hive can stage, defer, health-check, and roll back its own daemon coherently.

Homebrew has high initial user trust and low installation friction, with low Hive updater cost, but poor default automatic-update quality and no knowledge of live Hive state. Adding `brew autoupdate` improves scheduling only after extra user configuration and still leaves daemon safety to package hooks or Hive. It is a strong secondary route and a weak primary answer to this requirement.

npm/Bun global has familiar developer ergonomics and cheap notification, but low hands-off quality, competing ownership, permission/runtime/configuration variance, and no improvement to Gatekeeper: the same native artifact still needs signing and notarization. It is not worth the support surface.

Sparkle has excellent application update quality and macOS trust primitives, but very high cost and architectural mismatch for a bare CLI daemon. It wins only after Hive becomes a real macOS app.

The honest risk in the recommendation is supply-chain concentration: compromise of Hive's update signing process can reach every opted-in installation faster than a compromised manual channel. The mitigations are an offline manifest key, protected Apple signing identity, two-person release approval, immutable releases and provenance, deterministic/staged rollout, an emergency channel freeze, user-visible logs, and a hard disable switch. Automatic delivery should be the default; invisible, unaccountable mutation should not be.

## Documentation consequence and remaining decisions

README and SPEC should be updated if this recommendation is accepted, but this report deliberately does not edit them. README's installation promise should make the native installer primary, describe automatic updates and opt-out behavior in one sentence, and retain Homebrew under alternative installs. SPEC's tech-stack paragraph should replace “`brew install hive`” as the single-binary payoff with “one signed binary, no runtime,” add distribution/update ownership to the architecture, define the CLI/daemon protocol handshake and quiescent restart invariant, and either move the per-user LaunchAgent into v1 or narrow the promise to updates while Hive is active. The roadmap should name signing/notarization, release manifests, updater rollback, and compatibility testing as release blockers rather than polish.

Four decisions remain before implementation: whether dormant machines must update via LaunchAgent in v1; the stable-ring delay and promotion authority; the exact N/N−1 wire and schema compatibility policy; and whether Bun's required entitlements pass Apple's hardened-runtime/notarization checks without an unacceptable security exception. None changes the primary route. They determine how honest and safe its first implementation is.
