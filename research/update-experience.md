# The update experience for Hive

Verified against live documentation and source on 2026-07-10. This report builds on [distribution-auto-update.md](distribution-auto-update.md), whose recommendation — signed and notarized native install, immutable version directories under `~/.local/share/hive/versions`, atomic symlink activation, Homebrew as a secondary package-manager-owned channel, and a daemon-quiescence activation contract — is the accepted baseline. Nothing here relitigates that. This report designs what the user actually sees and types: the notice, the command, the versions.

## The experience in one paragraph

Hive should feel exactly like updating Claude Code, adapted for the fact that Hive has a daemon. The daemon quietly checks a signed channel manifest about once a day and stages verified downloads without asking. The user learns about it from a single dim line on the next human-invoked `hive` command — never a popup, never an interruption of a working session — and the new version activates on its own at the next daemon quiescence, or immediately when the user runs `hive update` with no team live. `hive update` also serves the impatient and the cautious: check without applying, pin an exact version, switch channels, disable the background behavior, roll back to the retained previous version, and see status. There is no LaunchAgent and no dormant-machine push in v1; the promise is Claude Code's promise — updates happen while you use the tool — and that resolves the baseline report's first open decision by the user's stated preference.

## What Claude Code actually ships

Claude Code is the target experience, so its behavior is worth recording precisely. The native install "checks for updates on startup and periodically while running. Updates download and install in the background, then take effect the next time you start Claude Code" ([Claude Code setup](https://code.claude.com/docs/en/setup)). The docs publish no numeric check interval, and they do not document the routine "update installed" banner wording or the activation mechanism between `~/.local/bin/claude` and `~/.local/share/claude/versions/VERSION` — those paths appear in the uninstall and binary-integrity instructions, but the symlink semantics are undocumented ([Claude Code setup](https://code.claude.com/docs/en/setup)). The visible contract is deliberately small: updating is something you notice happened, not something you do.

There are two channels. `latest` is the default; `stable` "is typically about one week old, skipping releases with major regressions," selected at install time (`curl -fsSL https://claude.ai/install.sh | bash -s stable`, or an exact version like `2.1.89`) or afterward via the `autoUpdatesChannel` setting ([Claude Code setup](https://code.claude.com/docs/en/setup)). The Homebrew route encodes the channel in the cask name (`claude-code` is stable, `claude-code@latest` is latest) and does not self-update by default; setting `CLAUDE_CODE_PACKAGE_MANAGER_AUTO_UPDATE=1` lets Claude Code run the package manager's own upgrade command in the background and show a restart prompt — it still never rewrites a brew-owned install itself ([Claude Code setup](https://code.claude.com/docs/en/setup), [env vars](https://code.claude.com/docs/en/env-vars)).

The control surface has two levels of off and a version floor. `DISABLE_AUTOUPDATER=1` stops background updates while manual `claude update` keeps working; `DISABLE_UPDATES=1` blocks everything including manual updates, for organizations that distribute through their own channels ([Claude Code env vars](https://code.claude.com/docs/en/env-vars)). In settings, `minimumVersion` is a floor that prevents downgrades but never blocks startup, while the managed-settings-only `requiredMinimumVersion`/`requiredMaximumVersion` gate startup itself and deliberately fail open on invalid values so a bad policy push cannot brick the tool ([Claude Code settings](https://code.claude.com/docs/en/settings)).

The commands are minimal. `claude update` is documented as exactly "Update to latest version," with no flags; it respects the channel and the version floors. `claude install [version|stable|latest]` reinstalls or pins. `claude --version` prints the version. `claude doctor` shows the result of the most recent update attempt, the channel, and why the autoupdater is disabled if it is ([Claude Code CLI reference](https://code.claude.com/docs/en/cli-reference), [setup](https://code.claude.com/docs/en/setup), [changelog](https://code.claude.com/docs/en/changelog)). Failure handling is documented but modest: downloads retry up to three attempts on drops, stalls, and checksum failures, with a ten-minute total deadline; there is no documented automatic rollback — recovery is rerunning `claude update` or `claude install <version>` ([Claude Code errors](https://code.claude.com/docs/en/errors#the-connection-dropped-while-downloading-the-update)).

Three lessons transfer directly. First, the default experience is silence plus activation-on-next-start; the user is told about updates far less often than the tool checks for them. Second, the escape hatches (channel, floor, two-level disable, exact version) exist but never appear unless sought. Third, even Anthropic does not attempt automatic post-activation rollback — the baseline report's daemon health-check-and-revert is Hive going beyond the precedent, justified because a daemon can be health-checked and a TUI cannot.

## What Codex CLI actually ships

Codex is distributed through an installer script (`curl -fsSL https://chatgpt.com/codex/install.sh | sh`), npm (`npm install -g @openai/codex`), a Homebrew cask (`brew install --cask codex`), and raw GitHub release binaries ([Codex README](https://github.com/openai/codex)); the official docs use the same installer script rerun as the update path ([Codex CLI docs](https://learn.chatgpt.com/docs/codex/cli)). There is a real `codex update` subcommand — "Check for and apply a Codex CLI update when the installed release supports self-update" — plus a `codex doctor` ([Codex CLI reference](https://learn.chatgpt.com/codex/developer-commands?surface=cli), [cli/src/main.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/cli/src/main.rs)). Crucially, the update action is install-method-aware: the TUI detects how Codex was installed and runs the matching command — `npm install -g @openai/codex` for npm, `bun`/`pnpm` equivalents, `brew upgrade --cask codex` for Homebrew, the installer script with `CODEX_NON_INTERACTIVE=1` for standalone — and offers nothing at all when the method is unrecognized ([update_action.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/tui/src/update_action.rs)). That is the same respect-the-owner principle the baseline report demands for Hive's Homebrew path, verified in a second vendor.

The notification design is more instructive than the command. The TUI checks at startup, asynchronously, against a cached result: a `version.json` in the Codex home directory stores `latest_version`, `last_checked_at`, and an optional `dismissed_version`, refreshed at most every 20 hours; the network fetch is spawned in the background so startup always renders the previous cached answer ([updates.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/tui/src/updates.rs), [updates_cache.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/tui/src/updates_cache.rs)). Homebrew installs check `formulae.brew.sh` rather than GitHub so the notice never precedes cask availability. The notice itself has two forms: an interactive "✨ Update available!" popup offering "Update now," "Skip," and "Skip until next version" — the last being a persistent per-version dismissal — and a passive history line, "Update available! X -> Y. Run {command} to update." ([update_prompt.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/tui/src/update_prompt.rs), [notices.rs](https://raw.githubusercontent.com/openai/codex/main/codex-rs/tui/src/history_cell/notices.rs)). A single config key, `check_for_update_on_startup` (default true, "set to `false` only if your Codex updates are centrally managed"), disables the whole thing ([config schema](https://raw.githubusercontent.com/openai/codex/main/codex-rs/core/config.schema.json), [config reference](https://learn.chatgpt.com/codex/config-file/config-reference)).

Codex versions as fast-moving 0.x semver — 0.144.1 and 0.144.0 shipped the same day, with 0.145.0-alpha prereleases running in parallel on GitHub releases ([Codex releases](https://github.com/openai/codex/releases)). The prerelease tags are its channel mechanism.

Two things transfer and one does not. Per-version dismissal ("skip until next version") is the politest rate limit in this survey and costs one field in a cache file. Install-method-aware update commands are mandatory, not optional. The startup popup does not transfer: Codex owns its TUI and can ask a question; Hive's orchestrator terminal belongs to Claude Code or Codex, not to Hive, and interrupting an agent session with Hive chrome would be exactly the annoyance this design must avoid.

## What the ecosystem agrees a notice looks like

The wider CLI ecosystem converged on one shape, and the convergence is strong enough to treat as a spec. GitHub's `gh` checks in a goroutine launched at command start and prints to stderr after the command finishes, at most once per 24 hours via a persisted state file; it skips CI, non-TTY output, and Codespaces entirely, is silenced by `GH_NO_UPDATE_NOTIFIER`, and — a detail worth stealing — delays the notice for Homebrew installs when the release is under 24 hours old so the cask has time to land ([gh cmd.go](https://raw.githubusercontent.com/cli/cli/trunk/internal/ghcmd/cmd.go), [gh update.go](https://raw.githubusercontent.com/cli/cli/trunk/internal/update/update.go)). npm fires an unawaited best-effort check and saves the message "for printing at process exit so it will not get lost," weekly for stable versions, color-coding the notice by semver distance (red major, yellow minor, cyan patch) and always printing the exact upgrade command; the `update-notifier` config key turns it off ([npm update-notifier.js](https://raw.githubusercontent.com/npm/cli/latest/lib/cli/update-notifier.js), [npm config](https://docs.npmjs.com/cli/v11/using-npm/config)). The `update-notifier` package that set the ecosystem convention checks in a detached unref'd child process and shows the result on the *next* run, defaulting to a one-day interval, with `NO_UPDATE_NOTIFIER`, a `--no-update-notifier` flag, and automatic CI/test-env suppression ([update-notifier readme](https://raw.githubusercontent.com/sindresorhus/update-notifier/main/readme.md)). Deno delays its startup check by 500 ms so it can never affect startup latency, caches for 24 hours, prints to stderr only when stderr is a terminal, and is disabled by `DENO_NO_UPDATE_CHECK` ([deno upgrade.rs](https://raw.githubusercontent.com/denoland/deno/main/cli/tools/upgrade.rs)). Homebrew is the contrast case — it acts (refreshing its own metadata before some commands, default every 86400 seconds) rather than notifying ([Homebrew manual](https://docs.brew.sh/Manpage)).

The consensus, every element verified above: check asynchronously so the user's command is never slower; cache the answer for about a day; print one line to stderr at a natural boundary (end of command, or start of a session); gate on TTY and skip CI; name the exact upgrade command for the user's install method; provide one dedicated opt-out. On security urgency there is essentially no terminal precedent: Tailscale accelerates security releases and marks them with a red arrow, but in its admin console, not in a CLI notice ([Tailscale updates](https://tailscale.com/kb/1067/update)); npm's red-for-major coloring signals semver breakage, not security. If Hive wants security-urgent notices — and it should, being an agent-control daemon — it will be setting the precedent, not following one.

## Versioning

The version story used to be four hardcoded copies of the string `"0.1.0"` — the daemon, the CLI's three MCP `Client` constructions, the channel bridge's `serverInfo`, the Codex app-server `clientInfo` — plus `package.json`, and no `hive --version` at all. Strings that drift independently cannot carry an N/N−1 handshake, so they were collapsed into one module. [docs/versioning-and-release.md](../docs/versioning-and-release.md) owns the resulting contract; this section records only what it means for the experience.

`src/version.ts` is the single source, with real values inlined at release build time. `bun build --compile --define 'process.env.HIVE_BUILD_VERSION="0.0.7"' ...` rewrites the expression into a string literal before the bundle is written ([Bun standalone executables](https://bun.com/docs/bundler/executables), [Bun bundler define](https://bun.com/docs/bundler)), so a release binary cannot be relabelled by exporting an environment variable at it. A checkout leaves the defines unset and falls back to `0.0.0-dev`, which never claims a release version and never nags. `hive --version` prints one line — `hive 0.0.7 (abc1234, 2026-07-10, darwin-arm64)` — because that is what every peer does and what bug reports need. The richer facts belong to `hive update status` and the handshake.

Semver policy: Hive is `0.0.x`, patch-only, starting at `0.0.1`, and every push to `main` bumps the patch by exactly one. The alternative was the honest reading of [semver's 0.x clause](https://semver.org/#spec-item-4) that Codex ships — patch for fixes, minor for features and anything breaking. It lost on a narrow point: it requires a human to classify every push, and a versioning scheme that depends on judgment at release time is a scheme that drifts the first time nobody is paying attention. Patch-only has one moving part and can be computed from the tags alone. It gives up the ability to signal breakage through the version, which is affordable precisely because breakage is signalled elsewhere — by the wire protocol and schema integers, which deliberately do not ride the semver. A marketing version can jump without breaking compatibility, and compatibility can break in a "minor" release; separate integers make the handshake mechanical instead of interpretive. When Hive is ready for `0.1.0` that is a deliberate decision, written down, not a side effect of a commit message.

The daemon handshake carries `productVersion`, a content-addressed `buildHash`, the wire-protocol range, the schema epoch, the capability set, and the `HiveUUID`. The build hash is the load-bearing one, and it is not the version: a rebuilt `0.0.7` with different code is a different daemon, and a version-only check would silently adopt it. It content-addresses the build's *inputs* — source tree, version, commit, target — because the output embeds the hash and cannot address itself. Hashing the MCP tool schemas instead of hand-bumping the wire integer was considered and rejected: a hash detects every change but cannot express "changed compatibly," which is the entire point of N/N−1.

## Where the notice surfaces

Hive has three possible surfaces and only one of them is right by default. The daemon has no UI. The orchestrator terminal belongs to a running Claude Code or Codex session, and injecting Hive chrome into it mid-session is both technically awkward and exactly the interruption this design forbids. That leaves the CLI — and the ecosystem consensus says precisely how to use it.

The division of labor: the **daemon owns checking**, the **CLI owns telling**. The daemon fetches the signed channel manifest shortly after it starts (delayed and asynchronous, Deno-style, so startup cost is zero) and then on a jittered ~24-hour timer while it runs; it verifies and stages the download per the baseline contract and records `{latestVersion, stagedVersion, checkedAt, securityCritical}` in the `meta` table. The CLI never touches the network for update purposes; on human-invoked commands it reads that state through the daemon it is already talking to and prints at most one line to stderr. This inverts gh/npm/Deno — they have no daemon, so each process must check — and having a resident process is Hive's structural advantage: checks actually happen every 24 hours of use rather than once per invocation burst, and the CLI notice is a local read with zero latency cost.

The notice prints after the command's normal output, on stderr, only when stderr is a TTY and no CI variable is set, in dim text, in exactly this shape:

```
hive 0.3.1 available (you have 0.2.4) — run hive update
```

When the daemon has already staged the download — the common case, since staging is automatic — the line says what is actually true: `hive 0.3.1 downloaded — activates when the current team finishes, or run hive update now`. When Hive detects it is running from a Homebrew-owned path, the command in the line becomes `brew upgrade hive`, per the baseline's ownership rule and Codex's verified practice. The line appears on the human commands — `hive claude`, `hive codex`, `hive status`, `hive quota`, `hive memory`, `hive watch`, `hive stop` — and never on `hive event` or the hidden app-server host command, which run inside agent turns where a surprise stderr line could pollute hook output. For `hive claude`/`hive codex` the line prints before the orchestrator launches, which is Hive's equivalent of a session-start banner and the last moment Hive owns the terminal.

`hive init` is the exception, and the most important surface. It is an idempotent request to resolve, create, attach, and focus a project — a session boundary, and the last moment Hive owns the terminal before an orchestrator takes it. It therefore always says something, including "could not check", because the user just asked Hive to start and deserves to know what it knows. Only there does silence become a worse answer than a line. Every other human command uses the passive notice below.

Frequency: the passive notice is shown at most once per 24 hours per new version, so a user running twenty commands a day sees one line a day, but a *newer* version resets the clock. Two refinements from the survey are worth their cost. First, per-version dismissal, Codex-style: `hive update skip` records the offered version and silences its notices until the next release. Second, a severity escalation Hive must set its own precedent for: when the manifest marks a release `securityCritical`, the notice ignores the 24-hour limit and the skip list, prints on every human command in yellow rather than dim, and says so — `hive 0.3.2 available — security release, run hive update`. A "you're N versions behind" counter was considered (npm colors by semver distance) and rejected: at Codex-like release cadence the number is noise, and the security flag plus the staged-and-waiting default already encode everything the count would imply.

The opt-outs mirror Claude Code's two levels because both levels are real: a config setting `updates.check = false` (or `HIVE_NO_UPDATE_CHECK=1`) stops background checks and notices while `hive update` keeps working; `HIVE_DISABLE_UPDATES=1` blocks even manual self-update for centrally managed machines. `NO_UPDATE_NOTIFIER`, the ecosystem-wide variable, is honored too — it costs one `||` and respects a convention users already export.

## `hive update`, precisely

The command family stays small; every peer that works well has fewer verbs than its users expect.

```
hive update                  check now, download+verify+stage, activate if safe
hive update check            check and report; exit 0 up-to-date, 10 update available
hive update status           version, commit, channel, staged/retained versions, last check, why-disabled
hive update 0.2.4            stage and activate an exact version (also the downgrade path)
hive update rollback         reactivate the retained previous version
hive update channel stable   switch channel (stable | beta); persists
hive update skip             silence notices for the currently offered version
hive update off | on         disable/enable background checks and notices
```

Claude Code splits pinning into a separate `claude install` command; Hive folds it into `hive update <version>` because a second installer-flavored verb earns its keep only when install and update genuinely differ, and under versioned directories they are the same operation: fetch, verify, stage, activate. `rollback` is sugar for `hive update <previous>` that requires no version lookup and is guaranteed local — it must work offline, since the moment you want it is the moment you distrust the network's newest offering. The baseline retains the previous version directory until the new one survives its grace period; `rollback` is the user-facing handle on that retention.

The interesting design is `hive update` while a team is live, and the answer is the quiescence contract made visible. The command always performs the safe part immediately — check, download, verify signature and hash, stage into `~/.local/share/hive/versions/<v>/` — and then tells the truth about activation:

```
$ hive update
hive 0.3.1 downloaded and verified.
3 agents live (leo, maya, sam) — 0.3.1 activates when the team finishes.
The running daemon and team are unaffected until then.
```

No flag forces activation over a live team. This was considered — `--now` with a scary prompt — and rejected: the daemon owns landing authority and approvals, "force" here means killing agents mid-write, and a user who truly wants immediate activation has an honest spelling already: `hive stop && hive update`. Making destruction a deliberate two-command act rather than a flag is the point. When the daemon is idle or not running, `hive update` activates immediately: it asks the daemon to prepare-for-update (or simply proceeds if no daemon is up), swaps the symlink, starts the new daemon, runs the bounded health check, and reports `hive 0.3.1 active` — or, on health failure, reports the automatic revert and keeps the staged version for diagnosis. When an update was staged by the background path and the team later drains, the daemon performs the same activation at quiescence on its own; the next CLI invocation's handshake sees the new daemon and the next notice slot says `hive updated to 0.3.1` once, closing the loop the way Claude Code's silent next-start activation never quite does.

Because activation can happen between two commands the user types, every command already carries the wire handshake from the versioning section; the mixed-version window (new CLI, old daemon, or the reverse) is N/N−1-compatible by policy, and the refusal message for a genuinely incompatible pair names the fix rather than failing mysteriously.

## The v1 cut

The wanted experience is two features: the dim line, and `hive update`. Version 1 ships exactly the slice that makes both honest:

- Build-time version embedding (`--define`), `hive --version`, the five constants, schema version in `meta`, and the `/health` + handshake extension. This is a prerequisite for everything else and useful even before any updater exists.
- The daemon-side manifest check (start + jittered 24h), staged verified downloads, `meta` state, and the CLI notice with TTY/CI gating, 24-hour rate limit, skip list, security escalation, and Homebrew detection.
- `hive update` with `check`, `status`, exact version, `rollback`, `channel`, `skip`, `off/on`, and quiescence-gated activation with health check and auto-revert, per the baseline contract.
- One channel live at launch (`stable`), with `beta` existing in the manifest format from day one even if unpopulated — channel switching is cheap to build now and disruptive to retrofit.

Deferred, deliberately: delta updates (Deno-style patches are bandwidth polish; full binaries are tens of megabytes); percentage-staged rollouts (the deterministic install-ID hash from the baseline — worthless below a few hundred installs); the LaunchAgent dormant-machine push (resolved out of v1 by the user's preference; the promise is check-and-notify while Hive is in use); enterprise floors/ceilings à la `requiredMinimumVersion` (no enterprise users yet); `CLAUDE_CODE_PACKAGE_MANAGER_AUTO_UPDATE`-style opt-in brew automation (say `brew upgrade hive` and stop); and any notice inside the orchestrator session itself.

Signing is the gap, and it is a real one. Nothing on this list is the updater code itself: an Apple Developer ID and a signing pipeline whose hardened-runtime entitlements are *proven* to pass notarization on a real compiled Bun binary (the baseline flags Bun's JIT entitlements as unverified risk); the notarization step wired into CI with Gatekeeper verification on a clean machine; the offline Ed25519 manifest key, its embedded public key, and the signed channel endpoint.

The original plan was to ship no updater at all until those existed — only the notice, pointing at the install script, because telling a user that a version exists requires the signed manifest and not the self-mutation machinery. That lost to a simpler observation: the notice tells the user to run an install script that performs exactly the same unsigned download the updater would, so withholding `hive update` did not withhold the risk. It only moved it somewhere less careful. `hive update` therefore ships, verifying every artifact's SHA-256 against the manifest and executing the candidate binary before activation, and it *says out loud* on every run that the release carries no Hive signature and that its integrity rests on GitHub's immutable release plus TLS. The cost is honestly stated: a compromised GitHub release reaches an opted-in installation with nothing but TLS between it and the user, which the manifest signature exists to prevent. `verifyManifest` is written, tested, and fails closed the moment a public key is embedded — a stripped signature is a refusal, not a downgrade. The unsigned window is a deliberate, loud, temporary state, not a design.

What did ship: build-time version embedding and `hive --version`; the handshake extension and its build hash; SHA-256-verified staging into immutable version directories; atomic symlink activation with a health check and automatic revert; `hive update` with `check`, `status`, `rollback`, and `skip`; the stale/foreign/busy daemon distinction and the restart path; and the notice on `hive init`. What has not: the daemon-owned background check, automatic activation at quiescence, `channel`, `off`/`on`, and the signed channel document. [docs/versioning-and-release.md](../docs/versioning-and-release.md) tracks those as open.

## Open decisions

1. **Beta-channel population.** The manifest format carries `beta` from day one, but whether Hive actually publishes beta builds before 1.0 — and who consumes them besides the author — is unresolved. Publishing a channel with one subscriber is ceremony; not publishing means the first channel switch happens untested.
2. **Security-notice severity source.** The `securityCritical` manifest flag is release-author judgment with no external check. Whether it should require something stronger than one person's tag — a second approver, or a linked advisory — before it overrides every rate limit on every install is a process decision, not a code one.
3. **Post-activation confirmation placement.** The one-shot `hive updated to 0.3.1` line assumes the next command comes soon after quiescent activation. If the daemon activates and then idles for days, the confirmation may arrive puzzlingly late or read as noise. Showing it only within some freshness window (say 48 hours) is plausible but unvalidated.
