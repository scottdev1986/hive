GAPS CLOSED BY THE NEW SUITES

- Mouse: before this bundle only any-motion (1003) and SGR (1006) had any
  test. Recorded now: X10 (9) reports a press and never a release; VT200
  (1000) reports both edges with the button-3 release sentinel; SGR
  discriminates the release by final byte 'm' instead, preserving the button;
  Shift+click under an active mouse mode writes zero bytes.
- Bracketed paste: only DECSET 2004 was tested, so a build that bracketed
  unconditionally passed. Recorded now: DECRST stops bracketing while a safe
  single-line body still reaches the encoder verbatim.
- Safe paste: with 2004 reset, a MULTILINE body is withheld entirely — an
  embedded newline would submit to the shell unseen. Attribution control: the
  same body does paste when bracketed, so the withholding is safe-paste and
  not a dead clipboard reader.
- Kitty stack: the pre-existing suite pinned PUSH only, so a terminal that
  latched Kitty mode permanently and ignored every pop passed it. Recorded
  now: CSI < u restores the legacy golden, two pushes require two pops (a
  boolean masquerading as a stack fails), and CSI ? u reports flags that
  TRACK the push — asserted as a round trip (baseline differs from pushed,
  and a matching pop returns to baseline) so a terminal that answers a
  constant fails.
- Special keys: the four arrows are pinned to CSI A/B/C/D; every navigation
  and function key encodes a non-empty, mutually distinct sequence. Non-empty
  is precisely the assertion that would have caught this codebase's earlier
  "zero writes for special keys" blocker.
- Mouse coordinate formats: DECSET 1005 (UTF-8) and 1015 (urxvt) had no rows.
  Recorded at column 125, NOT the column 2 the other mouse rows use — at
  column 2 the 1005 encoding is byte-identical to the default format, so a
  row recorded there would pass against a terminal that ignored 1005
  entirely. The suite positive-controls the COORDINATE CHOICE by asserting
  the two formats do coincide at the low column.
- Client unknown semantics: an unknown input receipt fences the client, so no
  further INPUT_SUBMIT is authored. See the matrix doc, "Row 10", for why
  fencing rather than resending is the correct reading of the clause.

CONTROLS

Each new suite brackets its reader through the same predicate the rows use:
- mouse: an unmoded gesture is silent AND an enabled mode is loud, so silence
  is attributable to the mode rather than to a dead callback.
- paste: set and unset 2004 must return DIFFERENT transcripts through the one
  reader, so a reader blind to the mode fails.
- kitty: the legacy and pushed goldens must DIFFER, or every push/pop row
  would pass vacuously whatever the terminal did.
- special keys: a plain printable key must be observable through the same
  reader, or the non-empty rows would be measuring nothing.
- unknown: a healthy receipt must permit a second submission, so the zero in
  the unknown row cannot come from a reader that never observes one.

Exact-count and wrote-nothing rows settle the run loop AFTER the expected
writes land, so silence is measured rather than assumed. Key-encoder bytes
are delivered asynchronously by the surface io thread, so every key row
drains rather than snapshots — a synchronous read is the documented
measurement error behind the earlier false "zero writes" report.
