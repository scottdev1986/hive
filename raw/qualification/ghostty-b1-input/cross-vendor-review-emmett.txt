FINDINGS

  F1 (RECORD ACCURACY — required correction, not an author defect)
  The "2x backing-scale mouse BUTTON coordinate regression donna fixed" has no
  production-code component. The coordinate expression
    engine.sendMousePos(x: p.x, y: bounds.height - p.y, ...)
  is byte-identical at 89969341 and at e6606a35; no committed revision ever
  scaled mouse coordinates. Gate 8 refactored forwardMouse to return Bool and
  extracted forwardMousePosition, leaving the arithmetic untouched, and added a
  guard test for behavior that was already correct.
  donna's own evidence claims COVERAGE ("the 2x backing-scale button path"),
  not a fix, so the "regression fixed" framing appears to have entered upstream
  of the author. The record should be corrected; nothing needs repairing in the
  code.

  F2 (REAL DEFECT, INTRODUCED BY THIS GATE — required follow-up)
  HiveTerminalView+Input.swift:398-400, firstRect(forCharacterRange:):
    viewRect.origin.x += viewRect.width * Double(range.location + range.length)
  Upstream advances by cellSize.width (SurfaceView_AppKit.swift, the #8493
  dictation fix). Two compounding divergences, both confirmed in engine source:
    - Surface.zig imePoint returns width = preedit_width * cell.width, so a
      multi-cell composition over-advances by the preedit width.
    - Surface.zig scales x, y and height by content_scale but deliberately does
      NOT scale width ("we don't apply content scale here ... on purpose"), so
      an unscaled backing-pixel width is added to a point-space origin —
      doubled at 2x.
  Correct only in the common single-cell-at-1x case. Newly introduced: the
  range.length == 0 block does not exist at 89969341. Enshrined by
  InputEncodingTests.swift:875-887, which asserts the wrong shape and so can
  never go red. Impact is IME candidate-window / dictation-indicator placement
  — cosmetic, no data or security consequence — but it is on the very axis this
  gate qualifies, and the weak multi-stage CJK coverage (F4) is why it survived.

  F3 (UNDOCUMENTED DIVERGENCE — pre-existing, carried through a rewrite)
  insertText omits upstream's `guard NSApp.currentEvent != nil else { return }`
  ("We must have an associated event"). The synchronous double-injection guard
  (keyTextAccumulator) is correct and I could not break it on any traced path —
  ctrl-modified, function, dead-key, IME-commit, encodeKey. But the eventless
  subset upstream rejects — dictation, Character Viewer, Services menu, an IME
  committing on a later runloop turn — reaches engine.sendText unguarded here.
  The guard was already absent at 89969341, so this is not a Gate-8 regression;
  Gate 8 did rewrite this function and did not restore it, and no "Not ported"
  comment marks the omission the way the file does elsewhere.
  Note: the committed tests call insertText directly under XCTest, where
  NSApp.currentEvent is nil — so restoring upstream's guard would turn several
  Gate-8 tests red. The test harness depends on the divergence; that coupling
  should be resolved deliberately, not by quietly keeping the gap.

  F4 (COVERAGE GAPS — recommended)
  - CJK "multi-stage" composition is single-stage: one setMarkedText(prefix(1))
    then commit, never through handleKeyDown. RTL is byte-identity only, with
    nothing RTL-specific. Dead keys and emoji/ZWJ are genuinely well covered.
  - Every Kitty golden calls encodeKey/flagsChanged directly; the real
    keyDown -> handleKeyDown -> interpretKeyEvents accumulator path has no
    real-engine byte coverage.
  - Goldens depending on characters(byApplyingModifiers:) re-translate through
    the runner's active keyboard layout and would go red on a non-US default,
    which undercuts the file's own layout-independence seam.
  - Non-load-bearing assertions: InputEncodingTests.swift:848-857 asserts only
    isFinite while its name claims a bounds fallback; :222-250 place every
    assertion inside the interpret closure and go green if it is never invoked;
    :252-261 is negative-only with no positive control.
  - pressureChange's stage latch is never executed by a test (the seam is called
    directly), and momentum never reaches scrollWheel through a real NSEvent.

  F5 (DEAD CONTROL — pre-existing, worth a comment fix)
  pendingAuthoringHeld is written twice and read nowhere in the entire workspace
  tree (grep-complete: decl HiveTerminalView.swift:69, writes +Input.swift:160,
  162). The comment "§22: first authoring key held until claim" describes gating
  that does not exist — handleKeyDown runs unconditionally. Behavior is
  unchanged from 89969341, where encodeKey was likewise called in both branches,
  but Gate 8 removed the comment that used to explain the gap ("production waits
  for CLAIM_RESULT"), leaving a claim of a control with no implementation.
