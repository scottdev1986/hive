# hive

## What this is

You `cd` into a project and type `hive claude`. A terminal opens with an AI orchestrator in it, scoped to that folder. You talk to it like a tech lead: "build this feature," "figure out why the tests are flaky," "have Codex take a second look at the auth code." It doesn't write the code itself. It decomposes the work, spawns agents — Claude Code here, Codex there, big model for the hard part, cheap model for the changelog — and each agent appears in its own terminal window where you can watch it work. Agents talk to each other when they need to, regardless of vendor. When the work is done, it gets merged, and the windows go away.

Why build this when 2026 is drowning in agent orchestrators? Because everything that exists is one of two things. Human-driven multi-tool: Conductor, Vibe Kanban — they'll run Claude and Codex side by side, but *you* are the orchestrator, dragging cards and assigning tasks. Or AI-driven mono-tool: Claude Code's agent teams — an AI lead that decomposes and delegates, but only ever to other Claude sessions. Nobody ships the intersection: **an AI orchestrator that routes work across vendors, picks the right tool and model per task, and lets its agents message each other across the vendor line.** That intersection is hive. Everything else in this document — worktrees, tmux, sandboxes, windows — is table stakes we build because we must. The cross-vendor brain and the cross-vendor nervous system are the product.

One honest strategic note before the architecture: we're building in the gap between Anthropic and OpenAI, and either could close it — agent teams adding non-Claude members would hurt. Our defensible ground is vendor neutrality, so the roadmap leans into adding a third tool early rather than polishing a two-tool duopoly. Speed matters. macOS only, by choice, for now.

## How it works

The design fell out of twelve questions, each of which had a way to go wrong. Walking them in order, because each answer constrains the next.

### 1. How agents talk to each other

The problem nobody states out loud: Claude Code and Codex are turn-based loops. They act, then stop and wait for input. Nothing interrupts them mid-task, and nothing polls while they sit idle. So "agents can message each other" needs two mechanisms, not one — a place for messages to live, and a way to *wake the recipient*.

The place to live: both tools speak MCP (Codex via `config.toml`, stdio or HTTP), and MCP is the only API surface identical across vendors. So hive runs a small local daemon that owns a SQLite message store and exposes itself to every agent as an MCP server: `hive_send`, `hive_inbox`, `hive_list_agents`. Same send path whether you're Claude or Codex.

Spawn and send can race: an orchestrator may name a recipient and another agent may send to it before worktree creation has produced an agent row. Hive therefore durably reserves the name in SQLite at the synchronous start of spawn. Sends to a reserved name are accepted into the same durable message table, then delivered on the recipient's session-start event or claimed by its first `hive_inbox` poll. The reservation is released whether spawn succeeds or fails. A name with neither a live agent row nor an active spawn reservation is rejected explicitly as unknown; terminal (`done`, `dead`, or `failed`) recipients are rejected too. This keeps typos from creating immortal orphan queues while closing the legitimate startup window.

The wake-up: tmux. Ordinary messages are `normal`: if the recipient is idle at its prompt, the daemon injects "📨 message from maya: …" into its pane; if it's mid-task, the message queues and a turn-boundary hook drains the inbox. Event-driven — no polling anywhere. The injection itself is `load-buffer`/`paste-buffer -p` (the body arrives as one bracketed paste, so embedded newlines become composer newlines instead of premature submits, and nothing passes through a shell), then a pause, then a single `send-keys Enter`. The pause is load-bearing: both TUIs coalesce input that arrives in a burst as a paste, so an Enter fired immediately after the text lands *inside* that window and becomes a literal newline — the message sits in the composer until a human presses Enter, which is exactly the failure the first field test hit. The obvious simpler shape — `send-keys -l` with the text and an immediate Enter — lost twice over: the Enter race above, plus raw keystroke newlines submitting partial messages. A verify-and-retry loop (capture the pane, re-send Enter if the text is still sitting there) was rejected because misreading a busy pane double-submits; one delayed Enter is the ecosystem-converged fix and duplicates nothing.

Directions that reduce authority cannot ride that ordinary queue. The incident that forced the distinction was simple: an agent began one long implementation turn, the user changed the assignment to "pause before coding," and Hive faithfully stored the message while the agent kept writing because the next turn boundary never came. The model did not disobey; it never saw the instruction. Messages therefore have explicit priority and intent. `urgent` waits for the nearest safe lifecycle boundary, injects before ordinary traffic, and requires an agent acknowledgement by a durable deadline. `critical` covers pause, stop, cancel, and write reduction: the daemon atomically advances the agent's capability epoch, revokes landing, denies its stale pending approvals, then kills only that agent's tmux session, preserving its worktree and transcript. It restarts the same identity and worktree in a fresh read-only process with the control at the top of context. The interruption also settles the run's quota reservation the way any stopped run settles (decision 6: started runs keep their conservative estimate), and swaps the agent's viewer — now a lens on a killed session — for a fresh window on the restarted process, re-tiling the wall (decision 10). The agent must acknowledge the matching message and epoch; stale acknowledgements cannot revive authority. Explicit structured intent wins. A deliberately narrow legacy phrase classifier catches old callers saying exactly "stop," "cancel," "pause before coding," or "do not modify" without turning conversational uses of those words into process kills.

Every message carries a monotonic recipient sequence and the durable states `queued`, `injected`, `agent-acknowledged`, and `applied`, with separate timestamps. `deliveredAt` survives as a compatibility alias for injection, but the send result exposes the real state; persistence is never reported as delivery. Optional sender idempotency keys make retries exact. Urgent and critical deadlines wake the orchestrator once when missed, and restart recovery resumes queued critical controls before normal work. This costs a process restart for the strongest control and cannot undo a shell command that finished before revocation, but it closes the dangerous future: after revocation the old process is gone, the replacement is read-only, and the daemon refuses a stale landing epoch. Prompt-only interruption was rejected because it repeats the original failure. Injecting critical text into an approval dialog was rejected because a TUI prompt is not a safe message boundary.

The root orchestrator uses the same substrate without becoming a fake agent. `orchestrator` is a reserved destination, absent from the agents table, and the root lives in the fixed attachable tmux session `hive-orchestrator`. An agent send to that destination durably inserts the full message, then injects one compact `hive.message` envelope into the root session. Successful injection acknowledges the row once; a missing root session leaves it unread for an explicit inbox recovery. Delivery and inbox claims share one serialized lane, while SQLite insertion order supplies a stable order for concurrent sends. The injected envelope is capped at 2 KiB and contains only id, sender, time, a bounded body preview, and a `hive_read_message` reference. Oversized reports therefore stay in SQLite until the orchestrator deliberately asks for one; full team state, pane output, task restatements, and previously acknowledged messages never ride along with a wake.

Alternatives we rejected: a pure file inbox ("agents check for messages") fails exactly when it matters, because an idle agent never checks. Routing everything through the orchestrator as a relay makes it a bottleneck and fills its context with traffic. What we give up: agents must run inside tmux. Which turns out to be a gift, not a cost — see below.

### 2. How the orchestrator knows anything

An agent in another terminal is an opaque process. The spec-fantasy version — "the orchestrator knows when agents are done, stuck, or running out of context" — needs real signals. They exist:

- Claude Code has lifecycle hooks: `Stop` fires on every turn end with the session's transcript path; `Notification` fires when the agent is *waiting on an approval* (that's the stuck-at-a-prompt signal); `SessionStart` announces birth.
- Codex has `notify` (fires a program on `agent-turn-complete` with JSON) and, as of this year, a broader hooks system. One sharp edge learned the hard way: Codex ignores `notify` in project-local `.codex/config.toml` and skips project config entirely for untrusted directories — so hive passes `notify` as a spawn-time `-c` override and pre-trusts each worktree the same way, rather than writing config into the worktree and hoping.
- Context usage is readable from disk: both tools write per-message token usage to their transcript/rollout files. No guessing.

So: at spawn, hive writes hook configs into each agent. Agents self-report into the daemon; the daemon maintains a live table — agent, status, last event, context %. A slow daemon-side `tmux capture-pane` heartbeat (~60s) remains a fallback stuck-detector for whatever hooks miss (hung tool call, dead process), but that maintenance never injects anything into the root.

Birth is the fragile part, and the first field test proved it: all four inaugural agents died in their first second (unsupported models, a broken CLI) while hive reported "spawning" forever and left orphaned windows. So "spawning" is a monitored phase, not a fire-and-forget: for 15 seconds after session creation the daemon watches the pane — session vanished or a launch-failure signature printed means status "failed" with the pane's last lines as the recorded reason, the session killed, and the fresh worktree cleaned up; the first hook event short-circuits the watch as proof of life. A 30-second reconciliation sweep separately marks any live-status agent whose tmux session has disappeared. The alternative — trusting that a created session equals a running agent — is exactly the silent-death mode the field test exposed, and "failed" with a captured reason is a first-class status precisely so the orchestrator can see it and retry a tier up. The orchestrator uses the daemon's status tool only on demand and never scrapes a terminal. The alternative — orchestrator eyeballing capture-pane dumps — burns its context on screenshots and turns the conductor into a babysitter.

The orchestrator is idle by default. It does not poll `hive_status`, `hive_inbox`, panes, logs, or worktrees. Only a user turn or a new message to the reserved destination starts a root turn. `hive_status` is an explicit user-driven read: the root requests its compact `active` view, which excludes terminal agents and omits paths, branches, and unbounded task text. The omitted detail preserves the full legacy response for the CLI and existing MCP callers. One known wart: Claude Code's Stop-hook `transcript_path` can go stale (anthropics/claude-code#8564); the daemon reads the most-recently-modified `.jsonl` in the project dir instead of trusting it.

Identity rides on this table too, and agents get **human first names** — maya, david, sam — drawn from a pool at spawn, never numbers. Everything keys off the name: tmux session `hive-maya`, worktree `.hive/worktrees/maya`, branch `hive/maya-auth-api`, the status row, message envelopes. The reason is that the user's entire interface is conversation with the orchestrator, and conversation wants names: "tell maya to reuse the middleware" and "what's david doing?" work; "message agent-3" forces the user to keep a mental numbering table the tool should be keeping for them. Numbered agents were the obvious default and lost for exactly that reason. Task-derived names ("auth-api") lost too: two agents can share a feature, and a name describing the task goes stale the moment the agent is respawned onto follow-up work — the worker's identity should outlive any one assignment. The cost is some whimsy in professional logs and a pool large enough to never collide within a session; both trivial.

### 3. How agents avoid destroying each other's work

Three agents editing one checkout doesn't produce merge conflicts — it produces silent overwrites, agents reading half-written files, and everyone fighting over `.git/index.lock`. The industry answer is unanimous and we take it: **worktree per writing agent.** Each writer gets its own branch and checkout (`.hive/worktrees/maya`, branch `hive/maya-auth-api`) sharing one object store. Read-only agents (research, review) share the main checkout free of charge.

But isolation is the floor, not the strategy. The real conflict-avoidance is planning, which is exactly the orchestrator's job: land shared interfaces on the base branch *before* fanning out, and give every agent an explicit file scope ("you own `lib/auth/`"). Merges still happen serially — maya lands, then david rebases against a world that includes maya's work — but merging is nobody's *separate* job: **every writer lands its own finished work on main.**

The reason is a failure mode we hit for real: task done, tests green, and the branch sitting in `.hive/worktrees/` forever because merging was somebody else's step. Completed work stranded on a worktree branch is indistinguishable from work never done. So the landing protocol rides in every writer's spawn prompt — it's composed into `buildAgentPrompt`, so no orchestrator ever has to remember to say it: when the task is done and tests are green, commit everything, `git rebase main` in the worktree, re-run the tests on the rebased branch, then call `hive_land` with the capability epoch issued at spawn. The daemon verifies that the epoch is still current and write authority has not been revoked before it performs `git merge --ff-only` in the primary checkout. Fast-forward-only is the whole trick for concurrency: the merge succeeds only if main hasn't moved since the rebase, so two agents landing at once can't interleave — the loser's merge is simply rejected, and it re-rebases and retries, three attempts before escalating instead of spinning. A rebase *conflict* means the planning failed and two agents genuinely touched the same code; the agent runs `git rebase --abort` and escalates to the orchestrator, which spawns an integrator agent — a judgment merge is a task, never a reflex, and nothing is ever forced. Red tests never merge, in any state, on any attempt. Direct primary-checkout merge instructions lost because they left no reliable way for a later critical control to revoke landing.

Alternatives that lost. Routing *all* merges through a spawned integrator agent was the first design: it kept writers pure and matched "the orchestrator can't write" (decision 11) cleanly, but it made landing asynchronous and optional — the integrator existed only if the orchestrator remembered to spawn it, which is exactly how green work strands. Integrators survive as the escalation path (conflicts, stranded recoveries), not the default. Plain merge commits instead of rebase-then-ff lost too: a merge commit means the tests you ran validated neither parent's final tree, while ff-only makes "the tests passed on exactly what main now is" structural. The accepted cost: the landing merge is the one sanctioned write outside an agent's worktree — a deliberate pinhole in the decision-4 sandbox. It's tolerable because ff-only moves a ref without rewriting history and git refuses it when the primary checkout has conflicting uncommitted state; for a sandboxed Codex agent the command surfaces in the approval queue like any other boundary crossing, one approval per landing, safely auto-approvable by policy.

Death gets the same guarantee: work is either merged or explicitly surfaced, never silently lost. `hive_kill` assesses the agent's branch and worktree before any cleanup — unmerged commits or uncommitted files make it refuse deletion, keep everything, and report the stranded work in its result, so the orchestrator must actively decide: spawn an integrator to land it, or pass `discardWork` to delete it on purpose (the discard is still reported, as the record of what was chosen). If the assessment itself fails, the worktree is kept — nothing is deleted on a guess, though the kill and the dead status always win. Cleanup after a successful landing is safe by construction: a landed branch tip is an ancestor of main, so deleting the branch and worktree deletes nothing.

The "david needs maya's work-in-progress" case gets a real mechanism instead of a hand-wave: maya commits to her branch and messages david "pull `hive/maya-auth-api` at abc123." Shared object store makes it instant. Uncommitted state is never shared; that way lies madness.

Consequences accepted: hive requires a git repo (we `git init` or degrade to single-writer if there isn't one), and practitioner consensus says 2–4 parallel writers is where coordination overhead starts eating the parallelism gains — "as many agents as it needs" has a soft ceiling and the orchestrator should know it.

### 4. Who answers the permission prompts

N agents in N windows each stopping for "allow this command?" gives you a fork: babysit every window (autonomy dies) or blanket `--dangerously-skip-permissions` (N unattended agents with shell access on your real laptop). Both tools now offer a third way, and hive builds on it:

**Sandboxed autonomy, one escalation queue.** Codex agents run `sandbox_mode = "workspace-write"` scoped to their worktree with `approval_policy = "on-request"`. Claude agents run auto mode (or `acceptEdits` plus a hive-maintained allowlist in the worktree's settings). Inside their box, agents never prompt. The rare escalation flows through the daemon — Claude's `Notification` hook, Codex's approval events — and surfaces in exactly one place: the orchestrator window. "maya wants to run `npm publish` — approve?" You supervise one window. The agent windows are for *watching*, not controlling.

The orchestrator can auto-resolve escalations by a policy you set at spawn ("network installs fine, nothing outside the repo, never push"), but anything irreversible or outward-facing — publish, push, deploy, spend — always reaches you. And there's a pleasant unification here: the sandbox boundary *is* the scope enforcement from decision 3. An agent physically cannot write outside its worktree, so "assigned file scopes" stops being an honor system.

### 5. What "memory" actually means

The original spec said "cross-platform memory" once and "keep every context as small as possible" twice, without noticing they fight: unbounded shared memory dumped into every prompt is how contexts bloat. The fix is realizing "memory" was three different things wearing one name:

- **Conventions** — how this codebase works, style, commands. Lives in a committed `AGENTS.md`. Both tools load it natively; zero hive machinery. If every agent always needs it, it goes here and nowhere else.
- **Run state** — who's building what right now, task assignments, "david finished the API." Lives in the daemon's SQLite and dies with the hive session. Calling this "memory" was the conflation to kill.
- **Durable knowledge** — gotchas discovered, decisions made, "the flaky test is X." Markdown facts, one per file, plus a short index. Two scopes: **per-repo** (`.hive/memory/`, committed, travels with the clone) and **global** (`~/.hive/memory/`, where cross-project lessons accumulate so the tool improves with every project). Exposed via daemon MCP as `memory_search` / `memory_write`; the daemon serializes writes; every fact carries a date; the orchestrator prunes stale ones as housekeeping. Promoting a repo lesson to global is a file move plus re-index.

The context-budget rule that reconciles memory with small contexts: agents are injected with the merged **index only** — a few hundred tokens — and pull full facts on demand from whichever scope owns them. The store never gets dumped into a prompt. Build-vs-buy note: open-source local memory servers (agent-memory, agentmemory) are close enough to this design that we may adopt one for the durable layer rather than build; the daemon integration is the only custom part either way.

### 6. Who picks the model

The trap: "the orchestrator is smart enough to know which model." No, it isn't — an LLM's pricing and capability knowledge froze at its training cutoff, and model lineups turned over twice in the last year. An orchestrator routing from memory confidently picks superseded models forever.

The split that works: **the orchestrator classifies; a table resolves.** The orchestrator's judgment call is "this subtask is `deep` / `standard` / `cheap`" — a thing LLMs are genuinely good at and that never goes stale. The table is a **tier × tool matrix**: each tier names a preferred tool plus a model-and-effort entry *per tool* — `cheap` prefers Codex at low effort, but also records that a cheap Claude agent means Haiku. That second dimension exists because the orchestrator can override the tool at spawn time ("use a Claude agent for this"), and the first shape we shipped — one tool+model per tier — had nothing to say then: the override discarded the model and a "cheap" Claude agent silently launched on the account's default (expensive) model. A route that can't survive a tool override isn't a route, it's a suggestion. Mechanically this is just spawn flags: `claude --model …`, `codex -c model_reasoning_effort=…`.

A corollary the first field test taught us: **routes must not assume account entitlements.** Naming a concrete Codex model in the defaults killed every Codex spawn on a ChatGPT-plan account ("model not supported"). The default Codex column therefore says model `"default"` — pass no model flag and let the CLI use whatever the account actually has, which is supported by definition; effort flags still steer cost per tier. The Claude column names model *aliases*, which every Claude plan resolves and which auto-update as models ship: `deep` uses `best` (Claude Code resolves it to the most capable model the account can reach — entitlement-adaptive by construction, exactly this principle as a vendor feature), `standard`/`review` use `sonnet`, `cheap` uses `haiku`. We considered pinning `deep` to `opus` instead — more predictable, never subject to `best`'s safety-classifier fallback — but it caps a top-tier account below the best model it's paying for. Pinning exact model IDs is opt-in via `routing.toml`, where the user knows their own plan.

Aliases are for *launching*, not for *looking at*. The first shape stored the route's model string on the agent record verbatim, so `hive_status` and every terminal title said `best` or `default` — which tells the user nothing about what's actually running and costing money. So at spawn time hive resolves the concrete model once and stores *that* on the record: a concrete route model passes through; `default` is read from the tool's own user config (`~/.codex/config.toml`, `~/.claude/settings.json` — the very file the CLI consults when hive omits the flag); Claude's `best` maps through a constant verified against the installed CLI, because the alias is resolved inside the CLI and no local file records it. Only when a tool's config names no model does the alias survive on the record — the CLI's built-in default genuinely isn't knowable locally, and we'd rather show an honest alias than a guessed model ID. Querying the CLI live at spawn was rejected: it costs a real API call and seconds of latency per agent for a label. The spawn command itself still passes the alias, so entitlement-adaptive launching is untouched. Downstream of this, a terminal viewer's title is exactly `<name> — <model>` (`maya — claude-fable-5`) and nothing else — no task text, path, tier, or session name, which leaked into titles in the first field test and made a wall of windows unreadable. Terminal.app's custom-title property does not override the profile's username, process, and dimensions fields, so Hive installs a versioned, bundled `hive-agent-v1` profile on first use and reuses it for viewers. That managed profile disables every generated title component without mutating the user's existing Terminal profiles.

The table auto-updates so new releases aren't missed: hive fetches a **curated routing manifest** published with hive releases (provider model-list APIs give you names, not judgment — curation is the value), and your local `~/.hive/routing.toml` overrides always win. Escape hatches: pin a vendor for a session; bump a tier automatically when an agent fails or stalls — retry-with-a-bigger-model is cheap and effective. And because we're multi-vendor, the table can encode a genuinely novel policy: route *review* to the other vendor from whoever wrote the code.

But capability is only half of routing. Claude and Codex subscription accounts have rolling and weekly capacity, models can draw from different or shared pools, and two simultaneous spawns can spend the same apparent headroom unless selection owns a reservation. So the resolver is quota-aware: it resolves both vendor candidates, maps each concrete model to a configured `(provider, account, pool)`, estimates the task in normalized planning units, and atomically reserves those units before creating a worktree or process. The account and pool are explicit because keying only by model double-counts a shared vendor limit, while keying only by vendor cannot represent a premium model with its own weekly cap. A completed turn reconciles the reservation to provider-reported consumption when a hook supplies it and otherwise commits the conservative estimate; a launch that never starts releases it, while a started agent that dies keeps the estimate. Expired reservations receive the same started-versus-unstarted treatment after a daemon restart.

The uncomfortable truth is that neither consumer CLI promises a machine-readable subscription-quota API. Their supported surfaces expose per-run tokens or credits in some modes and show account limits in interactive settings, but that is not the same as an authoritative remaining-capacity feed. Hive therefore records provenance instead of inventing precision. A fresh provider or gateway observation is `authoritative`; a user-entered dashboard reading is `reported`; locally reconciled run usage is `estimated`; absent, stale, or malformed observations are named as such. Operators can reconcile a pool from `hive quota reconcile`, and later provider adapters can write the same observation shape without changing routing. The rejected alternatives were scraping TUI text — brittle, account-sensitive, and liable to silently parse the wrong number — and presenting local token sums as the vendor's exact limit. This design gives up automatic exactness on consumer plans in exchange for honest, auditable decisions.

The deterministic policy is intentionally boring. Start with the table's preferred vendor (or the other vendor for a review when its author is known), then the other candidate; reject candidates whose estimated run would exceed either configured window; for `cheap` and `standard`, also reject a candidate that would invade the configured deep-capacity reserve. Compare eligible candidates by the worst post-reservation headroom, then preference, then vendor name. An explicit user vendor stays fixed when safe. If it is unsafe, hive does not silently disobey: spawn returns the remaining five-hour and weekly capacity, expected reset where known, and the safe fallback. If both vendors are exhausted, no process starts. With no configured allowance, existing installations enter compatibility mode: hive records estimates and emits one deduplicated `unknown quota` warning, but preserves the old static route until the operator configures real account pools. Failing every old spawn after an upgrade was rejected as a migration strategy; pretending an arbitrary default allowance was real lost for the same reason.

Warnings use the ordinary durable orchestrator message path, not stderr in a daemon nobody watches. Crossing configurable warning or critical remaining-capacity thresholds sends one message containing the provider, account, pool, both windows, data confidence and freshness, and known reset times. Alert state is durable. It rearms only after headroom rises above the threshold plus hysteresis or a reset boundary passes, so parallel reservations do not produce a wall of identical warnings. Corrupt state fails closed for the affected configured pool, stale external observations are combined with the more conservative local ledger, and status reports the reason without requiring anyone to inspect SQLite.

### 7. What happens when a context fills up

Sensing was solved in decision 2; this is the actuator. Field evidence says quality dies long before the window fills — models start repeating themselves and contradicting locked-in decisions around ~140K tokens, well inside technical limits. So the trigger is a quality line (~65%), not "almost full."

First, prevention, which is the real mechanism: the orchestrator sizes subtasks to finish inside one healthy context, and agents spawn with a minimal briefing — task, file scope, memory index — never repo dumps. Most agents should retire at task end without ever seeing the ceiling.

When one does cross the line: the daemon notifies the orchestrator, which injects "finish your current step, commit WIP to your branch, write a handoff." The handoff is structured and machine-targeted — goal, done/remaining, decisions made, **failed approaches** — stored in run state. Hive kills the session and respawns the same agent identity in the same worktree with the handoff preloaded. Fresh window, continuous work.

We explicitly do not rely on `/compact` for anything that matters — lossy self-summarization drops constraints and failed-approach history, which are the two things a successor most needs. What we give up: warm-agent speed. A warmed-up agent that never re-reads the repo iterates fast, so we recycle at the quality line, not eagerly. Bonus loop: failed approaches from handoffs are prime candidates for the durable memory layer — that's how "the tool gets smarter" stops being a slogan.

### 8. Skills, and when to build one

The tailwind that makes this feature possible: Agent Skills became an open standard (agentskills.io, Dec 2025) and OpenAI shipped Codex support within 48 hours. One `SKILL.md` folder serves every agent in the hive regardless of vendor, and progressive disclosure means a large library costs agents almost nothing until a skill is actually used.

Storage mirrors the memory scopes: per-repo skills in `.hive/skills/` and global skills in `~/.hive/skills/`. At spawn hive links each skill directory into the launched tool's project-native discovery path — `.claude/skills/` for Claude and `.agents/skills/` for Codex — with a same-named repo skill winning over a global one. Links are per skill rather than one link over the whole native directory, so vendor-specific skills can coexist; a conflicting same-named native entry fails the spawn instead of being overwritten. A skill that exists only in a vendor-native directory stays vendor-specific: silently promoting it would turn a tool-local customization into a cross-vendor contract its author never chose. The cost is that shared skills must be moved to the canonical hive directory once. Treating all native directories as additional sources was rejected because two editable copies would inevitably drift while appearing to be one skill. Write once, every vendor reads it.

The hole in the original spec was *detection* — "the orchestrator knows when a skill should be built" requires memory of past runs, which a single session doesn't have. The daemon does: it logs every task descriptor. At session end the orchestrator reviews history with a **rule of three** — same task shape done ~3 times across runs → propose a skill. Propose, not create: "I've formatted release notes three times; want a skill for it?" On yes, a cheap-tier agent authors the SKILL.md from those transcripts. Silent skill accumulation is how you get sprawl nobody audits, so creation is human-gated and skills get the same pruning policy as memory — unused for N runs, propose deletion. At spawn time, the orchestrator names the relevant skills in the briefing rather than hoping agents discover them.

### 9. Windows are viewers, not containers

The original framing — "each agent runs in a terminal window" — hides a failure mode: if the process lives in the window, closing david's window kills david mid-task. Invert it: **agents live in detached tmux sessions owned by the daemon; windows are disposable viewers.** Spawning a window means telling iTerm2 or Terminal.app (via osascript) to run `tmux attach -t hive-maya`. Closing it detaches; the agent keeps working headless; `hive watch maya` reopens the lens.

This costs nothing — decision 1 already made tmux the substrate for message injection and capture-pane monitoring — and it buys robustness plus a free `--headless` mode (SSH, CI). The root also runs in a fixed tmux session and attaches to the user's current terminal, which gives the daemon one precise wake target without scraping or polling it. macOS only means two emulator adapters, auto-detected, overridable in config. Nothing in the tmux+daemon core is macOS-specific except these adapters, so the platform door stays open without a rewrite.

Disposable viewers still need disposal: the first field test killed agents whose windows stayed open, attached to dead sessions. The fix is identity, not search. When hive opens a viewer it records a **native terminal handle** on the agent row — iTerm2's globally unique session ID, or for Terminal.app the app PID plus window ID plus TTY, all of which must still agree (and the window hold a single tab) before anything closes. Terminal.app handle capture follows the exact tab object returned by `do script` to its owning window; it must not assume the asynchronously activated front window is the new one. `hive_kill` kills the tmux session first, then atomically marks the agent dead while claiming the handle, and best-effort closes exactly that viewer; if a concurrent spawn loses that race it closes its own orphan. Finding the window by its title, agent name, or a session-name text search was rejected outright: titles are mutable and shared, and any fuzzy match is a loaded gun pointed at some unrelated terminal the user has open. A close failure never blocks the kill, the dead state, or worktree cleanup, and manually attached terminals and old handle-less rows are untouched.

### 10. The window wall lays itself out

Decision 9 made windows cheap; cheap windows pile up. macOS opens every new terminal in a default-offset cascade, so by the fourth agent the screen is a deck of overlapping cards and the orchestrator — the one window that matters — is buried under the workers. The fix is that hive owns the geometry: **whenever the set of hive windows changes, the daemon re-tiles them into a grid with the orchestrator centered and largest.** Every viewer-set change already flows through the daemon — spawn attaching a handle, `hive_kill` claiming one, the reconciliation sweep declaring an agent dead, `hive watch` reopening a lens — so each of those also fires a coalesced, best-effort layout pass. No polling, no timer: the same event edges that maintain the status table maintain the wall. Two shapes cover reality. The preferred one gives the orchestrator a full-height center column (~40% of the screen) with workers stacked in symmetric side columns, the odd worker going right, sides splitting into more columns as rows run out; by construction the center column is always the widest window on screen. When worker cells would fall below a minimum readable size — too many agents, too small a screen — the layout degrades honestly to a uniform grid in which every window is equal and the orchestrator holds the most central cell. Layout math is a pure function (screen rectangle in, frames out), so every count, parity, and degenerate screen is unit-tested without ever touching a window.

The orchestrator window is the one window hive didn't open — the user typed `hive claude` into it — so it has no spawn-time handle. At launch, hive resolves the window it's running in by TTY (`TERM_PROGRAM` names the emulator, the emulator's scripting dictionary maps TTY to window) and registers that handle with the daemon; on exit it unregisters. In an emulator hive can't script — VS Code, Alacritty, an existing tmux — registration silently yields nothing and the workers tile the full screen instead: participation is opt-out by circumstance, never an error. Registration alone also never moves anything; the wall only starts arranging once the first agent viewer exists, so a user who never spawns agents never sees their window jump.

Moving windows is the same trust problem as closing them, and it reuses the same answer from decision 9: a window is repositioned only through its recorded identity handle (iTerm2 session ID; Terminal.app PID + window ID + TTY), and the move is a bare `set bounds` — never `activate`, so focus stays wherever the user put it. A vanished window makes its move a silent no-op and never blocks the rest of the pass. The sweep that marks a vanished session dead closes its orphaned viewer too, because a lens on a dead session shows nothing worth keeping. Alternatives that lost: tmux's own tiling (`select-layout tiled`) collapses all agents into panes of one window — free, but it un-decides decision 9 by making one close destroy every lens, and N readable panes fit one screen far worse than N windows hive can size. Delegating to a window manager (Rectangle, yabai) was rejected as a dependency most users don't have, with yabai needing SIP disabled. Making the user arrange windows was the status quo being fixed. The accepted costs: layout is osascript and therefore macOS-only (it lives entirely inside the decision-9 adapter boundary), it repositions the user's own orchestrator window by design (`layout = "off"` in config restores manual control), and the "orchestrator is largest" guarantee holds only while minimum worker sizes fit — beyond that the equal grid is the honest ceiling.

### 11. The orchestrator physically cannot code

"The orchestrator does no coding" is load-bearing: an orchestrator that helpfully edits files is an unsandboxed writer in the main checkout (breaking decision 3), burning its own context on diffs (breaking decision 7), outside the approval flow (breaking decision 4). And both CLIs *will* drift into coding if merely asked nicely. A system prompt is a suggestion; permissions are a constraint.

So it's mechanical: a Codex orchestrator launches with `--sandbox read-only`; a Claude orchestrator gets deny rules on Edit/Write/NotebookEdit and read-only Bash. Its only actuators are the hive MCP tools — spawn, send, on-demand status, bounded inbox recovery and message reads, approve, memory. Reading is encouraged (good decomposition requires browsing the code); writing is impossible. The prompt explains the role and the reflex: you want to fix that typo → spawn a cheap agent. No small-edit carve-outs — carve-outs are how invariants erode. A side benefit: a read-only orchestrator triggers almost no approvals, so its window stays clean for the three streams that matter — your conversation, concise agent envelopes, and the escalation queue.

### 12. The name for all of this

Stepping back, the twelve decisions compress into one architecture:

- **The daemon** is the keystone: message bus, status table, approval router, memory store, run history. Everything speaks to it over MCP, which is what makes hive vendor-neutral.
- **tmux** is the substrate: sessions are where agents live, send-keys is how they're woken, capture-pane is the fallback sensor, windows are optional lenses.
- **The orchestrator** is a read-only classifier: it decomposes, scopes, tiers, and spawns; it never touches files.
- **Agents** are sandboxed, worktree-isolated, hook-instrumented, and disposable — recycled at the context quality line via handoffs.

## Tech stack

One language for the whole system: **TypeScript on Bun, compiled to a single binary.** The decisive constraint is MCP — the daemon *is* an MCP server, and the official TypeScript SDK is the reference implementation: most mature, first to get spec features, runs on Bun. Bun buys the three things a CLI-plus-daemon needs: fast cold start, built-in SQLite (`bun:sqlite`, no native-module pain), and `bun build --compile` for a single distributable binary — `brew install hive`, not "have the right Node version." Everything hive does is process-glue and JSON, not compute, so a systems language buys nothing here.

Alternatives considered: Go is the right shape (static binary, daemon ergonomics) and would win if the MCP TS SDK didn't exist, but its SDK is younger and we'd write more code for the same result in a TS-heavy ecosystem. Rust pays systems-language tax on a scripting-language problem. Python has a miserable CLI distribution story.

The concrete choices:

- **Daemon MCP transport:** Streamable HTTP on localhost — one daemon, one endpoint, N agents. Both Claude Code and Codex configure HTTP MCP servers natively; stdio would mean a process per agent.
- **Storage:** SQLite via `bun:sqlite` — inboxes, status table, run history, one file under `~/.hive/`.
- **tmux / git / osascript:** shell out via `Bun.spawn`. No wrapper libraries; the CLIs are the API.
- **Hook glue:** the hive binary itself — Claude hooks and Codex `notify` invoke `hive event …`, which POSTs to the daemon. One code path, no curl scripts.
- **Daemon lifecycle:** on-demand spawn with a pidfile; first `hive` command starts it. launchd is v2 polish, not v1 plumbing.
- **Schemas:** Zod, shared across messages, handoffs, and the routing table (the MCP SDK is Zod-native). CLI parsing via Commander; tests via `bun test`, with the interesting coverage being integration tests driving real tmux sessions.

One deliberate boundary: emulator adapters (osascript for iTerm2/Terminal) and tool adapters (spawn flags + hook-config writers for Claude/Codex) live behind small interfaces from day one. Those are the only two places vendor and platform specifics are allowed, and they're exactly where v2 grows (new terminals, Gemini CLI).

## Roadmap: the dogfooding arc

The sequencing principle: nothing in v1 learns. Learning features compound on top of a working loop and are wasted effort if the loop isn't proven. So: build the core loop until it's usable, then **use hive to build hive**, then add the learning layer with the tool's own help.

**v1 — prove the thesis.** Daemon (SQLite, MCP: spawn/status/send/inbox, send-keys delivery). `hive claude` / `hive codex` launching a read-only orchestrator with hive MCP attached. Agent spawning: named agents (human first names from a pool), worktree + branch per writer, sandboxed, tmux session + iTerm2/Terminal viewer window, the window wall auto-tiled around a centered orchestrator. Hooks wiring both directions; context % from transcripts, warn at 65% (no auto-respawn yet). Tier × tool routing with durable quota reservations, honest telemetry confidence, and proactive limit alerts. Committed `AGENTS.md`. Single approval queue. Landing protocol in every writer prompt (rebase → retest → ff-only merge), integrator agents only for escalated conflicts and stranded work. Done means: you can sit in one window, ask for a feature, watch a Claude agent and a Codex agent build it in parallel, approve one escalation, and see it merged.

**v1.1 — make it durable (built by hive).** Handoff-and-respawn automation. The two-scope memory store with index injection. `hive watch` and reattach polish. Stuck-detection heartbeat escalations. This phase is the dogfood gate: if hive can't comfortably build these features itself, that's the most important bug report we'll ever get.

**v2 — make it self-improving.** Skills rule-of-three proposals and authoring. Auto-updating routing manifest. Cross-vendor review routing. Third tool (Grok) to cement vendor neutrality.

## Open questions and deferred bets

Named honestly, not papered over:

- **Codex-side signal fidelity, the remaining half.** The delivery mechanics are settled (spawn-time `-c` overrides for `notify` and trust — see decision 2), but detecting "Codex agent awaiting approval" and measuring Codex context usage from rollout files is still assumed, not proven.
- **Injection fragility.** Pasting into a TUI is inherently timing-sensitive; the paste-then-delayed-Enter shape (decision 1) is the ecosystem's converged workaround, and the Enter delay is a constant we picked, not a guarantee. If either CLI ships a real programmatic input channel, we take it immediately.
- **Task-shape similarity** for the skills rule-of-three is undefined — embedding similarity? Orchestrator judgment over descriptors? Deferred to v2 with the feature.
- **Build-vs-buy on durable memory.** The three-layer design is settled; whether layer three is built or adopted from agent-memory/agentmemory is a v1.1 decision.
- **The parallelism ceiling.** We accept the 2–4 writer consensus but haven't tested where hive's coordination actually breaks. The orchestrator should learn to say "this task doesn't parallelize."
- **The existential bet.** Agent teams going multi-vendor, or Conductor growing an AI orchestrator, both compress our gap. The counter is speed and neutrality — being the layer neither vendor can be. If that stops being true, hive is a very nice personal tool and that's the honest floor.
