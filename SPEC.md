# hive

> **Architecture scope.** This document records the current Bun/tmux/Terminal.app system and the control-plane decisions learned from dogfooding it. The accepted destination is the native [Hive Workspace blueprint](docs/architecture/hive-workspace-blueprint.md), with restart order in the [handoff](docs/architecture/restart-handoff.md). Where the documents differ on windows, provider TUIs, tmux, process ownership, project identity, or IPC, the Workspace blueprint is authoritative for future work. No flagship implementation is authorized before the documented restart; Phase 0 control-plane authentication is the first proposed task afterward.

## What this is

You `cd` into a project and type `hive claude`. A terminal opens with an AI orchestrator in it, scoped to that folder. You talk to it like a tech lead: "build this feature," "figure out why the tests are flaky," "have Codex take a second look at the auth code." It doesn't write the code itself. It decomposes the work, spawns agents — Claude Code here, Codex there, big model for the hard part, cheap model for the changelog — and each agent appears in its own terminal window where you can watch it work. Agents talk to each other when they need to, regardless of vendor. When the work is done, it gets merged, and the windows go away.

Why build this when 2026 is drowning in agent orchestrators? Not because nobody ships a cross-vendor AI orchestrator — that claim is dead. AgentWrapper's agent-orchestrator (8.2k stars) combines an AI orchestrator, mixed Claude/Codex workers, worktree isolation, a Go+SQLite daemon, and PR-feedback loops; Superharness plays in the same category with a genuinely different architecture. The rest of the field still splits the familiar two ways — human-driven multi-tool (Conductor, Vibe Kanban: they run Claude and Codex side by side, but *you* are the orchestrator, dragging cards) and AI-driven mono-tool (Claude Code's agent teams: an AI lead that only ever delegates to other Claude sessions). What none of them ship — and what separates a demo from a tool you can walk away from while N agents hold shell access on your laptop — is the safety and control stack: **quota-aware tier×tool routing that durably reserves subscription-CLI capacity before spawning, durable peer delivery with critical controls that mechanically revoke authority rather than politely request compliance, an orchestrator that physically cannot write, and a rebase→retest→fast-forward-only landing protocol that makes "the tests passed on exactly what main now is" structural.** That stack is hive. Worktrees, tmux, sandboxes, windows are table stakes we build because we must; the control plane over cross-vendor agents is the product.

One honest strategic note before the architecture: we're building in a gap several parties could close — Anthropic or OpenAI shipping cross-vendor teams, or agent-orchestrator growing a real control plane. Our defensible ground is the safety stack plus vendor neutrality, so the roadmap leans into adding a third tool early rather than polishing a two-tool duopoly. Two things we watch without imitating: Cotal, an emerging cross-harness messaging standard (174 stars, pre-1.0, no confirmed Codex adapter) — if it matures, decision 1's daemon becomes an adapter target, not a casualty — and the learning layer, where the field is thinner than its marketing: Devin's knowledge store is manually curated, not an automatic loop, and Ruflo's cross-run learning claims are unverified, so the v2 self-improvement ambition is less pre-empted than a feature-matrix read suggests. Speed matters. macOS only, by choice, for now.

## How it works

The design fell out of fourteen questions, each of which had a way to go wrong. Walking them in order, because each answer constrains the next.

### 1. How agents talk to each other

The problem nobody states out loud: Claude Code and Codex are turn-based loops. They act, then stop and wait for input. Nothing interrupts them mid-task, and nothing polls while they sit idle. So "agents can message each other" needs two mechanisms, not one — a place for messages to live, and a way to *wake the recipient*.

The place to live: both tools speak MCP (Codex via `config.toml`, stdio or HTTP), and MCP is the only API surface identical across vendors. So hive runs a small local daemon that owns a SQLite message store and exposes itself to every agent as an MCP server: `hive_send`, `hive_inbox`, `hive_list_agents`. Same send path whether you're Claude or Codex.

Spawn and send can race: an orchestrator may name a recipient and another agent may send to it before worktree creation has produced an agent row. Hive therefore durably reserves the name in SQLite at the synchronous start of spawn. Sends to a reserved name are accepted into the same durable message table, then delivered on the recipient's session-start event or claimed by its first `hive_inbox` poll. The reservation is released whether spawn succeeds or fails. A name with neither a live agent row nor an active spawn reservation is rejected explicitly as unknown; terminal (`done`, `dead`, or `failed`) recipients are rejected too. This keeps typos from creating immortal orphan queues while closing the legitimate startup window.

The wake-up depends on what keeps the agent genuinely interactive. Claude Code (v2.1.80+) has Channels: an MCP server that declares the `claude/channel` capability can push `notifications/claude/channel` notifications into a running interactive session; a busy session queues the event for its next turn, and v2.1.81 adds a `claude/channel/permission` relay for approvals. Channels only load over stdio servers, and the hive daemon is an HTTP server, so the daemon cannot be the channel directly: each Claude spawn instead registers a thin `hive channel-bridge` stdio subprocess that declares both capabilities to the CLI and long-polls the daemon for queued deliveries. The bridge holds no state that matters. Codex has a programmatic app-server too, and the former claim that its host was necessarily headless was false: 0.144.0 documents `codex app-server --listen …` plus `codex --remote …` for the interactive TUI, while the open-source thread state holds multiple client connections. A driven proof attached that TUI, resumed the same thread from a second JSON-RPC connection, and injected a model-visible item that the model returned, all without altering the TUI composer. The planned Codex root adapter therefore uses app-server plus remote TUI and dual-client steering; it remains behind its deliberate-active-turn conformance gate. Until then legacy worker delivery stays on tmux.

Tmux injection is therefore the Codex primary and the fallback for an older Claude CLI or a disabled or unverified Channel. Ordinary messages are `normal`: if the recipient is idle at its prompt, the daemon injects "📨 message from maya: …" into its pane; if it's mid-task, the message queues and a turn-boundary hook drains the inbox. Event-driven — no polling anywhere. The injection itself is `load-buffer`/`paste-buffer -p` (the body arrives as one bracketed paste, so embedded newlines become composer newlines instead of premature submits, and nothing passes through a shell), then a pause, then a single `send-keys Enter`. The pause is load-bearing: both TUIs coalesce input that arrives in a burst as a paste, so an Enter fired immediately after the text lands *inside* that window and becomes a literal newline — the message sits in the composer until a human presses Enter, which is exactly the failure the first field test hit. The obvious simpler shape — `send-keys -l` with the text and an immediate Enter — lost twice over: the Enter race above, plus raw keystroke newlines submitting partial messages. A verify-and-retry loop (capture the pane, re-send Enter if the text is still sitting there) was rejected because misreading a busy pane double-submits; one delayed Enter is the ecosystem-converged fix. Tmux paste is timing-sensitive and cannot confirm receipt, so the native Claude channel wins wherever it is healthy. For Codex that mechanical weakness is the accepted price of leaving the human inside the real conversation; a reliable actuator that removes the reply box is not the better interface.

Directions that reduce authority cannot ride that ordinary queue. The incident that forced the distinction was simple: an agent began one long implementation turn, the user changed the assignment to "pause before coding," and Hive faithfully stored the message while the agent kept writing because the next turn boundary never came. The model did not disobey; it never saw the instruction. Messages therefore have explicit priority and intent. `urgent` waits for the nearest safe lifecycle boundary, injects before ordinary traffic, and requires an agent acknowledgement by a durable deadline. `critical` covers pause, stop, cancel, and write reduction: the daemon atomically advances the agent's capability epoch, revokes landing, denies its stale pending approvals, then kills only that agent's tmux session, preserving its worktree and transcript. It restarts the same worktree in a fresh read-only process with the control at the top of context, using the immutable execution identity recorded at the original launch rather than resolving today's route. The interrupted reservation settles once; the acknowledgement process receives a separate reservation tied to the control message, so concurrent routing sees its full cost without charging the interrupted run twice. The old viewer — now a lens on a killed session — is replaced and the wall re-tiles (decision 10). The agent must acknowledge the matching message and epoch; acknowledgement settles the control reservation, and stale acknowledgements cannot revive authority. Explicit structured intent wins. A deliberately narrow legacy phrase classifier catches old callers saying exactly "stop," "cancel," "pause before coding," or "do not modify" without turning conversational uses of those words into process kills.

Every message carries a monotonic recipient sequence and the durable states `queued`, `injected`, `agent-acknowledged`, and `applied`, with separate timestamps. `deliveredAt` survives as a compatibility alias for injection, but the send result exposes the real state; persistence is never reported as delivery. Optional sender idempotency keys make retries exact. Urgent and critical deadlines wake the orchestrator once when missed, and restart recovery resumes queued critical controls before normal work. This costs a process restart for the strongest control and cannot undo a shell command that finished before revocation, but it closes the dangerous future: after revocation the old process is gone, the replacement is read-only, and the daemon refuses a stale landing epoch. Prompt-only interruption was rejected because it repeats the original failure. Injecting critical text into an approval dialog was rejected because a TUI prompt is not a safe message boundary.

The root orchestrator uses the same substrate without becoming a fake agent. `orchestrator` is a reserved destination, absent from the agents table, and the root lives in one fixed attachable tmux session within its Hive instance. Tmux session names are machine-global, while a Hive instance is selected by `HIVE_HOME`, so every session ends in the first ten hex characters of SHA-256 over the resolved absolute home: `hive-orchestrator-<instance>` for the root and `hive-maya-<instance>` for an agent. An agent send to the reserved destination durably inserts the full message, then pushes one compact `hive.message` envelope through a verified native Claude Channels bridge registered as that reserved root. The bridge acknowledgement means only that it wrote the notification to Claude's transport, so Hive marks the row `injected`, never `applied`; Claude itself queues a busy-session notification for its next turn without Hive touching the human's chat area. Delivery and inbox claims share one serialized lane, while SQLite insertion order supplies a stable order for concurrent sends. The envelope is capped at 2 KiB and contains only id, sender, time, a bounded body preview, and a `hive_read_message` reference. Oversized reports therefore stay in SQLite until the orchestrator deliberately asks for one; full team state, pane output, task restatements, and previously acknowledged messages never ride along with a wake.

This makes Claude Channels the shipping root transport, not an optional acceleration. `hive claude` enables the bridge and refuses to launch unless the installed Claude supports Channels; Codex root support through app-server plus `--remote` is planned rather than incompatible. The fallback question has no clever answer: an unread indicator plus `hive_inbox` leaves the model unaware until somebody polls, which is delayed delivery under another name. Pane injection is forbidden because it owns the human's composer. The short-lived preflight-and-defer approach — capture the root composer and wait until it was empty — fixed draft clobbering, but lost because it still owned the chat area and delayed reports while the human typed. Verify-and-retry lost for the same double-submit risk it has in agent panes.

Alternatives we rejected: a pure file inbox ("agents check for messages") fails exactly when it matters, because an idle agent never checks. Routing everything through the orchestrator as a relay makes it a bottleneck and fills its context with traffic. Unscoped names such as `hive-orchestrator` and `hive-maya` were the original choice and lost after an isolated scratch daemon stopped the real orchestrator through their shared tmux server. Port scoping does not help because tmux does not know Hive ports; random suffixes isolate but make names unstable across restarts. Hashing the resolved home makes the existing instance boundary visible to tmux. The transition exception is deliberately one-way: only the default `~/.hive` installation may discover and stop legacy unscoped sessions, while a custom `HIVE_HOME` never matches them. This lets the normal installation migrate without reopening the scratch-instance footgun, at the cost that an old custom-home session needs manual cleanup. What we give up: agents must run inside tmux. Which turns out to be a gift, not a cost — see below.

### 2. How the orchestrator knows anything

An agent in another terminal is an opaque process. The spec-fantasy version — "the orchestrator knows when agents are done, stuck, or running out of context" — needs real signals. They exist:

- Claude Code has lifecycle hooks: `Stop` fires on every turn end with the session's transcript path; `Notification` reports informational events, including but not limited to permission prompts; `SessionStart` announces birth. A generic notification is only a lifecycle signal: it never enters the approval queue or changes agent status. Genuine permission requests reach that queue through an explicit `approval-request` event or the native permission relays described in decision 4. Treating every notification as an escalation looked convenient because approval waits are one notification subtype, but it lost immediately in practice: routine notifications blocked agents, starved their message inboxes, and buried real decisions.
- Codex runs its interactive **TUI** with `notify` (a program fired on `agent-turn-complete`) passed as a spawn-time override, plus rollout files as the broader sensor and tmux as the input path. The override and pre-trust are load-bearing: Codex ignores `notify` in project-local `.codex/config.toml` and skips project config entirely for untrusted directories. This path sees less lifecycle detail than a protocol connection and cannot intercept an approval into Hive's queue, but the Codex pane remains the real session: the human can answer questions and approvals where they appear.
- Context usage comes from the native stream where one exists and the tool's durable artifacts otherwise. Claude's transcript remains its source. The Codex app-server emits `thread/tokenUsage/updated`, including the context window; the default TUI driver instead reads rollout files, which are less immediate but do not replace the interactive surface.

The rejected default was the **app-server**, a first-party JSON-RPC control plane where `turn/start` begins a turn, `turn/steer` injects input, and `turn/interrupt` stops it while lifecycle, token usage, rate limits, and approval requests stream back over the same connection. This is a real and useful interface, not a side door, so Hive keeps it as a configured alternative and automatically replaces a failed handshake with the TUI. Its tmux session runs a small host around `codex app-server --stdio` and renders a human-scale event feed. That architecture won on machine control and observability, then lost as the default on the more basic dogfooding test: the feed is read-only. When Codex asks the human a question, the pane offers nowhere to respond. A viewer that shows the conversation but cannot participate in it violates the product's promise that each agent appears in a terminal where the user can work with it. The app-server alternative accepts that limitation in exchange for native steering, Hive-routed approvals, detailed lifecycle, and authoritative quota telemetry.

So: at spawn, hive wires each agent's signal path — hook configs for Claude, `notify` plus rollout sensing for the default Codex TUI, or the app-server connection when configured. Agents self-report into the daemon; the daemon maintains a live table — agent, status, last event, context %. A slow daemon-side `tmux capture-pane` heartbeat (~60s) remains a fallback stuck-detector for whatever hooks miss (hung tool call, dead process), but that maintenance never injects anything into the root.

Birth is the fragile part, and the first field test proved it: all four inaugural agents died in their first second (unsupported models, a broken CLI) while hive reported "spawning" forever and left orphaned windows. So "spawning" is a monitored phase, not a fire-and-forget: for 15 seconds after session creation the daemon watches the pane — session vanished or a launch-failure signature printed means status "failed" with the pane's last lines as the recorded reason, the session killed, and the fresh worktree cleaned up; the first hook event short-circuits the watch as proof of life. A 30-second reconciliation sweep separately handles any agent whose status claims a process but whose tmux session has disappeared — and because a vanished session is usually a crash, not a completion, that sweep tries to bring the agent back before declaring it dead (decision 13). The alternative — trusting that a created session equals a running agent — is exactly the silent-death mode the field test exposed, and "failed" with a captured reason is a first-class status precisely so the orchestrator can see it and retry a tier up. The orchestrator uses the daemon's status tool only on demand and never scrapes a terminal. The alternative — orchestrator eyeballing capture-pane dumps — burns its context on screenshots and turns the conductor into a babysitter.

The orchestrator is idle by default. It does not poll `hive_status`, `hive_inbox`, panes, logs, or worktrees. Only a user turn or a new message to the reserved destination starts a root turn. `hive_status` is an explicit user-driven read: the root requests its compact `active` view, which excludes terminal agents and omits paths, branches, and unbounded task text. The omitted detail preserves the full legacy response for the CLI and existing MCP callers. One known wart: Claude Code's Stop-hook `transcript_path` can go stale (anthropics/claude-code#8564); the daemon reads the most-recently-modified `.jsonl` in the project dir instead of trusting it.

Identity rides on this table too, and agents get **human first names** — maya, david, sam — drawn from a pool at spawn, never numbers. Everything keys off the name: tmux session `hive-maya-<instance>`, worktree `.hive/worktrees/maya`, branch `hive/maya-auth-api`, the status row, message envelopes. The reason is that the user's entire interface is conversation with the orchestrator, and conversation wants names: "tell maya to reuse the middleware" and "what's david doing?" work; "message agent-3" forces the user to keep a mental numbering table the tool should be keeping for them. Numbered agents were the obvious default and lost for exactly that reason. Task-derived names ("auth-api") lost too: two agents can share a feature, and a name describing the task goes stale the moment the agent is respawned onto follow-up work — the worker's identity should outlive any one assignment. The cost is some whimsy in professional logs and a pool large enough to never collide within a session; both trivial.

### 3. How agents avoid destroying each other's work

Three agents editing one checkout doesn't produce merge conflicts — it produces silent overwrites, agents reading half-written files, and everyone fighting over `.git/index.lock`. The industry answer is unanimous and we take it: **worktree per writing agent.** Each writer gets its own branch and checkout (`.hive/worktrees/maya`, branch `hive/maya-auth-api`) sharing one object store. Read-only agents (research, review) share the main checkout free of charge.

But isolation is the floor, not the strategy. The real conflict-avoidance is planning, which is exactly the orchestrator's job: land shared interfaces on the base branch *before* fanning out, and give every agent an explicit file scope ("you own `lib/auth/`"). Merges still happen serially — maya lands, then david rebases against a world that includes maya's work — but merging is nobody's *separate* job: **every writer lands its own finished work on main.**

The reason is a failure mode we hit for real: task done, tests green, and the branch sitting in `.hive/worktrees/` forever because merging was somebody else's step. Completed work stranded on a worktree branch is indistinguishable from work never done. So the landing protocol rides in every writer's spawn prompt — it's composed into `buildAgentPrompt`, so no orchestrator ever has to remember to say it: when the task is done and tests are green, commit everything, `git rebase main` in the worktree, re-run the tests on the rebased branch, then call `hive_land` with the capability epoch issued at spawn. The daemon verifies that the epoch is still current and write authority has not been revoked before it performs `git merge --ff-only` in the primary checkout. Fast-forward-only is the whole trick for concurrency: the merge succeeds only if main hasn't moved since the rebase, so two agents landing at once can't interleave — the loser's merge is simply rejected, and it re-rebases and retries, three attempts before escalating instead of spinning. A rebase *conflict* means the planning failed and two agents genuinely touched the same code; the agent runs `git rebase --abort` and escalates to the orchestrator, which spawns an integrator agent — a judgment merge is a task, never a reflex, and nothing is ever forced. Red tests never merge, in any state, on any attempt. Direct primary-checkout merge instructions lost because they left no reliable way for a later critical control to revoke landing.

Alternatives that lost. Routing *all* merges through a spawned integrator agent was the first design: it kept writers pure and matched "the orchestrator can't write" (decision 11) cleanly, but it made landing asynchronous and optional — the integrator existed only if the orchestrator remembered to spawn it, which is exactly how green work strands. Integrators survive as the escalation path (conflicts, stranded recoveries), not the default. Plain merge commits instead of rebase-then-ff lost too: a merge commit means the tests you ran validated neither parent's final tree, while ff-only makes "the tests passed on exactly what main now is" structural. The accepted cost: the landing merge is the one sanctioned write outside an agent's worktree — a deliberate pinhole in the decision-4 sandbox. It's tolerable because ff-only moves a ref without rewriting history and git refuses it when the primary checkout has conflicting uncommitted state; for a sandboxed Codex agent the command surfaces in the approval queue like any other boundary crossing, one approval per landing, safely auto-approvable by policy.

Death gets the same guarantee: work is either merged or explicitly surfaced, never silently lost. `hive_kill` assesses the agent's branch and worktree before any cleanup — unmerged commits or uncommitted files make it refuse deletion, keep everything, and report the stranded work in its result, so the orchestrator must actively decide: spawn an integrator to land it, or pass `discardWork` to delete it on purpose (the discard is still reported, as the record of what was chosen). If the assessment itself fails, the worktree is kept — nothing is deleted on a guess, though the kill and the dead status always win. Cleanup after a successful landing is safe by construction: a landed branch tip is an ancestor of main, so deleting the branch and worktree deletes nothing.

The "david needs maya's work-in-progress" case gets a real mechanism instead of a hand-wave: maya commits to her branch and messages david "pull `hive/maya-auth-api` at abc123." Shared object store makes it instant. Uncommitted state is never shared; that way lies madness.

Consequences accepted: hive requires a git repo (we `git init` or degrade to single-writer if there isn't one), and "as many agents as it needs" has a soft ceiling the orchestrator should know about. We default that ceiling to 2–4 parallel writers, and we label it honestly: a tunable heuristic, not a derived constant. There is no practitioner consensus to borrow — Claude Code's own docs suggest 3–5 teammates as their soft ceiling — so the number lives in config and the open question on parallelism (below) owns the harder truth that some tasks don't just parallelize slowly, they parallelize *badly*.

### 4. Who answers the permission prompts

N agents in N windows each stopping for "allow this command?" gives you a fork: babysit every window, or let the agents run unattended with shell access on your real laptop. Hive takes the second fork deliberately, and names the cost.

**Full autonomy for writers, by default.** A writer that stops for a prompt nobody is watching is a stalled writer, and the premise of this tool is that you walk away while N agents work. So `autonomy = "dangerous"` is the default: Claude writers launch with `permissions.defaultMode = "bypassPermissions"` in their worktree settings, Codex writers with `approval_policy = "never"` and `sandbox_mode = "danger-full-access"`. No writer ever prompts, and the escalation queue stays empty because nothing escalates.

One mechanism detail matters, because the obvious flag is wrong. Claude's `--dangerously-skip-permissions` raises a blocking acceptance dialog on every launch that no unattended spawn can answer; `--allow-dangerously-skip-permissions` does not suppress it, and accepting it does not persist. Setting `defaultMode` in the worktree's `settings.local.json` starts the session already in bypass mode with no dialog. Hive therefore never passes the flag.

The trade is real: an agent can now run any command anywhere on the machine, so the sandbox no longer enforces decision 3's file scopes — those revert to an honor system backed by the worktree each agent is briefed with, and the daemon's resource watchdog (decision 12) becomes the only mechanical floor left. `autonomy = "sandboxed"` in `~/.hive/config.toml` restores the previous posture in full: Codex at `workspace-write` scoped to its worktree with `approval_policy = "on-request"`, Claude at `acceptEdits` plus a hive-maintained allowlist, and the rare escalation flowing through the daemon — a Codex approval request arrives as an app-server JSON-RPC request the daemon answers directly; a Claude permission prompt becomes an explicit `approval-request` event, or, where Channels is live, arrives through the `claude/channel/permission` relay, which lets the daemon *answer* the prompt rather than merely observe it — surfacing in exactly one place, the orchestrator window, where the orchestrator may auto-resolve by policy but anything irreversible or outward-facing still reaches the human. Generic lifecycle notifications never share that path.

That sandboxed mode was the original default and it lost on the dogfooding test, not on principle: in practice the escalation queue interrupted long autonomous runs for approvals the user would have granted anyway, and a supervisor who must be present to unblock agents is precisely the babysitting this design exists to eliminate. Its pleasant unification — the sandbox boundary *being* the file-scope enforcement — was a real property we gave up, and the reason it was affordable is that landing is still gated: red tests never merge, and `hive_land` checks the capability epoch (decision 3). The mode stays maintained, one config line away, for anyone running untrusted work.

Two boundaries survive the change, mechanically rather than by convention. Autonomy applies to **writers only**: the read-only orchestrator (decision 11) and the read-only replacement a critical control spawns (decision 1) both pass `readOnly`, which short-circuits the autonomy branch in both tool adapters — a revoked agent cannot regain shell access through a config default. And a crash-recovered agent resumes with the posture it launched with, so recovery never drops a running agent into a prompt nobody will answer.

### 5. What "memory" actually means

The original spec said "cross-platform memory" once and "keep every context as small as possible" twice, without noticing they fight: unbounded shared memory dumped into every prompt is how contexts bloat. The fix is realizing "memory" was three different things wearing one name:

- **Conventions** — how this codebase works, style, commands. Lives in a committed `AGENTS.md`. Both tools load it natively; zero hive machinery. If every agent always needs it, it goes here and nowhere else.
- **Run state** — who's building what right now, task assignments, "david finished the API." Lives in the daemon's SQLite and dies with the hive session. Calling this "memory" was the conflation to kill.
- **Durable knowledge** — gotchas discovered, decisions made, "the flaky test is X." One Markdown file per fact: the filename minus `.md` is the fact's id, a short frontmatter block above the body carries title/date/tags (hand-rolled, no YAML dependency pulled in for three fields). Two scopes: **per-repo** (`.hive/memory/`, committed, travels with the clone) and **global** (`~/.hive/memory/`, where cross-project lessons accumulate so the tool improves with every project). Exposed via daemon MCP as `memory_search`; `memory_write`, which upserts — an explicit id overwrites that fact's file in place, an omitted id derives a slug from the title and disambiguates on collision (`flaky-test`, `flaky-test-2`); `memory_read`, which pulls one full fact by scope and id; `memory_delete`, the orchestrator's prune-stale-facts housekeeping; and `memory_reindex`, which forces a full rebuild of the search index from the files. `hive memory search|write|read|delete|reindex` mirrors the same five operations over the CLI. The daemon serializes writes; every fact carries a date. Promoting a repo lesson to global has no dedicated tool: it's `memory_read` the fact, `memory_write` it into the other scope, `memory_delete` the original — literally the "file move plus re-index" the mechanism always was, so a sixth `memory_promote` tool would only wrap three operations the interface already has, and the narrow-surface philosophy this section argues for says no.

The context-budget rule that reconciles memory with small contexts: agents are injected with the merged **index only** — a few hundred tokens, one line per fact (`- [scope] id (date): title`), newest-first and capped at 30 entries with an "N older facts omitted" trailer — built straight from the Markdown files, not from the search index. The orchestrator gets its index at launch; each spawned writer gets its own at spawn time, read from its own worktree rather than the primary checkout, the same convention decision 8 uses for skills — so a repo fact an agent hasn't actually pulled onto its branch never shows up in its context, which is what makes "committed, travels with the clone" a real enforcement instead of a promise. Global scope is always visible, since it isn't gated by git state. The store never gets dumped into a prompt; agents pull full facts on demand with `memory_read` or search wider with `memory_search`. Build-vs-buy is decided: the narrow core we built — Markdown facts, one per file, indexed by a single SQLite FTS5 table inside the daemon's existing database — ships in v1.1. Adopting an open-source memory server (agentmemory, agent-memory) looked attractive because the designs rhyme, and lost on weight: agentmemory ships 53 tools and 15 skills against the five operations hive needs, and bolting a subsystem larger than our daemon onto an index-only context budget is the exact bloat this section exists to prevent. The daemon integration was the only custom work under either choice, so buying saved almost nothing. The index stays deliberately disposable: it rebuilds unconditionally on every daemon start and re-syncs incrementally on every write or delete, so a `git pull` that changes committed repo facts without going through the daemon just needs one `memory_reindex` call to catch up.

### 6. Who picks the model

The trap: "the orchestrator is smart enough to know which model." No, it isn't — an LLM's pricing and capability knowledge froze at its training cutoff, and model lineups turned over twice in the last year. An orchestrator routing from memory confidently picks superseded models forever.

The split that works: **the orchestrator classifies; a table resolves.** The orchestrator's judgment call is "this subtask is `deep` / `standard` / `cheap`" — a thing LLMs are genuinely good at and that never goes stale. The table is a **tier × tool matrix**: each tier names a preferred tool plus a model-and-effort entry *per tool* — `cheap` prefers Codex at low effort, but also records that a cheap Claude agent means Haiku. That second dimension exists because the orchestrator can override the tool at spawn time ("use a Claude agent for this"), and the first shape we shipped — one tool+model per tier — had nothing to say then: the override discarded the model and a "cheap" Claude agent silently launched on the account's default (expensive) model. A route that can't survive a tool override isn't a route, it's a suggestion. Mechanically this is just spawn flags: `claude --model …`, `codex -c model_reasoning_effort=…`.

A corollary the first field test taught us: **routes must not assume account entitlements.** Naming a concrete Codex model in the defaults killed every Codex spawn on a ChatGPT-plan account ("model not supported"). The default Codex column therefore says model `"default"` — pass no model flag and let the CLI use whatever the account actually has, which is supported by definition; effort flags still steer cost per tier. The Claude column names model *aliases*, which every Claude plan resolves and which auto-update as models ship: `deep` uses `best` (Claude Code resolves it to the most capable model the account can reach — entitlement-adaptive by construction, exactly this principle as a vendor feature), `standard`/`review` use `sonnet`, `cheap` uses `haiku`. We considered pinning `deep` to `opus` instead — more predictable, never subject to `best`'s safety-classifier fallback — but it caps a top-tier account below the best model it's paying for. Pinning exact model IDs is opt-in via `routing.toml`, where the user knows their own plan.

`best` currently resolves to Fable 5, and Fable draws heavily on shared subscription capacity; on 2026-07-12 it moves to usage-only billing off the user's plan entirely. Auto-routing a tier onto a model about to become metered-and-unbudgeted is the wrong default, so `deep`'s auto-selected model changes on that date — but nothing about Fable is removed. Claude Opus 4.8 (`claude-opus-4-8`) is now a first-class supported model in exactly the same sense `best` and `sonnet` are: adapter model set, quota pool matching, and routing all treat it as an ordinary concrete Claude model, with the id verified against the live Claude Platform docs rather than trusted from training memory, since model lineups move faster than any model's own knowledge of itself. `defaultRoutingTable(now)` in `schemas/routing.ts` is a pure function of one cutoff constant, `FABLE_AUTO_ROUTING_CUTOFF`: before it, `deep.claude` stays `best`, exactly as it always has; on or after it, `deep.claude` pins directly to `claude-opus-4-8`, because by then `best` resolving to a now-metered Fable is no longer the entitlement-adaptive default it was designed to be. An earlier draft of this change went further — dropping Fable from the default immediately and requiring explicit opt-in to reach it at all — and was rejected outright: Fable stays fully supported forever, reachable at any time, before or after the cutoff, by pinning `deep.claude.model` to `best` or `claude-fable-5` in `routing.toml`. The cutoff only ever narrows what auto-routing picks by default; it does not narrow what the user can ask for.

Capacity pressure is a second, date-independent path to Opus 4.8. Whenever a spawn's resolved Claude candidate is Fable — whether via `best` before the cutoff or an explicit pin at any time — the spawner adds a second Claude candidate for Opus 4.8 to the quota resolver's candidate list, placed right after Fable. Nothing about the quota resolver itself changed to make this work: it was already generic over an arbitrary number of candidates, scoring each by worst post-reservation headroom and breaking ties by preferred-tool bonus and then array order. Placing Opus second means a tie — including the compatibility case where no pool has a live reading and every candidate ties at the same sentinel score — still resolves to Fable, preserving existing behavior wherever hive cannot yet see real headroom. But when Fable's pool has worse measured headroom than Opus's, the same unmodified scoring naturally prefers Opus instead. This makes quota pressure and explicit choice the two legitimate triggers for reaching Opus 4.8 ahead of the cutoff, without the wholesale default move the rejected draft attempted.

Aliases are useful for choosing a launch, but they are not an execution identity. The first shape stored the route's model string on the agent record verbatim, so `hive_status` and every terminal title said `best` or `default` — which tells the user nothing about what's actually running and costing money. Worse, replaying the tier through a changed route table could move a critical-control restart to another provider or model. Hive resolves the concrete model once, launches that concrete model, and durably records the immutable choices needed to reproduce it: provider/tool, concrete model, tier, and Codex reasoning effort where applicable. A concrete route model passes through; `default` is read from the tool's own user config (`~/.codex/config.toml`, `~/.claude/settings.json` — the very file the CLI consults when hive omits the flag); Claude's `best` maps through a constant verified against the installed CLI, because the alias is resolved inside the CLI and no local file records it. Dynamic facts — daemon port, worktree path, hooks, permissions, terminal choice — are rebuilt rather than copied into identity. This gives up alias adaptation between the initial launch and a control restart, deliberately: changing the model during a safety interruption is a larger lie than missing a newly entitled default.

Sometimes the concrete model cannot be known. A tool config may omit its model and leave `default` to a built-in the CLI does not expose; old agent rows also predate the immutable identity. Guessing from the current route, tier, or current tool default was rejected because each can silently change the model. These rows keep their worktree and revoked capability but do not get a read-only restart; Hive records the reason on the agent and sends a durable orchestrator alert naming the legacy or unresolved identity. The same fail-closed rule applies when a recorded model is no longer launchable. No alternate route is consulted. Downstream, a terminal viewer's title is exactly `<name> — <model>` (`maya — claude-fable-5`) and nothing else — no task text, path, tier, or session name, which leaked into titles in the first field test and made a wall of windows unreadable. Terminal.app's custom-title property does not override the profile's username, process, and dimensions fields, so Hive installs a bundled `hive-agent-v2` profile on first use and reuses it for viewers. That managed profile disables every generated title component and gives viewers a restrained blue-black background tint without mutating the user's existing Terminal profiles. Versioning matters because Terminal imports settings sets by name and does not update an already-imported profile when the bundle changes.

The table auto-updates so new releases aren't missed: hive fetches a **curated routing manifest** published with hive releases (provider model-list APIs give you names, not judgment — curation is the value), and your local `~/.hive/routing.toml` overrides always win. Escape hatches: pin a vendor for a session; name a model for a single spawn — `hive_spawn` takes an optional `model`, reserved for explicit user directives ("open an Opus 4.8 terminal"), which launches verbatim with no alias resolution, binds the spawn to its vendor for quota routing, and is never silently substituted (the Fable→Opus release valve included — substituting a user-named model is a lie, so under quota pressure the spawn fails with the capacity report instead). The orchestrator choosing models from its own knowledge stays rejected — that knowledge froze at training time, which is this section's founding observation; the `model` field is a pass-through for the user's words, not a judgment surface. And bump a tier automatically when an agent fails or stalls — retry-with-a-bigger-model is cheap and effective. And because we're multi-vendor, the table can encode a genuinely novel policy: route *review* to the other vendor from whoever wrote the code.

But capability is only half of routing. Claude and Codex subscription accounts have rolling and weekly capacity, models can draw from different or shared pools, and two simultaneous spawns can spend the same apparent headroom unless selection owns a reservation. So the resolver is quota-aware: it resolves both vendor candidates, maps each concrete model to a `(provider, account, pool)` that hive discovered from the provider itself, estimates the task as a fraction of each window, and atomically reserves it before creating a worktree or process. The account and pool are explicit because keying only by model double-counts a shared vendor limit, while keying only by vendor cannot represent a premium model with its own weekly cap — and both vendors do in fact publish exactly such per-model caps. A completed turn reconciles the reservation to provider-reported consumption when a hook supplies it and otherwise commits the conservative estimate; a launch that never starts releases it, while a started agent that dies keeps the estimate. Expired reservations receive the same started-versus-unstarted treatment after a daemon restart.

A critical interruption ends one run and starts another, even though the second run is read-only. Transferring the original reservation lost the boundary between two billable processes, while leaving the restart unreserved let concurrent spawns spend its headroom. Hive therefore uses a dedicated control-run reservation keyed to the critical message and pinned to the recorded provider/model. The original reservation settles before the replacement is considered. The control estimate uses the agent's existing tier estimate — conservative but provider-neutral — and may consume the last safe capacity because preserving deep-work reserve is less important than delivering revoked authority. It never borrows from another model. A repeated recovery after a daemon crash reuses the active message reservation and surviving tmux process; a failed launch settles or releases its attempt before a retry can reserve again. Acknowledgement/application, turn completion, death, `hive_kill`, missing-session reconciliation, launch failure, and reservation timeout all settle the active control reservation idempotently. Timeout also stops the read-only process and closes its viewer so no unreserved process remains alive.

If the recorded pool lacks enough five-hour or weekly headroom, Hive does not restore writes and does not drop the control. The old process stays stopped, the capability epoch stays revoked, the worktree stays intact, the critical message stays queued, and a durable orchestrator alert reports the exact model, remaining headroom, and required estimate. Recovery retries the same identity when capacity becomes available. Compatibility mode still creates an explicit unbounded control reservation with missing confidence; disabling quota-aware *routing* likewise does not disable control-run accounting. We rejected treating safety traffic as free because invisible usage is how the next concurrent route overcommits, and rejected switching vendors because an acknowledgement from a different model is not a faithful restart of the interrupted agent.

Nobody types a quota number into hive. Both vendors ship a program-readable limit surface, both are readable without starting a model turn, and hive reads them itself at every start. Codex's app-server answers `account/rateLimits/read` after its mandatory `initialize`/`initialized` handshake, with no thread and no turn: a snapshot of `primary` and `secondary` windows, each carrying `usedPercent`, `windowDurationMins`, and `resetsAt`, plus a `rateLimitsByLimitId` map of metered sub-limits. It is stable protocol, so hive calls it `authoritative`, and its `account/rateLimits/updated` notification keeps an app-server-driven agent current for free. Claude Code has no such thing on its statusLine path — that JSON only carries `rate_limits` after a session's first response — but its stream-json control protocol answers a `get_usage` request with `subscription_type`, `rate_limits_available`, and per-window `utilization` and `resets_at`, fetched fresh and costing nothing. The CLI marks `get_usage` experimental, so hive calls it `reported`. Both facts were verified by driving codex-cli 0.144.0 and claude 2.1.206, not by reading docs: the Codex methods are absent from the prose docs, and `get_usage` is absent from everything. Windows are matched by their reported duration, never by the position they arrive in, so a plan that lists its weekly bucket first cannot silently invert the two.

The load-bearing consequence is that a provider reports the *fraction* of a window consumed and never the window's size. A discovered pool is therefore denominated in percent, its allowance is exactly 100 by construction rather than by assumption, and hive needs no capacity figure from anyone to know its own headroom. What hive cannot get from a provider is how much of a window a *future* run will consume, so reservations still spend a hive-authored percent estimate, and every surface that shows one labels it an estimate. Two estimates, not one: a run is a far larger fraction of a five-hour bucket than of a week, and a week does not hold thirty-three five-hour buckets' worth of capacity.

The rejected alternative is the one hive shipped first, and it failed in the most instructive way. Pools used to come from an operator-written `~/.hive/quota.toml`, and every observation path began by looking up the configured pool for a model. An installation without that file had no pools, so the lookup returned nothing, so hive *discarded* the authoritative percentages Codex was handing it on every single turn — then warned the operator that quota data was unconfigured and routed on its own estimates. Requiring a human to transcribe a number the provider already publishes was the root error; discarding the published number for want of the transcription was its punishment. A manual pool survives only as an explicit override, denominated in the operator's own planning units, mapped onto by the provider's percentages, and never required for anything. Also rejected: scraping either TUI's display, brittle and liable to silently parse the wrong number; calling `api.anthropic.com/api/oauth/usage` ourselves, which is undocumented, rate-limits aggressively, and would break silently the day it changes — going through `get_usage` puts that contract on the CLI, where it belongs; and presenting local token sums as the vendor's exact limit.

So hive records provenance per fact rather than inventing precision, and it records it per *window*, because a statusLine payload can report the five-hour window while the weekly one is still absent, and stamping one timestamp across both would backdate freshness onto a fact nobody observed. Unknown is rendered as unknown. A window nobody has measured reports no number at all — not the zero sitting in its column — and a reading whose reset boundary has since passed is void rather than carried forward, because the human spends these accounts outside hive too, and "0% used" is a claim about the account, not about hive. An unmeasured pool cannot constrain a spawn, so the spawn proceeds under an explicit unbounded reservation and says so. Conservative local estimates remain the floor: an external reading and the local ledger combine by `max()`, so an optimistic provider number can never free capacity hive already knows it spent. Refresh is forced at daemon start, runs on the maintenance tick when the interval lapses or a passed reset has blinded a pool, and is skipped entirely while the free feeds — the app-server's notifications, Claude's statusLine — already keep a provider live. Model-scoped caps (Codex's metered `limitId`s, Claude's per-model weekly ceilings) arrive with a display name but usually no concrete model id. Hive records them, shows them, and refuses to route on them, because deciding that `"Fable"` means `claude-fable-5` would be a guess, and a wrong guess here refuses a spawn.

The deterministic policy is intentionally boring. Start with the table's preferred vendor (or the other vendor for a review when its author is known), then the other candidate; reject candidates whose estimated run would exceed either configured window; for `cheap` and `standard`, also reject a candidate that would invade the configured deep-capacity reserve. Compare eligible candidates by the worst post-reservation headroom, then preference, then vendor name. An explicit user vendor stays fixed when safe. If it is unsafe, hive does not silently disobey: spawn returns the remaining five-hour and weekly capacity, expected reset where known, and the safe fallback. If both vendors are exhausted, no process starts. When a provider's live numbers are missing — it is not signed in, its probe failed, no reading has landed yet — that pool enters compatibility mode: hive preserves the static route, records an explicit unbounded reservation, and emits one deduplicated warning that names the provider and the reason. The warning does not ask anyone to configure a file, because the remedy is for the provider to answer, and hive will adopt real numbers the moment it does. Failing every spawn while limits are unknown was rejected as a migration strategy; pretending an arbitrary default allowance was real lost for the same reason. Refusing a spawn is the sharp end of this subsystem — a wrong number here does not merely mislead a dashboard, it stops work — which is why the rule is accurate-or-unknown, never invented.

Warnings use the ordinary durable orchestrator message path, not stderr in a daemon nobody watches. Crossing configurable warning or critical remaining-capacity thresholds sends one message containing the provider, account, pool, both windows, data confidence and freshness, and known reset times. Alert state is durable. It rearms only after headroom rises above the threshold plus hysteresis or a reset boundary passes, so parallel reservations do not produce a wall of identical warnings. Corrupt state fails closed for the affected configured pool, stale external observations are combined with the more conservative local ledger, and status reports the reason without requiring anyone to inspect SQLite. One consequence of that combination rule is worth naming because it surprises people at the console: a manual observation reporting *lower* usage than the local ledger cannot free capacity inside the current five-hour window — the ledger and the supplemental observation combine by max(), so reconciliation only ever tightens the picture until a window resets. Conservative by design: an optimistic external number is exactly the input most likely to be stale or misread, and the cost of believing it is an overcommitted account.

### 7. What happens when a context fills up

Sensing was solved in decision 2; this is the actuator. Field evidence says quality dies long before the window fills — models start repeating themselves and contradicting locked-in decisions around ~140K tokens, well inside technical limits. So the trigger is a quality line, not "almost full" — and the 65% default is a tunable heuristic, not a derived constant. The ~140K figure is field lore, not a measurement hive has reproduced, and no practitioner consensus exists on where the line sits; the number lives in config, and we expect dogfooding to move it.

First, prevention, which is the real mechanism: the orchestrator sizes subtasks to finish inside one healthy context, and agents spawn with a minimal briefing — task, file scope, memory index — never repo dumps. Most agents should retire at task end without ever seeing the ceiling.

When one does cross the line: the daemon notifies the orchestrator, which injects "finish your current step, commit WIP to your branch, write a handoff." The handoff is structured and machine-targeted — goal, done/remaining, decisions made, **failed approaches** — stored in run state. Hive kills the session and respawns the same agent identity in the same worktree with the handoff preloaded. Fresh window, continuous work.

We explicitly do not rely on `/compact` for anything that matters — lossy self-summarization drops constraints and failed-approach history, which are the two things a successor most needs. What we give up: warm-agent speed. A warmed-up agent that never re-reads the repo iterates fast, so we recycle at the quality line, not eagerly. Bonus loop: failed approaches from handoffs are prime candidates for the durable memory layer — that's how "the tool gets smarter" stops being a slogan.

### 8. Skills, and when to build one

The tailwind that makes this feature possible: Agent Skills became an open standard (agentskills.io, Dec 2025) and OpenAI shipped Codex support within 48 hours. One `SKILL.md` folder serves every agent in the hive regardless of vendor, and progressive disclosure means a large library costs agents almost nothing until a skill is actually used.

Storage mirrors the memory scopes: per-repo skills in `.hive/skills/` and global skills in `~/.hive/skills/`. At spawn hive links each skill directory into the launched tool's project-native discovery path — `.claude/skills/` for Claude and `.agents/skills/` for Codex — with a same-named repo skill winning over a global one. Links are per skill rather than one link over the whole native directory, so vendor-specific skills can coexist; a conflicting same-named native entry fails the spawn instead of being overwritten. A skill that exists only in a vendor-native directory stays vendor-specific: silently promoting it would turn a tool-local customization into a cross-vendor contract its author never chose. The cost is that shared skills must be moved to the canonical hive directory once. Treating all native directories as additional sources was rejected because two editable copies would inevitably drift while appearing to be one skill. Write once, every vendor reads it.

The hole in the original spec was *detection* — "the orchestrator knows when a skill should be built" requires memory of past runs, which a single session doesn't have. The daemon does: it logs every task descriptor. At session end the orchestrator reviews history with a **rule of three** — same task shape done ~3 times across runs → propose a skill. Propose, not create: "I've formatted release notes three times; want a skill for it?" On yes, a cheap-tier agent authors the SKILL.md from those transcripts. Silent skill accumulation is how you get sprawl nobody audits, so creation is human-gated and skills get the same pruning policy as memory — unused for N runs, propose deletion. At spawn time, the orchestrator names the relevant skills in the briefing rather than hoping agents discover them.

### 9. Windows are viewers, not containers

The original framing — "each agent runs in a terminal window" — hides a failure mode: if the process lives in the window, closing david's window kills david mid-task. Invert it: **agents live in detached tmux sessions owned by the daemon; windows are disposable viewers.** Spawning a window means telling iTerm2 or Terminal.app (via osascript) to attach to `hive-maya-<instance>`. Closing it detaches; the agent keeps working headless; `hive watch maya` reopens the lens.

This costs nothing — decision 1 already made tmux the substrate for message injection and capture-pane monitoring — and it buys robustness plus a free `--headless` mode (SSH, CI). The root also runs in a fixed tmux session and attaches to the user's current terminal, which gives the daemon one precise wake target without scraping or polling it. macOS only means two emulator adapters, auto-detected, overridable in config. Nothing in the tmux+daemon core is macOS-specific except these adapters, so the platform door stays open without a rewrite.

Disposable viewers still need disposal: the first field test killed agents whose windows stayed open, attached to dead sessions. The fix is identity, not search. When hive opens a viewer it records a **native terminal handle** on the agent row — iTerm2's globally unique session ID, or for Terminal.app the app PID plus window ID plus TTY, all of which must still agree (and the window hold a single tab) before anything closes. Terminal.app handle capture follows the exact tab object returned by `do script` to its owning window; it must not assume the asynchronously activated front window is the new one. `hive_kill` kills the tmux session first, then atomically marks the agent dead while claiming the handle, and best-effort closes exactly that viewer; if a concurrent spawn loses that race it closes its own orphan. Finding the window by its title, agent name, or a session-name text search was rejected outright: titles are mutable and shared, and any fuzzy match is a loaded gun pointed at some unrelated terminal the user has open. A close failure never blocks the kill, the dead state, or worktree cleanup, and manually attached terminals and old handle-less rows are untouched.

### 10. The window wall lays itself out

Decision 9 made windows cheap; cheap windows pile up. macOS opens every new terminal in a default-offset cascade, so by the fourth agent the screen is a deck of overlapping cards and the orchestrator — the one window that matters — is buried under the workers. The fix is that hive owns the geometry: **whenever the set of hive windows changes, the daemon re-tiles them into a gapped wall with the orchestrator centered and largest.** Every viewer-set change already flows through the daemon — spawn attaching a handle, `hive_kill` claiming one, the reconciliation sweep declaring an agent dead, `hive watch` reopening a lens — so each of those also fires a coalesced, best-effort layout pass. No polling, no timer: the same event edges that maintain the status table maintain the wall. Two shapes cover reality. The preferred one gives the orchestrator a dominant center column with workers stacked in symmetric side columns, the odd worker going right, sides splitting into more columns as rows run out. The whole wall is inset vertically and horizontally, cells have visible gaps, and a final mechanical cap keeps every frame below a fixed fraction of the display. These constraints make every participant an ordinary floating window: none becomes a macOS full-screen space or an edge-to-edge half-screen split. When worker cells would fall below a minimum readable size — too many agents, too small a screen — the layout degrades honestly to an inset uniform grid in which every window is equal and the orchestrator holds the most central cell. Layout math is a pure function (screen rectangle in, frames out), so every count, parity, degenerate screen, and maximum-size guard is unit-tested without ever touching a window. The rejected first shape used full-height, edge-to-edge columns; it maximized pixels but made the wall feel like an involuntary window-manager mode rather than a tidy set of recognizable terminals.

The orchestrator window is the one window hive didn't open — the user typed `hive claude` into it — so it has no spawn-time handle. At launch, hive resolves the window it's running in by TTY (`TERM_PROGRAM` names the emulator, the emulator's scripting dictionary maps TTY to window) and registers that handle with the daemon; on exit it unregisters. In an emulator hive can't script — VS Code, Alacritty, an existing tmux — registration yields nothing and the workers occupy the inset wall themselves: participation is opt-out by circumstance. Capture and registration are cosmetic even in a supported emulator: automation denial, a transient process lookup failure, or an unavailable daemon produces a foreground warning but never blocks the orchestrator. `hive layout register` retries against a running root by asking tmux for the physical TTY of its one attached client, matching that exact TTY through the Terminal.app and iTerm2 adapters, and replacing the daemon's recorded handle. An explicit `--terminal` resolves the rare case where probing both applications is undesirable. Relaunching was rejected as the only recovery path because it destroys conversational continuity to repair window geometry; fuzzy title or front-window lookup remains rejected because recovery does not justify moving an unrelated window. The cost is that an ambiguously multi-attached root must be reduced to one client before registration. Registration alone also never moves anything; the wall only starts arranging once the first agent viewer exists, so a user who never spawns agents never sees their window jump.

Moving windows is the same trust problem as closing them, and it reuses the same answer from decision 9: a window is repositioned only through its recorded identity handle (iTerm2 session ID; Terminal.app PID + window ID + TTY), and the move is a bare `set bounds` — never `activate`, so focus stays wherever the user put it. Terminal.app's `do script` returns before its default cascade geometry has necessarily settled, so its adapter waits for stable bounds before publishing the handle; without that boundary, a correct initial layout can be overwritten moments later. A vanished window makes its move a silent no-op and never blocks the rest of the pass. The sweep that marks a vanished session dead closes its orphaned viewer too, because a lens on a dead session shows nothing worth keeping. Alternatives that lost: tmux's own tiling (`select-layout tiled`) collapses all agents into panes of one window — free, but it un-decides decision 9 by making one close destroy every lens, and N readable panes fit one screen far worse than N windows hive can size. Delegating to a window manager (Rectangle, yabai) was rejected as a dependency most users don't have, with yabai needing SIP disabled. Making the user arrange windows was the status quo being fixed. The accepted costs: layout is osascript and therefore macOS-only (it lives entirely inside the decision-9 adapter boundary), it repositions the user's own orchestrator window by design (`layout = "off"` in config restores manual control), and the "orchestrator is largest" guarantee holds only while minimum worker sizes fit — beyond that the equal grid is the honest ceiling.

### 11. The orchestrator physically cannot code

"The orchestrator does no coding" is load-bearing: an orchestrator that helpfully edits files is an unsandboxed writer in the main checkout (breaking decision 3), burning its own context on diffs (breaking decision 7), outside the approval flow (breaking decision 4). And both CLIs *will* drift into coding if merely asked nicely. A system prompt is a suggestion; permissions are a constraint.

So it's mechanical: a Codex orchestrator launches with `--sandbox read-only`; a Claude orchestrator gets deny rules on Edit/Write/NotebookEdit and read-only Bash. Its only actuators are the hive MCP tools — spawn, send, on-demand status, bounded inbox recovery and message reads, approve, memory. Reading is encouraged (good decomposition requires browsing the code); writing is impossible. The prompt explains the role and the reflex: you want to fix that typo → spawn a cheap agent. No small-edit carve-outs — carve-outs are how invariants erode. A side benefit: a read-only orchestrator triggers almost no approvals, so its window stays clean for the three streams that matter — your conversation, concise agent envelopes, and the escalation queue.

### 12. The machine outlives any one agent

Hive runs other people's code on your only computer. Agents write tests with infinite mock loops, background their own hung commands, and retry what didn't respond — each individually reasonable, and hive amplifies all of it by running many agents at once. The failure that proved this isn't hypothetical: one agent's unit test faked a long-poll with an instantly-resolving mock, turning a pump loop into a ~1.5 GB/s allocator; the hung run produced no output, so the agent retried, stacking three runaway processes to ~164 GB until macOS jetsammed the machine and killed every agent, the daemon, and the user's other work with it. The lesson is not "fix that test" — agents will write that test again. It's that **hive must degrade before the machine does.**

So the daemon's maintenance tick doubles as a resource sweep. It walks the process tree under every hive-owned tmux session (agents and orchestrator, roots from tmux pane pids, tree from one `ps` pass) and hard-kills any single process past a per-process memory ceiling — 12 GB by default, far above every legitimate workload observed and far below where the machine starts dying. It reads reclaimable system memory from `vm_stat`, and below a 4 GB floor it pauses `hive_spawn` until pressure clears, alerting at most hourly. Every kill and pause lands as a durable orchestrator message, so degradation is visible and attributable, never silent. The same sweep reaps codex app-server children whose host process died without cleanup (hosts drop a pidfile beside their socket; only children of agents this daemon knows to be dead, still running the expected binary, are killed — an unknown id may belong to another hive instance) and prunes settled history — old events, applied messages, resolved approvals — so the daemon's own tables aren't the slow leak. The transport buffers on the codex control path get ceilings for the same reason: a peer streaming garbage without newlines costs one dropped frame, not unbounded growth. The Channels bridge's own pump loop (`ChannelBridge.pumpLoop`) got a narrower guard at the source: three consecutive iterations under 5ms trip a 20ms floor delay before the loop continues. The real daemon long-poll either waits up to `pollWaitMs` (25s) or returns a backlog after genuine network latency, so this never fires on the success path — it only catches the exact shape of a fake or broken poller that resolves instantly and repeatedly, which is what turned the pump loop into the OOM's allocator in the first place. The process-ceiling sweep still catches a runaway after the fact; this stops that one failure shape before it spins up.

Alternatives that lost. OS-level limits (`setrlimit`, launchd caps) constrain address space, not resident memory, and on macOS a virtual-size cap either kills healthy Bun processes (which reserve enormous address ranges) or never fires; there is no cgroup equivalent to delegate to. Killing the whole agent session on breach punishes a working agent for one runaway child — killing just the process keeps the agent's context and lets the orchestrator decide whether to retry. Trusting the orchestrator to notice was rejected on arithmetic: the machine died about four minutes after the first runaway appeared, faster than any model-in-the-loop reaction. The accepted costs: a genuinely huge legitimate workload gets shot at the ceiling (it's config, `resources` in `~/.hive/config.toml`), the kill is a blunt SIGKILL with no drain, and the sweep reads `ps`/`vm_stat` output, which is macOS-shaped like the rest of the adapter layer.

### 13. What survives a crash

A machine-wide OOM made this concrete: every agent process died mid-task, and while the daemon's SQLite state survived intact — tasks, worktrees, branches, the message queue — each agent's in-session context did not have a recovery path. The orchestrator's only move was respawning agents from their stored task text, which throws away hours of accumulated conversation: the failed approaches, the files already read, the plan already half-executed. One agent died stuck in "spawning" and stayed there indefinitely while urgent messages to her sat queued past their deadlines, because the reconciliation sweep deliberately ignored spawning agents (that guard existed to avoid racing an in-flight monitored launch) and nothing else owned the case. The lesson generalizes: a process is disposable, a worktree is durable, and the conversation inside the tool is the expensive thing in between — so hive must be able to put a process back around a surviving conversation.

Both tools make that possible, natively. Claude Code resumes any session by id (`claude --resume <session-id>`), and every hook invocation carries the current `session_id` on stdin; Codex resumes any recorded rollout (`codex resume <session-id>`, taking the same `-c` config overrides as a fresh launch), and its `notify` payload carries the conversation's `thread-id`. Hive already owns both hook paths — Claude hooks and Codex notify both invoke `hive event` — so session identity capture is one field riding on traffic that already flows: the event CLI reads the hook's stdin JSON and the notify payload, and the daemon records the latest `toolSessionId` on the agent row. Capture is continuous rather than spawn-time because a Claude resume forks the conversation to a *new* session id — a one-shot record would go stale after the first recovery. When capture never happened (a Codex agent that crashed before its first turn ever completed), recovery falls back to disk discovery: Claude transcripts live under `~/.claude/projects/<munged-worktree-path>/` where the newest transcript is the session, and every Codex rollout under `~/.codex/sessions/` opens with a `session_meta` line naming its session id and cwd. Assigning the id ourselves at spawn (`claude --session-id <uuid>`) was rejected as the primary mechanism for the same fork reason: it only names the first session, so continuous capture is needed anyway and the flag adds nothing.

Recovery itself is the reconciliation sweep grown into classify-and-recover, running at daemon startup — after a machine crash, the daemon restart *is* the recovery moment — and on the same 30-second tick as before. Any agent whose status claims a process but whose tmux session is gone gets classified. Active work (working, idle, awaiting-approval, stuck) with an intact worktree and a known-or-discoverable session id is resumable: hive relaunches the tool with its native resume in a fresh tmux session under the same name and worktree, reopens the viewer, marks the agent idle (a resumed TUI sits at its prompt), tells the agent what happened through the ordinary message path — check your inbox, re-verify in-flight edits, continue — and flushes its queued messages. An agent that died during spawn has no conversation worth resuming and is marked dead with its worktree preserved and its stored task surfaced to the orchestrator for a respawn; the in-flight-spawn guard survives as a check on the spawn-name reservation, which only a live spawn in the current daemon process can hold because startup clears reservations stranded by a crash. Unresumable active work (worktree gone, no session found, resume launch failed) takes the same dead path. Every death also denies the agent's stale pending approvals and flags its queued messages once, so nothing waits forever on a corpse the way it did in the incident. Control-paused and write-revoked agents are explicitly excluded from resume: the critical-control machinery already owns their restart semantics, and recovery resuming around a revocation would undo decision 1's whole point.

Automatic-on-startup was chosen over a manual-only `hive recover` command because a machine that just crashed has nobody guaranteed at the console, and requiring a human step reproduces the stuck-forever failure. The manual command exists anyway — `hive recover [name]`, and `hive_recover` over MCP — for the after-the-fact cases automation refuses: it can resurrect an agent already marked dead, and it bypasses the auto-resume attempt cap because a human explicitly asked. That cap (three attempts, re-armed by a completed turn) is what keeps automatic resume from becoming a crash-loop amplifier: a process that keeps dying on its own is a bug, not a crash, and after three tries it surfaces as an explicit death instead of a fourth relaunch. Codex agents recover through the TUI resume path regardless of the configured driver — an app-server connection died with the daemon anyway, and the TUI is the maintained interactive default (decision 2).

What this gives up, honestly. A resumed conversation is not a checkpointed process: a tool call that was mid-flight at the crash simply never returns, which is why the recovery notice tells the agent to re-verify its in-flight edits rather than pretending continuity. Quota accounting treats the resumed process as the same logical run — the original reservation, kept-with-estimate at death per decision 6, is not re-reserved — so a manual resurrection of a long-dead agent runs briefly unaccounted; acceptable for a rare operator action, and the conservative ledger absorbs it. And resume trusts the tool's own session store: if `~/.claude` or `~/.codex` was lost with the machine, recovery degrades to exactly the respawn-from-task fallback it was built to avoid — which is the correct floor, not a failure of the design.

### 14. The name for all of this

Stepping back, the fourteen decisions compress into one architecture:

- **The daemon** is the keystone: message bus, status table, approval router, memory store, run history, resource watchdog. Agents speak to it over MCP; it speaks back through the best interactive path each driver supports — Claude Channels, tmux for the Codex TUI, or the configured Codex app-server.
- **tmux** is the substrate: sessions are where agents live, windows are optional lenses, and send-keys/capture-pane are the Codex TUI actuator and sensor as well as the fallback when a native channel is not there.
- **The orchestrator** is a read-only classifier: it decomposes, scopes, tiers, and spawns; it never touches files.
- **Agents** are sandboxed, worktree-isolated, hook-instrumented, and disposable — recycled at the context quality line via handoffs, and resumed with their conversation intact when a crash, not a decision, ended their process.

## Tech stack

One language for the whole system: **TypeScript on Bun, compiled to a single binary.** The decisive constraint is MCP — the daemon *is* an MCP server, and the official TypeScript SDK is the reference implementation: most mature, first to get spec features, runs on Bun. Bun buys the three things a CLI-plus-daemon needs: fast cold start, built-in SQLite (`bun:sqlite`, no native-module pain), and `bun build --compile` for a single distributable binary — one binary, no runtime, not "have the right Node version." Hive owns that binary's distribution: a `curl | sh` installer drops immutable version directories under `~/.local/share/hive` and activates one with an atomic symlink, `hive start` checks for a newer release and says so in one line, and `hive update` stages, verifies, and activates a new build only when no team is live. `brew install hive` was the original promise and remains a secondary channel, but it cannot be the primary one: Homebrew never upgrades a formula on its own, and it knows nothing about a running daemon holding approvals and landing authority. Because a Unix process keeps executing its open image after the symlink moves, an updated binary must never adopt the daemon it just replaced — the handshake's content-addressed build hash is what refuses it, and `hive update` restarts it. See [docs/versioning-and-release.md](docs/versioning-and-release.md), whose one rule everything else follows: one push to `main` is exactly one patch release, and no version number is ever typed by a human. Everything hive does is process-glue and JSON, not compute, so a systems language buys nothing here.

Alternatives considered: Go is the right shape (static binary, daemon ergonomics) and would win if the MCP TS SDK didn't exist, but its SDK is younger and we'd write more code for the same result in a TS-heavy ecosystem. Rust pays systems-language tax on a scripting-language problem. Python has a miserable CLI distribution story.

The concrete choices:

- **Daemon MCP transport:** Streamable HTTP on localhost — one daemon, one endpoint, N agents. Both Claude Code and Codex configure HTTP MCP servers natively; stdio would mean a process per agent.
- **Storage:** SQLite via `bun:sqlite` — inboxes, status table, run history, one file under `~/.hive/`.
- **tmux / git / osascript:** shell out via `Bun.spawn`. No wrapper libraries; the CLIs are the API.
- **Hook glue:** the hive binary itself — Claude hooks and Codex `notify` invoke `hive event …`, which POSTs to the daemon. One code path, no curl scripts.
- **Daemon lifecycle:** on-demand spawn with a pidfile; first `hive` command starts it. launchd is v2 polish, not v1 plumbing.
- **Schemas:** Zod, shared across messages, handoffs, and the routing table (the MCP SDK is Zod-native). CLI parsing via Commander; tests via `bun test`, with the interesting coverage being integration tests driving real tmux sessions.

One deliberate boundary: emulator adapters (osascript for iTerm2/Terminal) and tool adapters (spawn flags, hook-config writers, interactive drivers, and native control channels such as Channels and app-server) live behind small interfaces from day one. Those are the only two places vendor and platform specifics are allowed, and they're exactly where the roadmap grows (new terminals, OpenCode).

## Roadmap: the dogfooding arc

The sequencing principle: nothing in v1 learns. Learning features compound on top of a working loop and are wasted effort if the loop isn't proven. So: build the core loop until it's usable, then **use hive to build hive**, then add the learning layer with the tool's own help.

**v1 — prove the thesis.** Daemon (SQLite, MCP: spawn/status/send/inbox, send-keys delivery — decision 1's fallback path; the native channels land in v1.1). `hive claude` / `hive codex` launching a read-only orchestrator with hive MCP attached. Agent spawning: named agents (human first names from a pool), worktree + branch per writer, sandboxed, tmux session + iTerm2/Terminal viewer window, the window wall auto-tiled around a centered orchestrator. Hooks wiring both directions; context % from transcripts, warn at 65% (no auto-respawn yet). Tier × tool routing with durable quota reservations, honest telemetry confidence, and proactive limit alerts. Committed `AGENTS.md`. Single approval queue. Landing protocol in every writer prompt (rebase → retest → ff-only merge), integrator agents only for escalated conflicts and stranded work. Done means: you can sit in one window, ask for a feature, watch a Claude agent and a Codex agent build it in parallel, approve one escalation, and see it merged.

**v1.1 — make it durable (built by hive).** The Codex adapter has two proven drivers. The default interactive TUI keeps the pane writable by the human and uses notify, rollout sensing, and tmux delivery. The configured app-server alternative drives turns, steering, interruption, lifecycle, and approvals, while `account/rateLimits/read` and its update signal feed the quota store at authoritative provenance; a failed handshake falls back to the TUI. Claude-side Channels delivery is built the same way: the spawner version-gates the preview, the bridge subprocess carries messages and the permission relay both directions, and the statusLine hook feeds the subscriber's five-hour and weekly used-percentages into the quota store at `reported` provenance — so both vendors have a native control channel with tmux as the maintained fallback. Live quota discovery is built on top of both: every hive start probes Codex's app-server and Claude's `get_usage` control request for the account's real windows, so routing reserves against measured headroom on a machine that has never seen a `quota.toml`. The resource watchdog from decision 12 is built: per-process ceilings under hive sessions, spawn pause under system memory pressure, orphan reaping, and history pruning all run on the daemon's maintenance tick. Crash recovery (decision 13) makes agent sessions survive daemon and machine death: session identity rides the hook path, and the reconciliation sweep resumes vanished processes into their actual conversations. The remaining highest-ranked work is landing receipts, then handoff-and-respawn automation, the two-scope memory store with index injection (the built Markdown + SQLite-FTS core from decision 5), `hive watch` and reattach polish, stuck-detection heartbeat escalations. Task-DAG/WIP scheduling, worktree environment provisioning, and eval telemetry stay deliberately behind all of this, per the sequencing principle: each compounds on a proven loop and is wasted effort without one. This phase is the dogfood gate: if hive can't comfortably build these features itself, that's the most important bug report we'll ever get.

**v2 — make it self-improving.** Skills rule-of-three proposals and authoring. Auto-updating routing manifest. Cross-vendor review routing. Third tool (OpenCode — it is already the ecosystem's common third adapter, so it both cements vendor neutrality and tests our adapter boundary against the same bar everyone else set; Grok is a possible fourth).

## Open questions and deferred bets

Named honestly, not papered over:

- **Claude Channels is built on a research preview.** Decision 1's channel path is implemented and version-gated, but the dev flag, the allowlist, and the notification shape can all change under us without notice. The tmux fallback stays maintained rather than deleted for exactly this reason, and an env-gated live test (`HIVE_LIVE_CHANNELS=1`) drives a real CLI end-to-end as the canary for the day the contract moves.
- **Self-land versus a review gate.** Every writer lands its own finished work through rebase → retest → ff-only (decision 3), and industry practice leans hard the other way: Devin, CrewAI, Conductor, Vibe Kanban, and Intercom's agent PR pipeline all keep a separate review stage before merge. Our structural safety net — green tests on exactly the tree main becomes — is real, but it is not review: it catches breakage, not bad design, missing tests, or a plausible misreading of the task. Whether the orchestrator-plus-tests loop is enough, or whether landing needs a gate (v2's cross-vendor review routing exists partly because we suspect it), is genuinely open — and we are currently on the lonely side of the industry.
- **Task-shape similarity** for the skills rule-of-three is undefined — embedding similarity? Orchestrator judgment over descriptors? Deferred to v2 with the feature.
- **The parallelism ceiling.** The 2–4 writer default is a heuristic we chose, not a consensus we borrowed, and we haven't tested where hive's coordination actually breaks. The risk is worse than "slower": Google Research measured multi-agent setups degrading performance 39–70% on sequential tasks — some work gets *worse* when split, not merely less efficient. The orchestrator should learn to say "this task doesn't parallelize," and that judgment is unbuilt.
- **The existential bet.** Agent teams going multi-vendor, agent-orchestrator growing a real control plane, or Conductor growing an AI orchestrator would each compress our gap. The counter is speed and depth on the safety stack — being the layer no single vendor can be. If that stops being true, hive is a very nice personal tool and that's the honest floor.
