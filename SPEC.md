# hive

> **Architecture scope.** This document records the current Bun/tmux/Terminal.app system and the control-plane decisions learned from dogfooding it. The accepted destination is the native [Hive Workspace blueprint](docs/architecture/hive-workspace-blueprint.md). The two documents agree on the substrate: agents are the vendors' native TUIs living in daemon-owned tmux sessions, and every window — terminal or Workspace pane — is a disposable viewer over them. Where they still differ (process ownership, project identity, IPC), the blueprint is authoritative for future work; Phase 0 control-plane authentication is landed, and the flagship Workspace attaches to this system's daemon and tmux substrate rather than waiting on a restart (order preserved in the [handoff](docs/architecture/restart-handoff.md)).

## What this is

You `cd` into a project and type `hive claude` or `hive codex`. The Workspace app opens with the selected AI orchestrator in its master terminal pane, scoped to that folder. You talk to it like a tech lead: "build this feature," "figure out why the tests are flaky," "have Codex take a second look at the auth code." It doesn't write the code itself. It decomposes the work, spawns agents — Claude Code here, Codex there, big model for the hard part, cheap model for the changelog — and each agent appears in a satellite terminal pane where you can watch it work. Agents talk to each other when they need to, regardless of vendor. When the work is done, it gets merged, and the panes go away.

Why build this when 2026 is drowning in agent orchestrators? Not because nobody ships a cross-vendor AI orchestrator — that claim is dead. AgentWrapper's agent-orchestrator (8.2k stars) combines an AI orchestrator, mixed Claude/Codex workers, worktree isolation, a Go+SQLite daemon, and PR-feedback loops; Superharness plays in the same category with a genuinely different architecture. The rest of the field still splits the familiar two ways — human-driven multi-tool (Conductor, Vibe Kanban: they run Claude and Codex side by side, but *you* are the orchestrator, dragging cards) and AI-driven mono-tool (Claude Code's agent teams: an AI lead that only ever delegates to other Claude sessions). What none of them ship — and what separates a demo from a tool you can walk away from while N agents hold shell access on your laptop — is the safety and control stack: **quota-aware tier×tool routing that durably reserves subscription-CLI capacity before spawning, durable peer delivery with critical controls that mechanically revoke authority rather than politely request compliance, an orchestrator that physically cannot write, and a rebase→verify→fast-forward-only landing protocol that makes "the tests and the typechecker passed on exactly what main now is" structural.** That stack is hive. Worktrees, tmux, sandboxes, windows are table stakes we build because we must; the control plane over cross-vendor agents is the product.

One honest strategic note before the architecture: we're building in a gap several parties could close — Anthropic or OpenAI shipping cross-vendor teams, or agent-orchestrator growing a real control plane. Our defensible ground is the safety stack plus vendor neutrality, so the roadmap leans into adding a third tool early rather than polishing a two-tool duopoly. Two things we watch without imitating: Cotal, an emerging cross-harness messaging standard (174 stars, pre-1.0, no confirmed Codex adapter) — if it matures, decision 1's daemon becomes an adapter target, not a casualty — and the learning layer, where the field is thinner than its marketing: Devin's knowledge store is manually curated, not an automatic loop, and Ruflo's cross-run learning claims are unverified, so the v2 self-improvement ambition is less pre-empted than a feature-matrix read suggests. Speed matters. macOS only, by choice, for now.

## How it works

The design fell out of fourteen questions, each of which had a way to go wrong. Walking them in order, because each answer constrains the next.

### 1. How agents talk to each other

The problem nobody states out loud: Claude Code and Codex are turn-based loops. They act, then stop and wait for input. Nothing interrupts them mid-task, and nothing polls while they sit idle. So "agents can message each other" needs two mechanisms, not one — a place for messages to live, and a way to *wake the recipient*.

The place to live: both tools speak MCP (Codex via `config.toml`, stdio or HTTP), and MCP is the only API surface identical across vendors. So hive runs a small local daemon that owns a SQLite message store and exposes itself to every agent as an MCP server: `hive_send`, `hive_inbox`, `hive_list_agents`. Same send path whether you're Claude or Codex.

Spawn and send can race: an orchestrator may name a recipient and another agent may send to it before worktree creation has produced an agent row. Hive therefore durably reserves the name in SQLite at the synchronous start of spawn. Sends to a reserved name are accepted into the same durable message table, then delivered on the recipient's session-start event or claimed by its first `hive_inbox` poll. The reservation is released whether spawn succeeds or fails. A name with neither a live agent row nor an active spawn reservation is rejected explicitly as unknown; terminal (`done`, `dead`, or `failed`) recipients are rejected too. This keeps typos from creating immortal orphan queues while closing the legitimate startup window.

The wake-up depends on what keeps the agent genuinely interactive. Claude Code (v2.1.80+) has Channels: an MCP server that declares the `claude/channel` capability can push `notifications/claude/channel` notifications into a running interactive session; a busy session queues the event for its next turn, and v2.1.81 adds a `claude/channel/permission` relay for approvals. Channels only load over stdio servers, and the hive daemon is an HTTP server, so the daemon cannot be the channel directly: a Claude session instead registers a thin `hive channel-bridge` stdio subprocess that declares both capabilities to the CLI and long-polls the daemon for queued deliveries. The bridge holds no state that matters. Only the orchestrator gets one, for the reason decision 4 works out: a `server:` channel costs a launch dialog, and the orchestrator is the one Claude session with a human sitting in front of it. Codex has a programmatic app-server too, and the former claim that its host was necessarily headless was false: 0.144.0 documents `codex app-server --listen …` plus `codex --remote …` for the interactive TUI, while the open-source thread state holds multiple client connections. A driven proof attached that TUI, resumed the same thread from a second JSON-RPC connection, and injected a model-visible item that the model returned, all without altering the TUI composer. `hive codex` launches the root that way — an app-server authority behind a `--remote` TUI — because the proof closed the headless objection that used to block it. The daemon drives that second connection itself: `CodexRootDelivery` resumes the root's own thread and injects a model-visible item on every `hive_send` to the reserved orchestrator destination, dual-client steering exactly as the proof validated. It is lazy and disposable — a failed connect or delivery reports unconfirmed rather than throwing, the message stays durably queued, and the daemon falls through to Claude Channels, so a Claude-root machine (no Codex socket) never even attempts the connection and a stale socket left by a dead Codex root costs one failed connect, never a lost wake. The launcher and the daemon derive the same socket path from one function (`codexRootSocketPath` in `src/cli/orchestrator.ts`) so the two sides can never drift onto different sockets. Reaching that root at all still needs a credential: the operator's launcher asks the daemon for one through a dedicated, operator-only mint endpoint (60-second TTL, one sanctioned issuance outside the daemon's own spawn path), which the launcher writes to a 0600 file and hands to Codex on boot.

Tmux injection is therefore the primary path for every spawned agent of either tool — never for a root session, whose only wake path is a live native channel. Ordinary messages are `normal`: if the recipient is idle at its prompt, the daemon injects "📨 message from maya: …" into its pane; if it's mid-task, the message queues and a turn-boundary hook drains the inbox. Event-driven — no polling anywhere. The injection itself is `load-buffer`/`paste-buffer -p` (the body arrives as one bracketed paste, so embedded newlines become composer newlines instead of premature submits, and nothing passes through a shell), then a pause, then a single `send-keys Enter`. The pause is load-bearing: both TUIs coalesce input that arrives in a burst as a paste, so an Enter fired immediately after the text lands *inside* that window and becomes a literal newline — the message sits in the composer until a human presses Enter, which is exactly the failure the first field test hit. The obvious simpler shape — `send-keys -l` with the text and an immediate Enter — lost twice over: the Enter race above, plus raw keystroke newlines submitting partial messages. A verify-and-retry loop (capture the pane, re-send Enter if the text is still sitting there) was rejected because misreading a busy pane double-submits; one delayed Enter is the ecosystem-converged fix. Tmux paste is timing-sensitive and cannot confirm receipt, so the native Claude channel wins wherever it is healthy. For Codex that mechanical weakness is the accepted price of leaving the human inside the real conversation; a reliable actuator that removes the reply box is not the better interface.

Directions that reduce authority cannot ride that ordinary queue. The incident that forced the distinction was simple: an agent began one long implementation turn, the user changed the assignment to "pause before coding," and Hive faithfully stored the message while the agent kept writing because the next turn boundary never came. The model did not disobey; it never saw the instruction. Messages therefore have explicit priority and intent. `urgent` waits for the nearest safe lifecycle boundary, injects before ordinary traffic, and requires an agent acknowledgement by a durable deadline. `critical` covers pause, stop, cancel, and write reduction: the daemon atomically advances the agent's capability epoch, revokes landing, denies its stale pending approvals, then kills only that agent's tmux session, preserving its worktree and transcript. It restarts the same worktree in a fresh read-only process with the control at the top of context, using the immutable execution identity recorded at the original launch rather than resolving today's route. The interrupted reservation settles once; the acknowledgement process receives a separate reservation tied to the control message, so concurrent routing sees its full cost without charging the interrupted run twice. The old viewer — now a lens on a killed session — is replaced and the wall re-tiles (decision 10). The agent must acknowledge the matching message and epoch; acknowledgement settles the control reservation, and stale acknowledgements cannot revive authority. Explicit structured intent wins. A deliberately narrow legacy phrase classifier catches old callers saying exactly "stop," "cancel," "pause before coding," or "do not modify" without turning conversational uses of those words into process kills.

Every message carries a monotonic recipient sequence and the durable states `queued`, `injected`, `agent-acknowledged`, and `applied`, with separate timestamps. `deliveredAt` survives as a compatibility alias for injection, but the send result exposes the real state; persistence is never reported as delivery. Optional sender idempotency keys make retries exact. Urgent and critical deadlines wake the orchestrator once when missed, and restart recovery resumes queued critical controls before normal work. This costs a process restart for the strongest control and cannot undo a shell command that finished before revocation, but it closes the dangerous future: after revocation the old process is gone, the replacement is read-only, and the daemon refuses a stale landing epoch. If that replacement cannot launch, Hive cannot atomically resurrect the killed writer across an OS boundary, so it completes the operation into a terminal failed state, releases every reservation the ledger says the agent holds, and alerts once instead of retrying the same launch forever. Leaving the agent `control-paused` was rejected because it stranded revoked authority and scarce quota behind a recovery loop that had no new information. Prompt-only interruption was rejected because it repeats the original failure. Injecting critical text into an approval dialog was rejected because a TUI prompt is not a safe message boundary.

The root orchestrator uses the same substrate without becoming a fake agent. `orchestrator` is a reserved destination, absent from the agents table, and each root run starts fresh in the one instance-scoped tmux name. Tmux session names are machine-global, while a Hive instance is selected by `HIVE_HOME`, so every session ends in the first ten hex characters of SHA-256 over the resolved absolute home: `hive-orchestrator-<instance>` for the root and `hive-maya-<instance>` for an agent. The stable name identifies ownership, not conversation continuity: launch refuses to disturb a session with an attached client, kills an unattached stale session, then creates a new one without attach-if-exists behavior. Closing the terminal may strand a detached tmux server long enough for this launch guard to reap it; `hive stop` and clean daemon shutdown reap it eagerly. An agent send to the reserved destination durably inserts the full message, then pushes one compact `hive.message` envelope through a verified native Claude Channels bridge registered as that reserved root. The bridge acknowledgement means only that it wrote the notification to Claude's transport, so Hive marks the row `injected`, never `applied`; Claude itself queues a busy-session notification for its next turn without Hive touching the human's chat area. Delivery and inbox claims share one serialized lane, while SQLite insertion order supplies a stable order for concurrent sends. The envelope is capped at 2 KiB and contains only id, sender, time, a bounded body preview, and a `hive_read_message` reference. Oversized reports therefore stay in SQLite until the orchestrator deliberately asks for one; full team state, pane output, task restatements, and previously acknowledged messages never ride along with a wake.

The Workspace's private root child is therefore a supervisor, not a one-shot launcher. If the provider process exits, it asks the daemon for full status and starts a fresh root only when that read positively proves at least one nonterminal agent remains. An unreadable daemon is unknown, so the supervisor waits with bounded backoff rather than converting silence into either permission to restart or evidence that the session is over; zero live agents means a failed startup or a finished session and the original exit stands. Closing Workspace kills the supervisor itself, and `hive stop` closes agent holders before the root, so neither intentional boundary resurrects anything. A replacement declares itself a backup, receives the old process's exit code plus a bounded snapshot of every live agent's provider, model, status, branch, worktree, last event, and task, and sends each agent a non-destructive recovery request for its current files, blockers, and next action before it starts. It then re-reads status and inbox instead of trusting the snapshot. The vanished tmux session takes the dead root's view with it; the same Workspace master pane hosts the next fresh session. Resuming the old root conversation lost because Hive cannot prove why that process exited or whether its context contributed, while putting this loop in the daemon lost because the daemon does not own the attended pane and could relaunch after the user closed Workspace. Swift-only recovery lost for the inverse reason: the viewer does not hold durable agent truth. The accepted cost is root conversational context — team state is reconstructed from SQLite and measured agent replies — and a crash loop is slowed exponentially rather than allowed to spin.

This makes Claude Channels the Claude root transport, not an optional acceleration; the Codex root's equivalent is `CodexRootDelivery`, addressing the app-server authority's thread as a second client. `hive claude` enables the bridge and refuses to launch unless the installed Claude supports Channels. The fallback question has no clever answer when a root's channel is unconfirmed — a dead socket, a closed connection, a root that never finished coming up: an unread indicator plus `hive_inbox` leaves the model unaware until somebody polls, which is delayed delivery under another name. Pane injection is forbidden because it owns the human's composer. The short-lived preflight-and-defer approach — capture the root composer and wait until it was empty — fixed draft clobbering, but lost because it still owned the chat area and delayed reports while the human typed. Verify-and-retry lost for the same double-submit risk it has in agent panes.

Alternatives we rejected: a pure file inbox ("agents check for messages") fails exactly when it matters, because an idle agent never checks. Routing everything through the orchestrator as a relay makes it a bottleneck and fills its context with traffic. Unscoped names such as `hive-orchestrator` and `hive-maya` were the original choice and lost after an isolated scratch daemon stopped the real orchestrator through their shared tmux server. Port scoping does not help because tmux does not know Hive ports; random suffixes isolate but make names unstable across restarts and leave no bounded target for orphan cleanup. Hashing the resolved home makes the existing instance boundary visible to tmux. Reattaching the root to a surviving session under that stable name also lost: closing a terminal detaches its tmux client without ending the provider, so the next `hive claude` woke inside dirty context from the previous run. Fresh creation under a stable ownership name keeps cleanup exact without inheriting a conversation. The transition exception is deliberately one-way: only the default `~/.hive` installation may discover and stop legacy unscoped sessions, while a custom `HIVE_HOME` never matches them. This lets the normal installation migrate without reopening the scratch-instance footgun, at the cost that an old custom-home session needs manual cleanup. What we give up: agents must run inside tmux, and an active root must be closed explicitly before another can start. Which turns out to be a gift, not a cost — see below.

### 2. How the orchestrator knows anything

An agent in another terminal is an opaque process. The spec-fantasy version — "the orchestrator knows when agents are done, stuck, or running out of context" — needs real signals. They exist:

- Claude Code has lifecycle hooks: `Stop` fires on every turn end with the session's transcript path; `Notification` reports informational events, including but not limited to permission prompts; `SessionStart` announces birth. A generic notification is only a lifecycle signal: it never enters the approval queue or changes agent status. Genuine permission requests reach that queue through an explicit `approval-request` event or the native permission relays described in decision 4. Treating every notification as an escalation looked convenient because approval waits are one notification subtype, but it lost immediately in practice: routine notifications blocked agents, starved their message inboxes, and buried real decisions.
- Codex runs its interactive **TUI** with generated native lifecycle hooks beside the worktree's active config layer, plus rollout files as the broader sensor and tmux as the input path. `SessionStart`, `UserPromptSubmit`, `PostToolUse`, and `Stop` map to Hive's session-start, turn-start, tool-boundary, and turn-end events. The project trust override and one-run hook-trust bypass are load-bearing: Codex skips project config in untrusted directories and otherwise asks a human to approve newly generated command hooks, but Hive already generated and vetted these exact commands for an unattended agent. `PostToolUse` covers the shell, patch, and MCP tools Codex exposes to hooks, not every internal tool, so it supplies real mid-turn urgent boundaries without claiming every action is interruptible. The old `notify` override lost because Codex launched it as an unobserved child, discarded its output, and reported success without waiting; a failed child therefore made an otherwise healthy agent permanently deaf, with no session-start event or retry. Native command hooks wait and surface failure instead. This path still cannot intercept an approval into Hive's queue, but the Codex pane remains the real session: the human can answer questions and approvals where they appear.
- Context usage comes from whatever surface actually measures it, and from nowhere else. Codex states both halves outright: the app-server emits `thread/tokenUsage/updated` including the context window, and the rollout files the default TUI driver reads carry `model_context_window`. Claude states only the numerator. Its transcript records how many tokens a turn used but never the window those tokens fill, and the model id cannot supply the window either, because the 1M upgrade is a property of the account's *plan* rather than of the model — `claude-opus-4-8` is 200k on one plan and 1M on another, the string is byte-identical in both, and the CLI's `[1m]` marker never reaches the transcript. The window arrives on exactly one surface: the `statusLine` payload Claude Code pipes to `hive statusline` on every render, carrying `context_window_size` and Claude Code's own `used_percentage`. That is therefore where Hive reads it, and Hive re-derives neither number.

The rejected default was the **app-server**, a first-party JSON-RPC control plane where `turn/start` begins a turn, `turn/steer` injects input, and `turn/interrupt` stops it while lifecycle, token usage, rate limits, and approval requests stream back over the same connection. This is a real and useful interface, not a side door, so Hive keeps it as a configured alternative and automatically replaces a failed handshake with the TUI. Its tmux session runs a small host around `codex app-server --stdio` and renders a human-scale event feed. That architecture won on machine control and observability, then lost as the default on the more basic dogfooding test: the feed is read-only. When Codex asks the human a question, the pane offers nowhere to respond. A viewer that shows the conversation but cannot participate in it violates the product's promise that each agent appears in a terminal where the user can work with it. The app-server alternative accepts that limitation in exchange for native steering, Hive-routed approvals, detailed lifecycle, and authoritative quota telemetry.

So: at spawn, hive wires each agent's signal path — hook configs for Claude, `notify` plus rollout sensing for the default Codex TUI, or the app-server connection when configured. Agents self-report status and last-event through that path. Liveness is a pull: a maintenance-tick sweep reads each live agent's Claude transcript or Codex rollout — the same durable artifacts decision 13's crash recovery reads — for `lastActivityAt`. Occupancy is a push, and comes only from the surfaces above that measure it. A slow daemon-side `tmux capture-pane` heartbeat (~60s) remains a fallback stuck-detector for whatever hooks miss (hung tool call, dead process), but that maintenance never injects anything into the root.

**Read what the tool measures and hands you; infer only what nothing tells you.** This is the governing rule for every sensor here, and it was learned the expensive way. Hive used to compute Claude's context % itself, dividing the transcript's token count by a hardcoded 200,000 — a constant justified, in a comment, on the grounds that a larger window would merely read "conservatively high" and so err toward recycling early. Both halves were wrong. The window was not ours to know, and reading high is not the safe direction: recycling is not free, it discards the context we paid to build and re-pays, out of a quota pool that may be nearly exhausted, to rebuild it. Agents at ~22% of a 1M window reported 100% full.

The lesson is not "that constant was too small." It is that **the cost of a wrong telemetry number is never a wrong display — it is every decision downstream of it.** That inflated percentage fed the orchestrator's own agent-reuse rule ("reuse a live agent when its context % is under 65"), so for a full working day the orchestrator spawned fresh agents and re-paid their full briefings against a quota pool that reached 99%, precisely because the agents it should have reused looked full. Nobody saw an error. The same disease produced the unbound quota pool, the stale spawn-time model, and the readiness probe: in every case Hive was already being told the truth and chose to infer something else instead. A number the orchestrator acts on is never safe to guess in either direction.

Which is why **unknown is a first-class value here, and never a flattering default.** A measurement Hive does not have is reported as unknown — not 0, not 100, not a plausible-looking constant — and a consumer must treat unknown as "do not act," never as "empty" or "full." A missing number stops a bad decision; a wrong number causes one.

Birth is the fragile part, and the first field test proved it: all four inaugural agents died in their first second (unsupported models, a broken CLI) while hive reported "spawning" forever and left orphaned windows. So "spawning" is a monitored phase, not a fire-and-forget: the daemon watches until something proves the agent is alive, or until it has been silent long enough to prove it is not. Session vanished or a launch-failure signature printed means status "failed" with the pane's last lines as the recorded reason, the session killed, and the fresh worktree cleaned up. Proof of life is the row leaving "spawning", an advancing `lastEventAt`, a fresh Codex rollout write, or — the one that matters for a reasoning model — a redrawing screen *that the agent is the one redrawing*. There is deliberately **no wall-clock deadline**, and the rejected alternative is instructive because we shipped it: the first version killed any launch that had not produced a hook event or a rollout write within fifteen seconds, which measures *acting*, and a high-effort model thinks before it acts. Deep-tier Codex was therefore indistinguishable from a corpse and could not be spawned at all, while low-effort Codex, which greps within a second, sailed through. The variable was never health; it was reasoning effort. Worse, the rollout check had been added *as the fix for exactly this*, asserting that a fresh rollout mtime was the only early proof a Codex TUI could give — and measurement disproves it: Codex writes the rollout when the session opens and then nothing at all until the model emits, so it cannot prove life during reasoning either. What a thinking agent does emit is a redrawing screen; the TUI increments its `Working (12s)` timer once a second, measured on 24 of 24 consecutive polls through a pure reasoning phase.

But a redraw only proves that *something* is running an event loop, and hive's own launch guarantees that something need not be the agent: the provider runs inside a wrapper subshell (so a provider that calls `exit` cannot bypass the diagnostic hold), which makes the pane's process a shell and the agent merely its child. Anything that shell prints — a spinner, a clock, a progress line — redraws the pane whether the child behind it is alive or dead. Constructed and measured: a wrapper animating once a second over a child that had already exited changed the pane on 5 of 5 polls, and hive called that a healthy launch. **So a redraw counts only while the binary hive actually launched is still alive in that pane's process tree**, checked every poll. It is the launched `argv[0]`, never a provider name: the Codex app-server path runs `hive codex-app-server-host` and not `codex` at all, so a check looking for "codex" would kill every app-server agent it was meant to protect. The tempting discriminator loses on measurement — tmux reports `pane_current_command` as `zsh` for a perfectly healthy Codex agent and `bash` for the dead-child wrapper, because both are shells, so the foreground command cannot tell an agent from its wrapper. Only the tree names the agent instead of describing its screen. The residue we accept: a child that is alive but wedged *behind a wrapper that animates* still reads as alive, because no screen can distinguish it — hive's wrapper prints nothing while the child runs, so today there is nothing to animate it, and the honest bound is that this check proves the process exists, not that it is making progress.

Failure is therefore twelve consecutive silent polls — no event, no artifact write, and no redraw belonging to the agent — which bounds *redraw silence* (measured at 1 Hz) rather than *reasoning time* (unbounded, and no fixed number can be right). Silence means the agent gave us nothing, which is not the same as a still screen: a pane animated by a wrapper over a corpse is louder than silence and just as dead, and it counts here exactly as a frozen pane does. Both halves are verified against the real thing rather than argued — a live high-effort Codex reasoning for 26 seconds with no hook event, no rollout write and no tool call still reads alive, and the dead child behind the live wrapper no longer does. Exhausting that is still a failed launch, not tacit success: the earlier shape let an unproven process sit in "spawning" forever, silently holding its name, worktree, and quota reservation, so it fails exactly like a vanished session does — killed, cleaned up, reservation released, orchestrator alerted. This is stricter, not looser: a genuinely hung launch is now caught by a positive test for silence rather than by outliving a stopwatch, so it dies sooner than the timer used to allow. A 30-second reconciliation sweep separately handles any agent whose status claims a process but whose tmux session has disappeared — and because a vanished session is usually a crash, not a completion, that sweep tries to bring the agent back before declaring it dead (decision 13). The alternative — trusting that a created session equals a running agent — is exactly the silent-death mode the field test exposed, and "failed" with a captured reason is a first-class status precisely so the orchestrator can see it and retry a tier up. The orchestrator uses the daemon's status tool only on demand and never scrapes a terminal. The alternative — orchestrator eyeballing capture-pane dumps — burns its context on screenshots and turns the conductor into a babysitter.

The orchestrator is idle by default. It does not poll `hive_status`, `hive_inbox`, panes, logs, or worktrees. Only a user turn or a new message to the reserved destination starts a root turn. `hive_status` is an explicit user-driven read and an ownership surface agents may query themselves. Its default compact live view keeps the spawn brief as provenance rather than relabelling it as current truth, then reports how many later instructions Hive actually recorded and the latest instruction's bounded gist. It also derives touched paths from Git — dirty index/worktree entries plus the branch diff against main — and names any live agents whose observed paths overlap. This is evidence, not a declaration: it can say what files changed and what instructions were sent, but not claim which instruction the model currently obeys. Callers may project named fields, `full` means full records for live holders, and terminal history requires an explicit opt-in, so no ordinary status read can serialize the daemon's entire lifetime. Preserving the old unbounded omitted-detail response lost after it produced a 424 KB tool result; inventing a mutable “current task” lost because delivery is not proof that an instruction controls the current turn. One known wart: Claude Code's Stop-hook `transcript_path` can go stale (anthropics/claude-code#8564); the daemon reads the most-recently-modified `.jsonl` in the project dir instead of trusting it.

Identity rides on this table too, and agents get **human first names** — maya, david, sam — drawn from a pool at spawn, never numbers. Everything keys off the name: tmux session `hive-maya-<instance>`, worktree `.hive/worktrees/maya`, branch `hive/maya-auth-api`, the status row, message envelopes. The reason is that the user's entire interface is conversation with the orchestrator, and conversation wants names: "tell maya to reuse the middleware" and "what's david doing?" work; "message agent-3" forces the user to keep a mental numbering table the tool should be keeping for them. Numbered agents were the obvious default and lost for exactly that reason. Task-derived names ("auth-api") lost too: two agents can share a feature, and a name describing the task goes stale the moment the agent is respawned onto follow-up work — the worker's identity should outlive any one assignment. The cost is some whimsy in professional logs; trivial.

A name is only useful if it points at one agent, so **a name has exactly one live holder, and closure is durable.** Each holder is its own row with its own AgentUUID; `agents.name` is deliberately not unique, and a partial unique index over non-terminal rows is what mechanically forbids a second live holder. Reaching `done`, `dead`, or `failed` stamps `closedAt` once, and every surface that shows live and past agents together marks the ghosts: `maya (closed 14:11)` is never confusable with the maya who is working right now. A bare name always means the live holder — that is what message routing resolves, and a name whose holder has closed rejects sends exactly as an unknown name does. Allocation therefore prefers a name this Hive has never issued, walking a curated pool of ~600 short first names, no two of which are prefixes of each other or one edit apart (no mark/marc, no ana/anna); only when no fresh name remains does it reuse the least-recently-closed one. When every name has a live holder the spawn is refused with an error naming the fix. The pool started at 40, which recycled sam/maya/john/david across several dead agents inside a single session — the failure that motivated all of this.

Two alternatives lost. **Numeric suffixes** (`maya-2`) are never minted: they make a name that must be read carefully rather than heard, which surrenders the reason names exist. **Retiring every used name forever** was the first proposal and is stricter, but it buys nothing once closure is durable and reuse is pushed past ~600 spawns, while guaranteeing that a long-lived Hive eventually cannot spawn at all. What we accept in exchange is a residual ambiguity that no scheme fully removes: the user's scrollback is not a surface Hive controls, so after a reuse "maya reported X" refers to an agent that the live maya has never heard of. Rarity is the mitigation, not a proof. We also accept that `events` are keyed by name rather than AgentUUID, so a reused name blends its holders' event history; diagnostics only, and not worth a second migration until reuse is observed in the wild.

### 3. How agents avoid destroying each other's work

Three agents editing one checkout doesn't produce merge conflicts — it produces silent overwrites, agents reading half-written files, and everyone fighting over `.git/index.lock`. The industry answer is unanimous and we take it: **worktree per writing agent.** Each writer gets its own branch and checkout (`.hive/worktrees/maya`, branch `hive/maya-auth-api`) sharing one object store. Read-only agents (research, review) share the main checkout free of charge.

But isolation is the floor, not the strategy. The real conflict-avoidance is planning, which is exactly the orchestrator's job: land shared interfaces on the base branch *before* fanning out, and give every agent an explicit file scope ("you own `lib/auth/`"). Merges still happen serially — maya lands, then david rebases against a world that includes maya's work — but merging is nobody's *separate* job: **every writer lands its own finished work on main.**

The reason is a failure mode we hit for real: task done, tests green, and the branch sitting in `.hive/worktrees/` forever because merging was somebody else's step. Completed work stranded on a worktree branch is indistinguishable from work never done. So the landing protocol rides in every writer's spawn prompt — it's composed into `buildAgentPrompt`, so no orchestrator ever has to remember to say it: when the task is done and tests are green, commit everything, `git rebase main` in the worktree, re-run the tests *and* `bunx tsc --noEmit` on the rebased branch, then call `hive_land` with the capability epoch issued at spawn. The daemon verifies that the epoch is still current and write authority has not been revoked before it performs `git merge --ff-only` in the primary checkout. Fast-forward-only is the whole trick for concurrency: the merge succeeds only if main hasn't moved since the rebase, so two agents landing at once can't interleave — the loser's merge is simply rejected, and it re-rebases and retries, three attempts before escalating instead of spinning. A rebase *conflict* means the planning failed and two agents genuinely touched the same code; the agent runs `git rebase --abort` and escalates to the orchestrator, which spawns an integrator agent — a judgment merge is a task, never a reflex, and nothing is ever forced. Verification is tests *and* typecheck because `bun test` does not typecheck: a suite is green on trees the compiler rejects. That gap is not hypothetical — two agents each added a version module, each branch passed its own tests, and the merged main imported `HIVE_VERSION` from both and failed `tsc` outright. Tests-only verification lost precisely there: it lets "green" mean "the tests I happened to have passed," and no test either agent could have written would have caught a symbol collision that existed in neither branch. The rebase is the only moment the merged tree exists before it is main, so it is the only place the check is worth anything. Red tests never merge, and neither do type errors, in any state, on any attempt. Direct primary-checkout merge instructions lost because they left no reliable way for a later critical control to revoke landing.

Alternatives that lost. Routing *all* merges through a spawned integrator agent was the first design: it kept writers pure and matched "the orchestrator can't write" (decision 11) cleanly, but it made landing asynchronous and optional — the integrator existed only if the orchestrator remembered to spawn it, which is exactly how green work strands. Integrators survive as the escalation path (conflicts, stranded recoveries), not the default. Plain merge commits instead of rebase-then-ff lost too: a merge commit means the checks you ran validated neither parent's final tree, while ff-only makes "the tests and the typechecker passed on exactly what main now is" structural. The accepted cost: the landing merge is the one sanctioned write outside an agent's worktree — a deliberate pinhole in the decision-4 sandbox. It's tolerable because ff-only moves a ref without rewriting history and git refuses it when the primary checkout has conflicting uncommitted state; for a sandboxed Codex agent the command surfaces in the approval queue like any other boundary crossing, one approval per landing, safely auto-approvable by policy.

A refused landing has to say why, and the two things it must never do are guess and lie. It lied first: the ff-only merge ran under a 30-second deadline, and the helper reported *any* git failure as `git merge timed out after 30000ms` — because it read Bun's `Subprocess.killed`, which is true for a process that exits on its own, so the timeout branch fired on every non-zero exit. Git had already said exactly what was wrong, in about nine milliseconds, and we replaced it with a stopwatch reading of a wait that never happened. Every landing failure, of every kind, was reported as the one thing it was not. So the deadline is now a flag we set ourselves and nothing else, git's stderr is never discarded, and every blocker names the specific thing in the way: *which* file in the primary checkout has uncommitted changes the merge would overwrite, *which* branch does not exist, *how far* main has moved, *where* the held `index.lock` is. The guessing half is the harder discipline. Hive will not rebase for a writer — a rebase invalidates the green test run that earned the landing, and the whole point of ff-only is that the checks ran on exactly the tree that becomes main — and it will never discard uncommitted changes it did not write, however disposable they look. It has no way to tell a generated file from an hour of someone's unsaved thinking, and the cost of being wrong is not symmetric. Where a person must decide, the refusal ends in a single labeled `Fix:` line naming the command; where Hive could have acted itself, there is no line, because it acted (decision 14 states the rule and why the profile's nag failed it).

**A message is work, and the same guarantee covers it.** That was not true until ninety of them proved it: they had been handed to a recipient, honestly recorded as "injected", and then nothing ever looked at them again — twenty were Hive's own control alerts. Hive *knew* they were unconfirmed, which is exactly what a null `appliedAt` means, and told nobody, forever. So "injected" is now a promise with a deadline rather than a shrug: a sweep, running in the daemon's maintenance tick, either confirms the message reached the model, or surfaces it in one line naming the count and the recipients — never ninety lines dumped into the one context that has to stay clear, and never silence.

Two things make that sweep true rather than merely present, and both were learned by getting them wrong. The first is **where it reads the recipient's turn boundary**, which is decision 2's rule applied to delivery: read the surface that actually records it. A spawned agent carries `lastEventAt` on its row, but the orchestrator is not a spawned agent and *has no row at all*, so a sweep that asks the agents table for the root's last turn gets null back and concludes the root has never taken one. That is not an edge case — it is nearly the whole population. Measured against the live database: of 107 messages stuck in "injected", 105 were addressed to the orchestrator. A sweep reading the wrong surface therefore confirms nothing and announces that 102 messages were never confirmed applied — the ninety-line dump this mechanism exists to prevent, and a false one, because the root *had* reached a turn boundary after every one of them. The root's boundaries live in the events table its own hooks post, which is the only place they exist; reading that, the same sweep over the same data confirms all 105 and surfaces none. The second is that **the sweep must never surface its own output**: its alert is itself a message to the root, so if the alert can stall, the next sweep reports it, and that report stalls too — a loop with no fixed point that grows by a message every time the root is quiet. The alert is therefore born already surfaced. Nothing is lost by that, because an alert nobody read is not a new fact; it is the same fact, louder.

The third lesson arrived the night the sweep first ran at scale (2026-07-11): **"no boundary for five minutes" is one observation with two opposite causes, and an alert that cannot tell them apart trains its only reader to ignore it.** The sweep fired seven times in one evening — for three builders and the root — and every alarm was false: each recipient was mid-turn and working, one contradicted by its own completion report for the very message minutes later. A deep build turn routinely outlives any fixed deadline, so the deadline alone cannot be the alarm; but simply raising it was rejected, because that delays the real alert without curing the confusion — a busy builder and a deaf agent would still emit the same line, later. The sweep now triages before it speaks, on surfaces Hive already records: a recipient whose newest boundary is a `turn-start` has a turn open, and if it also shows a recent sign of life (every tool call refreshes the agent row's `lastEventAt`; the root's events do the same in the events table), it is BUSY and earns silence — unalerted, so every later sweep re-judges it. Everything else is named for what was measured: no turn events at all (the deafness signature), a closed turn that never followed the paste (a swallowed composer), a dead process, or an open turn silent past any legitimate single tool call (thirty minutes — one call can run a full test suite, and the historical deafness sat two hours). The same triage also watches **queued** messages, because the codex deafness incident blocked delivery before injection: a watchdog reading only the injected state was blind to the one incident it existed for. Queued-behind-a-busy-turn is routine and stays silent; queued at a recipient with no signs of life rings. Root-bound queued messages are exempt — the root's queue is its inbox, drained on its own turns, and the root is this alert's only audience.

And delivery claims only what it can prove. A normal message used to be marked "applied" — the strongest claim in the system, meaning the recipient acted on it — on the sole evidence that `tmux send-keys` did not throw, while the recipient's TUI was printing "messages to be submitted after next tool call" and holding the text unsubmitted. That is measuring bytes written to a pane and reporting that a mind changed. A paste now means "injected" and nothing more; "applied" is earned by the recipient reaching a turn boundary, which is the moment the TUI actually submits what it queued. Priority means something at the transport layer too: `urgent` and `critical` press escape first — the TUI advertises this itself — because an order revoking an agent's write authority must not sit unread in a composer while the agent keeps writing. The cost is paid deliberately and only there: an interrupt cancels the in-flight turn and the agent does not resume it, which is too destructive for routine coordination and exactly what the sender is asking for when they choose urgent.

Death gets the same guarantee: work is either merged, intentionally preserved, or explicitly surfaced, never silently lost. `hive_kill` assesses the agent's branch and worktree before any cleanup — unmerged commits or uncommitted files make it refuse deletion, keep everything, and report the stranded work in its result, so the orchestrator must actively decide: spawn an integrator to land it, mark the branch deliberately preserved, or pass `discardWork` to delete it on purpose (the discard is still reported, as the record of what was chosen). Preservation is a Git ref rather than an in-memory acknowledgement, so it survives daemon restarts and suppresses only the recurring stranded-branch alarm; it does not merge or delete anything. If the assessment itself fails, the worktree is kept — nothing is deleted on a guess, though the kill and the dead status always win. Cleanup after a successful landing is safe by construction: a landed branch tip is an ancestor of main, so deleting the branch and worktree deletes nothing. The idle reaper adds one boundary for knowledge Git cannot see: before deleting an idle, clean, fully merged agent, it sends that agent a durable warning to persist any context-only findings and waits for the warning to clear. A response, new edit, or commit naturally blocks deletion; an agent with nothing to keep clears the warning and is reaped on the next sweep. Refusing to reap every clean agent with no artifact lost because it turns the reaper off in practice, while immediate deletion lost because a clean worktree says nothing about valuable design still held in context.

The "david needs maya's work-in-progress" case gets a real mechanism instead of a hand-wave: maya commits to her branch and messages david "pull `hive/maya-auth-api` at abc123." Shared object store makes it instant. Uncommitted state is never shared; that way lies madness.

Consequences accepted: hive requires a git repo (we `git init` or degrade to single-writer if there isn't one), and "as many agents as it needs" has a soft ceiling the orchestrator should know about. We default that ceiling to 2–4 parallel writers, and we label it honestly: a tunable heuristic, not a derived constant. There is no practitioner consensus to borrow — Claude Code's own docs suggest 3–5 teammates as their soft ceiling — so the number lives in config and the open question on parallelism (below) owns the harder truth that some tasks don't just parallelize slowly, they parallelize *badly*.

### 4. Who answers the permission prompts

N agents in N windows each stopping for "allow this command?" gives you a fork: babysit every window, or let the agents run unattended with shell access on your real laptop. Hive takes the second fork deliberately, and names the cost.

**Full autonomy for writers, one deliberate setting away — and sandboxed out of the box.** A writer that stops for a prompt nobody is watching is a stalled writer, and the premise of this tool is that you walk away while N agents work. That is what `autonomy = "dangerous"` buys: Claude writers launch with `permissions.defaultMode = "bypassPermissions"` in their worktree settings, Codex writers with `approval_policy = "never"` and `sandbox_mode = "danger-full-access"`; no writer ever prompts, and the escalation queue stays empty because nothing escalates. But it is a setting, not the shipped default (flipped 2026-07-11, by user decision): the stranger who `curl | sh`'d Hive ten minutes ago has not yet decided to hand N unattended agents shell access to their laptop, and no default should decide it for them. The dial lives in the Workspace's Agents menu and in `hive autonomy <mode>`; both write through the daemon's operator-gated `/autonomy` endpoint — agent credentials cannot touch it, so no writer can raise its own autonomy — and both persist the choice to `~/.hive/config.toml` before applying it to future spawns and resumes. An absent key means the safe default; an explicit key means what it says, in both directions and across upgrades.

Autonomy is not just about the permission prompt, though, and this is where most of the mechanism lives. Claude Code opens a fresh agent worktree on up to three *separate* blocking dialogs before the model ever sees the task, each gated by different state, and every one of them is a deadlock for a session nobody is watching. Getting to the first turn means answering all three in advance — or not provoking them.

The first is folder trust: an unknown directory raises "Do you trust the files in this folder?". The CLI names its own escape hatch in the error it prints when it drops project settings — set `projects[<path>].hasTrustDialogAccepted` in `~/.claude.json` — so the spawner seeds that key for the agent's worktree before writing any config. Trust is load-bearing well past the dialog: an untrusted workspace makes the CLI *discard* the project-scoped hooks and permission rules Hive writes into the worktree, which would silently cost the daemon its event stream and cost a read-only agent its deny list. The seed is keyed on the worktree's own resolved path (resolved, because `/tmp` and `/var` are symlinks on macOS and the CLI matches the real path). That key is the narrow one: the CLI resolves trust by walking up from the session's directory, so seeding the worktree suffices, and Hive never has to trust the repository root — which would have changed the answer for the user's own sessions in their own repo. Accepting the dialog by hand records it against the main repo instead, which is exactly the broader write we decline to make.

The second is the bypass-permissions disclaimer, and here the obvious flag is wrong twice over. Claude's `--dangerously-skip-permissions` raises a blocking acceptance dialog on every launch; `--allow-dangerously-skip-permissions` does not suppress it, and accepting it does not persist. But setting `defaultMode` in `settings.local.json` does not avoid that dialog either — it raises the same one, because the CLI keys the disclaimer on the *mode*, not on how the mode arrived. What clears it is `skipDangerousModePermissionPrompt`, which the CLI honours from any settings source including the worktree's own `settings.local.json`. Hive writes it there, for dangerous writers only, so the acceptance lives inside the agent's worktree instead of depending on whatever the operator happens to have in `~/.claude`. Read-only sessions and `sandboxed` autonomy never get the key, because they never enter bypass mode.

The third cannot be pre-seeded at all, and that is what decides the Channels question. A `server:` channel — which is what the `hive channel-bridge` is — loads only behind `--dangerously-load-development-channels`, and that flag always raises "WARNING: Loading development channels" whose only exits are "I am using this for local development" and "Exit". Unlike the disclaimer, accepting it writes nothing: there is no flag, settings key, or config state anywhere that records the acceptance, so there is nothing for Hive to seed. Passing plain `--channels server:hive-channel` skips the dialog but the CLI then refuses to register the channel, which is a silent no-op — the worst of both. The only allowlist that could bless a channel, `allowedChannelPlugins`, reads exclusively from enterprise managed settings (`/Library/Application Support/ClaudeCode/managed-settings.json`, root-owned and machine-wide) and covers plugin channels only, never `server:` entries. So a spawned agent gets Channels or it gets an unattended launch, and it cannot get both. Hive chooses the unattended launch: the spawner never passes the flag, and agent messages ride the tmux fallback of decision 1. The orchestrator keeps Channels, because a human is sitting there to answer once.

The rejected alternatives are worth naming, because the pressure to take one will return. Driving the dialog by sending an Enter into the pane after launch would restore Channels for agents; it lost because it is a race against a TUI with no completion signal, and a mis-timed keystroke lands in the composer of a session that is already working. Writing `allowedChannelPlugins` into managed settings would suppress the dialog properly; it lost because it needs root, applies to every Claude session on the machine including the user's own, and would have Hive weakening a machine-wide safety default to buy itself a feature. Disabling Channels globally rather than per-spawn lost because the orchestrator's root transport (decision 1) genuinely depends on it and genuinely has a human present.

Whatever cannot be scoped to a Hive worktree, Hive does not touch. Global onboarding (`hasCompletedOnboarding`) and authentication are account-level state the operator owns; Hive assumes an already-onboarded, logged-in Claude and configures nothing outside the agent's own worktree and that worktree's single key in `~/.claude.json`.

The trade of dangerous mode is real: an agent can then run any command anywhere on the machine, so the sandbox no longer enforces decision 3's file scopes — those revert to an honor system backed by the worktree each agent is briefed with, and the daemon's resource watchdog (decision 12) becomes the only mechanical floor left. `autonomy = "sandboxed"` — the shipped default — is the posture that keeps the enforcement: Codex at `workspace-write` scoped to its worktree with `approval_policy = "on-request"`, Claude at `acceptEdits` plus a hive-maintained allowlist, and the rare escalation flowing through the daemon — a Codex approval request arrives as an app-server JSON-RPC request the daemon answers directly; a Claude permission prompt becomes an explicit `approval-request` event, or, where Channels is live, arrives through the `claude/channel/permission` relay, which lets the daemon *answer* the prompt rather than merely observe it — surfacing in exactly one place, the orchestrator window, where the orchestrator may auto-resolve by policy but anything irreversible or outward-facing still reaches the human. Generic lifecycle notifications never share that path.

The default has now flipped twice, and both flips taught something. Sandboxed was the original default and lost on the dogfooding test, not on principle: the escalation queue interrupted long autonomous runs for approvals the user would have granted anyway, and a supervisor who must be present to unblock agents is precisely the babysitting this design exists to eliminate. So dangerous shipped as the default — until the README audit (2026-07-11) found the public docs promising "sandboxed by default" while a fresh install launched writers with `bypassPermissions`. The resolution was the user's call and it reframed the question: the dogfooding pain was never an argument for defaulting *strangers* into full shell access; it was an argument for making the dial cheap. Ship safe, and put the switch in the Workspace menu where flipping it costs one click and persists. Dangerous mode stays fully maintained and fully available — the user's own workflow runs on it — it is just no longer an ambush. What made either mode affordable is unchanged: landing is still gated, red tests and type errors never merge, and `hive_land` checks the capability epoch (decision 3).

Two boundaries survive every flip of the dial, mechanically rather than by convention. Autonomy applies to **writers only**: the read-only orchestrator (decision 11) and the read-only replacement a critical control spawns (decision 1) both pass `readOnly`, which short-circuits the autonomy branch in both tool adapters — a revoked agent cannot regain shell access through a config default. And a crash-recovered agent resumes with the *current* setting — the same one the menu shows and the next spawn would get — read through a thunk at resume time. (The earlier wording promised "the posture it launched with", but that wire was never connected: the daemon accepted the option and forwarded nothing to recovery, so every resume actually came up sandboxed. The dial made the value observable, which is what exposed the gap.)

The capability that authorizes `hive_land` is itself one-shot: the credential a writer's process holds can merge exactly once, so landing a second, separately-authorized piece of follow-up work on the same credential — the ordinary case of a finished agent picking up more work in the same session — is refused outright rather than silently retried. The refusal is not a dead end. It durably files exactly one pending approval naming the agent (deduplicated, so a second refusal never queues a second approval), and the orchestrator approving it re-arms exactly one further landing (`capabilities.rearmOneShot`) — no integrator round trip, no killing and respawning the agent that did the work just to hand it a fresh grant. The original shape was a hard one-shot with no re-arm at all, and it lost the way it always does: in a single day it stranded five separately finished, gate-green landings behind a limit nobody could lift without an integrator. The one-shot itself stays — letting a single credential merge onto main indefinitely is the same unbounded authority decision 3's capability epoch exists to bound — only the dead end was replaced.

### 5. What "memory" actually means

The original spec said "cross-platform memory" once and "keep every context as small as possible" twice, without noticing they fight: unbounded shared memory dumped into every prompt is how contexts bloat. The fight is worse than token price — models degrade *inside* their advertised window, not just at its edge, and the drop is driven by how much semantically-crowded context you inject (the context-rot evidence in [docs/research/agent-memory-best-practices.md](docs/research/agent-memory-best-practices.md): 30–50% accuracy loss well before the limit, present even at perfect retrieval). A small injected surface is therefore an accuracy budget, not merely a cost budget, and that is why every mechanism below defaults to a tiny index with pull-on-demand rather than a dump. The fix is realizing "memory" was four different things wearing one name:

- **Conventions** — how this codebase works, style, commands, in a committed instructions file the *vendor* loads natively, zero hive machinery. There is no single filename to assume: Codex, Cursor, and Copilot read `AGENTS.md`, while Claude Code reads `CLAUDE.md` and sees `AGENTS.md` only through an `@AGENTS.md` import or a symlink (verified against the live vendor docs, not training memory). So hive neither picks the filename nor injects the contents — the vendor does, and *which* file a given repo uses is itself a repo-specific fact the profile records rather than a constant hive hardcodes.
- **Repo profile** — the *structured* answers an agent would otherwise burn a cold exploration rediscovering every run: build/test commands, entry points, which docs are briefable. Derived from the tree and cached in hive's own per-project state directory, regenerated whenever it drifts, and read *deterministically by product code*, not parsed out of prose. It is the load-bearing answer to a binding constraint — **nothing in hive may assume the hive repo's own layout, doc names, or conventions** — by moving every such assumption (today `brief.ts`'s hardcoded doc allowlist) out of hive's source and into a per-repo file. Its full shape and why it is a structured artifact rather than a reserved memory fact are decision 14; decision 5's only stake is the boundary — memory holds narrative truth, the profile holds structured truth, and neither stores the other's.
- **Run state** — who's building what right now, task assignments, "david finished the API." Lives in the daemon's SQLite and dies with the hive session. Calling this "memory" was the conflation to kill.
- **Durable knowledge** — the *narrative* lessons a `.toml` can't hold: gotchas discovered, decisions made and the reasoning behind them, "the flaky test is X and here's why." Immutable observations compile into owned articles in two scopes, indexed and injected as below. It never duplicates a profile field.

The durable-knowledge mechanics follow a compiler rather than a notebook. Each scope has `raw/<topic>/`, whose observations are immutable, and `wiki/<topic>/`, whose articles Hive owns and rewrites; topic directories go one level deep. `wiki/index.md` is one compact row per article and `wiki/log.md` is append-only. The same shape lives in **per-repo** `.hive/memory/` (private project knowledge, deliberately gitignored in Hive's own public repo) and **global** `~/.hive/memory/`. A raw observation records what was measured or supplied, when, by whom, its evidence, verification status, and what it supersedes. A compiled article states current truth, links every raw observation behind it, and keeps correction history or unresolved source disagreement out of the current claim. This costs more structure than one flat Markdown file, but it fixes the fatal property of the flat design: a wrong fact and `CORRECTED:` refutation could coexist forever with no machine-readable winner.

The five existing operations remain the surface rather than growing a parallel subsystem. `memory_write` records a new raw observation and creates or updates its compiled article; `memory_read` resolves an article by scope and id; `memory_search` searches article title/body only; `memory_delete` prunes the article while retaining immutable raw evidence; and `memory_reindex` migrates legacy flat files, rebuilds each Markdown index, then rebuilds disposable FTS. The first legacy migration snapshots each complete scope to the sibling `memory-backups/legacy-v1-<UTC timestamp>/` before writing inside the corpus and returns that path to the caller; a completion marker makes later rebuilds no-ops. The CLI mirrors those operations. The daemon serializes all mutations. An omitted article id derives and disambiguates a slug; an explicit id updates the canonical article. The write contract now requires `topic`, `source`, `evidence`, explicit `status` (`verified | unverified | stale | conflicted`), and a `supersedes` array; `verified` status also requires its confirmation date, and a conflicted body must actually annotate disagreement. Requiring those at the MCP schema and parsing the same schema again in the filesystem adapter is deliberate: the shipped `hive-memory` skill explains compilation and lint judgment, but telemetry already proved that progressive disclosure is not an enforcement path.

What earns a memory is still a policy, not a reflex, because an unbounded store is the accuracy problem this section opened with wearing a different hat. A lesson must be **durable**, **non-derivable**, and **load-bearing**. Chit-chat, profile restatements, and anything a `grep` answers do not qualify. Writing remains dedup-before-write: search first and target the existing article id. The mechanism now makes correction safe instead of merely asking for discipline. A correction creates another immutable raw observation, replaces the article's current account, records a supersedes edge, and retains the prior raw link; old reasoning remains auditable without masquerading as current truth. If evidence cannot decide, the article is explicitly conflicted with source attribution rather than silently picking a winner. The writer remains the agent at landing (`source: agent`); the orchestrator records its decisions; init seeds re-derivable articles; and silent auto-extraction remains deferred until this write policy is trusted.

Knowledge ages, so provenance and verification are first-class, not decoration. `source` is `init | agent | orchestrator | human | legacy`; `legacy` is the honest migration value when authorship is unknown, never invented precision. Status is explicit because absence proved too easy to ignore: `unverified` means never confirmed, `stale` means a newer write outran its last confirmation, `verified` names a confirmation date, and `conflicted` means sources still disagree. The rule those fields exist to enforce remains: **a recalled article that names a concrete path, command, or flag is re-checked against the repo before it drives an action**. The index is a pointer, the article is a claim, the repo and linked raw evidence are truth. Legacy migration leaves every flat source byte-for-byte in place, copies it verbatim into raw, derives its prior unverified/stale meaning without upgrading it, and compiles the wiki as a disposable derivative. The tempting cleanup step—deleting flat sources after copying—lost because it turned a re-runnable compile into a one-way migration for no storage benefit that matters. Validation ran against a read-only worktree copy of the 41-file live corpus, not against the live store; the live corpus may change or expose a shape that snapshot did not. A caught failure swaps the complete pre-write backup back into place before surfacing the error, while a process crash leaves every flat source and the backup intact with no completion marker so the next run can retry. If automatic restoration itself ever fails, stop and restore the reported `memory-backups/legacy-v1-<timestamp>/` directory over the affected memory scope before rebuilding again.

The context-budget rule still reconciles memory with small contexts: agents receive the merged **compiled index only** — `- [scope/topic] id (date) [status]: title`, newest-first, capped at 30 rows with an omission trailer. `buildMemoryIndex` rebuilds and then reads each scope's actual `wiki/index.md`; article bodies, evidence, and raw links never enter the spawn brief. The orchestrator gets that index at launch and each writer gets the index from its own worktree at spawn. Full articles remain pull-on-demand through `memory_read`, while FTS searches compiled articles rather than raw claims. Build-vs-buy remains decided in favor of the narrow Markdown plus SQLite FTS core: external memory servers and embeddings still add far more machinery than dozens-to-hundreds of repo articles justify. The cost of the wiki split is two Markdown writes and a tiny index rebuild per ingest; the benefit is that search, read, and spawn all agree on one converged current account while raw evidence remains auditable.

The payoff is that the second task in a repo costs less than the first, and the accounting is explicit — this is the layer's whole reason to exist. The injected index is a flat tax: 30 compact article rows plus a fixed header stay below roughly 900 tokens in the measured test no matter how large the store grows; topic and status make each row slightly wider than the old flat-fact row, but eliminate the much larger cost of opening the wrong correction. A `memory_read` is ~100–400 tokens, paid only when an article is relevant. Against that, a cold agent spends thousands rediscovering repo shape or hitting a gotcha live. The profile front-loads structured truth; compiled memory collapses the gotcha into one index row and one pull. The index tax stays flat while avoided rediscovery grows, so the tool still gets cheaper with every task — now without letting knowledge quality decay as the corpus compounds.

### 6. Who picks the model

The trap: "the orchestrator is smart enough to know which model." No, it isn't — an LLM's pricing and capability knowledge froze at its training cutoff, and model lineups turned over twice in the last year. An orchestrator routing from memory confidently picks superseded models forever.

The split that works: **the orchestrator classifies; a table resolves.** The orchestrator's judgment call is "this subtask is `deep` / `standard` / `cheap`" — a thing LLMs are genuinely good at and that never goes stale. The table is a **tier × tool matrix**: each tier names a preferred tool plus a model-and-effort entry *per tool* — `cheap` prefers Codex at low effort, but also records that a cheap Claude agent means Haiku. That second dimension exists because the orchestrator can override the tool at spawn time ("use a Claude agent for this"), and the first shape we shipped — one tool+model per tier — had nothing to say then: the override discarded the model and a "cheap" Claude agent silently launched on the account's default (expensive) model. A route that can't survive a tool override isn't a route, it's a suggestion. Mechanically this is just spawn flags: `claude --model …`, `codex -c model_reasoning_effort=…`.

A corollary the first field test taught us: **routes must not assume account entitlements.** Naming a concrete Codex model in the defaults killed every Codex spawn on a ChatGPT-plan account ("model not supported"). The default Codex column therefore says model `"default"` — pass no model flag and let the CLI use whatever the account actually has, which is supported by definition; effort flags still steer cost per tier. The Claude column names model *aliases*, which every Claude plan resolves and which auto-update as models ship: `deep` uses `best` (Claude Code resolves it to the most capable model the account can reach — entitlement-adaptive by construction, exactly this principle as a vendor feature), `standard`/`review` use `sonnet`, `cheap` uses `haiku`. We considered pinning `deep` to `opus` instead — more predictable, never subject to `best`'s safety-classifier fallback — but it caps a top-tier account below the best model it's paying for. Pinning exact model IDs is opt-in via `routing.toml`, where the user knows their own plan.

`best` currently resolves to Fable 5, and Fable draws heavily on shared subscription capacity; on 2026-07-12 it moves to usage-only billing off the user's plan entirely. Auto-routing a tier onto a model about to become metered-and-unbudgeted is the wrong default, so `deep`'s auto-selected model changes on that date — but nothing about Fable is removed. Claude Opus 4.8 (`claude-opus-4-8`) is now a first-class supported model in exactly the same sense `best` and `sonnet` are: adapter model set, quota pool matching, and routing all treat it as an ordinary concrete Claude model, with the id verified against the live Claude Platform docs rather than trusted from training memory, since model lineups move faster than any model's own knowledge of itself. `defaultRoutingTable(now)` in `schemas/routing.ts` is a pure function of one cutoff constant, `FABLE_AUTO_ROUTING_CUTOFF`: before it, `deep.claude` stays `best`, exactly as it always has; on or after it, `deep.claude` pins directly to `claude-opus-4-8`, because by then `best` resolving to a now-metered Fable is no longer the entitlement-adaptive default it was designed to be. An earlier draft of this change went further — dropping Fable from the default immediately and requiring explicit opt-in to reach it at all — and was rejected outright: Fable stays fully supported forever, reachable at any time, before or after the cutoff, by pinning `deep.claude.model` to `best` or `claude-fable-5` in `routing.toml`. The cutoff only ever narrows what auto-routing picks by default; it does not narrow what the user can ask for.

Capacity pressure is a second, date-independent path to Opus 4.8. Whenever a spawn's resolved Claude candidate is Fable — whether via `best` before the cutoff or an explicit pin at any time — the spawner adds a second Claude candidate for Opus 4.8 to the quota resolver's candidate list, placed right after Fable. Nothing about the quota resolver itself changed to make this work: it was already generic over an arbitrary number of candidates, scoring each by worst post-reservation headroom and breaking ties by preferred-tool bonus and then array order. Placing Opus second means a tie — including the compatibility case where no pool has a live reading and every candidate ties at the same sentinel score — still resolves to Fable, preserving existing behavior wherever hive cannot yet see real headroom. But when Fable's pool has worse measured headroom than Opus's, the same unmodified scoring naturally prefers Opus instead. This makes quota pressure and explicit choice the two legitimate triggers for reaching Opus 4.8 ahead of the cutoff, without the wholesale default move the rejected draft attempted.

Aliases are useful for choosing a launch, but they are not an execution identity. The first shape stored the route's model string on the agent record verbatim, so `hive_status` and every terminal title said `best` or `default` — which tells the user nothing about what's actually running and costing money. Worse, replaying the tier through a changed route table could move a critical-control restart to another provider or model. Hive resolves the concrete model once, launches that concrete model, and durably records the immutable choices needed to reproduce it: provider/tool, concrete model, tier, and Codex reasoning effort where applicable. A concrete route model passes through; `default` is read from the tool's own user config (`~/.codex/config.toml`, `~/.claude/settings.json` — the very file the CLI consults when hive omits the flag); Claude's `best` maps through a constant verified against the installed CLI, because the alias is resolved inside the CLI and no local file records it. Dynamic facts — daemon port, worktree path, hooks, permissions, terminal choice — are rebuilt rather than copied into identity. This gives up alias adaptation between the initial launch and a control restart, deliberately: changing the model during a safety interruption is a larger lie than missing a newly entitled default.

An execution identity is an *intention*, and an intention is not an observation. The user can type `/model` inside a running session, and nothing tells Hive: the row still says what the agent was launched with, forever. We found this the way you find these things — Hive believed two agents were on Fable and one on Sonnet while all three transcripts said Opus 4.8, so quota was observed against models nobody was running and `hive status` reported the same fiction back to the orchestrator, which routes off it. The fix is to stop asking one field to be both things. `model` stays the immutable launch identity, because a control restart must reproduce the launch it is interrupting; `live_model` is the observation, read from the transcript, which stamps every assistant turn with the model that produced it. Quota accounting and `hive status` read the observation and fall back to the intention. Overwriting `model` instead was the obvious move and would have been a serious bug: `restartForControl` fails closed when the recorded identity and the row disagree, so every agent whose user had typed `/model` would have become permanently unrestartable, capability revoked, at the next critical control. Conflating an intention with an observation caused the bug; the fix must not repeat it in the other direction. The observation comes from the transcript and deliberately not from the statusLine payload, which also carries a model: that report is only POSTed when the payload has a `rate_limits` block, so on an API-key account or before a session's first response it never arrives at all, and the live model would have been correct only when quota data happened to be present. Codex is not guessed at — its rollouts record no model name, so `live_model` stays null and the launch model stands.

Sometimes the concrete model cannot be known. A tool config may omit its model and leave `default` to a built-in the CLI does not expose; old agent rows also predate the immutable identity. Guessing from the current route, tier, or current tool default was rejected because each can silently change the model. These rows keep their worktree and revoked capability but do not get a read-only restart; Hive records the reason on the agent and sends a durable orchestrator alert naming the legacy or unresolved identity. The same fail-closed rule applies when a recorded model is no longer launchable. No alternate route is consulted. Downstream, a terminal viewer's title is exactly `<name> — <model>` (`maya — claude-fable-5`) and nothing else — no task text, path, tier, or session name, which leaked into titles in the first field test and made a wall of windows unreadable. Terminal.app's custom-title property does not override the profile's username, process, and dimensions fields, so Hive installs a bundled `hive-agent-v2` profile on first use and reuses it for viewers. That managed profile disables every generated title component and gives viewers a restrained blue-black background tint without mutating the user's existing Terminal profiles. Versioning matters because Terminal imports settings sets by name and does not update an already-imported profile when the bundle changes.

The table auto-updates so new releases aren't missed: hive fetches a **curated routing manifest** published with hive releases (provider model-list APIs give you names, not judgment — curation is the value), and your local `~/.hive/routing.toml` overrides always win.

That manifest design — and the shipped table underneath it — is **superseded** (2026-07-12, user directive): the binary now ships with **no model knowledge at all**. No compiled table, no candidate manifest, no alias constants; "curation is the value" was half right, and the half that was wrong was WHERE the curation lived — a document published with releases is still a hand-maintained list with a rebuild in its update path. What replaced it: routes derive per cell from **live discovery** (the account's own effective default, the one vendor-declared rank, vouched by a fresh capability record), the user's `routing.toml` (which still always wins), and a last-known-good snapshot that guards discovery outages; a cell none of those can author **refuses the spawn and names the vendor CLI hive needs**, never a baked-in guess. What still ships is policy that names no model — tier→vendor preference and tier→effort, user-overridable, the effort passed only when the resolved model's live record advertises that level. Model *judgment* (which model deserves which tier) belongs to external benchmarks consumed as data — LiveBench, the sole registered source by user ruling (2026-07-12), placing models in the LIVE derivation once the user-approved threshold/fit policy lands (his ruling: no shadow, no parallel path — the real thing, live) — and to the user's placements, never to the binary; `docs/research/dynamic-router-adoption.md` §6–7 carries the full contract, and the ten-pillar philosophy it serves.

Escape hatches: pin a vendor for a session; name a model for a single spawn — `hive_spawn` takes an optional `model`, reserved for explicit user directives ("open an Opus 4.8 terminal"), which launches verbatim with no alias resolution, binds the spawn to its vendor for quota routing — the model's vendor sets the tool when the caller names none, and an explicitly conflicting tool (a Claude model requested on a Codex spawn, say) refuses the spawn outright rather than silently picking a side — and is never silently substituted (the Fable→Opus release valve included — substituting a user-named model is a lie, so under quota pressure the spawn fails with the capacity report instead). The orchestrator choosing models from its own knowledge stays rejected — that knowledge froze at training time, which is this section's founding observation; the `model` field is a pass-through for the user's words, not a judgment surface. And bump a tier automatically when an agent fails or stalls — retry-with-a-bigger-model is cheap and effective. And because we're multi-vendor, the table can encode a genuinely novel policy: route *review* to the other vendor from whoever wrote the code.

But capability is only half of routing. Claude and Codex subscription accounts have rolling and weekly capacity, models can draw from different or shared pools, and two simultaneous spawns can spend the same apparent headroom unless selection owns a reservation. So the resolver is quota-aware: it resolves both vendor candidates, maps each concrete model to a `(provider, account, pool)` that hive discovered from the provider itself, estimates the task as a fraction of each window, and atomically reserves it before creating a worktree or process. The account and pool are explicit because keying only by model double-counts a shared vendor limit, while keying only by vendor cannot represent a premium model with its own weekly cap — and both vendors do in fact publish exactly such per-model caps. A completed turn reconciles the reservation to provider-reported consumption when a hook supplies it and otherwise commits the conservative estimate; a launch that never starts releases it, while a started agent that dies keeps the estimate. Expired reservations receive the same started-versus-unstarted treatment after a daemon restart.

A critical interruption ends one run and starts another, even though the second run is read-only. Transferring the original reservation lost the boundary between two billable processes, while leaving the restart unreserved let concurrent spawns spend its headroom. Hive therefore uses a dedicated control-run reservation keyed to the critical message and pinned to the recorded provider/model. The original reservation settles before the replacement is considered. The control estimate uses the agent's existing tier estimate — conservative but provider-neutral — and may consume the last safe capacity because preserving deep-work reserve is less important than delivering revoked authority. It never borrows from another model. A repeated recovery after a daemon crash reuses the active message reservation and surviving tmux process; a failed launch settles or releases its attempt before a retry can reserve again. Acknowledgement/application, turn completion, death, `hive_kill`, missing-session reconciliation, launch failure, and reservation timeout all settle the active control reservation idempotently. Timeout also stops the read-only process and closes its viewer so no unreserved process remains alive.

If the recorded pool lacks enough five-hour or weekly headroom, Hive does not restore writes and does not drop the control. The old process stays stopped, the capability epoch stays revoked, the worktree stays intact, the critical message stays queued, and a durable orchestrator alert reports the exact model, remaining headroom, and required estimate. Recovery retries the same identity when capacity becomes available. Compatibility mode still creates an explicit unbounded control reservation with missing confidence; disabling quota-aware *routing* likewise does not disable control-run accounting. We rejected treating safety traffic as free because invisible usage is how the next concurrent route overcommits, and rejected switching vendors because an acknowledgement from a different model is not a faithful restart of the interrupted agent.

Nobody types a quota number into hive. Both vendors ship a program-readable limit surface, both are readable without starting a model turn, and hive reads them itself at every start. Codex's app-server answers `account/rateLimits/read` after its mandatory `initialize`/`initialized` handshake, with no thread and no turn: a snapshot of `primary` and `secondary` windows, each carrying `usedPercent`, `windowDurationMins`, and `resetsAt`, plus a `rateLimitsByLimitId` map of metered sub-limits. It is stable protocol, so hive calls it `authoritative`, and its `account/rateLimits/updated` notification keeps an app-server-driven agent current for free. Claude Code has no such thing on its statusLine path — that JSON only carries `rate_limits` after a session's first response — but its stream-json control protocol answers a `get_usage` request with `subscription_type`, `rate_limits_available`, and per-window `utilization` and `resets_at`, fetched fresh and costing nothing. The CLI marks `get_usage` experimental, so hive calls it `reported`. Both facts were verified by driving codex-cli 0.144.0 and claude 2.1.206, not by reading docs: the Codex methods are absent from the prose docs, and `get_usage` is absent from everything. Windows are matched by their reported duration, never by the position they arrive in, so a plan that lists its weekly bucket first cannot silently invert the two.

Capacity and money are separate gates. The same Codex reply also carries `credits: { hasCredits, unlimited, balance }`, and Claude's usage reply carries `extra_usage.is_enabled` plus `spend.enabled`; both are enough to ask the useful question, "would this automatic spawn go beyond the plan into paid capacity?" The rule is provider-neutral: headroom launches without a prompt; an exhausted pool with paid overflow asks through the existing approvals queue; overflow that the provider proves disabled cannot charge and therefore does not ask; a billing state Hive cannot measure is excluded from automatic routing, while an explicit model pin still launches because it is the user's direct instruction. Codex exposes one irreducible uncertainty: `hasCredits: false` and `balance: "0"` prove only that no credits are sitting there now, while OpenAI supports server-side auto-top-up and the CLI exposes no auto-top-up setting. An exhausted Codex pool therefore asks even in the zero-balance state, and the prompt says that proceeding may purchase credits. Treating zero balance as disabled lost because it could silently trigger a purchase; refusing outright lost because uncertainty is not proof the model cannot run. Asking preserves both truths at the cost of one deliberate prompt only at exhaustion. A spawn that begins with headroom can still cross the boundary mid-turn because neither vendor predicts a future run's usage; asking on every launch was rejected because a prompt that fires constantly becomes a reflex and protects nothing.

The load-bearing consequence is that a provider reports the *fraction* of a window consumed and never the window's size. A discovered pool is therefore denominated in percent, its allowance is exactly 100 by construction rather than by assumption, and hive needs no capacity figure from anyone to know its own headroom. What hive cannot get from a provider is how much of a window a *future* run will consume, so reservations still spend a hive-authored percent estimate, and every surface that shows one labels it an estimate. Two estimates, not one: a run is a far larger fraction of a five-hour bucket than of a week, and a week does not hold thirty-three five-hour buckets' worth of capacity.

The rejected alternative is the one hive shipped first, and it failed in the most instructive way. Pools used to come from an operator-written `~/.hive/quota.toml`, and every observation path began by looking up the configured pool for a model. An installation without that file had no pools, so the lookup returned nothing, so hive *discarded* the authoritative percentages Codex was handing it on every single turn — then warned the operator that quota data was unconfigured and routed on its own estimates. Requiring a human to transcribe a number the provider already publishes was the root error; discarding the published number for want of the transcription was its punishment. A manual pool survives only as an explicit override, denominated in the operator's own planning units, mapped onto by the provider's percentages, and never required for anything. Also rejected: scraping either TUI's display, brittle and liable to silently parse the wrong number; calling `api.anthropic.com/api/oauth/usage` ourselves, which is undocumented, rate-limits aggressively, and would break silently the day it changes — going through `get_usage` puts that contract on the CLI, where it belongs; and presenting local token sums as the vendor's exact limit.

So hive records provenance per fact rather than inventing precision, and it records it per *window*, because a statusLine payload can report the five-hour window while the weekly one is still absent, and stamping one timestamp across both would backdate freshness onto a fact nobody observed. Unknown is rendered as unknown. A window nobody has measured reports no number at all — not the zero sitting in its column — and a reading whose reset boundary has since passed is void rather than carried forward, because the human spends these accounts outside hive too, and "0% used" is a claim about the account, not about hive. An unmeasured pool cannot constrain a spawn, so the spawn proceeds under an explicit unbounded reservation and says so. Refresh is forced at daemon start, runs on the maintenance tick when the interval lapses or a passed reset has blinded a pool, and is skipped entirely while the free feeds — the app-server's notifications, Claude's statusLine — already keep a provider live.

A measurement beats an estimate. A provider's reading already counts everything spent before it was taken, hive's own runs included, so the only spend it cannot know about is what came *after* it — and that alone is what hive adds to it. The label follows the number actually published, not the reading it was built from: a figure containing any estimate of unreported spend is `estimated`, and `authoritative` is reserved for one the provider alone produced. The rejected alternative is what hive shipped first, and it failed loudly: usage was `max(hive's whole ledger for the window, the reading plus spend since)`, so that "an optimistic provider number can never free capacity hive knows it spent." Every word of that holds except *knows* — hive never knew, it guessed from a static per-tier table, and the floor let the guess outrank the measurement. When Codex reported 0% of the weekly window consumed and hive's estimates summed to 12%, `hive quota` published 12% under `source: provider, confidence: authoritative`, while the user could read the real number off his own screen. An estimate wearing a measurement's badge is worse than either an honest estimate or an admitted unknown. The floor's real fear, a provider that lags, is answered without it: spend recorded after the reading is exactly the spend the reading cannot have seen, and it is still counted. "After" means later in the ledger's own commit sequence, not later on a wall clock — a clock cannot order two events that share a millisecond, and comparing timestamps silently dropped the spend that landed in a reading's own millisecond, under-counting in the one direction that lets a spawn through a limit the user has really hit. At that boundary the tie goes to counting: over-counting refuses a spawn that would have fit and is corrected by the next reading, while under-counting spends quota that does not exist and nothing corrects it.

Model-scoped caps — Codex's metered `limitId`s, Claude's per-model weekly ceilings — arrive naming a model and withholding its id; Claude literally sends `scope.model: { id: null, display_name: "Fable" }`. But both vendors name these pools exactly as their own model catalogs name their models, and both publish that catalog for free: Claude in the `initialize` control response hive already awaits, Codex through `model/list`. So the binding is *discovered*, by joining the pool's provider-given name against the provider's own display names, and every id form of a model — `claude-opus-4-8`, its `[1m]` variant, `opus`, `default` — binds to the same meter, so a spawn cannot dodge an exhausted pool by spelling the model differently. A pool matching nothing in the catalog binds to nothing and says so. Hive used to record these pools, show them, and refuse to route on them, on the grounds that deciding `"Fable"` meant `claude-fable-5` was a guess. That was right about the payload and wrong about the system — the id is absent from the *quota* surface but present in the *catalog* surface — and the caution cost everything it was meant to protect: on 2026-07-11 two deep-tier agents launched onto Fable while Fable's own weekly pool sat at 99%, measured, fresh, provider-sourced, and joined to nothing. A pool that gates nothing is not a safe pool, it is an ignored one. A model is therefore gated by *every* pool that meters it — the account-wide one and its own — and the tighter governs; consulting only the first matching pool is precisely what waved that spawn through on the general pool's 39% while the model's own 1% went unasked. A model with no cap of its own is metered by the general pool, never by nothing: Opus has no dedicated meter, and minting a phantom one for it that reported "usage unknown, routing unconstrained" did the worst possible thing, since an unconstrained model is the most attractive route a router can be offered.

The deterministic policy is intentionally boring. Start with the table's preferred vendor (or the other vendor for a review when its author is known), then the other candidate; reject candidates whose estimated run would exceed *any* window of *any* pool that meters them; for `cheap` and `standard`, also reject a candidate that would invade the configured deep-capacity reserve. Compare eligible candidates by the worst post-reservation headroom, then preference, then vendor name.

Eligibility is headroom **and** viability, because headroom alone ranked a route that could not run. A route hive has just watched fail to produce a working agent is passed over for automatic selection, whatever its capacity says — a fallback target that cannot start is not a fallback, and a gate that refuses an exhausted model only to hand the work to a dead route has protected nothing. Ranking on headroom alone made this inevitable: an idle Codex pool at 0% used outscores a working Claude pool at 63% every time, so the emptiest pool silently outranked the question of whether that pool's agents could start at all — which, on 2026-07-11, deep-tier Codex could not. This is an observation and never a belief: nothing in hive names a vendor or a tier as broken, since such a fact expires the moment someone fixes it. A launch that never proves life is recorded against its route; any successful launch clears the route instantly; and the exclusion lapses on a cooldown so the route is always retried. That expiry is load-bearing, not a nicety — a route excluded forever could never produce the success that would clear it, and the guard would quietly become the outage it was built to prevent. A quarantined route is still used when it is the only candidate or when the user pinned it explicitly, warning loudly: hive's own recent bad luck is a weaker fact than a human's explicit ask.

When an exhausted pool refuses a spawn and the provider is holding unspent usage-limit reset grants, the refusal says so — the count, not the redemption. Hive never spends one. Burning a human's finite credit to admit a spawn is the human's decision, and an agent that can quietly spend the user's scarce resources to get its own way is a bad agent; that it would look helpful while doing so is exactly what makes it dangerous. An explicit user vendor stays fixed when safe. If it is unsafe, hive does not silently disobey: spawn returns the remaining five-hour and weekly capacity, expected reset where known, and the safe fallback. If both vendors are exhausted, no process starts. When a provider's live numbers are missing — it is not signed in, its probe failed, no reading has landed yet — that pool enters compatibility mode: hive preserves the static route, records an explicit unbounded reservation, and emits one deduplicated warning that names the provider and the reason. The warning does not ask anyone to configure a file, because the remedy is for the provider to answer, and hive will adopt real numbers the moment it does. Failing every spawn while limits are unknown was rejected as a migration strategy; pretending an arbitrary default allowance was real lost for the same reason. Refusing a spawn is the sharp end of this subsystem — a wrong number here does not merely mislead a dashboard, it stops work — which is why the rule is accurate-or-unknown, never invented.

Warnings use the ordinary durable orchestrator message path, not stderr in a daemon nobody watches. Crossing configurable warning or critical remaining-capacity thresholds sends one message containing the provider, account, pool, both windows, data confidence and freshness, and known reset times. Alert state is durable. It rearms only after headroom rises above the threshold plus hysteresis or a reset boundary passes, so parallel reservations do not produce a wall of identical warnings. Corrupt state fails closed for the affected configured pool, and status reports the reason without requiring anyone to inspect SQLite. A reading that reports *lower* usage than hive's own ledger is believed, and it gives the capacity back: the provider measured the account, hive only estimated its own share of it, and the reading is the better fact. Reconciliation therefore corrects in both directions rather than only tightening — which is the whole point, since a gate refusing spawns on the strength of a fiction the provider has already contradicted is not being conservative, it is being wrong in the expensive direction.

### 7. What happens when a context fills up

Sensing was solved in decision 2; this is the actuator. Quality does die long before the window fills — but **not at a threshold, and not at a percentage**, and the two constants this decision used to rest on were both wrong. The evidence is in [docs/research/context-degradation-and-agent-recycling.md](research/context-degradation-and-agent-recycling.md); the short version is that the primary triggers are **task completion** and **repeated failure**, and the token ceiling is only a backstop for the agent that hits neither.

Two rejected alternatives, preserved because the way a fabricated constant survives is by nobody writing down that it was fabricated.

**Rejected: "quality dies around ~140K tokens."** This was field lore with no measurement anywhere behind it. It entered this spec in the initial commit and was never derived. Its most-repeated public attribution — to an engineer at Sourcegraph — is a **hallucinated citation**: the post it names contains no token numbers at all. Its likeliest true origin is arithmetic, 200K × 70% ≈ 140K, which is a *budget* observation about a 200K window retconned into a *quality* claim. Nothing in the published literature lands at 140K.

**Rejected: a percentage of the context window, at any value.** Not mis-tuned — wrong unit. Degradation onset is an absolute token count with a large model-specific constant, so the window size predicts almost nothing about where quality dies. The measurement that settles it: a 1M-window model (Gemini-3-Flash) is already at 40% success by 32K while a 200K-window model (Claude-4.5-Opus) still holds 84% at the same absolute depth. **The model with the five-times-larger window degrades earlier in absolute tokens.** A bigger window buys no headroom, so "recycle at N% of the window" is unsound in both directions and gets *more* dangerous as windows grow. The 65% default only ever looked defensible because 65% × 200K ≈ 130K sat near the (invented) 140K line; on a 1M window it means 650,000 tokens, and the correction to decision 2's denominator exposed that rather than caused it.

So the ceiling that remains is **absolute tokens, configured per model**, and it is a backstop rather than the mechanism. Hive does not yet know its own curve — every published number was measured on someone else's workload — so the number lives in config and the doc names the experiment that would set it honestly.

The percentage has to be able to say **unknown**, and for a long time it could not. `contextPct` was a plain number, so an agent Hive could not observe fell back to the 0 it was spawned with — and 0 does not mean "empty", it means "no idea", while reading to every consumer as an invitation to load more work on. A live Codex agent that had done real work sat at 0% for exactly this reason, and the user caught it from the display: "that cannot be true." Null is now a first-class value end to end — schema, column, sensors, display — and it propagates as itself: a sweep that observes nothing *writes* null rather than leaving the last number standing, because "we no longer know" is a finding and not the absence of one. Everything downstream reads null as **full, not free**: the recycle line does not fire on it and the orchestrator's reuse rule refuses it, since loading more work onto an agent whose remaining room you cannot see is precisely the mistake the line exists to prevent. The display renders it `—`, never a number, and never 0. The cost is real and accepted: an agent Hive cannot sense is an agent it will not reuse, so a broken sensor now shows up as lost reuse rather than as a confident wrong number. That is the trade we want — the previous design bought its cheerful 0% by lying, and a number you cannot trust is worse than no number, because you act on it.

**The real trigger is error state, not depth.** The best-evidenced predictor of an agent's next mistake is not how many tokens it holds but whether its own previous mistakes are in its context — *self-conditioning*, which is measured, and which model scale does not fix. Claude Code's own guidance restates it without citing it: corrected more than twice on the same issue, clear and start fresh. So the retire conditions, in priority order, are **task completion** (the normal path, and free), **repeated failure at the same sub-goal** (the quality path), and only then the token ceiling (the backstop). An agent that has failed twice is compromised no matter how much room it has left; an agent with a clean history is fine far deeper than any folklore admits.

**Two lines, not one, and the gap between them is the point.** A single number cannot do this job, because "still good enough to keep working" and "has room to take on something new" are different questions, and Hive was answering both with 65. That let an agent accept a fresh task at 64% and get killed at 66% — recycled mid-task, holding work it had just been handed, which is the worst possible moment to lose it. So:

- **Admit** new work to a live agent only when its resident tokens plus the task's estimated cost plus a handoff reserve still fit under the ceiling, it has no unresolved repeated failure, and its cache is warm. The estimate exists already: the orchestrator tiers every task.
- **Retire** on task completion, on repeated failure, or on the ceiling.

The gap between admitting and retiring is deliberate: it is the room an agent needs to *finish what it accepted*. Sizing work so it fits in that gap is prevention, and prevention is the real mechanism — the orchestrator sizes subtasks to finish inside one healthy context, and agents spawn with a minimal briefing (task, file scope, memory index) rather than repo dumps. Most agents should retire at task end without ever seeing the ceiling.

**The handoff protocol.** The decisive measurement is that a successor given *nothing but the worktree* still solves 46.4% of interrupted coding tasks, while one given structured notes solves ~51% at 60% fewer tokens. So a handoff is load-bearing for **cost, not correctness** — and a *bad* handoff is actively worse than none, scoring below the no-context baseline. That sets the whole design: carry little, and carry only what the repo cannot say.

> Anything reconstructible from the worktree is a **pointer**. Anything not reconstructible is carried **verbatim and pinned**.

The outgoing agent's document has two halves, and the split matters because only one of them can lie. The **computed half** is extracted from the worktree and cannot hallucinate: branch, HEAD, `git diff --stat`, changed files, the last validation command with its exit code and output tail, todo state. The **written half** is short and interpretive: goal, work completed, **failures observed with their reasons**, remaining uncertainty, the single next concrete action, rollback notes. Separately and above both sits the **pinned block** — the user's constraints, prohibitions, and amendments to the goal, re-injected byte-identical and never passed through a summarizer. That block is the highest-value item in this decision: with a constraint present, violation rate is 0%; when a summarizer drops it, 38%, and roughly 47 pinned tokens restore full compliance. The model does not disobey the rule, it never sees it.

Deliberately **dropped**: the transcript, raw tool outputs, file contents, and the exploration path. All are reconstructible, carrying them buys nothing measurable, and unlabelled dead ends actively waste a successor's effort. Failed approaches survive as *labelled negative knowledge* — "tried X, failed because Y, do not retry" — never as raw trace; the value is in the reason, not the record. The incoming agent reads the pinned block, the computed half, and the written half, and is told to **verify the predecessor's claims against the worktree rather than trust them**, because over-trusting a predecessor is a measured failure mode. And because Hive knows the successor's remaining goal, the document is written as extraction *toward that goal*, not as a neutral summary of the past.

**Revised: `/compact` is not the enemy; the cold respawn is.** This decision used to reject `/compact` outright. The quality half of that objection was right and now has hard evidence behind it — lossy self-summarization does drop exactly the constraints and failed-approach history a successor most needs. But it was never a *cost* objection, and on cost the old reasoning ran backwards: a cache read costs 0.1× and a cache write 1.25×, so **re-paying a token costs 12.5× what carrying it warm costs**, and a Hive recycle re-pays the entire briefing while `/compact` keeps the cached prefix and rebuilds only the conversation. The synthesis takes both: **write the durable handoff artifact to disk first, keep the pinned constraints out of the lossy path entirely, then compact in place.** That preserves everything the old rejection was protecting and never pays the cold-spawn toll. Kill-and-respawn remains correct when an agent has gone *cold* (a fat idle agent is more expensive to resume than to replace) or when its context is poisoned by its own failures and shedding the trajectory is the whole point.

**Recycling is not the safe direction to err, and believing it was is what caused the 200K bug.** Both errors cost. Recycling too late lands degraded work — and worse, degraded work that a green test suite cannot catch, since agentic code quality measurably rots while pass rates hold. Recycling too early throws away a context that is ~96% cache-read, re-pays a ~33K-token cold start out of a quota pool that has hit 99%, and drops constraints on the way through. Below roughly 200K resident tokens a respawn needs a dozen turns of remaining work just to break even; below 120K, closer to thirty. The instinct that early recycling is merely wasteful while late recycling is dangerous is what justified hardcoding a 200K denominator on the reasoning that it would only ever read "conservatively high" — and that instinct is what this decision now explicitly rejects. Bonus loop: failed approaches from handoffs are prime candidates for the durable memory layer — that's how "the tool gets smarter" stops being a slogan.

### 8. Skills, and when to build one

The tailwind that makes this feature possible: Agent Skills became an open standard (agentskills.io, Dec 2025) and OpenAI shipped Codex support within 48 hours. One `SKILL.md` folder serves every agent in the hive regardless of vendor, and progressive disclosure means a large library costs agents almost nothing until a skill is actually used.

Storage mirrors the memory scopes: per-repo skills in `.hive/skills/` and global skills in `~/.hive/skills/`. At spawn hive links each skill directory into the launched tool's project-native discovery path — `.claude/skills/` for Claude and `.agents/skills/` for Codex — with a same-named repo skill winning over a global one.

Hive also has skills of its own to give — how to work inside a hive, and how to write code that does not have to be rewritten — and shipping them turns out to be a packaging question, not an authoring one. Hive reaches a user as a `bun build --compile` binary: only what `src/cli.ts` imports exists on their machine, and there is no checkout out there to read a `skills/` directory from. So **the directory decides what ships.** `skills/` is imported into the binary as text (`src/skills/shipped.ts`) and is therefore inside it, wherever it goes; `.hive/skills/` and `.claude/skills/` are Hive's own development kit — instructions about Hive's source tree, alongside `.hive/memory/`, which is a log of what *this* project learned — and they are nobody else's business. That is not a promise, it is a test: `src/skills/packaging.test.ts` compiles the real binary and reads its bytes, failing if a shipped skill is missing from it or if a Hive memory or dev-only skill is found in it. No new release artifact carries any of this; a skill rides inside the executable that already ships.

A skill reaches an agent one way, through one installer that never destroys a file a human owns. Absent, it is written; identical, it is left; **different, the human's copy wins** and the drift is reported, with `--force` the only way to take Hive's version back. `hive init` runs that installer in the user's repo, for each vendor CLI actually present on the machine — no `.claude/` for someone who has no Claude Code — and the spawner runs the same installer into each worktree, so an agent has Hive's skills even in a repo that never heard of Hive. The user's own skills are linked first and a linked name is never written through, which makes the precedence readable off the code: a skill you wrote beats a skill Hive ships. Links are per skill rather than one link over the whole native directory, so vendor-specific skills can coexist; a conflicting same-named native entry fails the spawn instead of being overwritten. A skill that exists only in a vendor-native directory stays vendor-specific: silently promoting it would turn a tool-local customization into a cross-vendor contract its author never chose. The cost is that shared skills must be moved to the canonical hive directory once. Treating all native directories as additional sources was rejected because two editable copies would inevitably drift while appearing to be one skill. Write once, every vendor reads it.

The hole in the original spec was *detection* — "the orchestrator knows when a skill should be built" requires memory of past runs, which a single session doesn't have. The daemon does: it logs every task descriptor. At session end the orchestrator reviews history with a **rule of three** — same task shape done ~3 times across runs → propose a skill. Propose, not create: "I've formatted release notes three times; want a skill for it?" On yes, a cheap-tier agent authors the SKILL.md from those transcripts. Silent skill accumulation is how you get sprawl nobody audits, so creation is human-gated and skills get the same pruning policy as memory — unused for N runs, propose deletion. At spawn time, the orchestrator names the relevant skills in the briefing rather than hoping agents discover them.

### 9. Windows are viewers, not containers

The original framing — "each agent runs in a terminal window" — hides a failure mode: if the process lives in the window, closing david's window kills david mid-task. Invert it: **agents live in detached tmux sessions owned by the daemon; windows are disposable viewers.** Spawning a window means telling iTerm2 or Terminal.app (via osascript) to attach to `hive-maya-<instance>`. Closing it detaches; the agent keeps working headless; `hive watch maya` reopens the lens.

This costs nothing — decision 1 already made tmux the substrate for message injection and capture-pane monitoring — and it buys robustness plus a free `--headless` mode (SSH, CI). The root also runs in a fixed tmux session and the Workspace master pane hosts its interactive client, which gives the daemon one precise wake target without scraping or polling it. macOS only means two emulator adapters, auto-detected, overridable in config. Nothing in the tmux+daemon core is macOS-specific except these adapters, so the platform door stays open without a rewrite.

Disposable viewers still need disposal: the first field test killed agents whose windows stayed open, attached to dead sessions. The fix is identity, not search. When hive opens a viewer it records a **native terminal handle** on the agent row — iTerm2's globally unique session ID, or for Terminal.app the app PID plus window ID plus TTY, all of which must still agree (and the window hold a single tab) before anything closes. Terminal.app handle capture follows the exact tab object returned by `do script` to its owning window; it must not assume the asynchronously activated front window is the new one. `hive_kill` kills the tmux session first, then atomically marks the agent dead while claiming the handle, and best-effort closes exactly that viewer; if a concurrent spawn loses that race it closes its own orphan. Finding the window by its title, agent name, or a session-name text search was rejected outright: titles are mutable and shared, and any fuzzy match is a loaded gun pointed at some unrelated terminal the user has open. A close failure never blocks the kill, the dead state, or worktree cleanup, and manually attached terminals and old handle-less rows are untouched.

### 10. The window wall lays itself out

Decision 9 made windows cheap; cheap windows pile up. macOS opens every new terminal in a default-offset cascade, so by the fourth agent the screen is a deck of overlapping cards and the orchestrator — the one window that matters — is buried under the workers. The fix is that hive owns the geometry: **whenever the set of hive windows changes, the daemon re-tiles them into a gapped wall with the orchestrator centered and largest.** Every viewer-set change already flows through the daemon — spawn attaching a handle, `hive_kill` claiming one, the reconciliation sweep declaring an agent dead, `hive watch` reopening a lens — so each of those also fires a coalesced, best-effort layout pass. No polling, no timer: the same event edges that maintain the status table maintain the wall. Two shapes cover reality. The preferred one gives the orchestrator a dominant center column with workers stacked in symmetric side columns, the odd worker going right, sides splitting into more columns as rows run out. The whole wall is inset vertically and horizontally, cells have visible gaps, and a final mechanical cap keeps every frame below a fixed fraction of the display. These constraints make every participant an ordinary floating window: none becomes a macOS full-screen space or an edge-to-edge half-screen split. When worker cells would fall below a minimum readable size — too many agents, too small a screen — the layout degrades honestly to an inset uniform grid in which every window is equal and the orchestrator holds the most central cell. Layout math is a pure function (screen rectangle in, frames out), so every count, parity, degenerate screen, and maximum-size guard is unit-tested without ever touching a window. The rejected first shape used full-height, edge-to-edge columns; it maximized pixels but made the wall feel like an involuntary window-manager mode rather than a tidy set of recognizable terminals.

The orchestrator viewer is no longer a special window hive has to discover. `hive claude` and `hive codex` run the same repo resolution and session boundary as bare `hive`, then launch the installed Workspace with `--orchestrator claude|codex` beside its project, port, and exact hive binary. The Workspace master pane invokes a private CLI boundary that starts the selected read-only root in its fixed tmux session; closing the app detaches the viewer without killing the root. Bare `hive` keeps Claude as its default master. Running the public commands directly in whichever terminal the user typed into was the earlier design and lost because it bypassed the Workspace surface, made the orchestrator the one viewer Hive did not own, and made `hive codex` and `hive claude` structurally different from the ordinary project entry. The accepted cost is that both explicit orchestrator entries require an installed release Workspace, under the same deliberate no-development-fallback rule as bare `hive`; a source checkout cannot quietly substitute a debug app.

Moving windows is the same trust problem as closing them, and it reuses the same answer from decision 9: a window is repositioned only through its recorded identity handle (iTerm2 session ID; Terminal.app PID + window ID + TTY), and the move is a bare `set bounds` — never `activate`, so focus stays wherever the user put it. Terminal.app's `do script` returns before its default cascade geometry has necessarily settled, so its adapter waits for stable bounds before publishing the handle; without that boundary, a correct initial layout can be overwritten moments later. A vanished window makes its move a silent no-op and never blocks the rest of the pass. The sweep that marks a vanished session dead closes its orphaned viewer too, because a lens on a dead session shows nothing worth keeping. Alternatives that lost: tmux's own tiling (`select-layout tiled`) collapses all agents into panes of one window — free, but it un-decides decision 9 by making one close destroy every lens, and N readable panes fit one screen far worse than N windows hive can size. Delegating to a window manager (Rectangle, yabai) was rejected as a dependency most users don't have, with yabai needing SIP disabled. Making the user arrange windows was the status quo being fixed. The accepted costs: layout is osascript and therefore macOS-only (it lives entirely inside the decision-9 adapter boundary), it repositions the user's own orchestrator window by design (`layout = "off"` in config restores manual control), and the "orchestrator is largest" guarantee holds only while minimum worker sizes fit — beyond that the equal grid is the honest ceiling.

### 11. The orchestrator physically cannot code

"The orchestrator does no coding" is load-bearing: an orchestrator that helpfully edits files is an unsandboxed writer in the main checkout (breaking decision 3), burning its own context on diffs (breaking decision 7), outside the approval flow (breaking decision 4). And both CLIs *will* drift into coding if merely asked nicely. A system prompt is a suggestion; permissions are a constraint.

So it's mechanical: a Codex orchestrator launches with `--sandbox read-only`; a Claude orchestrator gets deny rules on Edit/Write/NotebookEdit and read-only Bash. Its only actuators are the hive MCP tools — spawn, send, on-demand status, bounded inbox recovery and message reads, approve, memory. Reading is encouraged (good decomposition requires browsing the code); writing is impossible. The prompt explains the role and the reflex: you want to fix that typo → spawn a cheap agent. Every root-only behavior is launch-scoped. Claude receives settings and MCP files from Hive's own home through `--settings` plus `--setting-sources user` and strict `--mcp-config`, and receives the assembled brief through `--append-system-prompt` before the variadic Channels terminator. Codex receives its MCP endpoint through a `-c` override and its brief through the remote TUI's positional prompt, after the read-only flag; its authority app-server has no startup-prompt option. Project-root `.claude`, `.mcp.json`, and `.codex` runtime state was the earlier choice and lost because a crash before the restore made every ordinary Claude or Codex launch in that repo inherit Hive. A signature-limited migration removes those stale generated entries without touching user-owned neighbors. The cost is a small runtime directory under Hive's home; the benefit is that the repo root remains an ordinary repo even while Workspace is open. No small-edit carve-outs — carve-outs are how invariants erode. A side benefit: a read-only orchestrator triggers almost no approvals, so its window stays clean for the three streams that matter — your conversation, concise agent envelopes, and the escalation queue.

### 12. The machine outlives any one agent

Hive runs other people's code on your only computer. Agents write tests with infinite mock loops, background their own hung commands, and retry what didn't respond — each individually reasonable, and hive amplifies all of it by running many agents at once. The failure that proved this isn't hypothetical: one agent's unit test faked a long-poll with an instantly-resolving mock, turning a pump loop into a ~1.5 GB/s allocator; the hung run produced no output, so the agent retried, stacking three runaway processes to ~164 GB until macOS jetsammed the machine and killed every agent, the daemon, and the user's other work with it. The lesson is not "fix that test" — agents will write that test again. It's that **hive must degrade before the machine does.**

So the daemon's maintenance tick doubles as a resource sweep. It walks the process tree under every hive-owned tmux session (agents and orchestrator, roots from tmux pane pids, tree from one `ps` pass) and hard-kills any single process past a per-process memory ceiling — 12 GB by default, far above every legitimate workload observed and far below where the machine starts dying. It reads reclaimable system memory from `vm_stat`, and below a 4 GB floor it pauses `hive_spawn` until pressure clears, alerting at most hourly. Every kill and pause lands as a durable orchestrator message, so degradation is visible and attributable, never silent. The same sweep reaps codex app-server children whose host process died without cleanup (hosts drop a pidfile beside their socket; only children of agents this daemon knows to be dead, still running the expected binary, are killed — an unknown id may belong to another hive instance) and prunes settled history — old events, applied messages, resolved approvals — so the daemon's own tables aren't the slow leak. The transport buffers on the codex control path get ceilings for the same reason: a peer streaming garbage without newlines costs one dropped frame, not unbounded growth. The Channels bridge's own pump loop (`ChannelBridge.pumpLoop`) got a narrower guard at the source: three consecutive iterations under 5ms trip a 20ms floor delay before the loop continues. The real daemon long-poll either waits up to `pollWaitMs` (25s) or returns a backlog after genuine network latency, so this never fires on the success path — it only catches the exact shape of a fake or broken poller that resolves instantly and repeatedly, which is what turned the pump loop into the OOM's allocator in the first place. The process-ceiling sweep still catches a runaway after the fact; this stops that one failure shape before it spins up.

Alternatives that lost. OS-level limits (`setrlimit`, launchd caps) constrain address space, not resident memory, and on macOS a virtual-size cap either kills healthy Bun processes (which reserve enormous address ranges) or never fires; there is no cgroup equivalent to delegate to. Killing the whole agent session on breach punishes a working agent for one runaway child — killing just the process keeps the agent's context and lets the orchestrator decide whether to retry. Trusting the orchestrator to notice was rejected on arithmetic: the machine died about four minutes after the first runaway appeared, faster than any model-in-the-loop reaction. The accepted costs: a genuinely huge legitimate workload gets shot at the ceiling (it's config, `resources` in `~/.hive/config.toml`), the kill is a blunt SIGKILL with no drain, and the sweep reads `ps`/`vm_stat` output, which is macOS-shaped like the rest of the adapter layer.

### 13. What survives a crash

A machine-wide OOM made this concrete: every agent process died mid-task, and while the daemon's SQLite state survived intact — tasks, worktrees, branches, the message queue — each agent's in-session context did not have a recovery path. The orchestrator's only move was respawning agents from their stored task text, which throws away hours of accumulated conversation: the failed approaches, the files already read, the plan already half-executed. One agent died stuck in "spawning" and stayed there indefinitely while urgent messages to her sat queued past their deadlines, because the reconciliation sweep deliberately ignored spawning agents (that guard existed to avoid racing an in-flight monitored launch) and nothing else owned the case. The lesson generalizes: a process is disposable, a worktree is durable, and the conversation inside the tool is the expensive thing in between — so hive must be able to put a process back around a surviving conversation.

Both tools make that possible, natively. Claude Code resumes any session by id (`claude --resume <session-id>`), and every hook invocation carries the current `session_id` on stdin; Codex resumes any recorded rollout (`codex resume <session-id>`, taking the same `-c` config overrides as a fresh launch), and its `notify` payload carries the conversation's `thread-id`. Hive already owns both hook paths — Claude hooks and Codex notify both invoke `hive event` — so session identity capture is one field riding on traffic that already flows: the event CLI reads the hook's stdin JSON and the notify payload, and the daemon records the latest `toolSessionId` on the agent row. Capture is continuous rather than spawn-time because a Claude resume forks the conversation to a *new* session id — a one-shot record would go stale after the first recovery. When capture never happened (a Codex agent that crashed before its first turn ever completed), recovery falls back to disk discovery: Claude transcripts live under `~/.claude/projects/<munged-worktree-path>/` where the newest transcript is the session, and every Codex rollout under `~/.codex/sessions/` opens with a `session_meta` line naming its session id and cwd. Assigning the id ourselves at spawn (`claude --session-id <uuid>`) was rejected as the primary mechanism for the same fork reason: it only names the first session, so continuous capture is needed anyway and the flag adds nothing.

Recovery itself is the reconciliation sweep grown into classify-and-recover, running at daemon startup — after a machine crash, the daemon restart *is* the recovery moment — and on the same 30-second tick as before. Any agent whose status claims a process but whose tmux session is gone gets classified. Active work (working, idle, awaiting-approval, stuck) with an intact worktree and a known-or-discoverable session id is resumable: hive relaunches the tool with its native resume in a fresh tmux session under the same name and worktree, reopens the viewer, marks the agent idle (a resumed TUI sits at its prompt), tells the agent what happened through the ordinary message path — check your inbox, re-verify in-flight edits, continue — and flushes its queued messages. An agent that died during spawn has no conversation worth resuming and is marked dead with its worktree preserved and its stored task surfaced to the orchestrator for a respawn; the in-flight-spawn guard survives as a check on the spawn-name reservation, which only a live spawn in the current daemon process can hold because startup clears reservations stranded by a crash. Unresumable active work (worktree gone, no session found, resume launch failed) takes the same dead path — and "resume launch failed" means the same fail-loud proof-of-life watch a fresh spawn runs (decision 2): a hook event, an advancing `lastEventAt`, a fresh Codex rollout write, or a redrawing screen all count as proof the resume took, and a screen that has stayed frozen through the silence budget fails the resume exactly as it fails an ordinary launch, rather than leaving a falsely-resumed row nobody re-checks. Every death also denies the agent's stale pending approvals and flags its queued messages once, so nothing waits forever on a corpse the way it did in the incident. Control-paused and write-revoked agents are explicitly excluded from resume: the critical-control machinery already owns their restart semantics, and recovery resuming around a revocation would undo decision 1's whole point.

Automatic-on-startup was chosen over a manual-only `hive recover` command because a machine that just crashed has nobody guaranteed at the console, and requiring a human step reproduces the stuck-forever failure. The manual command exists anyway — `hive recover [name]`, and `hive_recover` over MCP — for the after-the-fact cases automation refuses: it can resurrect an agent already marked dead, and it bypasses the auto-resume attempt cap because a human explicitly asked. That cap (three attempts, re-armed by a completed turn) is what keeps automatic resume from becoming a crash-loop amplifier: a process that keeps dying on its own is a bug, not a crash, and after three tries it surfaces as an explicit death instead of a fourth relaunch. Codex agents recover through the TUI resume path regardless of the configured driver — an app-server connection died with the daemon anyway, and the TUI is the maintained interactive default (decision 2).

What this gives up, honestly. A resumed conversation is not a checkpointed process: a tool call that was mid-flight at the crash simply never returns, which is why the recovery notice tells the agent to re-verify its in-flight edits rather than pretending continuity. Quota accounting treats the resumed process as the same logical run — the original reservation, kept-with-estimate at death per decision 6, is not re-reserved — so a manual resurrection of a long-dead agent runs briefly unaccounted; acceptable for a rare operator action, and the conservative ledger absorbs it. And resume trusts the tool's own session store: if `~/.claude` or `~/.codex` was lost with the machine, recovery degrades to exactly the respawn-from-task fallback it was built to avoid — which is the correct floor, not a failure of the design.

### 14. Starting into a repo

Starting is one boundary regardless of which interactive root the user wants. Bare `hive`, `hive claude`, and `hive codex` all resolve the repo, refresh and start through the shared session boundary, and open the installed Workspace against the resulting daemon; the explicit forms differ only in the orchestrator selector passed to the app. The app receives the exact launching hive binary, so its feed and private orchestrator child cannot drift to another installation on `PATH`.

Every mechanism above is built to make a spawn cheap, and every one of them quietly assumes it knows this repo's shape. The scoped brief extracts the doc sections a task names — but `src/adapters/brief.ts` ships a literal `["SPEC.md", "README.md", "CLAUDE.md"]` allowlist and a bare-name `SPEC §6` rule that means nothing in a repo whose design doc isn't called SPEC. The orchestrator brief teaches agents to cite "SPEC §6" and "docs/research/x.md" by name. The landing gate tells an agent to "re-run the tests" without knowing the command. Point hive at a repo that keeps its design doc in `rfcs/DESIGN.md` and tests with `pytest`, and the token economics we built for *this* repo simply do not happen. Scott's binding constraint — hive works with any repo, nothing hive-repo-specific in the product — is violated by exactly these hardcodes.

The fix is to stop compiling in what should be discovered. Hive profiles the repo and writes the answer down; every spawn reads the answer instead of a hardcoded guess. The survey behind this is [docs/research/repo-startup-and-init.md](docs/research/repo-startup-and-init.md) — Claude Code's `/init`, Codex's `AGENTS.md` bootstrap, aider's tree-sitter/PageRank repo map, Cursor's example-driven rule generation — and two of its lessons shape everything here: derive-then-refine and never block, and cheap deterministic facts are a different tier from expensive model-authored ones.

The profile is a structured artifact, not a memory fact. It holds what product code reads deterministically: the briefable-doc allowlist and which doc is *primary* (the one that earns the bare-name §-selector treatment, whatever it is called); the build/test/typecheck/lint/run commands; a conventions pointer (whether an `AGENTS.md` exists, where, plus primary language, package manager, monorepo layout); the top-ranked entry points; and a staleness fingerprint. That list is exactly the set of things something reads — the membership rule, and it has teeth. The profile also used to carry an `index_budget`: a file count and a derived `map_tokens` cap, aider's `--map-tokens` analogue, sized so a monorepo's entry-point index could not drown every context it touched. It was a good idea for a feature nobody ever built, so nothing ever read it, and a cached number with no reader is not free — the count changed on every commit that added a file, which invalidated the profile, which is precisely how a correct profile came to look stale. It is gone. A repo-size-aware cap belongs at the point of use, derived when something finally needs one. We also considered making the profile a reserved memory fact — scope `repo`, id `repo-profile` — so it would ride the existing store, index, and injection machinery for free. Attractive, and rejected: `brief.ts`, the router, and the spawner would then have to parse a Markdown fact body to recover a typed allowlist, coupling structured config to the memory schema and to LLM free-text. Structured truth deserves a structured reader. The seam with memory (decision 5) runs the other way — repo-specific structured answers come from the profile, read by product code (the router, `brief.ts`) rather than by the memory store, and memory never duplicates a profile field: commands and layout live in the profile, and only narrative durable knowledge lives in memory.

The profile is a **cache, not a document**, and it lives in Hive's own per-project state directory — `~/.hive/projects/<hiveUuid>/profile.toml`, keyed by the identity the project registry already mints, so it survives the repo being moved or renamed and every linked worktree of a repo reads the one project profile rather than quietly profiling its own branch. It is not in the repo, not in anyone's diff, and not anyone's business. Machine-specific things — absolute worktree paths, the daemon port — were never in it and are rebuilt at runtime.

We shipped it committed at `.hive/profile.toml` first, on the reasoning that repo facts travel with the clone, the same reason committed memory and skills live under `.hive/`: a teammate who clones would get token-cheap spawns without re-profiling, and a fact about the committed tree belongs in the committed tree. That lost, and it lost to a measurement. Generating the profile is a `git ls-files`, a handful of `stat`s, and a read of the repo's markdown: **56 ms and zero model tokens** on this repo. What committing it spared a teammate was 56 ms. What it cost was a file in every diff, and — because a cached artifact in a tree that moves must be checked — a staleness concept, a staleness message, and a `hive init --refresh` command for a human to run. We built all three, and they were the whole disease: the user updated, started Hive, and got told the profile was "20 commits stale" and that they should go fix it by hand. Regenerating it produced byte-identical doc names and commands. The nag was real and the staleness was not. The principle the old design got backwards: **a derived artifact that is cheap to rebuild should never be something a human maintains, and should never be somewhere a human has to look at it.**

So there is no staleness state a user can be in. Every session boundary calls `ensureProfile`, which compares a fingerprint of the inputs that actually determine the profile — the doc inventory, the manifests, the lockfiles, the conventional entry files, by path and size — and rebuilds when they drift. It prints nothing on any path. The fingerprint deliberately does not hash the Git tree, which is what it used to do, and that one line was the bug: hashing the tree makes *every commit to any file* — a test fixture, a typo in a comment — mark the profile stale. Staleness now means the profile is *wrong*, and a profile that is wrong gets rewritten in the milliseconds before the caller sees it, by the daemon or the CLI, both idempotent and converging on identical bytes. `hive init --refresh` survives only as a debugging escape hatch for a detection you want to watch rerun. It is never required, and nothing ever tells you to run it.

The one half a human owns is committed: `.hive/profile.override.toml`. Detection is a heuristic and heuristics are sometimes wrong — a repo whose real test command is `make test-ci` will be read as `npm test` forever — and before this file there was nowhere to say otherwise that a regeneration would not erase. So the derived profile is a cache Hive rewrites at will, and the override is a small, hand-edited file that layers over it and Hive never writes. It is the *only* part of the profile that should be committed, and it is the inverse of what we committed before: a correction is a team fact, earned once and shared, while everything it corrects is a derivation any clone reproduces in milliseconds. Absent file means no override; a field it does not mention means the detection was right.

What happens the first time hive meets a repo decides whether this is portable or theatre. Generation is a **deterministic scan** — zero model tokens, instant: read the package manager and commands out of the manifests, inventory docs by scanning the repo root and common doc directories and ranking them by inbound links and role, and pick the primary doc as the most-cited one. That alone un-hardcodes the brief mechanism and delivers most of the economics on any repo, with no quota, before the daemon is even up. It is not something a user asks for, because there is no version of this a user would ever decline.

That leaves `hive init` owning only what the scan cannot derive and what Hive must therefore be *asked* for, because it writes into the user's repo: the shipped skills (decision 8), an optional conventions scaffold, and model-authored narrative gotchas supplied through the seeding path. The skills go where each vendor actually reads them, for the vendors the user actually has — detection is `claude` or `codex` being on `PATH`, and a vendor that is absent gets no directory, because an empty `.claude/` in a stranger's repo is litter and not a feature. Init seeds no memory of its own: the facts it writes are the caller's, about the user's repo. Hive's development memories stay in Hive's repo, and the packaging test is what says so. The seam matters: hive's model is its *agents*, not the `init` binary, so the CLI does not itself call a model. When explicitly requested, it writes an `AGENTS.md` scaffold only if none exists and seeds caller-supplied narrative facts — and it indexes those facts on the way out rather than printing `hive memory reindex` at someone, which is Hive asking to be finished by hand. Codex caps the `AGENTS.md` chain at 32 KiB and truncates it silently, so a blind append could evict a human's instructions. Whatever the source, a seeded fact carries `source: init` and a `verified` date (decision 5's provenance fields), marked derived and re-derivable, distinct from the earned facts an agent learns, which stay load-bearing until disproven. Structured facts never become memory — they are already in the profile.

We considered keeping `hive init` for profiling and `hive start` for the daemon. It lost because it makes first use a two-command onboarding ritual without creating a meaningful safety boundary: both operations are deterministic, local, and required before work begins. We also considered retaining `hive start` as a deprecated forwarding alias. That lost because a legacy spelling preserves the ambiguity the merge is meant to remove: onboarding has one command and one name. The profiling half of that argument has since collapsed entirely: `hive init` does not own profiling at all, because nothing does — it happens.

The payoff is at the spawn. `brief.ts` stops hardcoding: its briefable docs and directories come from the profile, and the bare-name selector rule applies to whatever the profile names `primary_doc` — so a task citing "DESIGN §3" in a repo that has a `DESIGN.md` gets exactly the extraction hive gets for "SPEC §6," and a repo with no primary doc simply loses a special case it never needed. The orchestrator brief stops teaching hive's own doc names as examples; it is fed the profile's load-bearing doc list and primary doc at launch, so it learns to cite *this* repo's docs. The landing gate's "re-run the tests" resolves to the profile's concrete command. The profile itself is mostly read by product code, not dumped into prompts — conventions still reach agents through `AGENTS.md`'s native load — so the context budget stays intact.

The general rule this decision is one instance of: **never name a command for work Hive could have done itself.** Every follow-up command in Hive's output has to answer for itself, and the test is whether Hive could safely have done the thing. If it could, it does, silently — a stale profile, an unindexed seeded fact. If it genuinely could not, because the user must choose or because acting would be destructive — `hive update` refusing to kill a live agent team so it can restart the daemon — the command survives, as one labeled `Fix:` line, never as prose. Claude Code draws the line in the same place and the same way, and it is the only line worth drawing: a `Fix:` the user must act on is information, and a `Fix:` for something we could have done ourselves is an admission.

Sequencing follows the derive-then-refine split. The deterministic bootstrap is v1 — it is the portability floor that makes the already-shipped brief mechanism work on a repo that is not hive's own, and it costs nothing to run. The richer `hive init` — model-authored conventions, seeded memory facts, `AGENTS.md` scaffolding — is v1.1, alongside the memory store it seeds, because seeded facts cannot exist before the store does.

### 15. The name for all of this

Stepping back, the fifteen decisions compress into one architecture:

- **The daemon** is the keystone: message bus, status table, approval router, memory store, run history, resource watchdog. Agents speak to it over MCP; it speaks back through the best interactive path each driver supports — Claude Channels, tmux for the Codex TUI, or the configured Codex app-server.
- **The repo profile** is the portability seam: a derived, silently self-healing cache in hive's own state directory that tells every mechanism above this repo's doc names, commands, and shape — so the token economics hold on any codebase, not just hive's own, and nobody is ever asked to maintain it.
- **tmux** is the substrate: sessions are where agents live, windows are optional lenses, and send-keys/capture-pane are the Codex TUI actuator and sensor as well as the fallback when a native channel is not there.
- **The orchestrator** is a read-only classifier: it decomposes, scopes, tiers, and spawns; it never touches files.
- **Agents** are sandboxed, worktree-isolated, hook-instrumented, and disposable — recycled at the context quality line via handoffs, and resumed with their conversation intact when a crash, not a decision, ended their process.

## Tech stack

One language for the whole system: **TypeScript on Bun, compiled to a single binary.** The decisive constraint is MCP — the daemon *is* an MCP server, and the official TypeScript SDK is the reference implementation: most mature, first to get spec features, runs on Bun. Bun buys the three things a CLI-plus-daemon needs: fast cold start, built-in SQLite (`bun:sqlite`, no native-module pain), and `bun build --compile` for a single distributable binary — one binary, no runtime, not "have the right Node version." Hive owns that binary's distribution: a `curl | sh` installer drops immutable version directories under `~/.local/share/hive` and activates one with an atomic symlink, `hive init` checks for a newer release and says so in one line, and `hive update` stages, verifies, and activates a new build only when no team is live. `brew install hive` was the original promise and remains a secondary channel, but it cannot be the primary one: Homebrew never upgrades a formula on its own, and it knows nothing about a running daemon holding approvals and landing authority. Because a Unix process keeps executing its open image after the symlink moves, an updated binary must never adopt the daemon it just replaced — the handshake's content-addressed build hash is what refuses it, and `hive update` restarts it. See [docs/versioning-and-release.md](docs/versioning-and-release.md), whose one rule everything else follows: one push to `main` is exactly one patch release, and no version number is ever typed by a human. Everything hive does is process-glue and JSON, not compute, so a systems language buys nothing here.

Alternatives considered: Go is the right shape (static binary, daemon ergonomics) and would win if the MCP TS SDK didn't exist, but its SDK is younger and we'd write more code for the same result in a TS-heavy ecosystem. Rust pays systems-language tax on a scripting-language problem. Python has a miserable CLI distribution story.

The concrete choices:

- **Daemon MCP transport:** Streamable HTTP on localhost — one daemon, one endpoint, N agents. Both Claude Code and Codex configure HTTP MCP servers natively; stdio would mean a process per agent.
- **Storage:** SQLite via `bun:sqlite` — inboxes, status table, run history, one file under `~/.hive/`.
- **tmux / git / osascript:** shell out via `Bun.spawn`. No wrapper libraries; the CLIs are the API.
- **Hook glue:** the hive binary itself — Claude hooks and Codex `notify` invoke `hive event …`, which POSTs to the daemon. One code path, no curl scripts.
- **Daemon lifecycle:** on-demand spawn with a pidfile; first `hive` command starts it. launchd is v2 polish, not v1 plumbing.
- **Schemas:** Zod, shared across messages, handoffs, and the routing table (the MCP SDK is Zod-native). CLI parsing via Commander; tests via `bun test`, with the interesting coverage being integration tests driving real tmux sessions.

One deliberate boundary: emulator adapters (osascript for iTerm2/Terminal) and tool adapters (spawn flags, hook-config writers, interactive drivers, and native control channels such as Channels and app-server) live behind small interfaces from day one. Those are the only two places vendor and platform specifics are allowed, and they're exactly where the roadmap grows (new terminals, OpenCode).

## Roadmap: the dogfooding arc

The sequencing principle: nothing in v1 learns. Learning features compound on top of a working loop and are wasted effort if the loop isn't proven. So: build the core loop until it's usable, then **use hive to build hive**, then add the learning layer with the tool's own help.

**v1 — prove the thesis.** Daemon (SQLite, MCP: spawn/status/send/inbox, send-keys delivery — decision 1's fallback path; the native channels land in v1.1). `hive claude` / `hive codex` opening Workspace with a read-only orchestrator and hive MCP attached. Agent spawning: named agents (human first names from a pool), worktree + branch per writer, sandboxed, daemon-owned tmux session + Workspace terminal pane. Hooks wiring both directions; context size from transcripts, warn on the per-model token ceiling (no auto-respawn yet). Tier × tool routing with durable quota reservations, honest telemetry confidence, and proactive limit alerts. Committed `AGENTS.md`. A deterministic repo profile (decision 14), derived on first use and regenerated silently thereafter, so the scoped-brief allowlist and orchestrator citations work on any repo, not hive's own. Single approval queue. Landing protocol in every writer prompt (rebase → verify → ff-only merge), integrator agents only for escalated conflicts and stranded work. Done means: you can sit in one window, ask for a feature, watch a Claude agent and a Codex agent build it in parallel, approve one escalation, and see it merged.

**v1.1 — make it durable (built by hive).** The Codex adapter has two proven drivers. The default interactive TUI keeps the pane writable by the human and uses notify, rollout sensing, and tmux delivery. The configured app-server alternative drives turns, steering, interruption, lifecycle, and approvals, while `account/rateLimits/read` and its update signal feed the quota store at authoritative provenance; a failed handshake falls back to the TUI. Claude-side Channels delivery is built the same way, but scoped to the attended root: the bridge subprocess carries messages and the permission relay both directions for the orchestrator, while spawned agents stay on tmux because the preview's launch dialog cannot be answered by an unattended session (decision 4). The Codex root's equivalent is built too: `CodexRootDelivery` dual-client-steers the app-server authority's own thread (decision 1), so a `hive_send` to a Codex orchestrator wakes it the same way a Claude Channels notification wakes a Claude one. The statusLine hook feeds the subscriber's five-hour and weekly used-percentages into the quota store at `reported` provenance. Live quota discovery is built on top of both: every daemon start probes Codex's app-server and Claude's `get_usage` control request for the account's real windows, so routing reserves against measured headroom on a machine that has never seen a `quota.toml`. The resource watchdog from decision 12 is built: per-process ceilings under hive sessions, spawn pause under system memory pressure, orphan reaping, and history pruning all run on the daemon's maintenance tick. Crash recovery (decision 13) makes agent sessions survive daemon and machine death: session identity rides the hook path, and the reconciliation sweep resumes vanished processes into their actual conversations. The remaining highest-ranked work is landing receipts, then handoff-and-respawn automation, the two-scope memory store with index injection (the built Markdown + SQLite-FTS core from decision 5) and the richer `hive init` that seeds it (decision 14), `hive watch` and reattach polish, stuck-detection heartbeat escalations. Task-DAG/WIP scheduling, worktree environment provisioning, and eval telemetry stay deliberately behind all of this, per the sequencing principle: each compounds on a proven loop and is wasted effort without one. This phase is the dogfood gate: if hive can't comfortably build these features itself, that's the most important bug report we'll ever get.

**v2 — make it self-improving.** Skills rule-of-three proposals and authoring. Auto-updating routing manifest. Cross-vendor review routing. Third tool (OpenCode — it is already the ecosystem's common third adapter, so it both cements vendor neutrality and tests our adapter boundary against the same bar everyone else set; Grok is a possible fourth).

## Open questions and deferred bets

Named honestly, not papered over:

- **The codex root token mint is a bearer secret, not a bound exchange.** `POST /codex-root-token` (decision 1) hands the launcher a 60-second-TTL orchestrator credential over plain request/response — it is not single-use, and nothing binds the minted token to the specific launch that requested it, only to the operator role and a clock. That is not a bug so much as a ceiling: MCP's stateless transport gives the daemon no session to bind a token to, and single-use/exchange binding would need a session-ful protocol hive does not have. The short TTL and the operator-only mint action are the mitigation today; whether that is enough, or whether the codex root needs its own session-bound handshake, is open.
- **Claude Channels is built on a research preview, and its launch dialog costs us the agent path.** Decision 1's channel path is implemented for the orchestrator, but the dev flag, the allowlist, and the notification shape can all change under us without notice. The tmux fallback stays maintained rather than deleted for exactly this reason, and an env-gated live test (`HIVE_LIVE_CHANNELS=1`) drives a real CLI end-to-end as the canary for the day the contract moves. The bet we are carrying: if Anthropic ever lets a `server:` channel record its acceptance, or admits one to the allowlist without machine-wide managed settings, spawned agents get the reliable transport back and decision 4's third dialog stops mattering. Until then `src/daemon/launch-watch.test.ts` fails the moment a spawn starts passing the flag again.
- **Self-land versus a review gate.** Every writer lands its own finished work through rebase → retest → ff-only (decision 3), and industry practice leans hard the other way: Devin, CrewAI, Conductor, Vibe Kanban, and Intercom's agent PR pipeline all keep a separate review stage before merge. Our structural safety net — green tests and a clean typecheck on exactly the tree main becomes — is real, but it is not review: it catches breakage, not bad design, missing tests, or a plausible misreading of the task. Whether the orchestrator-plus-tests loop is enough, or whether landing needs a gate (v2's cross-vendor review routing exists partly because we suspect it), is genuinely open — and we are currently on the lonely side of the industry.
- **Task-shape similarity** for the skills rule-of-three is undefined — embedding similarity? Orchestrator judgment over descriptors? Deferred to v2 with the feature.
- **The parallelism ceiling.** The 2–4 writer default is a heuristic we chose, not a consensus we borrowed, and we haven't tested where hive's coordination actually breaks. The risk is worse than "slower": Google Research measured multi-agent setups degrading performance 39–70% on sequential tasks — some work gets *worse* when split, not merely less efficient. The orchestrator should learn to say "this task doesn't parallelize," and that judgment is unbuilt.
- **The existential bet.** Agent teams going multi-vendor, agent-orchestrator growing a real control plane, or Conductor growing an AI orchestrator would each compress our gap. The counter is speed and depth on the safety stack — being the layer no single vendor can be. If that stops being true, hive is a very nice personal tool and that's the honest floor.
